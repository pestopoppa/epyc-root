# Progress Report

**Date:** 2026-01-12
**Focus:** Fix MoE + Speculative Decoding Failures for Qwen3-Thinking Models
**Status:** Complete

---

## Summary

Fixed benchmark failures (`exit=1`) for `moe4_spec` configs on Qwen3-Thinking models. Root cause was missing BOS token metadata in GGUF files.

---

## Root Cause Analysis

### Problem
Benchmark was failing with:
```
main: draft model special tokens must match target model to use speculation
```

### Investigation
1. Ran `check_draft_compatibility.py` - showed `BOS token mismatch: draft=151643, target=None`
2. Used `llama-gguf` to verify: Qwen3-Thinking models have `tokenizer.ggml.eos_token_id` but **no** `tokenizer.ggml.bos_token_id`
3. All draft models have BOS tokens defined → llama.cpp rejects spec decode

### Affected Models
- `Qwen3-30B-A3B-Thinking-2507-Q8_0` (unsloth)
- `Qwen3-30B-A3B-Thinking-2507-Q4_K_S` (unsloth)
- `Qwen3-4B-Thinking-2507-Q8_0` (lmstudio-community)

**Note:** Qwen3-Coder models are NOT affected - they use `draft_qwen3_coder_0_75b` (jukofyork) which has matching BOS=None.

## Lessons Learned

1. **Compatibility script gaps**: `check_draft_compatibility.py` detects BOS mismatch but only as WARNING. When target has `bos_token_id=None`, this should be treated as incompatible.

2. **Multiple draft models**: Registry had 4 draft models with `qwen3-30b-a3b-thinking` in compatible_targets - all needed updating.

3. **Three forbidden_configs formats**: Registry supports `forbidden_configs`, `constraints.forbid`, and `acceleration.disallowed` - code now checks all three.
