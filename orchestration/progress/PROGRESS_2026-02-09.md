### Summary

Implemented pre-extracted question pool to eliminate ~20-30s HF dataset loading on every seeding run. All ~45K questions from 16 adapters + YAML suites are extracted once into a single JSONL file. Runtime sampling reads this file (~100ms) instead of deserializing 16 Arrow/Parquet datasets.

## REPL System Prompt Compression + File-Based Hot-Swap

### Summary

Compressed the REPL system prompt tools section by 88% (5103 → 591 chars) and added file-based hot-swap for prompt engineering without code deploys. The REPL tools description was the single largest token overhead in the inference pipeline — 30+ tool descriptions that 90%+ of benchmark questions never use.

### Approach

1. **COMPACT_ROOT_LM_TOOLS**: New constant with 9 core REPL functions (peek, grep, FINAL, CALL, list_tools, etc.) instead of 30+ verbose tool descriptions. The model calls `list_tools()` on demand when it needs extras — a local function call, not an LLM round-trip.

2. **PromptStyle.MINIMAL**: Graph node execution (`_execute_turn` in nodes.py) now uses MINIMAL style, which selects COMPACT tools. The module-level `build_root_lm_prompt()` backward-compat function stays STRUCTURED.

3. **File-based hot-swap**: New `tools_file` and `rules_file` fields on `PromptConfig`. When set, prompts are read from disk on every request (~1ms). Falls back gracefully to style-based selection on file read errors.

4. **External prompt files**: `orchestration/prompts/tools.md` and `orchestration/prompts/rules.md` — edit and the next request picks up changes. No restart needed.

### Impact

| Metric | Before | After (MINIMAL) | Change |
|--------|--------|-----------------|--------|
| Tools section | ~1450 tokens | ~140 tokens | **-90%** |
| Rules section | ~550 tokens | ~550 tokens | unchanged |
| Total REPL overhead | ~2000 tokens | ~690 tokens | **-65%** |
| Prompt eval (cold est.) | 6.2s | ~2.5s | **-60%** |
| Gen speed (est.) | 13 t/s | ~18-20 t/s | **+40-54%** |

## Architect Deliberation Loop Fix + Token Budget

### Summary

Fixed architect models (Qwen3-235B, Qwen3-Coder-480B) entering infinite self-doubt loops during 3-way seeding. Model would reach a conclusion (e.g., `D|C`), then continue generating: "Incorrect, need to be bolder..." — wasting minutes deliberating instead of committing. Two fixes: (1) commitment instruction in prompts, (2) per-role token budget.

### Root Cause

`_architect_delegated_answer()` called `llm_call()` with default `n_tokens=-1` (unlimited). The model produced its answer token (`D|C`) but generation continued uncapped. The response parsing at line 248 (`stripped.startswith("D|")`) only runs AFTER the entire generation completes — by then the model has wasted 3000+ tokens second-guessing itself.

### Fix 1: Commitment Instruction

Added to `build_architect_investigate_prompt()`:
"After reaching a conclusion, COMMIT to it. Do not second-guess, revise, or restart your analysis. Your first well-reasoned answer is your final answer."

Added to `build_architect_synthesis_prompt()`:
"Commit to your best answer; do not second-guess or restart analysis."

### Fix 2: Token Budget (baseline_tps × 500s)

Per-role token budgets derived from model speed:

| Role | Speed | Budget (tokens) |
|------|-------|-----------------|
| `architect_general` | 6.75 t/s | 3375 |
| `architect_coding` | 10.3 t/s | 5150 |

Added `_ARCHITECT_TOKEN_BUDGET` dict in `chat_delegation.py`. Passed as `n_tokens` to `llm_call()` in `_architect_delegated_answer()`.

### Fix 3: MCQ Newline Preservation (TUI)

`textwrap.wrap()` collapsed all `\n` into spaces, destroying A/B/C/D choice formatting in GPQA questions. Fixed by splitting on `\n` first and wrapping each paragraph individually.

### Files Modified

| File | Change |
|------|--------|
| `src/api/routes/chat_delegation.py` | `_ARCHITECT_TOKEN_BUDGET` dict; `n_tokens=` param on architect `llm_call()`; `_extract_toon_decision()` with MCQ-aware regex; `<think>` tag stripping; `import re` |
| `src/prompt_builders/review.py` | Commitment instruction in investigate + synthesis prompts; simplified D| formatting instruction ("ON ITS OWN LINE. Stop generating after D|answer.") |
| `scripts/benchmark/seeding_tui.py` | Per-paragraph wrapping preserving MCQ newlines |

### Fix 4: MCQ-Aware TOON Extraction

Model embeds `D|C` mid-sentence: `"Decision: D|CTo confirm this conclusion..."`. The old parser checked `stripped.startswith("D|")` which fails when the model prefixes reasoning or `<think>` tags.

New `_extract_toon_decision()` function with tiered strategy:
1. **MCQ shortcut**: `D\|([A-D])(?=[^a-z]|$)` — catches `D|C` even when followed by `To confirm...` (C followed by uppercase T, not lowercase)
2. **Own-line**: `^D\|(.+)$` with MULTILINE — catches clean formatting
3. **General**: `D\|(.+?)(?:\.\s|\n|D\||$)` — text until sentence boundary
4. **I| delegation**: `I\|(brief:.+?)(?:\n|$)` — investigation patterns

Also strips `<think>...</think>` tags before parsing (reasoning model compatibility).

## FINAL() Rescue When REPL Execution Fails

### Summary

When the model generates `FINAL("C")` inside code that has syntax errors before the FINAL line, the REPL crashes at the error and FINAL() never executes — causing unnecessary escalation. Added regex rescue that extracts the answer directly from raw LLM output when REPL fails but FINAL() is present.

### Changes

| File | Change |
|------|--------|
| `src/graph/nodes.py` | Added `_FINAL_RE` regex and `_extract_final_from_raw()` helper; rescue logic after REPL execution |

### How It Works

1. Save `raw_llm_output` before `extract_code_from_response()` transforms it
2. After REPL execution, if `not result.is_final and result.error`:
   - Search raw output for `FINAL("answer")` or `FINAL(answer)`
   - If found, override result with `is_final=True, final_answer=<extracted>`
3. This prevents escalation to coder_escalation for code that was "close enough"

## Architect Decision Budget + Early-Stop Streaming

### Summary

### Fix 1: Two-Tier Token Budget

| Turn | Budget | Purpose |
|------|--------|---------|
| Turn 0 (routing decision) | **500 tokens** | Enough for `<think>` (300) + D\|answer or I\|brief (200) |

### Fix 2: Streaming Early-Stop

Architecture: three layers cooperating via `StopIteration`:

### Result

Before: Architect generates 3833 tokens in 124s for `D|B` answer.

## Question Pool Sampling Fix + Debug Reseen

### Summary

`sample_from_pool()` sampled only `3 * sample_per_suite` (30) from pools of thousands, then filtered against the seen set. With 53K questions in the pool but only 30 sampled per suite, even a small seen set (12 questions) could overlap enough to leave only 3 unseen — causing the seeding script to stop prematurely.

### Root Cause

```python
# Old: sample 30 from 14,000 → filter 12 seen → often <10 remain
oversample = sample_per_suite * 3
sampled = rng.sample(questions, min(oversample, len(questions)))
fresh = [q for q in sampled if q["id"] not in seen]
```

### Fix

Shuffle the full suite list and walk it until `sample_per_suite` unseen are found. This guarantees we draw from all 53K questions.

```python
# New: shuffle all 14,000 → take first 10 unseen
rng.shuffle(shuffled)
for q in shuffled:
    if q["id"] not in seen:
        fresh.append(q)
        if len(fresh) >= sample_per_suite:
            break
```

### Debug Reseen Mode

Added `allow_reseen` parameter. In `--debug` mode, when a suite is exhausted (all questions seen), backfills remaining slots with seen questions so the debugger always has questions to analyze. Normal seeding skips exhausted suites.

### Changes

| File | Change |
|------|--------|
| `scripts/benchmark/question_pool.py` | `sample_from_pool()`: full shuffle + `allow_reseen` param |
| `scripts/benchmark/seed_specialist_routing.py` | `sample_unseen_questions()`: `allow_reseen` param; fallback oversample 3x→20x; both callers pass `debugger is not None` |

### Test Results

- 3124/3124 tests pass (48 workers, 16s)
- Verified: 53K pool, 500 seen → 10/10 per suite (was ~1-3)
- Verified: 44/44 seen → 0 normal, 10 debug mode
- Verified: 40/44 seen → 4 normal, 10 debug mode (backfill)
