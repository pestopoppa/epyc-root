## Heterogeneous Worker Pool Implementation

### Summary

Implemented a heterogeneous worker pool for parallel task execution. Workers are specialized by task type with HOT/WARM tiering for resource efficiency.

### Architecture

```
HOT Tier (~9GB, always resident):
├── explore (port 8082) → Qwen2.5-7B-Instruct (24 threads)
└── code (port 8092) → Qwen2.5-Coder-7B-Instruct (24 threads)

WARM Tier (~2GB, on-demand):
├── fast_1 (port 8102) → Qwen2.5-Coder-1.5B (12 threads)
└── fast_2 (port 8112) → Qwen2.5-Coder-1.5B (12 threads)
```

### Task Type Routing

| Task Type | Worker | Expected Speed |
|-----------|--------|----------------|
| explore, summarize, understand | explore (7B) | ~15 t/s |
| code_impl, refactor, test_gen | code (7B) | ~18 t/s |
| boilerplate, transform | fast (1.5B) | ~60 t/s |

### Key Features

1. **Auto-expansion**: WARM workers spin up when parallelism > 4 concurrent tasks
2. **Auto-shutdown**: WARM workers stop after 5 min idle
3. **Prompt lookup**: Enabled for all workers (ngram-min=3)
4. **Round-robin**: Load balancing across same-type workers
5. **Fallback**: Degrades to ThreadPoolExecutor if pool unavailable

### Model Choice: Qwen2.5 vs Qwen3

Chose Qwen2.5 for workers because:
- **No `<think>` tags**: Clean, concise output (workers execute, don't reason)
- **Code specialization**: Qwen2.5-Coder-7B exists at this size
- **Draft compatibility**: Same tokenizer as Qwen2.5-0.5B draft
- **Lower latency**: No thinking token overhead
