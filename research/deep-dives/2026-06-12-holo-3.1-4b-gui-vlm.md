# Holo-3.1-4B — GUI-grounding / computer-use VLM (deep-dive refinement)

- **Date:** 2026-06-12
- **Intake:** intake-691 — "Holo-3.1-4B" (https://huggingface.co/Hcompany/Holo-3.1-4B)
- **Prior verdict:** `worth_investigating`, relevance **low**, "gated on whether GUI/computer-use automation enters the roadmap; score against intake-481 (VLAA-GUI) as a candidate grounding backbone."
- **Refined verdict:** **PARKED — roadmap-gated AND runnability-gated.** No concrete creative near-term EPYC deployment surfaced that beats what is already deployed. Relevance stays **low**. Not a clean drop-in; the 4B is not GGUF-runnable today and its grounding capability is largely redundant with the deployed Qwen3-VL-30B (:8087).

## TL;DR / Refined recommendation

- **Still roadmap-gated, and now additionally blocked on runnability.** Stress-testing for a creative cheap-first / draft / grounding-tool / vision-tool role per project policy did **not** find a near-term EPYC use that the deployed stack can't already cover.
- **llama.cpp-CPU runnability verdict: NOT runnable today as a vision worker.** The 4B ships **safetensors BF16 only** — there is **no official GGUF** and **no mmproj** for the 4B (the only official Holo3.1 GGUF is for the *35B-A3B*, not this model). Architecture is `Qwen3_5ForConditionalGeneration` (Qwen3.5-family multimodal, *not* mainline `Qwen3-VL`); llama.cpp has the **text** arch (`LLM_ARCH_QWEN35`) but Qwen3.5 **vision/mmproj** conversion is immature and community-dependent (cf. ggml-org/llama.cpp #21268 / #21271, ollama #15898). Standing up the 4B would mean a transformers-CPU eager/SDPA path, not a llama-server stack role.
- **Capability is largely redundant.** GUI/UI grounding (return bbox/point for "where is the email field") is already served by the deployed **Qwen3-VL-30B-A3B (:8087)** — itself a ScreenSpot-Pro grounding baseline — and **Qwen2.5-VL-7B (:8086)**. This is the *same* conclusion the intake-680 (LocateAnything-3B) deep-dive reached: don't add a grounding specialist until the deployed VLMs are measured and shown inadequate on the actual form-fill / element-localization loop.
- **One honest near-term hook, but it's a measurement, not a deployment:** IF the operator wants to push the intake-680 "brain + eyes" autopilot form-fill / screenshot-grounding pattern, Holo-3.1-4B becomes a **precision A/B candidate** *alongside* LocateAnything-3B — but only after benchmarking deployed Qwen-VL first, and only via a throwaway transformers-CPU worker, never a GGUF stack role. This does **not** unblock anything that isn't already gated behind the same "does a GUI-agent workload exist?" question.
- **No new handoff/index work warranted.** A cross-ref line on `multimodal-pipeline.md` is sufficient; this entry rides the intake-680 grounding-tool gate, it does not open a new one.

## What it is

**H Company's Holo3.1 family** — open-weight computer-use / GUI-agent VLMs (web, desktop, mobile automation; UI grounding; native function-calling). Apache-2.0. Family: 0.8B, 4B, 9B, 35B-A3B (MoE). Holo-3.1-4B is the "cost-efficient deployment" tier.

**Exact architecture (from `config.json`):**
- `model_type: qwen3_5`, `architectures: ["Qwen3_5ForConditionalGeneration"]` — a **Qwen3.5-family multimodal** model, **not** mainline `Qwen3-VL` / `Qwen2.5-VL`. (The intake "Qwen3-VL lineage" note is directionally right — Qwen3.5 multimodal — but the precise arch ID is `qwen3_5`, which matters for GGUF conversion.)
- Text: 32 layers, hidden 2560. Vision encoder: 24 layers, hidden 1024, patch 16, out-hidden 2560 (projects to the text dim). Image/video tokens wired (`image_token_id` 248056, `video_token_id` 248057).
- ~4–5B params total ("Model size: 5B params" on the card). Weights: `model-0000{1,2}-of-00002.safetensors` (~10.4 GB total) → **BF16 safetensors**, no quantized 4B checkpoint in-repo.

**Real benchmark numbers (absolute, from the H Company blog — note these are H Company self-reported):**
- **AndroidWorld:** 4B = **72%** (improved from 58% on Holo3-class baseline) — i.e. the intake "58→71%" is the 4B mobile-automation gain (blog says 72%).
- **AndroidWorld:** 35B-A3B = **79.3%** (from 67%); mobile gains across the family cited at +11.3–14 pp.
- **OSWorld / ScreenSpot-Pro / WebVoyager / Holotab** evaluated; the headline "**>25% over Holo3**" is a *cross-harness* improvement measured inside H Company's own **Holotab** product harness, **not** a clean OSWorld delta. Treat the "OSWorld >25%" intake phrasing as harness-relative, not an absolute OSWorld score.
- Quantization quality (35B): FP8 and NVFP4 ≈ **−2 points** vs BF16 on OSWorld. NVFP4 throughput 1.41× FP8 / 1.74× BF16 — but these are **DGX Spark (GB10) GPU** numbers, not CPU/EPYC, and the DGX Spark is **not acquired** (`project_dgx_spark_target`).
- **Native function-calling + native execution** added in 3.1: near-parity with structured-JSON output across OSWorld + internal benchmarks; improved third-party agent-stack compatibility. This is the genuinely new 3.1 capability.

**License:** Apache-2.0 (clean — no flag needed; cf. LocateAnything's NVIDIA non-commercial license, which Holo *avoids*).

**Quant / GGUF status (the decisive fact):**
- **Holo-3.1-4B: BF16 safetensors only. No GGUF, no mmproj, no FP8/NVFP4 for the 4B.**
- The Holo3.1 collection ships **`Hcompany/Holo-3.1-35B-A3B-GGUF`**, `-FP8`, `-NVFP4` — all for the **35B-A3B**, not the 4B.
- The blog's "Q4 GGUF for local inference / llama.cpp / LM Studio / Jan / Ollama" claim refers to the **35B-A3B** local-deployment story (and Apple-Silicon reference numbers), **not** the 4B. The intake "quantized checkpoints" note conflated the family-level claim with the 4B specifically.

## Fit to EPYC

### llama.cpp CPU runnability

**Verdict: not runnable as a vision worker today; would require a transformers-CPU path or a GGUF-conversion spike with an unverified vision projector.**

- **Text arch:** llama.cpp has `LLM_ARCH_QWEN35` / `LLM_ARCH_QWEN35MOE`, so the *language* tower of a Qwen3.5 model is convertible. The deployed **frontdoor Qwen3.6-35B-A3B** already runs as GGUF, so Qwen3.5/3.6-text on our fork is a solved path.
- **Vision tower is the blocker.** A computer-use VLM is worthless without its **mmproj/vision projector**, and:
  - No mmproj ships for Holo-3.1-4B; you'd have to convert the safetensors vision encoder yourself.
  - Qwen3.5-family **mmproj/vision conversion is immature and known-fragile** in the ecosystem: ggml-org/llama.cpp **#21268** (Qwen3.5-122B mmproj CLIP graph uses unsupported operators / FA-unavailable fallback; closed via **#21271** but as a perf/fallback fix, on CUDA), and ollama **#15898** (`qwen35moe` mmproj/vision loading fails — vendored llama.cpp missing the arch). The picture is "text yes, vision-projector still settling."
  - Even where Qwen3.5 vision *does* load, community reports note crashes processing images through the server path. Not a green-light for a production stack role.
- **Net:** to evaluate Holo-3.1-4B on EPYC near-term you would run **HF transformers on CPU (eager/SDPA, BF16, ~10.4 GB)** — exactly the throwaway-worker pattern the intake-680 deep-dive prescribed for LocateAnything-3B. That is a *precision A/B harness*, **not** an orchestrator-stack llama-server role. A GGUF stack role is blocked until (a) someone converts the Qwen3.5 vision projector for this checkpoint and (b) the llama.cpp Qwen3.5-mmproj path is shown stable on CPU.

### Creative-use stress-test (cheap-first worker / draft / grounding-tool / vision-tool)

Per project policy (`feedback_dont_dismiss_creative_uses`, `feedback_dont_dismiss_creative_uses`), each candidate role was examined honestly:

- **Cheap-first general worker?** No. It's a *specialist* GUI/computer-use VLM fine-tune, not a general chat/tool worker; it would underperform the deployed gemma-4-26B-A4B / Qwen3.6 on our text+tool eval suite, and it's not even GGUF-runnable. Reject.
- **Draft / speculative-decode head?** No. No deployed Qwen3.5-multimodal *target* in the stack to draft for (frontdoor is Qwen3.6-35B-A3B; the purpose-built draft head story is the Gemma4Assistant case, not this). Structurally invalid, same reason the intake-682 "drafter for 26B-A4B" framing was rejected.
- **Grounding-tool ("eyes" for the brain) — the strongest candidate?** This is the intake-680 LocateAnything pattern: frontdoor (brain) calls a grounding model that returns bbox/point for UI elements, enabling form-fill / GUI-agent loops. Holo-3.1-4B *can* do this and adds **native function-calling** (cleaner tool integration than LocateAnything's custom decode). **But:**
  - The capability is **already served** by the deployed **Qwen3-VL-30B-A3B (:8087)** (a ScreenSpot-Pro grounding baseline) and **Qwen2.5-VL-7B (:8086)** — both return bounding boxes via prompt. The standing instruction (intake-680 correction) is: **measure the deployed VLMs on the form-fill IoU loop FIRST**; only add a specialist if they're inadequate.
  - Holo brings Apache-2.0 (vs LocateAnything's non-commercial license) and native function-calling — so **if** a grounding specialist is ever warranted, Holo-3.1-4B is a **legitimate A/B contender against LocateAnything-3B**. But it does not *create* the need; it rides the same unanswered gate ("does a GUI-agent / screenshot-automation workload exist on EPYC?").
- **Vision-tool for our doc pipeline / PaddleOCR-VL line?** Weak fit. Our document pipeline does **layout/OCR/table extraction** (LightOnOCR / OpenDataLoader-PDF / OCR-2-1B), not natural-language *element localization on rendered UIs*. Holo is screenshot/GUI-grounding-shaped, not document-OCR-shaped. The overlap is the generic "return a bbox" primitive, which Qwen3-VL already covers. Reject as a doc-pipeline tool.
- **Autopilot screenshot grounding?** Plausible *if* autopilot ever needs to parse our own dashboards/UI visually — but autopilot today reads structured artifacts (journal/state JSON/logs), not screenshots. No current screen-grounding need in the orchestrator or autopilot. This is a hypothetical, not a workload.

**Conclusion of the stress-test:** the only non-rejected use (grounding-tool) is *gated behind the exact same precondition* as LocateAnything and the GUI-roadmap question. No creative role lets Holo-3.1-4B do useful work on EPYC *today* that the deployed stack can't already do. Honest verdict: **park it.**

## Decision gates & next steps

**Flip to active investigation only if ALL of:**
1. A concrete **GUI-agent / computer-use / screenshot-automation workload** enters the EPYC roadmap (autopilot dashboard parsing, automated UI form-fill, browser/desktop automation eval), AND
2. A precision benchmark of the **deployed** Qwen3-VL-30B (:8087) and Qwen2.5-VL-7B (:8086) on that workload's element-localization / field-placement IoU shows them **inadequate**.

**If both hold, the next step is a *measurement*, not a deployment:**
- Spike a **throwaway HF-transformers CPU worker** (eager/SDPA, BF16, ~10.4 GB RAM — trivial on the 1.1 TB host) running Holo-3.1-4B, **A/B'd against LocateAnything-3B and the deployed Qwen-VLs** on the same grounding loop. Compare field-placement IoU, per-call latency, and tool-call ergonomics (Holo's native function-calling is a plus here).
- **Do NOT** attempt a GGUF stack role until the Qwen3.5 vision-projector / mmproj path is shown stable on CPU in llama.cpp **and** an mmproj is produced for this checkpoint. Track llama.cpp Qwen3.5-mmproj status (#21268/#21271 lineage) as the unblock signal.

**Otherwise:** keep parked. No handoff stub, no index churn (per CLAUDE.md: no index changes without explicit operator approval). A single cross-ref line on `multimodal-pipeline.md` records the grounding-A/B option next to intake-680.

## Risks

- **Runnability mismatch:** treating the blog's "Q4 GGUF / llama.cpp" claim as applying to the 4B is wrong — that's the 35B-A3B story. Acting on it would mean discovering only at conversion time that the 4B has no GGUF and no mmproj. (Flagged here to prevent that.)
- **Self-reported benchmarks:** AndroidWorld 72% / 35B 79.3% / ">25% over Holo3" are **H Company first-party**, and the ">25%" is *Holotab-harness-relative*, not a clean OSWorld absolute. No independent replication found. Treat as a prior, not a result.
- **GPU-shaped efficiency story:** every speedup number (NVFP4 1.41×/1.74×, 6.8s→3.3s step time) is **DGX Spark (GB10) GPU** — irrelevant to CPU/EPYC and predicated on hardware we don't have (`project_dgx_spark_target`).
- **Redundancy risk:** adding Holo as a grounding worker before measuring the deployed Qwen-VLs would repeat the over-provisioning the intake-680 correction explicitly warned against (Qwen3-VL-30B already grounds; Qwen2.5-VL-7B already grounds).
- **Vision-projector immaturity:** Qwen3.5 mmproj on llama.cpp is settling, not settled; a premature GGUF stack role risks the image-processing crashes reported upstream.
