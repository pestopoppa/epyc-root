# AGENT REPORT: autopilot

# SUMMARY
The autopilot's evidence architecture consists of ~17 persistent stores with NO single chokepoint: the append-only-in-name journal (JSONL/TSV) is the closest thing to a source of truth, but five derived stores (Pareto archive + HV history inside autopilot_state.json, baseline state, AP-22 short_term_memory.md, strategies.db+FAISS, and the resumable Claude CLI planner session) are each written independently at trial time rather than derived from the journal, so every historical contamination required a multi-store manual purge. The planner prompt is assembled from 20+ sections in autopilot.py:1628-1900, mixing measured facts (q/s/c/r, memory_count, host-health), policy (blacklist, action availability), and persisted narrative (hypotheses, falsifiers, self-criticism, STM, distilled strategies) with no provenance tags on most numbers. Write-time gating via classify_learning_exclusion is consistently applied to the Pareto archive, baseline, and AP-22 memory, but NOT to the distill_knowledge→strategy-store path (reads raw all_entries() with only a legacy-scale text scrubber), and a benign-exclusion representative that lands on the frontier still triggers a strategy-store write at autopilot.py:2470 despite the log claiming strategy learning is excluded. The 12 incidents I tabulated show fixes scattered across ≥9 files, with a partial late convergence (src/autopilot_core contract extraction 2026-06-05; all-read-sources scrub script 2026-06-01), but every narrative-class incident (gate-lock 9.900, falsifier recurrence, distill re-injection, @708 session fixation) leaked through a persisted-narrative store that write-time fixes alone could not clean — the system's recurring failure mode is narrative that outlives the facts that generated it. Provenance is partial (trial_id/config_fingerprint/source_trial_id chains exist for archive and baseline, enforced by the archive-first invariant) but absent for strategy insights, STM lines, and session-resident reasoning. Five structural invariants (event-sourced ledger with supersession instead of in-place scrubs; derived-store rebuild from ledger; fact/policy/narrative type separation with regenerate-don't-replay narrative; schema/unit validation at store boundaries; provenance IDs on every prompt number) would have prevented all tabulated incidents.

## 5. VERDICT

**Scattered, with a late partial chokepoint.** There are effectively three evidence planes with different integrity properties:
- **Facts**: journal rows + Pareto/`repro_clusters`/HV in state. Strongest plane — exclusion-gated at 6 of 8 read/write surfaces, atomic writes, WAL crash recovery, and `journal_reconstruction.py` proves the archive is *re-derivable* from the journal. But the journal itself is mutated in place by scrubs/rewinds (history editing, not supersession), and the runtime archive is NOT actually derived from it — they are dual-written and have desynced (scrub doesn't touch archive `scrub_journal.py:39-41`; rewinds needed separate archive prunes).
- **Policy**: blacklist, program.md, baseline, durable-halt latches. Reasonably centralized (blacklist enforced at one dispatch point; baseline now invariant-guarded both load- and write-side).
- **Narrative**: journal free-text fields (hypothesis/falsifier/self_criticism/failure_analysis), STM, strategies.db, claude session. This plane has **no exclusion gating on two paths (distill input; frontier strategy auto-store), no provenance, and persists across the facts it was derived from** — it caused or amplified incidents 1, 2, 3, and 8, and every fix is a sanitizer bolted onto a replay channel.

**Provenance**: partial. Traceable: Pareto entry → `trial_id`/`config_fingerprint`/`n_reproductions`; baseline promotion → `source_trial_id` must be on the frontier (archive-first invariant `safety_gate.py:941-959`); journal `eval_details.learning_exclusion` records why a trial was excluded; per-trial git tags. NOT traceable: a strategy insight to its evidence trials (only the distill action's counter is stored, `evolution_manager.py:154`); an STM "running hypothesis" beyond a `[tN]` tag in free text; any number the resumed session remembers; `memory_count`, batch telemetry, and `model_quality_signatures.yaml` values (manually maintained, dated 2026-04-16) entering the prompt. So the answer to "can any number in the planner prompt be traced to the measurement that produced it" is **no — roughly half can (journal/archive-derived sections), half cannot (narrative + telemetry + manually-maintained YAML sections)**.

**Minimal structural invariants that would have prevented ALL tabulated incidents**:
1. **Event-sourced single ledger with supersession**: the journal is the only durable evidence store; corrections are appended supersession events (the existing `bug_corrupted_by` tagging done as append, never rewrite); archive/baseline/STM/strategy views are rebuilt from ledger + exclusion log on load (the `journal_reconstruction.py` machinery promoted from dashboard-only to runtime). Prevents: #1 (baseline is a fold over the ledger, nothing to clobber), #3/#5 (scrub = one event, all views re-derive), the scrub/archive desync, and the rewind-must-purge class.
2. **Narrative is regenerated, never replayed**: hypotheses/falsifiers/insights are recomputed each trial from current facts; no persisted free-text store (STM, strategies, resumed session) feeds the prompt without passing through the exclusion gate + a provenance check. Prevents #2, #3, #8, and the residual 9.900 citations.
3. **Unit/scale/resolution contracts at every store boundary**: quality∈[0,3], thresholds expressed relative to sampling resolution (3/n), validator caps surfaced in the same contract the planner reads. Prevents #1 (9.900), #10, #11, #12.
4. **Baseline as a pure function of the archive** (max-of-frontier with eligibility proof) rather than an independently persisted number — the existing archive-first + load-guard invariants taken to their conclusion. Prevents #1, #10.
5. **Provenance ID on every prompt number** (measurement trial_id or named instrument + timestamp), with the critic instructed to reject claims lacking one. Prevents #6 and #7 from becoming planner-level narratives, and makes #8-style session fixation detectable (stale IDs).

The codebase is already converging on (3) and (4) piecemeal; (1), (2), (5) are the missing architecture — and notably, every component needed for (1) already exists (`journal_reconstruction.py`, `action_identity.py`, `learning_exclusions.py`, `pareto_math.py`).

# SUMMARY
The autopilot's eval tower is a 3-tier instrument whose production gate (T1, used by every trial via hybrid_eval) is a FIXED 43-question set: 38 pool questions (2 per suite from 19 suites, stratified with hard-coded seed=42) plus 5 tool-use sentinels, scored binary per question and aggregated as fraction_correct×3. The aggregate quality quantum is 3/43≈0.070 and the per-suite quantum is 1.5 (n=2), so the smallest observable suite move is 50 percentage points; the live journal shows the entire 60-trial quality range spans only 4 question-flips (1.674–1.953) with sd 0.058 and a MAD significance bar of 0.103 (~1.5 flips), meaning the planner is chasing 1-flip deltas the instrument cannot certify. I found a live misfire in progress: trial 775 (a config-neutral seed_batch) raised the T1 baseline to the all-time noise maximum 1.953 (28/43), after which the modal honest outcome 1.814 reads as a −7.1% regression violation — a baseline-ratchet that the MAD filter's z=2.01 marginal pass let through. ## 4. GAMING / MISFIRE CATALOG

### (a) Scoring a "win" without genuine improvement
1. **Baseline ratchet to the noise maximum (LIVE, happening now).** `update_baseline` is monotonic-only and accepts any clean trial whose quality is a frontier max; the MAD bar is 0.103 and a 2-flip draw (z≈2.01 on the live window) clears it. Trial 775 — a `seed_batch`, config-neutral — raised the T1 baseline to 1.953 (log:115569). There is no decay/refresh; the baseline now equals the best single observation ever, and the modal outcome (1.814) trips the −5% gate. Net effect: lucky noise is permanently enshrined as "production quality". Evidence: `safety_gate.py:935-941`, journal stats above.
2. **MAD==0 escape hatch.** With a quantized 10-deep window, MAD collapses to 0 whenever >5 entries are identical; then ANY change ≠ median counts as significant (`safety_gate.py:604`) — a single 1-flip draw becomes a certified improvement and (via 1) a baseline raise.
3. **Fixed question set → overfitting surface.** seed=42 forever (`eval_tower.py:726-752`): prompt/flag mutations can tune to the specific 38 questions (e.g. `<answer>`-tag discipline, comma formatting) without any pool-level gain. The substring scorer's history (factorial comma fix, `debug_scorer.py:488-495`; memory `feedback_substring_scorer_comma_brittle.md`) shows scores move on formatting alone.
4. **Speed-axis frontier manufacturing.** Clean-pass trials enter the archive with raw single-trial speed at CV 9.1% (`autopilot.py:2424-2435`); a +1σ speed draw (~+4.5 t/s) on unchanged config is non-dominated → "frontier" → strategy-store write (`autopilot.py:2470-2479`). The median-cluster defense applies only to within-noise-quality trials. The 2026-06-04 analysis (memory: 8/11 mad_noise exclusions were non-dominated) is what motivated the representative path, but the symmetric hazard (lucky clean trials) remains.
5. **Tool-use suite as a 5-question lever.** tool_use is 5/43 = 11.6% of aggregate quality, scored substring on a runtime secret. Historical bypasses: secrets hardcoded in repo+YAML (greppable — confirmed FALSE-secure, tool-use-eval-contract.md:33), `io.open` uncounted reads (gate run 1, :88-96), cross-request telemetry pollution (:34). ## 5. STATISTICAL POWER VERDICT

**The instrument is underpowered for the effect sizes being optimized, by roughly an order of magnitude — but the deficit is policy/aggregation, not just n.**

Numbers: target effects = 1-2 flips (0.07-0.14 quality, 2.3-4.7pp); single-trial repeatability sd = 0.058; MAD certification bar = 0.103; per-suite resolution = 50pp; baseline promotion happens off ONE quantized observation; only 4 T2 validations have ever run. At ~13 min per T1 eval (wall ≈ 800s, journal `eval_wall_s`) the loop gets ~20 trials/day, and 9/120 recent trials reached frontier with zero certified learning signal (24 mad_noise + 69 noise-labeled "reverts").

What a statistician would prioritize, in order:
1. **Paired per-question analysis (the single most important change).** The question set is already fixed — the design is a repeated-measures experiment, but per-question outcomes are discarded at aggregation (`eval_tower.py:621-683`; the journal stores only aggregates). Persisting the 43-bit outcome vector per trial and testing config-vs-baseline with McNemar on discordant questions converts sd≈0.83-flip trial noise into a test on the 2-4 actually-stochastic questions; a true 2-question improvement becomes detectable in 1-2 trials instead of never. Cost: ~zero inference, storage only.
2. **Sequential testing on reproduction clusters.** The infrastructure exists (`pareto_archive.upsert_representative` keeps per-fingerprint clusters with medians) — replace the single-shot MAD z>2 keep/promote rule with an SPRT/grouped-sequential rule over the cluster: promote baseline only when the cluster median exceeds baseline with k≥3 reproductions. This directly kills the trial-775 ratchet (one lucky draw can no longer move the baseline) and rescues sub-threshold honest wins (evidence accumulates instead of being individually discarded as mad_noise).
3. **More questions where they discriminate.** Raising t1_n 50→200 (~10/suite) cuts the quantum to 0.015 and the per-suite quantum to 0.3, at ~4× wall cost — worth it only for promotion/validation evals, not every trial. Cheaper: drop the ~10 permanently-0 and ~16 saturated questions (zero Fisher information) and reallocate to borderline suites; a 43-question budget concentrated on discriminating items would outperform the current spread.
4. **Baseline hygiene**: make the aggregate regression gate resolution-aware like the per-suite gate (threshold ≥ max(5%, 2 quanta)), recompute the baseline as a rolling median of recent passed trials rather than a monotonic max, and fix the double `gate.check()` and the dead throughput floor (re-anchor `frontdoor_speed` to the current aggregate-tps regime; `safety_gate.py:493-505` never updates it).

Verdict: as built, T1 can reliably detect ≥2-flip (≥4.7pp) single-shot changes and — via the reproduction machinery — eventually confirm ~1.5-flip sustained gains; everything smaller is noise-traded. The planner's current action space (flag toggles, seed batches) produces effects below that line, so the loop's recent output is dominated by measurement artifacts: the keep/revert stream, the baseline ratchet, and mad_noise exclusions are all consequences of instrument resolution, not model behavior.

# SUMMARY
The orchestrator's /chat serving path contains ~25 distinct decision points spread across 4 layers (role routing, mode selection, execution-time escalation/review, NUMA placement), backed by at least 8 distinct optimization paradigms and ~12 learned-artifact stores — but the live production decision stack is far thinner than the codebase suggests. Verified against the running process (PID 1884727, env via /proc + logs/orchestrator.log): the actual live deciders are (1) the MemRL KNN+Q-value HybridRouter (75% of 2026-06-11 routing decisions logged "learned"), (2) the cheap-first speculative pre-filter (config default-on, Phase A), (3) rule-based mode selection and EscalationPolicy with risk-aware think-harder, (4) the WP-2 placement state machine + cross-role disjoint placement + reverse migration (all env-on), and (5) the factual-risk canary (25% enforce on frontdoor). Everything else — MLP classifier fast-path (flag ON but weights file MISSING since the 2026-05-25 episodic-memory reset; boot log proves "fast-path disabled"), frontdoor verifier head (loaded but only reachable via the dead classifier path, and shadow-mode anyway), GraphRouter GNN, bilinear scorer (DAR-4), SPO+/epsilon-greedy (DAR-3), risk-abstain gate, failure-graph veto (inert because specialist_routing flag is off, which also reduces the keyword specialist classifier to constant-frontdoor), plan review, proactive/architect delegation, Trinity tri-role (shadow only), swarm fanout, model fallback, and shape-aware contention — is built but dark. The latent unifying structure is real: nearly every learner approximates the same Q(prompt, action)≈P(success) minus a hand-tuned cost penalty, but the cost penalty appears in at least 5 independently-tuned λ forms and the project's own DAR-1 audit found Q-values 96% uniform, meaning cost/similarity/prior terms — not learning — actually drive routing, which lands 77% of traffic on frontdoor. Routing has accreted past a unifying principle in code, while production de facto runs a much simpler system: KNN-memory router + cheap-first + rules + topology-safe placement.

## 3. COMPOSITION SEMANTICS

**Override chain (role axis, strict order in `_route_request`)**: forced/explicit > classifier fast-path (dead) > risk-abstain (off) > learned-KNN (if confident) > rules; then *post-hoc mutations*: failure-veto can revert specialist→frontdoor (inert), ingest-guard can demote→worker_general (off), plan-review can rewrite routing (off), cheap-first can bypass the routed role entirely (ON), and runtime escalation can move up-chain regardless of the original decision. Provenance is preserved by strategy-string concatenation (`"cheap_first:<strategy>"`, `"<strategy>:ingest_triviality_guard"`, `"binding:<strategy>"`).

**Where two learners can disagree & arbitration**:
- *Classifier vs KNN*: per-class confidence threshold; below-threshold falls through (hybrid_router.py:331-417). Verifier adds a second veto for frontdoor only — enforcing mode would silently fall to KNN.
- *Priors (heuristic) vs Q (learned)*: additive blend `posterior = selection + prior_strength × prior` — a prior can flip the ranking (tested: `test_retriever_risk_gate.py::test_hybrid_router_prior_blend_can_flip...`). Since priors are degenerate-frontdoor (finding above), this is a systematic frontdoor pull on learned routing.
- *GNN vs KNN*: convex blend with store-size-annealed weight (off).
- *Factual-risk vs everything*: high band simultaneously (a) blocks cheap-first, (b) tightens the (inert) veto, (c) accelerates think-harder, (d) would force plan review. It cannot change the routed role directly.
- *Trinity role vs model role*: declared orthogonal axes; today shadow-only, no conflict possible.
- *Routing vs placement*: fully decoupled — routing picks a role; `ConcurrencyAwareBackend` picks the physical instance. ContentionDenied/timeout surfaces as an error, not a re-route; there is **no feedback from placement pressure into role choice** (queue depth is logged into routing_meta but consumed by nothing).

**Feedback loops (online)**:
1. Outcome → progress JSONL → QScorer reward → TD update on the routing memory → next KNN retrieval. Escalations *during* execution feed back as −0.15 penalties on the original routing decision (q_reward.py:58-59) — the only place execution-layer decisions teach the routing layer.
2. `retriever.update_last_role` → cache-affinity p_warm on the *next* decision (a 1-step warm-cache prior; retriever.py:88-117,302-308).
3. Review-gate verdicts and plan-review episodes are stored as memories (chat_review episode storage), nudging future review thresholds via the same Q store.
4. The MLP/GAT/SkillBank are *offline distillations of the same episodic store* — currently broken because the store was reset 2026-05-25 (retrain-routing-models.md: need ~500+ fresh routing memories; 06-11 rate ≈ 81 scored/day).
5. Autopilot can hot-swap routing flags + tune cheap-first/escalation/retrieval knobs via POST /config + env (config_applicator.py:26-79) — an *outer* meta-loop over the whole stack.

## 5. VERDICT

**Latent unifying structure — yes, and it is explicit in the code**: at least seven components are estimators of the *same* value function `V(prompt, action) ≈ P(success) − λ·cost`:
KNN Q-table (decision-time `selection_score = Q − 0.15·cost_ratio`, retriever.py:256), the reward that trains it (learning-time `reward − 0.15·max(0,cost_ratio−1) − 0.10·quality_gap − 0.05·mem_tier`, q_reward.py:88-106), the MLP (distilled argmax of that table), the GAT (cold-start prior of it), the bilinear scorer (factored parameterization of it), the verifier head (its frontdoor column), the cheap-first Phase-B gate and the review gate (threshold reads of the same Q), route_3way (Q/cost re-aggregation of it), and the dead CONF| protocol (an LLM self-estimate of it). The handoffs already converge on this framing (DAR-4/4b/5 = one bilinear `Q(prompt, model)` with an inference-time preference vector ω=(perf,cost); tri-role adds the orthogonal `(model, role)` action axis; LRC adds independent heads per decision surface). A single decision-theoretic model would be: **one calibrated success-probability model Q(features(prompt), identity(model), role) + one explicit cost model c(model, placement-state) + one scalarization ω**, with escalation as sequential re-evaluation of the same Q after evidence (a failure observation), cheap-first as the ω-cost-extreme of the same argmax, and the risk/difficulty bands as features rather than parallel mode machines. The only genuinely *different* decision family is placement/contention — a feasibility/safety constraint problem (topology vetoes + measured throughput matrix), correctly kept separate.

**What components actually disagree about in practice**: almost nothing, for a degenerate reason. DAR-1 measured 96% uniform Q-values (decision-aware-routing.md) — so the "learned" router's ranking is dominated by the cost term, similarity, and the (frontdoor-degenerate, because specialist_routing is off) heuristic priors. Net live behavior: 77% of requests route frontdoor; the disagreements that remain are (a) forced-role eval traffic vs learned, (b) the factual-risk canary occasionally blocking cheap-first/accelerating think-harder on 25% of frontdoor traffic, and (c) escalation policy overriding the original route after failures (24×/day). The five offline learners cannot disagree with anything because three weight files don't exist and the other is unreachable.

**Dead / deletable with zero production impact** (proof cited): `get_confidence_routing` + `_parse_confidence_response` + `_select_role_by_confidence` (chat_routing.py:283-448 — zero callers); `HybridRouter.route_3way` (zero callers); `dispatch_swarm_fanout` (zero callers + flag default-off both modes, features.py:186). **Dark-but-planned (disable-able today, deletion would contradict active handoffs)**: GraphRouter/GAT stack (flag off features.py:101 + weights absent), bilinear scorer (BILINEAR_SCORER_ENABLED unset + weights absent; DAR-4 still queued), SPO+/ε-greedy (env unset; DAR-3 unblocked by P19.6 audit), risk-abstain gate (config-off), SkillBank (flag off + stale skills.db), binding_routing/script_interception/personas/staged_rewards/deep_research_mode/batch_edit/edit-transaction (flags off), shape-aware contention C-phase (`select_backfill_candidate` pure function, no behavioral caller until flag flip). **Surprise findings the architect should act on**: (1) production runs on *test* feature defaults — specialist_routing/plan_review/architect_delegation/parallel_execution/model_fallback are silently OFF despite `default_prod=True`, making large code regions (failure graph, proactive delegation, plan review, keyword specialist routing) unreachable; (2) the classifier→verifier fast-path chain has been dead since the 05-25 memory reset while its flags say ON — the system's stated architecture (MLP common-case + KNN fallthrough, learned-routing-controller.md) is not the running architecture; (3) the Q feedback loop is live but re-learning from scratch at ~81 updates/day against a 500-memory retrain gate.

## 5. MI210 HYPOTHESIS CRITIQUE

### Evidence state (all DESIGN-STAGE, hardware-gated)
- Card NOT acquired; expected ~July 2026 (`agentic-rocm-kernel-authoring.md:6`). Every stage of `gpu-drafter-mi200-investigation.md` is "observation-and-design, no hardware change" (`:282`).

- ROCm maturity / fork status: the fork's HIP backend should **compile** on gfx90a (CUDA-derived, ROCm 6.1+), and GEAK is gfx90a-PROVEN for *compile compatibility* — but **all AMD perf numbers are vendor-reported until reproduced on the actual MI210** (`agentic-rocm-kernel-authoring.md:44-45`; `rocm-verify-profile-backend.md:48-49`). gfx90a predicts compile, **not** performance equivalence (single-GCD bandwidth, ROCm version, autotune space differ).

### The thesis and its gating measurement
The handoff thesis: MI210 adds a **latency tier** (frontdoor + matched-vocab Qwen3-1.7B drafter co-resident → 2-3×) on top of the already-competitive CPU tier; architect stays CPU (doesn't fit), workers stay CPU (throughput-amortized) (`gpu-drafter-mi200-investigation.md:10-28`).

**The single highest-leverage gate is α(Qwen3-1.7B→Qwen3.6) at γ=3, and it is UNMEASURED** (`:157-175`). Three downstream investments (cascade, custom drafter training, adaptive-K) all key off this one number. **Critique: α is CPU-testable TODAY** — frontdoor currently launches with zero spec-dec (`:190`), so adding `-md`+`--spec-type` to the CPU frontdoor would produce α before any GPU purchase. This is the cheapest discriminating experiment and is not yet run.

Stage 0 already found worker MTP **saturated at 76.9%** acceptance, which DOWNGRADED the MTP-head-split EV from +15-35% to **+10-15%** (`:185-190,212-216`) — i.e. the most-engineering-heavy stage is also the lowest-EV.

### PCIe / interconnect economics
- Draft+target co-resident on ONE GPU → no PCIe in the spec loop (clean; `:60-64`).
- CPU-target + GPU-draft (Stage 1) and any prefill/decode disaggregation pay EPYC's **xGMI/PCIe ~64 GB/s**, proportionally far worse than NVLink ~900 GB/s — exactly why the disaggregation handoff defaults NOT-PURSUED for single-user (`numa-prefill-decode-disaggregation.md:34-36`).
- Decode-phase breakdown: short-context decode is 85-92% weight-GEMM, BW-bound, where **CPU is already competitive and GPU gain is marginal**; GPU wins are concentrated in prefill (always compute-bound) and long-context decode (`gpu-acceleration-path.md:195-204`).

### Strongest ALTERNATIVE uses + discriminating evidence
1. **Full frontdoor RESIDENCY on-GPU** (under-weighted in the handoff): 37 GB Q8 fits in 64 GB; BW alone (1.6 TB/s vs the ~one-NUMA-node frontdoor sustains) implies a ~85 t/s solo upper bound vs the deployed 27 t/s — **~3× from residency before any spec-dec**. The handoff cites ~85 t/s solo (`:60-64`) but frames it as a drafter-co-location detail rather than the headline. **Discriminator**: a single solo-throughput bench, frontdoor-on-GPU vs CPU.
2. **Prefill offload** (compute-bound, GPU's strength). **Discriminator**: measure per-role prefill/decode split — decode-bound short-context → GPU marginal; prefill/long-context → GPU large (`gpu-acceleration-path.md:195-204`).
3. **Embedding / reranker / classifier / verifier host** — batch-friendly compute-bound work where a GPU dominates not just on BW: 6 BGE embedder instances (`model_registry.yaml:218-225`), NextPLAID ColBERT code+docs (`:677-721`), routing classifier + frontdoor-specialist verifier (`orchestrator_stack.py:1120,1152-1153`). **Discriminator**: these are already multi-instance/batchable, so GPU utilization is high and they free CPU cores for the heavy roles.
4. **Batch eval acceleration** — autopilot's 6 GEPA workers + seeding/eval runs are throughput, not latency, workloads.

**Discriminating-evidence summary**: (a) measure α on the CPU frontdoor (no hardware needed); (b) measure prefill/decode split per role; (c) compare frontdoor solo on-GPU residency (~85 t/s) vs the drafter-split (1.3-3×) vs the batch-host alternative. The handoff's own staged plan (Stage 1 gate ≥1.3×, Stage 2 target 2-3×) is sound, but the **residency play and the α-pre-measurement are the cheapest, highest-information experiments and should precede acquisition**.

### MI210 kernel-authoring posture (context)
Net-new work on the MI210 is scoped to an agentic HIP/Triton kernel-authoring loop adopting AMD-native GEAK/Apex/AgentKernelArena (MIT/Apache-2.0), with C4 (gfx90a profiler-metric) + C6 (anti-reward-hacking) the only genuinely net-new pieces (`agentic-rocm-kernel-authoring.md:21-28`; `rocm-verify-profile-backend.md:16-24`). This is complementary (authors the kernels), not a placement decision.

## 7. Implications for the proposed {fixed paired core + rotating audit + promotion eval + item analytics} design

- **Fixed paired core**: keep pairing, but curate via `load_questions_by_ids` instead of seed-42 luck; drop/replace the 8 dead items (usaco×2, ip×2, vl×2, bcb×2) and the 4 pegged-zero items; balance mid-difficulty (tier mix) using existing tier metadata.
