## Question Pool Pre-extraction

### Summary
Seeding script spent ~20-30s at startup loading 16 HuggingFace datasets sequentially. Each `get_adapter()` returned a fresh instance, and `_ensure_loaded()` deserialized Arrow/Parquet from the HF cache (1-10s each) with no module-level caching. Implemented pre-extraction of all ~45K questions into a single JSONL file so runtime sampling drops to ~100ms.

### Problem
- 16 HF datasets loaded sequentially via `datasets.load_dataset()` on every run
- `get_adapter()` returns `cls()` — fresh instance each time, no caching
- ~20-30s startup cost before any evaluation begins
- Especially painful during iterative development and short dry runs

### Solution

#### 1. `BaseAdapter.extract_all()` method (`dataset_adapters.py`)
- Added to `BaseAdapter`: iterates full dataset through `_row_to_prompt()`
- Handles multi-dataset adapters (Math, Coder, Thinking) which use idx-based `_row_to_prompt()`
- `VLAdapter` override: delegates to `_vl_adapter.sample(n=total)` since VL uses a different loading mechanism

#### 2. `question_pool.py` module (new, ~220 lines)
- **`build_pool(output_path)`**: Iterates all `ADAPTER_SUITES` + `YAML_ONLY_SUITES`, instantiates each adapter, calls `extract_all()`, writes JSONL. Header line contains metadata (timestamp, total count, per-suite counts).
- **`load_pool(pool_path)`**: Reads JSONL, returns `dict[suite_name, list[dict]]`. Warns if >30 days old.
- **`sample_from_pool(pool, suites, sample_per_suite, seed, seen)`**: Mirrors existing `sample_unseen_questions()` logic — oversample 3x, dedup against seen, truncate to `sample_per_suite`, round-robin interleave across suites.
- **`pool_header(pool_path)`**: Reads just the first line to check metadata without loading full pool.
- **CLI**: `python scripts/benchmark/question_pool.py --build` (build pool), `--stats` (print pool info).

#### 3. `seed_specialist_routing.py` integration
- **`sample_unseen_questions()`** gains `use_pool` kwarg (default `True`):
  - Tries pool first via `load_pool()` + `sample_from_pool()`
  - Auto-builds pool if JSONL file missing (one-time cost)
  - Falls back to HF adapters on any pool failure
- **`run_batch()` and `run_batch_3way()`** gain `use_pool` kwarg, threaded from `main()`.
- **New CLI args**:
  - `--rebuild-pool`: Rebuild pool from all adapters, print stats, exit.
  - `--no-pool`: Bypass pool, use original HF adapter loading (for debugging/comparison).
- All 4 call sites (`run_batch` × 2, `run_batch_3way` × 2) pass `use_pool=not args.no_pool`.

## Early-Stop Regex Bug Fix

### Problem
During 3-way seeding, architect_general answered `D|7` for a USACO problem. The early-stop regex `D\|.{2,}` did NOT match:
- `D\|[A-D]` — `7` not in `[A-D]`
- `D\|.{2,}` — only `7` after `D|` (1 char, needs 2+)
- `.` doesn't match `\n`, so even `D|7\nThe problem...` never matched

The model generated 100-3000 wasted tokens of post-answer rambling.

### Fix
Changed `D\|.{2,}` to `D\|.+` in `chat_delegation.py:313`. Now catches any `D|X` with 1+ characters after `D|`. Bare `D|` (partial output during streaming) still correctly does NOT match.

### Edge Case Verification
| Input | Old Regex | New Regex |
|-------|-----------|-----------|
| `D\|` | no match | no match |
| `D\|7` | **no match** | match |
| `D\|A` | match | match |
| `D\|42` | match | match |
| `D\|yes` | match | match |

## REPL TPS Telemetry Fix (generation_ms=0)

### Problem
79/99 REPL results showed `generation_ms=0` and `predicted_tps=0` while `tokens_generated` was correct (e.g., 672). Direct mode had correct timing. Both use the same `LLMPrimitives` object.

### Root Cause
Early-stop streaming in `llama_server.py:infer_stream_text()`. When `on_chunk` raises `StopIteration` (REPL's FINAL() detection), the SSE loop breaks immediately. But the `timings` dict is only populated by the `stop: True` SSE event which never arrives. So `generation_ms = timings.get("predicted_ms", 0.0)` → 0.

REPL sets `_early_stop_check` (nodes.py:319) for every LLM call. Direct mode does not. This explains the pattern: 79/99 REPL = 0, 3/105 direct = 0.

### Fix
When early-stop breaks the SSE stream, compute timing from wall clock elapsed time:
```python
if early_stopped:
    _es_elapsed = (time.perf_counter() - http_start) * 1000
    timings = {
        "predicted_ms": _es_elapsed,
        "predicted_per_second": tokens / (elapsed_s) if elapsed_s > 0 else 0,
    }
```

This is wall-clock time (includes prompt eval + HTTP overhead), not pure generation time, but it's far better than 0 for TPS estimation.

## REPL Answer Truncation on Code Suites

### Problem
REPL mode underperforms direct on code suites (coder: 6/6→3/6, debugbench: 3/6→0/6). Model generates code + `FINAL("Code implementation complete")` instead of `FINAL(the_actual_code)`. Direct mode returns the full LLM output which contains the code; REPL extracts only the FINAL() argument.

### Root Cause
REPL prompt instruction: "Complete the task. Output Python code with FINAL(answer):" — the model interprets "answer" as a status message, not the code artifact. `auto_wrap_final()` already handles bare function definitions (wraps in `FINAL('''code''')`), but when the model explicitly writes `FINAL("done")`, auto_wrap_final can't intervene.

### Fix
Modified REPL instruction in `builder.py` to clarify code-artifact expectations:
```
"Complete the task. Output Python code with FINAL(answer). If asked to write/implement code, FINAL must contain the code itself, not a status message like 'done' or 'implemented':"
```

### Architectural Note
REPL mode is fundamentally mismatched for code-artifact benchmark tasks. Direct mode returns the LLM's full output (containing the code). REPL extracts code, executes it, and captures only FINAL(). The 3-way eval correctly captures this — binary rewards (pass/fail) will train the hybrid router to prefer SELF:direct over SELF:repl for code tasks.

| File | Change |
|------|--------|
| `src/prompt_builders/builder.py:139` | Clarified FINAL() instruction for code tasks |

## Claude-in-the-Loop Seeding Debugger

### Summary
Implemented an automated debugging pipeline that invokes a persistent Claude Code session during MemRL seeding runs. When anomalies are detected in model answers (repetition loops, template echo, format violations, etc.), Claude receives diagnostic batches, analyzes root causes, and applies fixes — prompt hot-swaps (immediate) or code patches (with API hot-restart).

### Architecture
```
Seeding script (--debug)
    ├─ runs answer → compute anomaly signals
    ├─ accumulate diagnostic → JSONL log
    ├─ [batch full OR critical anomaly]
    │   └─ subprocess: claude -p <batch> --resume SESSION_ID
    │       ├─ reads diagnostics + tap sections
    │       ├─ edits prompts (.md) or code (.py)
    │       └─ returns {session_id, result, actions_taken}
    ├─ change audit log (git diff capture)
    ├─ auto-commit (optional, easy revert)
    └─ hot-restart API if .py changed
```

### 12 Anomaly Signals
| Signal | Detection | Weight |
|--------|-----------|--------|
| `repetition_loop` | Trigram unique ratio < 0.4 | 1.0 |
| `comment_only` | All code lines are `#` comments | 0.5 |
| `template_echo` | Both `D\|` and `I\|` in answer | 1.0 |
| `self_doubt_loop` | >3 restart phrases | 0.5 |
| `format_violation` | Architect role, no `D\|`/`I\|` prefix | 1.0 |
| `think_tag_leak` | `<think>` in final answer | 0.5 |
| `near_empty` | tokens < 5, no error | 1.0 |
| `excessive_tokens` | tokens > 2000 for MCQ | 0.5 |
| `delegation_format_error` | `I\|` but missing `brief:` | 1.0 |
| `self_escalation` | Consecutive dupes in role_history | 0.5 |
| `vision_blindness` | Vision role but tokens < 10 | 1.0 |
| `silent_execution` | tools_used > 0, answer empty, no error | 0.5 |

Score = max(triggered weights), normalized to [0,1].

### Key Design Decisions
1. **Persistent session**: Same `session_id` reused across all batches via `--resume`. Claude accumulates context about bugs seen and fixes applied.
2. **Batch processing**: Not every answer triggers Claude. Default every 5 answers, or immediately on critical anomaly (score >= 1.0).
3. **Hot-swap**: Prompt `.md` edits take effect on next `/chat` call (uncached `resolve_prompt()`). No restart needed.
4. **Code fixes**: `.py` edits trigger `orchestrator_stack.py reload orchestrator`. Seeding survives via infrastructure retry.
5. **Audit trail**: `logs/debug_changes.jsonl` — full git diff per batch, easy `git revert` for rollback.
6. **Dry-run**: `--debug-dry-run` computes diagnostics and logs what would be sent, without invoking Claude.

## Anomaly Signal False-Positive Fixes

### Problem
First live `--debug` run (3 questions, 12 diagnostics, 8 Claude batches) revealed two systematic false positives:

1. **`format_violation`**: Fired on every architect answer because `RoleResult.answer` contains the *extracted* answer (D|/I| prefix already stripped by `_parse_architect_decision`), not raw model output. Short post-extraction answers like "C" or "42" have no prefix.
2. **`near_empty`**: Fired on every MCQ single-letter answer ("A", "C") since `len(tokens) < 5`.

### Fixes Applied
- `anomaly.py:detect_format_violation()`: Skip answers <50 chars (post-extraction results). Skip when `role_history` has >1 entry (delegation occurred, prefix already stripped).
- `anomaly.py:detect_near_empty()`: Skip when `scoring_method == "multiple_choice"` — single-letter answers are valid for MCQ.
- Updated test fixtures in `test_anomaly_signals.py` and `test_diagnostic_writer.py` to use >50 char answers.

## Test Suite Hang Fix

### Problem
`pytest tests/ -n 48` hung at 98-99% every time. Root cause: 3 tests in `tests/integration/test_knowledge_tools_live.py` make live HTTP calls to Semantic Scholar and Wikipedia APIs with no timeout. When APIs are slow/unreachable, xdist workers block indefinitely.

### Hanging Tests
- `TestSemanticScholarSearch::test_search_papers_returns_results`
- `TestSemanticScholarSearch::test_search_papers_with_year_filter`
- `TestWikipediaSearch::test_search_wikipedia_returns_results`

### Why Timeouts Are Hard
- `socket.setdefaulttimeout()` gets overridden by library-level timeouts
- `ThreadPoolExecutor` timeout raises exception but daemon thread keeps blocking, holding xdist worker alive
- `SIGALRM` doesn't work across xdist forked workers
- TCP probe passes (port 443 open) but HTTP API hangs after connection

### Fixes Applied
1. **`src/tools/knowledge.py`**: Added `_socket_timeout()` context manager (15s default) wrapping all 5 API functions. Provides baseline protection when libraries don't override it.
2. **`tests/integration/test_knowledge_tools_live.py`**: Added `_http_probe()` using `curl --max-time` subprocess — guaranteed killable. Skips test classes when their API endpoint doesn't respond within 5s. Semantic Scholar probe tightened to 2s.
3. **Result**: Full suite completes in ~14s with `-n 48`. No hang. Unreachable APIs skipped cleanly.

## Debug Seeding Script Root Cause & Fix

### Problem
First `--debug` run stopped after only 3 questions (12 diagnostics, 8 Claude batches). Script should have kept running continuously.

### Root Cause: Two Cascading Bugs

**Bug 1: `capture_git_diff()` captured entire dirty working tree.** `git diff` returned 30+ files from prior uncommitted work (chat.py, nodes.py, etc.). `_detect_and_apply_code_changes()` saw `.py` files and ran `orchestrator_stack.py reload orchestrator` — **killing the API on every Claude invocation** (8 times in 28 minutes). None of those `.py` files were changed by Claude.

**Bug 2: Every critical anomaly (score >= 1.0) triggered a solo Claude invocation.** With 4 configs/question, 2-3 per question hit score 1.0 (false-positive `near_empty` on MCQ + `format_violation` on extracted answers). Produced 8 blocking `subprocess.run(timeout=300)` calls instead of 2-3 batched calls. Each blocked seeding for 30-120s.

**Cascade**: Claude invocation → spurious API restart → next question's HTTP call fails → `run_batch_3way` raises → continuous loop health gate fails → consecutive failures accumulate → script breaks out.

### Evidence from Logs
- `logs/seeding_diagnostics.jsonl`: 12 records, 3 questions × 4 configs
- `logs/debug_changes.jsonl`: 8 records, each listing 30+ "files_modified" (all pre-existing dirty files)
- Timeline: 19:38 → 20:06 (28 min for 3 questions)
- Every batch logged the same 30+ files — Claude changed none of them

### Fixes Applied

| Fix | File | Description |
|-----|------|-------------|
| **Snapshot diffing** | `change_log.py` | New `capture_git_diff_stat()` hashes file content (SHA1 of first 8KB) before/after Claude invocation. `diff_snapshots()` compares — only files Claude *actually* changed trigger action. Pre-existing dirty files ignored. |
| **No batch bypass** | `claude_debugger.py` | Critical anomalies (score >= 1.0) set `_urgent` flag instead of immediate dispatch. New `end_question()` method flushes urgent batches after all configs for a question are submitted. Reduces 8 solo invocations → 1-2 batched invocations. |
| **Background Popen** | `claude_debugger.py` | `_dispatch()` uses `subprocess.Popen` (non-blocking) instead of `subprocess.run` (blocking). Seeding continues while Claude analyzes in background. Results collected on next `add_diagnostic()` call or `flush()`. |
| **Try/except guard** | `seed_specialist_routing.py` | Debugger calls wrapped in `try/except`. Errors logged as warnings, never kill the eval loop. `end_question()` called after all configs. |

### API Changes
- `add_diagnostic()`: No longer returns a value. Non-blocking. Sets `_urgent` on critical anomaly.
- `end_question()`: New method. Called after all configs for one question. Dispatches if urgent.
- `flush()`: No longer returns a value. Dispatches remaining + waits for background.
- `invoke_claude()`: Removed (replaced by private `_dispatch()` + `_process_result()`).

### Verification
93 pipeline_monitor tests pass. 3126 unit tests pass (12s).
