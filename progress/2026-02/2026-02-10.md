# Progress Report: 2026-02-10

## Debugger Tap Visibility: Inline Inference Tap in Claude Debugger

### Summary

The Claude debugger could see the final answer and role chain names (e.g., `architect_general → frontdoor → architect_coding → frontdoor`) but not WHY each hop happened — the architect's raw TOON decision, the specialist's intermediate code/REPL output, or tool calls. The inference tap (`src/inference_tap.py`) already logs every `llm_call()` with role, prompt, and full response to `/mnt/raid0/llm/tmp/inference_tap.log`. The gap: the seeding script never recorded tap byte offsets around each API call.

### Approach

Two changes: (1) seeding script captures tap file size before/after each `/chat` call, stores in `RoleResult`, passes through to `build_diagnostic()`; (2) debugger reads the tap bytes inline in the prompt instead of just displaying the offset.

### Changes

| File | Change |
|------|--------|
| `scripts/benchmark/seeding_types.py` | Added `tap_offset_bytes: int = 0` and `tap_length_bytes: int = 0` to `RoleResult` dataclass |
| `scripts/benchmark/seeding_eval.py` | Added `_TAP_PATH`, `_tap_size()` helper; wrapped `_call_orchestrator_with_slot_poll()` with `tap_before`/`tap_after`; sets `rr.tap_offset_bytes` and `rr.tap_length_bytes` on every `RoleResult` (including retry path) |
| `scripts/benchmark/seed_specialist_routing.py` | Both `build_diagnostic()` calls (primary + retry) now pass `tap_offset_bytes` and `tap_length_bytes` from `RoleResult` |
| `src/pipeline_monitor/claude_debugger.py` | Added `_read_tap_inline(offset, length)` helper (12K char truncation); `_build_prompt()` inlines tap content as fenced code block; updated `DEBUGGER_SYSTEM_PROMPT` — removed `read_tap_section.py` instruction, replaced with guidance on using inlined inference log |
| `tests/unit/test_claude_debugger.py` | 4 new tests: tap inlined when present, not inlined when zero, graceful fallback when file missing, truncation when too large |

### Key Details

- `_tap_size()` uses `os.path.getsize()` with `OSError` fallback to 0 (file may not exist)
- `_read_tap_inline()` reads up to `_MAX_TAP_INLINE * 4` raw bytes (UTF-8 worst case), decodes with `errors="replace"`, truncates text at 12K chars (~3K tokens)
- Tap range covers the entire API call including any infra retries (tap_before set once, tap_after updated after retry)
- Condition uses `tap_len > 0` (not `tap_off and tap_len`) since offset=0 is a valid start position
- `build_diagnostic()` already had `tap_offset_bytes` and `tap_length_bytes` params (added during debugger creation) — this change threads them from the actual tap file

### Budget Impact

Typical delegation chain (2-3 hops) generates ~4-8KB of tap output. With 12K char limit, the debugger gets the full chain for most questions. A batch of 5 diagnostics x 12KB = 60KB max tap content — fits comfortably in Claude's context.

| `tests/unit/test_claude_debugger.py` | 4 new REPL tests: inlined when present, not inlined when zero, graceful on missing file, both taps together |

### Bug Fixes

1. `_read_tap_inline()` had `path: str = _TAP_PATH` as default parameter. Python evaluates defaults at function definition time, so patching `_TAP_PATH` in tests didn't affect the default. Fixed to `path: str | None = None` with `if path is None: path = _TAP_PATH` inside the function body.
2. **Missing tap passthrough** (found during self-review): `build_diagnostic()` call in `seed_specialist_routing.py:2302` was missing all four tap fields (`tap_offset_bytes`, `tap_length_bytes`, `repl_tap_offset_bytes`, `repl_tap_length_bytes`). Data flowed from `seeding_eval.py` → `RoleResult` but was silently dropped at the hub. Would have resulted in zero tap content in the debugger at runtime.

## Adaptive Memory Viability Pilot (Graph-Derived Episodic Memory)

### Summary

Implemented a standalone adaptive pilot harness to test whether audit-graph-derived memory improves small-model strategic quality, explicitly independent from `seed_specialist_routing.py`.

### Implemented

| File | Purpose |
|------|---------|

| `scripts/experiments/memory_viability_report.py` | Decision/report helpers |

| `configs/memory_viability/search_space.yaml` | Adaptive mutation/search bounds |
| `docs/experiments/memory_viability.md` | Runbook |

### What the runner does

- Loads benchmark questions from debug suites (`thinking`, `coder` selectable).
- Builds memory blocks from `logs/audit_graph/*` (raw/coarse graphs + failure/hypothesis motifs).
- Evaluates fixed controls (`baseline`, `raw_subgraph`, `category_coarse`) plus adaptive variants.
- Scores deterministically via `benchmark.debug_scorer`.
- Produces per-run artifacts:
  - `results.jsonl`
  - `round_summary.csv`
  - `variants.jsonl`
  - `arm_decisions.json`
  - `decision.md`
  - `run_config.json`
