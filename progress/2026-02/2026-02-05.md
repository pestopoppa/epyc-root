## Session 25: Fix Reward Injection Timeouts & Replay Missing Rewards

### Problem

Eval script logs showed reward injections timing out during 3-way evaluation. Two patterns observed:

1. **Sporadic**: 1 out of 4 rewards times out (3/4 injected)
2. **Cascading**: After architect_coding 600s/0-token timeout, all reward injections fail (0/3 injected)

16 out of 44 evaluated questions had incomplete reward injection (22 missed out of 51 total rewards).

### Root Cause

The `/chat/reward` endpoint (`src/api/routes/chat.py:120`) was declared `async` but called `store_external_reward()` **synchronously**, blocking the FastAPI event loop. That function performs three blocking operations:

1. **HTTP POST to embedder** (port 8090) — sync, 10s timeout
2. **FAISS similarity search** + SQLite read
3. **SQLite write with `conn.commit()`**

Combined with uvicorn running a **single worker** (no `--workers` flag in `orchestrator_stack.py`), any blocking call freezes all concurrent requests. The eval script's 10-second reward injection timeout (`seed_specialist_routing.py:896`) was too tight for this.

### Fix 1: `asyncio.to_thread()` for `/chat/reward`

Offloaded the blocking `store_external_reward()` call to a thread, matching the pattern already used by `score_completed_task()`:

```python
# Before (blocking event loop):
success = store_external_reward(state, ...)

# After (thread-offloaded):
success = await asyncio.to_thread(store_external_reward, state, ...)
```

### Fix 2: Replay Script for Missing Rewards

Created `scripts/benchmark/replay_missing_rewards.py` to backfill failed reward injections from checkpoint JSONL data. Checkpoints store the computed reward values but don't record which specific rewards succeeded — so the script re-injects all rewards for affected questions (duplicates are harmless, they just update existing Q-values incrementally).

Features:
- Scans all `3way_*.jsonl` checkpoint files for `rewards_injected < len(rewards)`
- Uses 30s timeout (vs original 10s)
- Marks source as `3way_eval_replay` for traceability
- `--dry-run` mode for inspection
- `--file` flag for specific checkpoint files

### Fix 3: Identified "seen" tracking gap

Questions where reward injection fails are still marked as "seen" in `seen_questions.jsonl`. The `record_seen()` call happens unconditionally after `_checkpoint_3way()`, regardless of reward injection success. This means re-running the eval script will NOT re-sample these questions — the replay script is required.

### Replay Results

```bash

# Found 16 questions with missing rewards (22 missed out of 51 total)
# Replay complete: 51/51 rewards injected.
```

All 16 questions backfilled successfully. Zero timeouts with the `asyncio.to_thread()` fix active.

## Session 26: Parallel BGE Embedder Migration (896 → 1024 dims)

### Problem

Reward injection timeouts were still occurring because the single embedder server (port 8090 running Qwen2.5-Coder-0.5B) could become slow/unresponsive during load.

### Solution: Parallel BGE Embedder Pool

Replaced single Qwen embedder with 6 parallel BGE-large instances using a "probe-first" architecture:

1. **Model switch**: Qwen2.5-Coder-0.5B (896 dims) → BGE-large-en-v1.5 (1024 dims)
2. **Parallel instances**: 6 identical servers on ports 8090-8095
3. **Probe-first approach**:
   - Phase 1: Lightweight `/health` probes to all servers (~1ms)
   - Phase 2: Embedding request sent only to first responder
   - Avoids wasting compute on 5 unused servers

## Session 28: Canvas Control Plane & Research Context Tracker Architecture

### Problem

The JSON Canvas infrastructure (Session 27) provided export/import but lacked integration with the inference pipeline. User identified that canvas should serve as a control plane for:
1. Routing confidence steering
2. MemRL retrieval filtering
3. Task decomposition visualization
4. Failure veto overrides
5. Live monitoring/debugging

### Discovery: Multi-Step Planning Already Exists

Investigation revealed extensive planning infrastructure already in place:

| Component | Location | Status |
|-----------|----------|--------|
| TaskIR schema | `orchestration/task_ir.schema.json` | ✓ Production |
| Wave executor | `orchestration/wave_executor.py` | ✓ Production |
| Architect decomposition | model_registry `architect_general` | ✓ Production |
| Step dependencies | TaskIR `depends_on` field | ✓ Supported |

Gaps identified:
- Review loop skipped in parallel mode
- No dynamic step insertion mid-execution
- Fixed 2-5 step granularity

### Architecture Decision: Profile + Plan Canvas, Text Steering

After analysis, chose **Option B**:

| Mode | Implementation | User Interface |
|------|----------------|----------------|
| **Profile Canvas** | User knowledge graph as system context | Obsidian editing |
| **Plan Canvas** | TaskIR DAG visualization | Obsidian editing |
| **Steering** | Text commands, NOT canvas | `prefer_role()`, `veto_role()`, `force_role()` |

**Rationale**: Canvas excels at spatial/relational tasks (profile graphs, TaskIR DAGs). Simple overrides are faster via text commands than canvas round-trips.

### Research Context Tracker Design

New handoff for tracking REPL tool results as a DAG:

**Core concept**: When a model uses tools repeatedly, structure results as a DAG with:
- Node IDs (`G1`, `R1`, `F1` for Glob/Grep/Read)
- Parent-child relationships (which result led to which query)
- Cross-references (semantic similarity between results)

**Key decision**: Semantic cross-reference detection is **ON by default** using existing BGE embedder pool (ports 8090-8095). ~500ms overhead per node is acceptable. Flag `use_semantic=False` to disable.

```python
class ResearchContext:
    def __init__(
        self,
        use_semantic: bool = True,  # ON by default
        embedder: "ParallelEmbedderClient | None" = None,
    ):
        self.nodes: dict[str, ResearchNode] = {}
        self.use_semantic = use_semantic
        self.embeddings: dict[str, "np.ndarray"] = {}
```

## Session 29: Research Context Tracker Implementation

### Summary

Implemented the Research Context Tracker from the handoff created in Session 28. This provides DAG-based tracking of REPL tool invocations with automatic ID assignment, parent-child relationships, and cross-reference detection (string + semantic).

### Files Created

| File | Lines | Purpose |
|------|-------|---------|
| `src/research_context.py` | ~270 | Core ResearchContext class with nodes, refs, render |
| `tests/unit/test_research_context.py` | ~480 | 36 unit + integration tests |

### Files Modified

| File | Changes |
|------|---------|
| `src/repl_environment/environment.py` | Added `_research_context`, `_last_research_node` attributes |
| `src/repl_environment/context.py` | Hooked `_invoke_tool()` to track TOOL calls |
| `src/repl_environment/file_exploration.py` | Added `_track_research()` helper, hooked peek/grep/list_dir |
| `src/repl_environment/state.py` | `get_state()` includes research context when >= 3 nodes; checkpoint/restore support |

### Core Features

**Node ID Assignment**:
- Tool-specific prefixes: G (grep), P (peek), L (llm_call), T (TOOL), D (list_dir), etc.
- Auto-incrementing: G1, G2, G3...

**Parent Tracking**:
- `_last_research_node` automatically set as parent for sequential calls
- Explicit `parent_id` parameter for manual override
- `get_lineage(node_id)` returns ancestor chain

**Cross-Reference Detection**:
- **String refs**: Regex finds explicit mentions like "see G1", "based on P2"
- **Semantic refs**: Cosine similarity > 0.7 using existing BGE embedder pool (ports 8090-8095)
- Graceful fallback if embedder unavailable

**Context Injection**:
- `get_state()` includes rendered tree when >= 3 nodes
- Tree shows: node ID, tool, query, summary, refs, status (analyzed/pending/stale)

### Example Output

```

Progress: 1 analyzed, 2 pending, 0 stale

[+] G1: grep(pattern='error')

  [?] P1: peek(n=100)

  [?] L1: llm_call(summarize)
```

### Design Decisions

| Decision | Implementation |
|----------|----------------|
| State injection threshold | >= 3 nodes (avoids noise on trivial sessions) |
| Parent inference | Uses `_last_research_node` as default |
| Semantic default | ON (`use_semantic=True`), graceful fallback |
| MCP tools | Skipped in v1 (render() injection sufficient) |

## Session 30: Condense Tool Timing Log Output

### Problem

Eval logs were printing one line per tool call, creating 28+ lines of output for a single REPL turn:

```
      web_search: 190ms (ok)
      web_search: 162ms (ok)
      search_papers: 1ms (ok)
      ... (25 more lines)
```

### Solution

Modified `_format_tool_timings()` in `scripts/benchmark/eval_log_format.py` to aggregate by tool name:

```
      web_search: 14x, 9145ms total (14 ok)
      search_wikipedia: 6x, 1264ms total (6 ok)
      fetch_wikipedia: 3x, 494ms total (3 ok)
      search_papers: 2x, 2ms total (2 ok)
      search_arxiv: 1ms (ok)
      search_books: 1ms (ok)
```

**Format rules**:
- Single calls: compact `name: Xms (ok/fail)`
- Multiple calls: `name: Nx, Xms total (N ok)` or `(N ok, M fail)` if mixed
