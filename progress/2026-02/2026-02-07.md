## `/refactor` Slash Command — Codebase Refactoring Analysis (Session 5)

### Summary
Designed and implemented a `/refactor` slash command that analyzes a code scope for technical debt and produces a recurring refactoring handoff. Started from a ~500-line enterprise consulting prompt template, stripped it down to a focused 267-line command aligned with project conventions.

### Design Decisions

1. **Slash command, not agent definition** — existing `agents/` files were never used; a command in `.claude/commands/` integrates directly with Claude Code's `/` invocation
2. **Recurring handoff** — output goes to `handoffs/active/refactoring.md` which is overwritten (never archived) each cycle, creating a continuous improvement loop
3. **Analysis only, no implementation** — the command produces an actionable handoff for a separate implementation session
4. **Risk-weighted prioritization** — `Priority = (Severity × Freq × Risk_Multiplier) / Effort` where Risk_Multiplier comes from test coverage analysis (1.0/1.5/2.0)
5. **Phase 0 safety net** — untested code gets tests added before refactoring in later phases
6. **Test & coverage analysis** integrated as Step 4 — maps source→test coverage, test quality, test-to-source coupling from git history, CI/gate constraints

### Prompt Review Findings (from user's draft)
- Enterprise framing removed (team turnover, GDPR, Gantt charts, budget)
- 10-section consulting deliverable replaced with handoff format
- XML wrapper tags replaced with Markdown sections
- Scope narrowed from security+performance+compliance+refactoring to refactoring only
- "Not a code rewriter" constraint reframed: agent produces actionable handoff with code sketches, not just generic advice

### Analysis Pipeline (8 steps)
1. Scope Discovery — file inventory, line counts, import graph, public API surface
2. Complexity Analysis — structural complexity, duplication, dead code, naming
3. Dependency & Coupling — circular imports, god modules, shotgun surgery, layer violations
4. Test & Coverage Analysis — coverage mapping, test quality, git coupling, CI constraints
5. Risk-Prioritized Issue List — scored with test coverage risk multiplier
6. Design Refactoring Plan — phased, independently deployable, dependency-aware
7. Write Handoff — to `handoffs/active/refactoring.md` using project template
8. Validate — verify referenced files exist, run gates

## REPL Prompt Tuning — Response Depth & Consult Escalation (Session 10)

### Summary
Analyzed 3-way seeding eval results showing SELF:repl producing 13 tokens vs SELF:direct's 219 for the same HellaSwag question, and ARCHITECT failing entirely (465s, delegated to coder). Root-caused three distinct issues and implemented prompt-level fixes to address response depth and escalation ergonomics.

### Analysis Findings

1. **SELF:repl brevity** — REPL system prompt listed "reasoning" in the short-circuit list ("For simple questions (factual, reasoning, multiple-choice), answer directly with FINAL(answer)"), causing the model to emit `FINAL("D")` instead of reasoning through answers. The code-generation framing + terse FINAL() examples reinforced this.

2. **ARCHITECT failure on simple MC** — The architect prompt forces a meta-routing decision (`D|answer` vs `I|brief:|to:role`) before answering. For HellaSwag, the 235B model over-delegates to `coder_primary` (a code specialist) for a commonsense reading comprehension question. This is expected behavior given the architect role's design — it's optimized for complex multi-step tasks, not simple MC.

3. **Missing consult path** — Escalation was binary: either handle it yourself or `escalate()` (full control transfer with TOON scaffolding). No lightweight "ask a stronger model" option existed in the prompt, even though `llm_call(prompt, role="architect")` was already available as a REPL primitive.

### Changes Made

#### 1. REPL System Prompt — Response Depth Matching (`builder.py:92-98`)
**Before**: "For simple questions (factual, reasoning, multiple-choice), answer directly with FINAL(answer)."
**After**: "Match your response depth to the user's request. For factual lookups and multiple-choice, answer concisely. For explanations or reasoning, think thoroughly before FINAL(answer)."

#### 2. REPL Rules — Split Direct/Reasoning/Tools (`constants.py:94-98`)
**Before**: Single "Answer directly for: factual, reasoning, multiple-choice, math, general knowledge"
**After**:
- **Answer directly** for: factual lookups, multiple-choice, short math
- **Reason thoroughly** for: explanations, analysis, multi-step problems, "why" questions
- **Use tools** for: file access, web search, current events, running code
- **Match depth to request**: concise for simple, detailed for complex

#### 3. REPL Examples — Added Verbose Patterns (`constants.py:106-111`)
- Kept terse examples: `FINAL("Paris")`, `FINAL("B")`, `FINAL("42")`
- Added multi-step reasoning example: `FINAL("Step 1: ... Step 2: ... Therefore ...")`
- Added analysis example: `FINAL("The function has O(n log n) complexity because...")`

#### 4. Escalation — Consult vs Handoff (`constants.py:121-127`)
New `## ESCALATION (two modes)` section replacing the old single-line `escalate()` guidance:
- **Consult**: `llm_call("Be concise. " + question, role="architect")` — frontdoor keeps control, formats answer in FINAL(). Prompt kept minimal (no REPL state).
- **Handoff**: `escalate(reason)` — full control transfer for multi-step investigation/delegation.

### Design Decisions
- **Did NOT add escalation hint to "Reason thoroughly" bullet** — considered and rejected. Coupling escalation to reasoning uncertainty would cause over-escalation, turning 18 t/s frontdoor answers into 6.75 t/s architect answers unnecessarily.
- **Did NOT add new eval method** — consult is a tool call within SELF:repl, not a new routing path. Shows up as `1 tools` in eval logs.
- **Did NOT modify architect prompt** — architect failure on simple MC is by design (routing scaffolding overhead). Production routing correctly avoids sending simple questions to the architect.

## REPL Import Unblocking & Dynamic Cost Routing (Session 11)

### Summary
Discovered that RestrictedPython executor blocks ALL imports (including scipy, numpy, math), preventing the REPL from doing numerical computation. Switched production to the standard AST-blocklist executor which only blocks dangerous modules. Also replaced static cost tiers in MemRL routing with actual elapsed-seconds data from seeded memories. Fixed lookup script to accept suite-prefixed question IDs from seeding logs.

### Analysis: Import Blocking

The REPL had two executor backends:

| Executor | Import policy | When used |
|---|---|---|
| `REPLEnvironment` (standard) | AST blocklist — blocks `os, sys, subprocess` etc., allows `scipy, numpy, math` | `restricted_python: false` |
| `_RestrictedREPLEnvironment` | Removes `__import__` entirely — blocks ALL imports | `restricted_python: true` (was production default) |

**Evidence**: On math/math500 question (find minimum of x^4 + 4y^2 + 4z^4 given xyz=2), SELF:repl tried `from scipy.optimize import minimize` — correct approach, but the import was blocked. Burned 411s and 1485 tokens writing code that couldn't execute. Meanwhile ARCHITECT solved it analytically in 87s.

### Changes Made

#### 1. Switch Production to Standard Executor (`features.py:315`)
Changed `restricted_python` production default from `True` to `False`. The AST-blocklist executor (`security.py`) already blocks dangerous modules (`os, sys, subprocess, socket, shutil, pathlib, tempfile, multiprocessing, threading, ctypes, pickle, importlib`) while allowing scientific libraries.

#### 2. Update REPL Import Rules (`constants.py:101-102`)
**Before**: `"NO IMPORTS - import/from are BLOCKED"`
**After**: `"SAFE IMPORTS ONLY"` with explicit whitelist (`math, json, re, numpy, scipy, itertools, collections, functools, statistics, datetime, fractions, decimal`) and blocklist (`os, sys, subprocess, socket`).

#### 3. Dynamic Cost Routing (`retriever.py:393-452`)
**Before**: `score = Q(action) / static_cost_tier` where tiers were arbitrary integers (SELF=2, ARCHITECT=4)
**After**: `score = Q(action) / avg_elapsed_seconds` using actual compute time from seeded memories. Falls back to static tiers only when no elapsed data is stored yet.

|---|---|---|---|---|

| HellaSwag: SELF:repl | 1.0 | 6.8s | 0.147 | **wins** (both pass, faster) |
| HellaSwag: ARCHITECT | 0.0 | 465.3s | 0.000 | |
| Math T3: SELF:direct | 0.0 | 66.2s | 0.000 | |
| Math T3: SELF:repl | 0.0 | 411.6s | 0.000 | |
| Math T3: ARCHITECT | 1.0 | 86.9s | 0.012 | **wins** (only one that passes) |

#### 4. Lookup Script Fix (`lookup_question.py`)
Fixed `parse_question_id()` to strip suite prefix from seeding log format (`thinking/hellaswag_21243` → `hellaswag_21243`). The `PREFIX_MAP` already maps `hellaswag_` → `thinking` suite.

### Design Decisions
- **Standard executor over RestrictedPython** — The AST blocklist is sufficient because the code is generated by our own LLM, not adversarial users. The security model protects against accidental dangerous operations, not sandbox escapes.
- **Elapsed seconds over token count** — Elapsed time captures the full cost (model speed + delegation overhead + tool calls). Token count misses the 465s architect delegation chain that produced only 2403 tokens.
- **Fallback to static tiers** — New deployments with empty episodic memory still get reasonable routing before any seeding data exists.

## Seeding Question Interleaving (Session 12)

### Summary
Changed seeding script question iteration from sequential-by-suite to round-robin interleaving across suites, so the orchestrator sees diverse question types early in each run.

### Problem
`sample_unseen_questions()` returned all questions grouped by suite: all 30 thinking, then all 30 general, then all 30 math, etc. During debugging, this meant seeing 30 identical-type questions before any variety. It also meant bursty workload patterns on model servers.

### Change
**`seed_specialist_routing.py:358-393`** — Collect per-suite samples into separate lists, then interleave round-robin:
- Before: `thinking_q1, thinking_q2, ..., thinking_q30, general_q1, ...`
- After: `thinking_q1, general_q1, math_q1, agentic_q1, ..., thinking_q2, general_q2, ...`

Suites with fewer questions get skipped in later rounds. Total count and content identical — only order changes.

### Files Modified
| File | Change |
|------|--------|
| `scripts/benchmark/seed_specialist_routing.py` | `sample_unseen_questions()`: round-robin interleaving |
