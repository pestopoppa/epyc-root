## Session 4: Seeding Run Fix + Memory Cleanup

### Problem

First seeding run with mode-advantage tasks revealed architect_general (port 8083) timing out with 0 tokens generated. After a timeout, the llama-server continued grinding on the stale request, causing cascading timeouts on subsequent combos targeting the same port.

### Changes

**Slot erase on timeout** (`scripts/benchmark/seed_specialist_routing.py`)
- Added `_erase_slots(port)` helper: queries `/slots` on a llama-server port, finds slots with `is_processing=True`, and POSTs `/slots/{id}?action=erase` to force-cancel
- Called automatically after error+timeout on heavy ports (8083, 8084, 8085, 8087) when `tokens_generated == 0` (server never started generating — stuck in prompt eval or hung)
- Best-effort (exceptions swallowed) — doesn't interfere with normal flow
- Safety analysis confirmed: `force_role` bypasses escalation, seeding is sequential per question, no risk of erasing a legitimate concurrent request
