## Session 3: Fix orchestrator_eval.py Pre-Launch Bugs

### Problem

Pre-launch review of `orchestrator_eval.py` found three bugs that would silently corrupt MemRL evaluation results:

1. **`mode` field missing from ChatResponse** — The Q-scorer builds action strings as `"role:mode"` (e.g., `"frontdoor:direct"`). Without a `mode` field, eval read `routing_strategy` instead, producing `"frontdoor:learned"` — fragmenting identical execution paths into different Q-values depending on whether MemRL or heuristics chose the route.

2. **`ORCHESTRATOR_REACT_MODE=1` missing from env vars** — Both `orchestrator_stack.py` and `orchestrator_eval.py` restart helpers set all feature flags except react mode. Without it, the heuristic fallback can never route to react mode during cold start.

3. **Direct mode lock-in** — `_should_use_direct_mode()` returns True for ~90% of benchmark prompts (anything with context ≤ 20KB and no file-operation keywords). Direct mode = single LLM call with no tools, no REPL, no self-escalation. MemRL never sees non-direct modes, so it can never learn when tools/escalation help.

### Changes

6 files modified:

1. **`src/api/models/responses.py`** — Added `mode: str = Field(default="", description="Execution mode: direct, react, repl, delegated, or mock")` to `ChatResponse`.

2. **`src/api/routes/chat_pipeline.py`** — Set `mode=` in all 6 `ChatResponse(...)` constructors:
   - `_execute_mock()` → `mode="mock"`
   - `_execute_delegated()` → `mode="delegated"`
   - `_execute_react()` → `mode="react"`
   - `_execute_direct()` → `mode="direct"`
   - `_execute_repl()` (two-stage) → `mode="repl"`
   - `_execute_repl()` (main) → `mode="repl"`

3. **`scripts/benchmark/orchestrator_eval.py`** — Changed `result.get("routing_strategy", "unknown")` to `result.get("mode", "unknown")`. Added `ORCHESTRATOR_REACT_MODE=1` to `restart_orchestrator_api()` env vars.

4. **`scripts/server/orchestrator_stack.py`** — Added `env["ORCHESTRATOR_REACT_MODE"] = "1"` to `start_orchestrator()`.

5. **`src/api/routes/chat_routing.py`** — Changed `_select_mode()` heuristic fallback from `direct → react → repl` to `react → repl`. REPL is a superset of direct (model can `FINAL("answer")` immediately for simple questions, or use tools if needed). This gives MemRL exposure to non-direct modes during cold start.

### Design Rationale

- **Why "repl" as default, not "direct"?** REPL mode is strictly a superset. The model can call `FINAL("answer")` on the first turn for simple questions (equivalent to direct), but it also has the *option* to use tools, escalate, or explore. MemRL learns from outcomes — if direct-style answering works, Q-values will converge to favor immediate FINAL(). But the model needs the opportunity to try tools first.
- **`_should_use_direct_mode()` kept but uncalled from `_select_mode()`**. The function is still defined and importable (test_api_imports.py validates this). It may be useful later when MemRL explicitly learns to choose direct mode.

## Session 6: Eval Launch Fixes — ORCHESTRATOR_REPL Collision + Output Improvements

### Problem 1: ORCHESTRATOR_REPL=1 crashes pydantic-settings

`start_orchestrator()` set `ORCHESTRATOR_REPL=1` as a feature flag. But `OrchestratorSettings` in `config.py` has `repl: REPLSettings` (a nested model) with `env_prefix="ORCHESTRATOR_"`. Pydantic-settings tried to parse `1` as a `REPLSettings` model → `ValidationError`. The orchestrator API wouldn't start.

**Fix**: Removed `ORCHESTRATOR_REPL` from env vars in both `orchestrator_stack.py` and `orchestrator_eval.py`. The `repl` feature flag defaults to `True` in `features.py` already — no env var needed.

### Problem 2: t/s includes prompt eval time

`evaluate_prompt()` calculated `tps = tokens_generated / elapsed_seconds`, which includes prompt evaluation. For long prompts this severely underreports generation speed (e.g., 1.2 t/s reported when actual generation speed is 23+ t/s).

**Fix**: Use `predicted_tps` from ChatResponse (clean generation-only metric from llama.cpp). Falls back to wall-clock calculation only when `predicted_tps` is unavailable.

### Problem 3: No tool or escalation visibility

Live output only showed route and score. No way to see which tools were invoked, how many, or whether escalation/delegation occurred.

**Fix**: Added `tools_called` and `role_history` to `EvalResult`. Live output now shows a detail line below each result when escalation or tool use occurred:

```

                                           chain: frontdoor → coder_escalation  tools(3): python_eval, web_search, lint
```

Summary section now includes mode distribution and per-tool usage counts.
