### Design Decisions
- **Virtual memory over lossy summarization**: Original `_maybe_compact_context()` truncated at 8K chars and used lossy `worker_summarize`. New design externalizes full context (zero loss) and generates a navigable index with line coordinates for random access via `read_file()`.
- **worker_explore over worker_summarize**: `worker_summarize` is in `SERIAL_ROLES` and would serialize compaction. `worker_explore` uses the same 7B server but is not serial-gated.
- **Line coordinates in index**: Per user feedback, the compaction index includes line ranges so the model can one-shot `read_file(path, offset=N, limit=M)` instead of browsing.
- **Two-stage clearing**: C3 fires at 40% context, C1 at 60%. C3 is cheap (regex), C1 requires LLM call. This gives a graduated response to context pressure.

## Phase 2B-Sidecar: Corpus Draft Source in llama.cpp Speculation Loop

Instead of injecting corpus snippets into prompts (Phase 2A), the sidecar feeds token-level n-grams directly into the speculation loop's `nc_static` cache via the pluggable `common_speculative_state` architecture.

### Integration Benchmark Results ŌĆö CLOSED

Benchmarked all four sidecar modes on 30B (Qwen3-Coder-30B-A3B) with V3 corpus on port 9081:

| Mode | Avg t/s | vs Baseline | vs Phase 2A |
|------|---------|-------------|-------------|
| **Phase 2A (prompt injection)** | **29.9** | **+16%** | ŌĆö |
| Pre-query (`--corpus-refresh 0`) | 26.0 | +1% | -13% |
| Async (`--corpus-refresh -1`) | 25.8 | 0% | -14% |
| Blocking (`--corpus-refresh 64`) | 24.9 | -3% | -17% |
| Blocking + ngram_cache | 23.6 | -8% | -21% |

#### Root Cause Analysis
- **Blocking mode**: SQLite queries across 16 shards took 24.3s per request for only 41 accepted tokens (760ms/accepted token). I/O stalls the entire speculation pipeline.
- **Pre-query/Async**: Eliminated I/O blocking, performance returns to baseline (~26 t/s). But corpus n-grams in `nc_static` produce only ~25-30% sidecar acceptance rate ŌĆö not enough to improve over draft-model-only speculation.
- **Why Phase 2A wins**: Prompt injection puts corpus tokens directly into the context. The model's own prompt lookup builds n-gram matches from those tokens at near-zero cost. The sidecar tokenizes snippets externally and injects into `nc_static`, but those token-level n-grams don't match what the draft model proposes.

#### Decision
Phase 2B-Sidecar CLOSED. Phase 2A prompt injection remains the production approach (+16%/+72% on 30B/32B). Branch `feature/corpus-sidecar` preserved on `llama.cpp-experimental` for reference. No cherry-pick to production.

## Q5: SoftMatcha v2 GloVe/FastText Code Vocabulary Coverage

### Summary
Empirically evaluated GloVe and FastText vocabulary coverage on V3 code corpus (76.6M snippets, Python/C++/Rust) to determine if SoftMatcha v2 soft matching is viable for code retrieval.

### Results ŌĆö Hypothesis WRONG

| Model | Overall | C++ | Python | Rust |
|-------|---------|-----|--------|------|
| GloVe (400K vocab) | **79.2%** | 79.3% | 78.0% | 84.4% |
| FastText known (2M vocab) | **74.0%** | 76.7% | 69.8% | 80.6% |
| FastText subword (magnitude > 0.5) | **86.1%** | 87.1% | 84.6% | 88.4% |

Original hypothesis was <15% coverage. Actual coverage is 74-86%.

### Per-Category Breakdown (the nuanced picture)

| Category | GloVe | FT Known | FT Subword |
|----------|-------|----------|------------|
| operators/punctuation | 100% | 100% | 100% |
| keywords | 97.9% | 100% | 100% |
| common identifiers | 99.0% | 100% | 100% |
| type names | 96.1% | 98.6% | 100% |
| code-specific | 99.7% | 100% | 99.5% |
| compound identifiers | 66.8% | **2.9%** | 13.2% |
| numeric literals | 90.7% | 94.2% | 98.7% |
| other | 57.5% | 65.7% | 93.9% |

**Key insight**: High overall coverage is dominated by trivially matchable tokens (operators, English keywords). The actual code-specific compound identifiers (`self.assertEqual`, `camelCase`, `getData`) have <3% coverage in FastText known vocab. GloVe's 66.8% compound coverage is inflated by the `_` token (19,960 of 30,109 compound tokens).

### Moses Tokenizer Artifact
Top OOV tokens are XML entities (`&quot;`, `&apos;`, `&gt;`, `&#91;`) from sacremoses XML-escaping ŌĆö not actual code vocabulary gaps.

### Step 2: SoftMatcha Test Index + Query Comparison

Built HDF5 inverted file index from 10K V3 snippets using GloVe embeddings:
- 2,496,007 tokens, 19,211 vocabulary, 55.9MB index (5.6x corpus), 23.6s build time

**NL query results ŌĆö 0 matches at ALL thresholds (1.0 to 0.5)**:

| Query | Exact (1.0) | Soft (0.5) |
|-------|-------------|------------|
| calculate loss predictions | 0 | 0 |
| async retry exponential backoff | 0 | 0 |
| binary search tree insert delete | 0 | 0 |
| thread safe lru cache | 0 | 0 |
| recursive descent parser | 0 | 0 |
| shortest path graph dijkstra | 0 | 0 |

**Root cause**: SoftMatcha is a sequential pattern matcher ŌĆö ALL query tokens must appear consecutively. Natural language phrases never appear as consecutive sequences in code.

**Code-pattern diagnostic queries**:

| Query | Exact | Soft (0.5) | Notes |
|-------|-------|------------|-------|
| `def` | 5,827 | 5,857 | +30 noise |
| `return` | 9,955 | 57,691 | `for` matches at 0.53 (meaningless) |
| `for i in` | 354 | 3,210 | soft sample: `to it .` (garbage) |
| `if err` | 5 | 369 | `,` matches `if` at 0.53 |
| `int main` | 101 | 119 | small legitimate expansion |

**Moses tokenizer issues**: `BinarySearchTree` ŌåÆ `binarysearchtree` (1 lowercase token), `self.search(query)` stays joined, `_` splits from identifiers.

### Q5 Decision: CLOSED

SoftMatcha v2 soft matching is **architecturally unsuitable for code retrieval**:
1. Sequential pattern matching doesn't work for code (no NL phrases in code)
2. GloVe word similarity is meaningless between code tokens (`return` Ōēł `for`)
3. Moses tokenizer destroys code structure (lowercasing, joining method calls)
4. Even if it worked, V3 exact n-gram matching already delivers +16%/+72% on 30B/32B

## Q3 CLOSED: First-20-Token Re-query vs Keyword-only Retrieval (2026-02-19)

### Method
Ablation comparing V3 corpus 4-gram hits for three retrieval strategies across 6 quality gate prompts (using 32B baseline outputs from Feb 15 quality gate):

| Strategy | Gram Hits | vs Keyword |
|----------|-----------|------------|
| A) Keyword NL extraction (production) | 21 | baseline |
| B) First-20-token re-query | 53 | +152% |
| C) Full output n-grams (theoretical ceiling) | 981 | +4571% |

### Key Findings
1. **V3 n-gram index works correctly for code** ŌĆö `"for i in range"` ŌåÆ 24,987 matches, `"from typing import optional"` ŌåÆ 58,161. NL queries hit when they overlap with code comments (`"binary search tree in"` ŌåÆ 57, `"token bucket rate limiter"` ŌåÆ 18).
2. **First-20-token re-query doubles gram hits** (53 vs 21) because actual code tokens match the code corpus better than NL keywords.
3. **Full output dwarfs both** (981 hits) ŌĆö the model's own generated code is the best n-gram source by 47x over re-query.
4. **Re-query latency: 185ms** (cached V3 retriever, 13 shards touched). At 12.6 t/s, that's 2.3 tokens of generation time lost.

### Decision
Q3 CLOSED ŌĆö keyword-only retrieval is sufficient. Prompt lookup already self-matches against the model's own output during generation, providing 47x more n-gram material than any corpus re-query. The 185ms latency cost is not justified by the marginal +32 gram hits.
