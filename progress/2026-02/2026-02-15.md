Phase 2A proved corpus injection is a speed optimization — outputs are mostly identical with/without it. The model ignores the `## Reference Code` section because there's no instruction to use it. Phase 2B-Quality changes injection from passive (n-gram fodder) to active (quality RAG).


## Qwen3-TTS Phase 4: llama.cpp Native C++ Pipeline

Built custom `llama-tts-qwen3` binary in `llama.cpp-experimental` (branch: `feature/qwen3-tts-support`).

### Architecture

Three-model pipeline: Talker GGUF (Q4_K_M, 462MB) + Code Predictor GGUF (Q8_0, 395MB) + sidecar weights (float32, 1.5GB). Sidecar holds text_embedding, text_projection MLP, codec_embedding, 15 CP embeddings, 15 CP lm_heads.

### What Works

- Binary compiles, loads all models, generates 16 codec tokens per frame
- Multi-head Code Predictor with correct per-step lm_head (via `llama_set_embeddings` + `llama_get_embeddings_ith`)
- Talker hidden state extraction for CP `past_hidden` input
- Temperature/top-k/repetition-penalty sampling
- Stale-frame EOS detection heuristic
- End-to-end pipeline: C++ binary → codec tokens → Python Tokenizer Decoder → WAV (24kHz)
- Codec generation speed: 1.5x real-time (vs PyTorch 0.9x RT)

### What's Broken: Audio Output is Noise

**Whisper transcription of 4s output**: "Assistant. Hello? This is the day of assistance!" (expected: "Hello, this is a test of the text to speech system.")

Root cause NOT yet identified. Suspects:
1. **Chat template leaking**: "assistant" heard in output suggests `<|im_start|>assistant\n` prefix is being spoken
2. **Text embedding/projection mismatch**: CPU float32 text_projection may not match PyTorch bf16 computation
3. **Codec embedding sum formula wrong**: The way 16 code embeddings + text hidden are combined for next Talker input may diverge from reference
4. **Wrong prompt format**: Talker may need different formatting than `<|im_start|>assistant\n{text}<|im_end|>\n<|im_start|>assistant\n`

### Bugs Fixed (6 total)

1. Sidecar header size mismatch (C 36B vs Python 32B)
2. CP vocab size mismatch (2048 not 3072)
3. llama_kv_cache_clear API removed in upstream
4. Logging not initialized
5. Single lm_head for all CP steps → multi-head via embeddings API
6. CP past_hidden using codec_embedding → now uses Talker hidden states

### Next Step

Generate PyTorch reference codec tokens for same text, compare token-by-token against C++ output to find divergence point. This will isolate whether the issue is in text projection, embedding injection, generation loop, or prompt format.

## Nightshift Integration (Automated Overnight Maintenance)

Implemented [nightshift](https://github.com/marcus/nightshift) for automated overnight code maintenance via Claude Code CLI. Full integration took ~6 hours across multiple iterations solving permission, isolation, and architecture issues.

### Architecture

```
systemd timer (02:30 daily)
  → run_wrapper.sh
    → inference_guard.sh (checks llama-server RSS)
    → nightshift run
      → PATH shadow claude (scripts/nightshift/bin/claude)
        → docker exec into devcontainer (bypassPermissions)
          → claude in worktree (/mnt/raid0/llm/claude-nightshift)
```

### Key Design Decisions

1. **Dedicated git worktree** (`/mnt/raid0/llm/claude-nightshift`): Nightshift creates branches for PRs. Running in the main checkout disrupted parallel agents. Worktree is detached HEAD, synced to latest `main` before each run.

2. **Devcontainer routing**: Nightshift's `--dangerously-skip-permissions` flag didn't work reliably for file writes. The devcontainer already has `defaultMode: bypassPermissions` with 1037 allow rules. PATH shadowing routes nightshift's `claude` invocations through `docker exec`.

3. **Inference guard**: Pre-run check of llama-server RSS via `/proc/*/status`. When >200GB RAM detected (inference active), restricts to analysis-only tasks (lint-fix, td-review, docs-backfill, dead-code, doc-drift). Tested live: correctly detected 557GB RSS.

4. **Budget management**: 90% daily budget cap, 5% reserve for morning interactive use. Nightshift's built-in budget system prevents token waste when out of tasks.

### Task Set (Aggressive, 11 tasks)

`lint-fix`, `bug-finder`, `auto-dry`, `td-review`, `docs-backfill`, `skill-groom`, `dead-code`, `test-gap`, `security-footgun`, `perf-regression`, `doc-drift`

### Issues Encountered & Fixed

| Issue | Root Cause | Fix |
|-------|-----------|-----|
| Nested session block | `CLAUDECODE` env var set | Unset env var (systemd has clean env) |
| No file writes despite skip-permissions flag | Claude Code permission system unreliable headless | Route through devcontainer with bypassPermissions |
| Branch switching disrupts parallel agents | Nightshift creates branches in working dir | Dedicated git worktree |
| Git refs permission denied in container | Container runs as `node`, repo owned by `daniele` | `core.sharedRepository group` + chmod g+w |
| `.gitconfig` readonly bind mount | Docker bind mount | XDG git config fallback (`/home/node/.config/git/config`) |

## Bug Fixes: Seeding TUI & Kill Script

### TUI Inference Stream Not Resetting Between Questions

**Problem**: When the 3-way seeding TUI (`--tui`) moved to a new question, the inference stream panel showed stale output from the previous question. The REPL panel correctly reset but the inference stream did not.

**Root cause**: In `seeding_tui.py:504-509`, `update_progress()` called `self._repl_tailer.reset_section()` on question change but omitted the equivalent call for `self._tailer` (inference stream). The `TapTailer._current_section` deque retained old content until a new `========` separator appeared in the tap file.

**Fix**: Added `self._tailer.reset_section()` alongside the existing REPL reset when `qid` changes.

**File**: `scripts/benchmark/seeding_tui.py` (1 line added)

### Kill Script Fails to Terminate TUI (SIGTERM Ignored by Curses Threads)

**Problem**: `scripts/benchmark/kill_seeding.sh` sent SIGTERM, waited 1s, and moved on — but the TUI's curses daemon threads ignored SIGTERM, keeping the process alive as a zombie. Required manual SIGKILL.

**Fix**: Added 3-step escalation to step 1 of the kill script:
1. SIGTERM (graceful) → poll up to 3s checking each second
2. If still alive: SIGKILL (handles curses threads that ignore SIGTERM)
3. Reap zombie children (`stat =~ /^Z/`) left by the TUI

The `--force` flag still skips straight to SIGKILL as before.

**File**: `scripts/benchmark/kill_seeding.sh` (step 1 rewritten)
