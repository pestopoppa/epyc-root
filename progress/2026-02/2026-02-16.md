## REPL Output Spill-to-File with Rolling Summary

Large REPL execution output (>8192 chars) was being hard-truncated, leaving the model blind to test results, error messages, and computed values. Now output spills to a file and a worker-generated rolling summary replaces the truncation.

### Design

- **Spill**: Output exceeding `output_cap` written to `{spill_dir}/{session_id}/turn_{N}.txt`
- **Rolling summary**: Worker model (Qwen2.5-7B, ~44 t/s) maintains a compressed scratchpad across turns:
  - Turn 1: summarize tail of output (max 10 lines)
  - Turn N: pass previous summary + new tail → worker updates summary, preserving key findings, dropping superseded details
- **Fallback**: Static head (15 lines) + tail (5 lines) when worker unavailable (unit tests)
- **Access**: Model can `peek(spill_path)` or `grep(spill_path, pattern)` for full details on demand
- `max_output_preview` bumped 500 → 1500 to accommodate summary + header + footer

`claude` sessions were alive and accepting input, but startup hooks were failing and degrading UX during startup:

- `SessionStart:startup hook error`
- plugin hook chain came from `claude-mem` hook config
- failures included unresolved `bun` in hook execution path and noisy installer output during startup

### Actions Taken

1. Inspected tmux window/pane/process state and confirmed windows were not dead.
2. Captured pane scrollback + Claude debug logs to isolate hook failure source.
3. Closed tmux windows `2` and `6` per user request.
4. Hardened plugin hook commands:
   - replaced `bun ...` with absolute `/home/daniele/.local/bin/bun ...`
   - guarded `smart-install.js` behind `if [ ! -x /home/daniele/.local/bin/bun ]`
   - made installer path non-fatal (`|| true`) and bounded timeout
5. Applied same patch to both plugin hook locations to prevent path-dependent regressions:
   - `/home/daniele/.claude/plugins/marketplaces/thedotmack/plugin/hooks/hooks.json`
   - `/home/daniele/.claude/plugins/cache/thedotmack/claude-mem/9.0.12/hooks/hooks.json`

### Verification

- Latest debug log shows `Hook SessionStart:startup (SessionStart) success` entries.
- No remaining hook commands starting with relative `bun ` across discovered plugin `hooks.json` files.

### Notes

- Backups were created for edited hook files with timestamped `.bak.*` suffixes.
- No repo code changes were required for runtime behavior correction; this was user-environment/plugin hook remediation.
