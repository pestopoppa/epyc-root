## GraphRouter MemRL Augmentation

### Summary

Implemented GNN-based parallel routing signal for MemRL HybridRouter cold-start optimization. Based on GraphRouter (ICLR 2025, arXiv:2410.03834). Solves new model onboarding time: hours of organic data accumulation → minutes of scripted inductive prediction.

### Implementation (7 Phases, 49 Tests)

| Phase | File | Description |
|-------|------|-------------|
| A | `orchestration/repl_memory/routing_graph.py` | Kuzu bipartite graph: TaskType, QueryCluster, LLMRole nodes. MiniBatchKMeans clustering from EpisodicStore. PERFORMANCE_ON edges with success_rate, avg_q, avg_latency. |
| B | `orchestration/repl_memory/lightweight_gat.py` | Pure numpy 2-layer heterogeneous GAT (1024→32×4→32). Multi-head attention with scatter aggregation via `np.add.at`. Save/load `.npz`. |
| C | `orchestration/repl_memory/graph_router_predictor.py` | TTL-cached inference wrapper (<0.5ms warm). Finds nearest QueryCluster, runs GAT forward, softmax over LLM roles. |
| D | `orchestration/repl_memory/retriever.py` | HybridRouter: `graph_router` + `graph_router_weight` params. `_blend_graph_router_scores()` + adaptive weight (0.1→0.3 by store size). Telemetry in `_record_decision_meta`. |
| E | `src/features.py`, `src/api/services/memrl.py` | Feature flag `graph_router` (requires `specialist_routing`). Init in `ensure_memrl_initialized()`. |
| F | `scripts/graph_router/train_graph_router.py` | Offline training: sync graph, edge masking 20%, BCE loss, SGD with cosine LR, early stopping. |
| G | `scripts/graph_router/onboard_model.py` | Inductive new model onboarding: embed capability → LLMRole node → GAT predicts routing. |

### Architecture

```
HybridRouter.route(task_ir):
  ├── TwoPhaseRetriever (EXISTING, unchanged)
  ├── GraphRouterPredictor (NEW, parallel signal)
  │   └── Kuzu bipartite graph → 2-layer GAT → edge scores per role
  ├── Ensemble blending (NEW)
  │   └── posterior = (1-w) × retriever + w × graph_router
  │       w anneals 0.1→0.3 by episodic store size
  └── Fallback: RuleBasedRouter (EXISTING, unchanged)
```

### Key Design Decisions

- **Max 30% GNN influence**: LLMRouterBench found routing gains are coarse-grained. Conservative blend.
- **Pure numpy GAT**: No PyG/DGL dependency. ~130 nodes × 32-dim → <0.1ms forward pass.
- **Decoupled from seeding**: `seed_specialist_routing.py` unchanged. Training pipeline reads EpisodicStore offline.
- **Graceful degradation**: No weights / empty graph / not ready → silent skip, zero behavioral change.

## OpenAI-Format Tool Call Translation Fix

### Problem

Qwen3-Coder-30B emits OpenAI-format JSON tool_calls when it wants to use REPL tools:
```
[{"id":"call_...","function":{"name":"web_search","arguments":"{\"query\":\"...\"}"},"type":"function"}]
```
instead of the expected REPL syntax:
```
result = CALL("web_search", query="...")
```

The REPL code extractor (`extract_code_from_response`) didn't recognize this format, so:
1. Tool calls were never executed
2. No tool results returned to the model
3. Model generated 20+ identical tool calls in a degenerate repetition loop
4. `\n```\n` stop sequence never fired (no code blocks generated)

Discovered during 3-way eval (`seed_specialist_routing.py --3way`) on GPQA Organic Chemistry question — SELF:repl hung for 3+ minutes generating identical `web_search` calls.

### Root Cause

Qwen3-Coder's instruct training on chat-completion tool-calling format overrides prompt instructions even in raw `/completion` mode. The model sees tool descriptions in the system prompt text and spontaneously generates structured JSON tool_calls instead of inline Python `CALL()` syntax. The llama-server uses `chat_format: Content-only` with raw `/completion` endpoint, so no chat template is applied — this is purely model behavior.

### Fix

Added `translate_openai_tool_calls()` in `src/prompt_builders/code_utils.py`:

1. **Detection**: Regex `_TOOL_CALL_RE` matches `"function":{"arguments"|"name"` or `"id":"call_..."` patterns
2. **Extraction**: `_extract_json_arrays()` — balanced-bracket scanner handles nested braces in `"arguments"` fields (simple regex `[^]]+` fails on nested JSON)
3. **Deduplication**: Identical tool calls (same name + args) collapsed to single invocation — kills the 20+ repetition loop
4. **Translation**: Converts to `result = CALL("web_search", query="...")` + `print(result)` — no auto-FINAL, model continues reasoning with tool output in next turn
5. **Integration**: Hooked as preprocessing step in `extract_code_from_response()` — runs before any other code extraction logic

Also added `CALL(` to `code_starters` list so properly-formatted CALL invocations are detected.

### Additional Fix: Stale Orchestrator Instance

Killed duplicate uvicorn orchestrator (PID 2557207, 8+ hours old, 6 workers at ~100% CPU each). The stale instance was starving llama-servers of CPU time, causing the frontdoor to generate tokens extremely slowly.

### Files Modified

| File | Change |
|------|--------|
| `src/prompt_builders/code_utils.py` | Added `_extract_json_arrays()`, `translate_openai_tool_calls()`, `CALL(` in code_starters |
| `src/prompt_builders/__init__.py` | Exported `translate_openai_tool_calls` |

### Test Results

- 129 existing tests pass (code_utils, prompt_builder, extract_code)

## Slot-Erase-on-Timeout + Delegation Timeout Fix

### Problem

During 3-way eval, the orchestrator hangs when:
1. ARCHITECT delegates to coder_escalation, which holds the inference lock for 40s+
2. Next question's SELF:direct (frontdoor) can't acquire the lock, times out after 15s
3. The coder_escalation backend keeps generating tokens nobody reads (resource leak)
4. No cancellation signal is sent to the backend
5. Delegated models get squeezed by parent's remaining deadline instead of their standard role timeout

### Fix 1: Slot Erase on Lock Timeout (`src/inference_lock.py`)

Ported `_erase_slots` logic from `scripts/benchmark/seeding_orchestrator.py` into production as `_erase_port_slots(port)`. Lightweight, fire-and-forget, caches working erase strategy per port.

- **On lock acquisition timeout**: `_lock_holder_ports()` reads `/proc/{pid}/cmdline` to map holder PIDs to llama-server ports, then erases processing slots on those ports
- **On error inside lock (finally block)**: erases slots on the current role's port to free it for the next request
- New `port` parameter on `inference_lock()` enables cleanup when provided

### Fix 2: Port Propagation (`src/llm_primitives/inference.py`)

- Added `_extract_port(url)` helper to parse port from backend URLs
- `_real_call_single()`, `_call_caching_backend()`, and `_real_call_monitored()` now pass `port=` to `inference_lock()`

### Fix 3: Delegation Timeout Budget (`src/api/routes/chat_delegation.py`)

Before calling `_run_specialist_loop`, wraps the call in `primitives.request_context(deadline_s=...)` with a deadline computed as `max(role_timeout, specialist_budget_s)`. This gives the specialist its full configured role timeout instead of being squeezed by the parent's remaining deadline. The specialist loop already enforces wall-clock limits via elapsed checks.

### Files Modified

| File | Change |
|------|--------|
| `src/inference_lock.py` | Added `_erase_port_slots()`, `_lock_holder_ports()`, `_SLOT_ERASE_CAPABILITY` cache. Added `port` param to `inference_lock()`. Timeout/error slot cleanup in except + finally. |
| `src/llm_primitives/inference.py` | Added `_extract_port()`. Pass `port=` to all `inference_lock()` calls. |
| `src/api/routes/chat_delegation.py` | Nested `request_context()` before `_run_specialist_loop` with extended specialist deadline. |

### Test Results

- 62 inference_lock tests: all pass (4 skipped)
- 140 delegation tests: all pass (3 pre-existing stale constant failures excluded)
- Import verification: all new functions accessible

### Fix 4: llama.cpp Server Slot Erase Endpoint (production-consolidated)

Initial 3-way eval revealed `slot erase unsupported on port 80XX; disabling erase attempts` on all ports. Two server-side issues:

1. **`--slot-save-path` gate blocked erase**: The `POST /slots/:id?action=erase` endpoint required `--slot-save-path` even though erase doesn't touch disk. Moved the check to only gate save/restore actions.

2. **Erase deferred on processing slots**: The erase handler deferred (queued for later) when a slot was `is_processing()`. Changed to force-release the processing slot immediately (`slot->release()`) then clear KV cache. This is the critical fix — without it, erase can't cancel actively-generating inference.

**Files modified (llama.cpp fork)**:
- `tools/server/server-context.cpp`: Two changes — erase allowed without `--slot-save-path`, force-release on processing slots

## Architect Inference Hang — SSE Stream Stall Root Cause Investigation

### Problem

During 3-way eval, architect inference requests hang for 4+ minutes. TUI shows empty RESPONSE section, log shows "still waiting for ARCHITECT (241s elapsed, 26 tokens so far)". Slot state: `is_processing: false`, `n_decoded: 26`, `has_next_token: false` — server finished but Python client blocks.

### Root Cause Found

**Server-side**: The `SLOT_ERASE` handler in `server-context.cpp:1872` calls `slot->release()` without sending any result to the queue. The HTTP streaming handler for the original request is orphaned — `rd.next()` blocks forever because no final result arrives and `should_stop()` only returns true when TCP closes.

**Trigger chain**: Lock timeout → `_erase_port_slots(8084)` → server erase → `slot->release()` (no result sent) → HTTP handler orphaned → Python httpx blocks for full read timeout (600s).

### Investigation Path

1. Verified `queue_results.send()` silently drops results when task ID not in `waiting_task_ids` (no logging)
2. Confirmed task IDs are monotonic (`id++`), never reused — eliminated ID reuse hypothesis
3. Confirmed `add_waiting_tasks()` called before `queue_tasks.post()` — eliminated registration race
4. Traced full SSE path: `send_partial_response()` → `queue_results.send()` → `recv_with_timeout(1s)` → HTTP chunked_content_provider → httplib socket write
5. Examined `process_token()` stop conditions (EOS, stop strings, budget, context, time) — all correctly call `send_final_response()`
6. Examined speculative decoding path (`common_sampler_sample_and_accept_n`) — correctly calls `send_final_response()` on stop
7. **Found root cause**: `SLOT_ERASE` handler is the ONLY path that calls `release()` without `send_final_response()`

### Fixes Applied

| Layer | Fix | File | Effect |
|-------|-----|------|--------|
| Server | Erase sends error to original handler before release | `server-context.cpp` | Immediate unblock |
| Server | Debug log for silently dropped results | `server-queue.cpp` | Observability |
| Client | Read timeout capped at 120s | `llama_server.py` | 120s max stall (safety net) |
| Client | Graceful ReadTimeout recovery | `llama_server.py` | Returns partial content |

### Files Modified

| File | Change |
|------|--------|
| `llama.cpp/tools/server/server-context.cpp` | Erase: capture task ID, send_error() before release |
| `llama.cpp/tools/server/server-queue.cpp` | Added debug log for dropped results |
| `src/backends/llama_server.py` | 120s read timeout cap, ReadTimeout recovery |
| `handoffs/active/architect-inference-hang-bug-report.md` | Full investigation report with confirmed root cause |

## Orchestrator Worker CPU Spin Fix + Startup Stabilization

### Problem

After restarting the orchestrator, all 6 uvicorn workers ran at 75-80% CPU indefinitely. The seeding script (`seed_specialist_routing.py`) crashed between 30min-2hrs into runs with no traceback captured.

### Root Causes

1. **Background cleanup thundering herd**: All 6 workers independently ran `background_cleanup()` every 30s, each parsing ~180MB of progress logs via `get_unscored_tasks(days=7)`. Total: ~1GB JSONL parsing every 30 seconds.

2. **Per-task trajectory re-parsing**: Even after fixing (1), the primary worker burned 79% CPU because `_score_task()` called `get_task_trajectory(task_id)` which called `read_recent(days=30)` — scanning 30 days (~468MB) per task. With batch_size=10, that's ~4.7GB of JSONL parsing per scoring cycle.

3. **No crash logging**: The seeding script's `main()` had no top-level exception handler, so fatal errors were lost to stderr of a backgrounded process.

4. **No startup stabilization**: Workers take ~70s to load FAISS (148K vectors) and Kuzu databases during lifespan init. The seeding script's preflight check hit `/health` once — which succeeds as soon as 1 worker is ready — then immediately started sending inference requests while 5 workers were still CPU-hot.

### Fixes Applied

| Fix | File | Detail |
|-----|------|--------|
| Worker election via file lock | `src/api/__init__.py` | `fcntl.LOCK_EX \| LOCK_NB` on `.bg_cleanup.lock` — first worker wins, others skip background tasks |
| Shared read cache | `orchestration/repl_memory/progress_logger.py` | `_read_recent_cached(days=1)` with 120s TTL, shared by `get_unscored_tasks` and `get_task_trajectory` |
| Reduced lookback | `progress_logger.py` | `get_unscored_tasks`: 7→1 day, `get_task_trajectory`: 30→1 day |
| Single-pass scan | `progress_logger.py` | Replaced two-pass completed+routing scan with single-pass |
| Startup stabilization | `scripts/benchmark/seeding_infra.py` | `_wait_for_workers_ready()`: polls worker CPU via `ps --ppid`, waits until all below 10% for 2 consecutive checks |
| Crash logging | `scripts/benchmark/seed_specialist_routing.py` | Top-level try/except captures tracebacks to `logs/seeding_crash.log` |

### Results

| Metric | Before | After |
|--------|--------|-------|
| Non-primary workers CPU | 75-80% each | 0.0-0.2% |
| Primary worker CPU | 79% | 1.0% |
| Primary CPU time per 60s | +70s | +1s |
