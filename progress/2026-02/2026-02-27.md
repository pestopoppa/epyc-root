# Progress Report — 2026-02-27

## Seeding Diagnostics Review & Fixes

Reviewed 324 records from `seed_specialist_routing` run (12:04–18:51 UTC). Overall pass rate: 124/324 (38%).

### Issues Found & Fixed

#### 1. SimpleQA 0% Pass Rate — Stale Question Pool

**Root cause**: `SimpleQAAdapter` was updated to use `f1` scoring (threshold=0.8), but the pre-extracted question pool (`question_pool.jsonl`) was built before that fix and still had `exact_match` for all 4,326 simpleqa entries. The seeding script's fast-path pool loader served the stale config.

**Impact**: Correct-but-differently-formatted answers (e.g. "Oxford University" vs "University of Oxford") all scored as failures.

**Fix**: Rebuilt question pool. All simpleqa entries now use `f1` scoring with `extract_pattern: ####\s*(.+)`, `threshold: 0.8`, `normalize: true`.

#### 2. USACO/LiveCodeBench Specialist Timeout (10–15% pass rate)

**Root cause**: Two compounding issues:
- `specialist_max_seconds` default of 45s too short for competitive programming — specialist killed mid-generation
- `total_max_seconds` default of 110s too tight to recover after specialist timeout

**Fix** (`chat_delegation.py`):
- Specialist time budget: ×2.5 for coding tasks (45s → 112.5s), detected via question content signals (usaco, codeforces, leetcode, stdin, etc.)
- Total delegation budget: ×2.0 for coding tasks (110s → 220s)

#### 3. Coder `input()` Loop — Code-Report Rescue Misfiring

**Root cause**: Coder writes competitive programming solution using `input()` → code-report rescue (≥6 lines without FINAL) intercepts it before REPL execution → returns broken code to architect as "prose report" → architect re-delegates → same loop.

The coder never gets to see the `input()` violation error or the nudge telling it to use `CALL("run_python_code", code=..., stdin_data=...)`.

**Fix** (`chat_delegation.py`): Code-report rescue now skips code containing `input()` (without `CALL()`), allowing it to reach REPL execution where the violation error + nudge are surfaced.

#### 4. 504 Gateway Timeouts (38 records, ~12%)

**Root cause**: All 504s are exactly 120.0–120.1s elapsed, 0 tokens, role=frontdoor. The orchestrator's exclusive inference lock (`heavy_model.lock`) serializes all heavy role inference. When a delegation cycle (frontdoor → architect → coder) holds the lock across multiple sequential inferences, the next request's frontdoor call blocks until timeout.

**Status**: Not fixed — the exclusive lock is a deliberate constraint to prevent concurrent heavy inference from thrashing CPU/memory. The specialist timeout fixes above should reduce wasted time per request, improving effective throughput.
