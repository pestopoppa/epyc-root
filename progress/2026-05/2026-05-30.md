## DeepSeek-V4 llama.cpp bug-report audit — conversation-mode cluster

### Trigger

Operator requested an audit of `bug-reports/llama-cpp-deepseek-v4/02-llama-completion-auto-cnv-silent-semantic-change.md` for fixability. The path was provided twice; both references resolved to the same original report. The report covered two user-visible problems in the antirez DeepSeek-V4 llama.cpp fork at `2f2d44052`: `llama-completion` auto-enabling conversation mode when a GGUF embeds a chat template, and `llama-cli --no-conversation` falling into interactive behavior.

### Audit scope

Read the original bug report, the active V4 handoff, local llama.cpp source paths, and runnable help output from non-stale local builds. No application code was changed in the llama.cpp checkout. The audit stayed in review/documentation mode because the local llama.cpp repo carries an upstream contributor policy requiring human-owned implementation decisions.

### Findings

| Issue | Verdict | Root cause / code path | Recommended fix |
|---|---|---|---|
| `llama-cli --no-conversation` continues after warning | Real bug, HIGH | `tools/cli/cli.cpp` accepts the shared `--no-conversation` parse result, prints an error when `params.conversation_mode == COMMON_CONVERSATION_MODE_DISABLED`, then falls through into model loading. | Minimal: `return 1` immediately after the warning. Better: remove CLI from the shared conversation-mode option and add a CLI-only deprecated handler that parse-fails before model load. |
| `llama-cli` infinite `> ` loop on stdin EOF | Real bug, HIGH; broader than `--no-conversation` | `tools/cli/cli.cpp` treats empty input as "skip empty message" and `continue`; `console::readline()` returns a boolean that does not distinguish EOF from normal line completion. | Minimal: after empty-buffer read, check EOF and break with an `EOF by user` message. Better: change `console::readline()` to return an enum (`MORE`, `DONE`, `EOF`). |
| `llama-completion` auto-cnv for chat-template GGUFs | Real behavior, LOW UX issue after audit | `tools/completion/completion.cpp:213` enables conversation mode automatically when `common_chat_templates_was_explicit()` is true and params are AUTO. | Keep auto mode, but change the runtime notice from `LOG_INF` to `LOG_WRN` and explicitly say raw completion changed to chat mode; use `-no-cnv` for raw prompt completion or `-st` for a single chat turn. |

### Correction to the original report

The original report overstated the help-text gap. Current source and generated help text already document `(default: auto enabled if chat template is available)` for `-cnv` / `-no-cnv`. The remaining auto-cnv problem is runtime visibility and semantic clarity, not absence from help.

### Result

Per §Throughput gate "provisional evidence rule" (added 2026-05-29 to handle externally-blocked quality reference), this counts as provisional pending the quality gate. If quality later passes, the provisional FAIL stands. If quality fails, both axes fail.

Artifacts: `epyc-inference-research/data/cpu_optimization/2026-05-30-v4-throughput-gate-provisional/20260530T000217.{stdout,stderr,summary.md}`.

Initial summary.md had `<parse failed>` for the t/s metrics — sed regex bug (anchored t/s number immediately after `(` instead of after `ms_per_token,`). Re-parsed against same stderr; verdict landed.

### Calibration logic critique (informs Strategy A decision)

The 18 t/s floor was derived as `76.5 × (4/13) = 23.5` (gemma4-26B-A4B's 4B active × inverse param ratio) with a ~23% conservatism discount to 18. The methodology ignores V4's per-token overhead from CSA + HCA + indexer + compressor + manifold-constrained Hyper-Connections — components that don't exist in gemma4. Our 9.13 t/s suggests V4's effective compute density is ~2.5× gemma4's per active param. The 18 t/s floor was set under a heuristic that doesn't model V4's arch overhead. If Strategy A is pursued, the floor should be recalibrated against a V4-arch-aware baseline.
