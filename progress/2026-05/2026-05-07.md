# 2026-05-07 — sd-server replaces ComfyUI for image generation

## Summary

Started as "explore IPEX as cheap CPU speedup before committing to a multi-week ggml port"; ended as "stable-diffusion.cpp upstream already has full ERNIE-Image-Turbo support — swap ComfyUI for sd-server, +2.54× wall-clock at production resolution, no quality cost."

The swap is shipped: orchestrator stack now manages `sd_server` instead of `comfyui` (port 8190 in place of 8188), `start_sd_server()` registered, registries updated, Hermes plugin chain unchanged (still routes through `ImageGenerator` → which now calls `SDServerClient` instead of building ComfyUI workflow JSON).

## Performance trajectory (1024² 8-step extrapolated to production)

| Stage | Wall clock | Speedup vs ComfyUI baseline |
|---|---|---|
| ComfyUI-GGUF + PyTorch (2026-05-06 baseline) | 478 s | 1.0× |
| sd-server stock (`--diffusion-fa` only) | ~258 s | 1.85× |
| **sd-server + `--diffusion-conv-direct` + `--vae-conv-direct`** | **~188 s** | **2.54×** |
| sd-server Q4_K_M + conv-direct (DEPRECATED — quality loss) | ~160 s | 2.99× — but visibly degrades Korean text |

**Production-shipped point**: sd-server Q8_0 + conv-direct flags. ~3 min/image at 1024² 8 steps on EPYC 9655 CPU.

## A/B tests run

### Test 1: IPEX 2.8 + torch.compile (sidecar venv)

Built `/mnt/raid0/llm/comfyui-ipex/ComfyUI` on a torch-2.8.0+cpu venv with `intel-extension-for-pytorch==2.8.0`. Wrote a custom node `ApplyIpexOptimize` that wraps `model.model.diffusion_model` in `ipex.optimize(..., dtype=torch.bfloat16, weights_prepack=False)`. Ran 512² 4-step A/B with same prompt + same seed.

**Result**: **0–4% speedup — measurement noise, no real win.**

| | Baseline | IPEX | Δ |
|---|---|---|---|
| Wall clock | 70.03 s | 70.03 s | 0% |
| `Prompt executed` | 66.56 s | 65.69 s | −1.3% |
| Sampler avg s/iter | 15.24 | 14.65 | −3.9% |

Reason: ComfyUI-GGUF's `GGMLLayer.forward` overrides `nn.Linear.forward` and dequants Q8 weights to BF16 just-in-time per call. IPEX's `optimize()` works on `nn.Module` parameters at optimize time — for `GGMLTensor` parameters with custom forward, IPEX has nothing to bind onto. The BF16 matmul that runs is already in PyTorch's stock oneDNN dispatch. **The dequant tax IS the bottleneck and IPEX can't reach it.**

### Test 2: stable-diffusion.cpp recon

Cloned `leejet/stable-diffusion.cpp` master-596-90e87bc. **Found `src/ernie_image.hpp` (441 lines) — full ERNIE-Image-Turbo implementation already exists upstream**: `ErnieImageAttention`, `ErnieImageFeedForward`, `ErnieImageSharedAdaLNBlock`, `ErnieImageAdaLNContinuous`, `ErnieImageModel`, `ErnieImageRunner`. `model.h` has `VERSION_ERNIE_IMAGE` enum, `model.cpp:459` registers ERNIE arch detection, `diffusion_model.hpp:549` has the runtime wrapper. `docs/ernie_image.md` literally points at our exact files (`unsloth/ERNIE-Image-Turbo-GGUF` + `Comfy-Org/ERNIE-Image`). The C++ port we'd estimated as multi-week was already done by upstream maintainers.

CMake build with `-march=native` picked up everything Zen 5 has — AVX-512F/BW/DQ/VL/VBMI/IFMA/CD + VNNI. Binary contains 9,067 zmm/vnni instructions, 387 explicit `vpdpbusd`/`vpdpwssd` dot-products. **The "rebuild with explicit AVX-512BW VNNI" lever was already on by default.**

### Test 3: sd-server smoke test

512² 4-step ERNIE-Image-Turbo run via `sd-cli`:

| | ComfyUI-GGUF (PyTorch) | sd.cpp (ggml native) | Speedup |
|---|---|---|---|
| Wall clock | 70.03 s | **40.24 s** | **1.74×** |
| Sampler s/iter | 15.24 | **4.45** | **3.43×** |
| VAE decode | (mixed in) | 20.65 s | (newly visible bottleneck) |

DiT forward 3.4× faster on native ggml — exactly the dequant tax recovered. VAE became the new dominant cost.

### Test 4: production scale + conv-direct flags

Full perf prompt (~600 words, Korean baseball broadcast aesthetic with bilingual text and BubbleBrain signature) at 832×1248 (2:3 vertical) 8 steps. Two runs at same prompt + same seed (`1639522513`):

| Run | Sampling | VAE | Wall clock |
|---|---|---|---|
| Stock (only `--diffusion-fa`) | 181.15 s (22.6 s/iter) | 76.02 s | 258.31 s |
| **+ `--diffusion-conv-direct` + `--vae-conv-direct`** | **180.89 s** | **10.71 s** | **192.76 s** |

**`--vae-conv-direct` is a 7.10× win on the VAE alone**, +1.34× on wall-clock, no quality impact (same seed = visually identical output). `--diffusion-conv-direct` was within noise on sampling. Updated the launcher to enable both flags by default.

### Test 5: Q4_K_M (distilled-model penalty)

Pulled `ernie-image-turbo-Q4_K_M.gguf` (4.7 GB) and re-ran the same prompt + seed at 832×1248 8 steps:

| | Q8 (production) | Q4_K_M | Δ |
|---|---|---|---|
| Sampling | 181 s @ 22.6 s/iter | 152.7 s @ 19.1 s/iter | 1.18× faster |
| Wall clock | 192.76 s | 164.33 s | 1.17× faster |

Lower than my 30–50% estimate — at 96-thread DiT the workload is compute-bound enough that Q4's bandwidth savings hit a ceiling.

**Visual quality**: Korean jersey text "잠실" (Jamsil) **rendered cleanly at Q8, corrupted at Q4** (distorted glyphs, partial gibberish on adjacent English brand text). BubbleBrain signature still legible at Q4 but fuzzier. General composition / pose / lighting / grain preserved.

**Verdict**: Distilled-model quantization-penalty hypothesis (deep dive §4.4) **empirically confirmed for ERNIE-Image-Turbo at 8 steps**. The penalty hits the model's signature capability — bilingual long-text rendering — first. **17% wall-clock for visible degradation of the differentiator** is a bad trade. Q8 stays as the production point. Q4 file deleted post-test.

## Empirical findings worth recording

**ggml CPU backend is at the kernel ceiling on Zen 5 stock**. With `-march=native` enabled by sd.cpp's CMake, the binary is already emitting AVX-512BW + VNNI — 9,067 zmm instructions + 387 dot-product instructions. The "rebuild with explicit feature flags" lever I considered in earlier triage was already closed. **The remaining sampler win — port our llama.cpp fork's AVX-512BW 8x8 Q8_0 kernel** (`project_q8_8x8_avx512bw_outcome.md`: +31.8% at 1t / +1-3% at 96t for LLM decode) — is unmeasured for DiT workloads. DiT has higher arithmetic intensity than LLM decode (16K tokens per matmul instead of 1), so the kernel's compute-side win could propagate further (estimate +10-20% on sampling), but that's a 2–4 day port to sd.cpp's diverged ggml. Deferred.

**`--vae-conv-direct` is the high-ROI flag** I missed in the initial sd.cpp deployment. 7.1× speedup on VAE decode alone, no quality cost, same seed produces visually identical output. `--diffusion-conv-direct` was within noise but enabled defensively (no downside). Both belong in the launcher always. Worth checking other ggml-using projects for similar "direct conv" flags that aren't on by default.

**The distilled-model quantization-penalty hypothesis is real**. Deep dive §4.4 had it as community folklore; this session has the receipt. ERNIE-Image-Turbo at 8 steps + Q4_K_M loses its differentiator (Korean/dense-text rendering) before generic quality degrades visibly. Mechanism prediction held: short error budgets + dequant noise that 50-step samplers integrate over many steps cannot be averaged out in 8.

**stable-diffusion.cpp ships ERNIE-Image-Turbo support out of the box**. The "weeks-long C++ port" estimate I gave the user was wrong — the port was already done upstream. Lesson: when a project of leejet/sd.cpp's pace is the most-likely home for a new arch, scan the source before planning the port. Spent 30 min instead of 2 weeks because of one `find ... ernie ...`.

**Hermes plugin path survived the swap unchanged**. Plugin imports `src.services.image_generator.ImageGenerator` — that class kept its public `generate(request) -> result` interface. Internals swapped from ComfyUI workflow JSON to sd-server HTTP, no change visible to plugin or to Hermes' tool registry. This is exactly the value of the thin-interface pattern; one of the few clean wins of the OCR-pipeline-mirroring discipline.
