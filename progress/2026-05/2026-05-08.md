## What it is

- **bucketMul**: transpose W, sort each row into fixed-size buckets by |w| (size 16 for FP16, size 8 for Q8 on Apple GPU), tag positions, store per-bucket avg|w| as "bucket stats". At inference: `v × probes → topK → cutoff → dispatch list → multiply only surviving buckets`. The "effort" dial is the topK percentage.
- **"Ad-hoc distillation"**: each row's buckets are sorted descending by importance, so the trailing 10–30% can simply not be loaded at mmap time — no conversion needed.
- **Q8 packing variant**: 1 sign + 3 position + 4 value bits per weight, with per-bucket min/max ranges (the sort makes per-bucket spans tight enough that 4 value bits suffice).

## Why it landed at `worth_investigating` rather than `not_applicable`

Per `feedback_dont_dismiss_creative_uses.md` — the implementation itself does not fit our stack, but two ideas are portable:

1. **Load-time trailing-bucket skip = free distillation lever** in any sorted-bucket repack format we might build for unrelated reasons.
2. **Runtime "effort dial"** as a per-token quality-budget primitive that could compose with `routing-intelligence.md` / `per-request-reasoning-budget.md`.

Both fragments are too small to justify a dedicated handoff against active CPU2 / wdata-aware-MUL_MAT / TQ3 / CPU18 priorities, so the entry is logged with explicit re-surface trigger conditions and no follow-up.

## Addendum — deep-dive expansion (2026-05-08 PM)

User asked for a real engineering audit beyond the blog read. Fetched the actual Swift/Metal source (`bucketMul.metal`, `convert.swift`, `convert.metal`, `bucketMul.swift`) and produced `research/deep-dives/kolinko-effort-engine-deep-dive.md`.

### New findings surfaced from source-code audit

| Finding | Why it matters |
|---|---|
| **Probes are stride-sampled diagonal weights** of W: `probes[i] = w[i + i·cols]`, NOT `input·W` expectations as the blog prose implies | A genuinely cheap O(1)/row magnitude estimator vs. Deja Vu's learned predictor — could inform any future dynamic-sparsity work |
| **FP16 bucket size of 16 is forced** by 4-bit position encoding into FP16 mantissa LSBs (`as_type<ushort>(w) & 15`) | Effective FP16 mantissa drops 10 → ~6 bits; the design choice cannot be relaxed without a different storage format. |
| **Q8 packing is `[SvvvvvPPP]`** = 1 sign + 4 value + 3 position bits with separate top-2% / bottom-1% outlier store | The "4 value bits suffice" claim is contingent on the per-bucket-slice min/max range tightening from the sort. |
| **4 kernel launches per matmul** (cutoff bisection + dispatch + bucketMul + integrate) plus atomic dispatch contention | On EPYC the launch tax likely dominates the BW savings — open question without a port. |
| **Last code commit `46f5269` 2024-04-25; issue #5 still open on Mac-only compilation** | No third party has even attempted a Linux port in 25 months; lack of replication is itself a negative signal. |
| **Three portable algorithmic artefacts** (load-time row-skip, per-token effort dial, probe-as-diagonal-sample) — none individually justifies a port | All three re-surface triggers documented in intake-528 verdict_justification |

### Deep-dive verdict

`worth_investigating` (kept). Direct-port cost estimated **~1 staff-month** (new ggml repack + AVX-512 kernel + dispatch + tests). Realistic upside on EPYC: **1.2–1.5× at 30–50% quality cost vs Q4_K_M** — below already-shipped MTP 2.98× (intake-527) and CPU2 31.8% @ 1t. **No scenario beats the existing roadmap on quality-per-staff-hour.** The deep dive itself is the closure record for any future "should we look at Effort?" question.

### Three structural blockers (compounding) — none surfaced in initial intake

1. **MoE vs dense FFN**: production stack is MoE (Qwen3 30B-A3B, Coder, Next, REAP-246B); Sakana models are dense gated-MLP. TwELL skips post-ReLU zeros within a single FFN; MoE skips entire experts. Compute-saving regimes do not orthogonal-stack on the same model — paper does not demonstrate it.
2. **SwiGLU vs ReGLU**: production drafters/targets use SiLU/SwiGLU. SparseLM uses `nn.ReLU` in actual model code (`hidden_act:"silu"` in `config.json` is a leftover overridden by `model_type: "llama_sparse_relu"`). Draft-target compatibility is broken — SparseLM cannot be a drafter for any Qwen target (guaranteed logit drift).
3. **No quantized baseline**: Q4_K_M ≈ 0.5 bytes/weight; sparse-BF16 at 99% sparsity ≈ 0.32 effective bytes/weight, but Q4_K_M's 256-element super-block layout is broken by indirect-addressed sparse loads. Either duplicate the super-block scale per non-zero index (memory blow-up) or read full super-blocks anyway (no BW saving).
