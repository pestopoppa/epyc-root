### Root cause (two layers)

The headline and the expandable task list are fed by **two decoupled data
paths** with different semantics and windows:

| Surface | Source | Window | Meaning |
|---|---|---|---|
| Headline (`N req Â· X t/s Â· Nm ago`) | `inference_tap.log` sections + recently-*completed* chat tasks (`/dashboard/api/topology_activity`) | 600s | time since last **completed** request |
| Accordion body (clickable cards) | started-minus-terminal `chat-*` events in progress JSONL (`scan_orchestrator_tasks` â†’ snapshot `in_flight_tasks`) | 300s age cutoff | tasks **currently** started-not-terminal |

In the screenshot the top bar read `0/39 slots active` â€” genuinely nothing was
running. The headline was correctly showing requests that *completed* seconds
ago; `"4s ago"` is age-since-completion, not "in progress". So the panel was
technically correct but **misleading**, and two genuine latent bugs meant the
list *would* under-report real work:

1. **Single 300s age cutoff < 600s headline window.** A task generating at
   0.4â€“0.5 t/s (`ingest_long_context`, `architect_general`) routinely runs
   >300s for one generation and was silently dropped as a "restart-orphan"
   while still occupying a slot â€” invisible mid-run.
2. **Role-key mismatch.** In-flight grouped by `chosen_action`; headline by tap
   `role` / `final_role`; topology rows by base label. A task could land in a
   bucket with no matching badge.

Verified against today's live log: 636 / 2755 chat tasks (23%) had a
`task_started` with no terminal event â€” all hours old (restart-orphans from API
restarts dropping `task_completed`/`task_failed`). All correctly excluded; none
were recent, confirming the screenshot was "headline = recent completed, list =
nothing live" rather than a dropped live task.

### Autopilot orphan-pollution question â€” answered: NO scrub needed

User asked whether the 23% orphan rate could pollute autopilot's journal /
hypothesis strategy. **It cannot.** Autopilot is decoupled from the orchestrator
progress JSONL by design: it scores trials synchronously via its own
`EvalTower` (`scripts/autopilot/eval_tower.py`) and writes its own journal
(`orchestration/autopilot_journal*.{tsv,jsonl}`) + Pareto archive
(`autopilot_state.json`). A grep of `scripts/autopilot/` for any
`progress`/`task_started`/`task_completed` consumption comes back empty. The
orphans live only in the operational progress log, so they can't be counted as
failures, timeouts, or skew any trial metric. (The scrub machinery â€”
`scripts/autopilot/scrub_journal.py` + `trustworthiness_score()` â€” exists for
*real* orchestrator-bug corruption, but is not triggered here.)
