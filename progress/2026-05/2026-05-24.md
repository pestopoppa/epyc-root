## Session 1 — dashboard topology panel: in-flight tasks now track live slots

### Problem

In the autopilot dashboard's **TOPOLOGY** panel, the per-role accordions showed
`no in-flight tasks for this role` even when the accordion headline hinted at
activity (e.g. `frontdoor — 2 req · 4.1 t/s · 4s ago`). The user read the
headline as "tasks in progress" and the empty body as a bug.

### Root cause (two layers)

The headline and the expandable task list are fed by **two decoupled data
paths** with different semantics and windows:

| Surface | Source | Window | Meaning |
|---|---|---|---|
| Headline (`N req · X t/s · Nm ago`) | `inference_tap.log` sections + recently-*completed* chat tasks (`/dashboard/api/topology_activity`) | 600s | time since last **completed** request |
| Accordion body (clickable cards) | started-minus-terminal `chat-*` events in progress JSONL (`scan_orchestrator_tasks` → snapshot `in_flight_tasks`) | 300s age cutoff | tasks **currently** started-not-terminal |

In the screenshot the top bar read `0/39 slots active` — genuinely nothing was
running. The headline was correctly showing requests that *completed* seconds
ago; `"4s ago"` is age-since-completion, not "in progress". So the panel was
technically correct but **misleading**, and two genuine latent bugs meant the
list *would* under-report real work:

1. **Single 300s age cutoff < 600s headline window.** A task generating at
   0.4–0.5 t/s (`ingest_long_context`, `architect_general`) routinely runs
   >300s for one generation and was silently dropped as a "restart-orphan"
   while still occupying a slot — invisible mid-run.
2. **Role-key mismatch.** In-flight grouped by `chosen_action`; headline by tap
   `role` / `final_role`; topology rows by base label. A task could land in a
   bucket with no matching badge.

Verified against today's live log: 636 / 2755 chat tasks (23%) had a
`task_started` with no terminal event — all hours old (restart-orphans from API
restarts dropping `task_completed`/`task_failed`). All correctly excluded; none
were recent, confirming the screenshot was "headline = recent completed, list =
nothing live" rather than a dropped live task.

`scripts/autopilot/scrub_journal.py` + `trustworthiness_score()` — exists for

**Critical engineering decisions vs upstream `engram_demo_v1.py`:**
1. **Global config state removed.** Upstream reads `engram_cfg` and `backbone_config` as module-level singletons; ours takes them as ctor args, so one process can host multiple Engram modules with different settings (needed for ablations).
2. **The TWO missing zero-init lines added.** Upstream demo omits zero-init despite the paper requiring it. Without these, `engram.forward(...)` is non-zero at step 0 and a frozen-backbone retrofit perturbs the backbone immediately. Our `apply_identity_init(engram)` zeros: `value_proj.weight`, `value_proj.bias`, AND `short_conv.conv.weight`. (Embedding table + key_projs + RMSNorm weights deliberately NOT zeroed — they need default init so gradient signal flows once training begins.)
3. **Tokenizer module split.** `CompressedTokenizer` (HF transformers-dependent) is separate from the canonicalization logic (pure-Python). This lets unit tests run with zero network/disk deps.
4. **Numpy hash kept as source of truth.** Skipped the buggy PR #15 torch port — the upstream's `torch.from_numpy(...)` runs hash on CPU and transfers indices to the embedding's device. Our `Engram._hash_to_device` does this cleanly without the device-mismatch bug.
5. **HF integration via `EngramLayerWrapper` subclass + `set_input_ids` cache.** HF DecoderLayer signatures only pass hidden_states + masks, so a plain forward_pre_hook loses input_ids. The wrapper caches input_ids per training-batch and validates shape on each forward (raises on stale ids or batch-size mismatch, prevents silent miscomputation).

### Critical result — identity-at-step-zero invariant PASSES

The `test_identity_init_yields_zero_engram_output` test asserts that with `apply_identity_init(engram)` applied, `engram.forward(non-zero hidden states, real token ids)` produces an all-zeros output (atol = 1e-6). It **passes**. A companion sanity test (`test_default_init_does_NOT_preserve_identity`) confirms default init does NOT produce zero output — so we'd catch any future refactor that accidentally made Engram identity-by-default (which would be a bug).

**What this means for Track B:** the architectural surgery is now known to be well-posed at step 0. A frozen-backbone retrofit can plug in an Engram layer and the backbone's behavior is bit-identical to the un-augmented model until training begins. The paper-side concern (§6.3: co-trained backbones learn to delegate to Engram, frozen backbones haven't done that learning) still applies — but at least the architectural surgery itself doesn't shock the backbone. This is the load-bearing piece of evidence we needed before going to GPU for Phase 0 proxy training.
