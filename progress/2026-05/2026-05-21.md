# 2026-05-21 — global KMP_BLOCKTIME=10 fix kills idle OMP busy-spin across the stack

## Session 1 (morning) — delegation latency root cause

### Problem

User restarted autopilot, observed multi-second latency when models delegated/escalated to each other (e.g. `frontdoor → worker_explore`). Asked: "We didn't seem to have this previously. What's the matter?"

### Root cause

**16 of 17 llama-server instances were launched without `KMP_BLOCKTIME` set.** Audit:

| Server | Port(s) | KMP_BLOCKTIME |
|---|---|---|
| worker_general (gemma4 MTP) | 8072 + 8082/8182/8282/8382 | ✓ 10 |
| frontdoor (full + 4 quarters) | 8070 + 8080/8180/8280/8380 | ✗ |
| architect_general | 8083 | ✗ |
| ingest_long_context | 8085 | ✗ |
| worker_vision | 8086 | ✗ |
| vision_escalation | 8087 | ✗ |
| embedder (×6) | 8090–8095 | ✗ |

Under `OMP_WAIT_POLICY=active` without `KMP_BLOCKTIME`, AOCC's libomp keeps worker teams **busy-waiting between OMP regions**. With 5 frontdoor instances + architect + ingest + 2 visions + 6 embedders all idle, `%Cpu(s) us` sat at **49.7%** with no inference — ~48 cores' worth of pure idle spin. When delegation fired, the receiving server had to evict spinning idle teams to claim CPU on cores 0-95, manifesting as multi-second per-delegation latency.

**Why this is the first time it bit hard**: per memory `feedback_ik_llamacpp_omp_idle_spin.md` (2026-05-09), `KMP_BLOCKTIME=10` was wired in **only on the `worker_pool` branch** of `orchestrator_stack.py` to fix gemma4 MTP idle spin. The other launch branches (default frontdoor, architect, ingest, vision, embedding) were never updated. When I restarted frontdoor as `--only frontdoor` in last night's Session 6, all 5 frontdoor instances came up *fresh and idle simultaneously* — 5× spinning OMP teams overlapping on cores 0-47/96-143 made the latency visible in this morning's autopilot run.

### Change

| Repo | File | Change |
|---|---|---|
| epyc-orchestrator | `scripts/server/orchestrator_stack.py:908-936` | Added `KMP_BLOCKTIME: "10"` to `_CANONICAL_OMP_ENV` so all launch branches inherit it via `build_launch_env`. Left the existing `env["KMP_BLOCKTIME"] = "10"` in the binary_override branch (line 1977) as idempotent + documentation continuity. |

### Restart + verification

- Full stack stop/start cycle (`orchestrator_stack.py stop --all` + `start`).
- Audited all 17 llama-server PIDs post-restart: **17/17 carry `KMP_BLOCKTIME=10` in env**.
- Idle CPU verification:

| Metric | Before fix | After fix |
|---|---|---|
| `%Cpu(s) us` (idle) | 49.7 | **0.3** |
| `%Cpu(s) id` (idle) | 49.7 | **99.7** |
| `loadavg 1-min` | 79.84 | 0.82 |
| API status | ok | ok (post-restart) |
| frontdoor probe | ok | ok (15.0ms) |
| architect_general probe | ok | ok (6.5ms) |

The chip is genuinely quiet at idle now. Delegation latency should be dominated by inference cost, not OMP scheduler contention.

### Outcome

- Autopilot was killed by user before the restart, so no autopilot run was interrupted mid-trial.
- Stack came back up healthy with the fix in place.
- User cleared to restart autopilot.
