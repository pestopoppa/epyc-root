# 2026-05-21 — global KMP_BLOCKTIME=10 fix kills idle OMP busy-spin across the stack

## Session 1 (morning) — delegation latency root cause

### Problem

User restarted autopilot, observed multi-second latency when models delegated/escalated to each other (e.g. `frontdoor → worker_explore`). Asked: "We didn't seem to have this previously. What's the matter?"

### Root cause

**16 of 17 llama-server instances were launched without `KMP_BLOCKTIME` set.** Audit:

| Server | Port(s) | KMP_BLOCKTIME |
|---|---|---|
| worker_general (gemma4 MTP) | 8072 + 8082/8182/8282/8382 | ✓ 10 |
| frontdoor (full + 4 quarters) | 8070 + 8080/8180/8280/8380 | ✗ |
| architect_general | 8083 | ✗ |
| ingest_long_context | 8085 | ✗ |
| worker_vision | 8086 | ✗ |
| vision_escalation | 8087 | ✗ |
| embedder (×6) | 8090–8095 | ✗ |

Under `OMP_WAIT_POLICY=active` without `KMP_BLOCKTIME`, AOCC's libomp keeps worker teams **busy-waiting between OMP regions**. With 5 frontdoor instances + architect + ingest + 2 visions + 6 embedders all idle, `%Cpu(s) us` sat at **49.7%** with no inference — ~48 cores' worth of pure idle spin. When delegation fired, the receiving server had to evict spinning idle teams to claim CPU on cores 0-95, manifesting as multi-second per-delegation latency.

**Why this is the first time it bit hard**: per memory `feedback_ik_llamacpp_omp_idle_spin.md` (2026-05-09), `KMP_BLOCKTIME=10` was wired in **only on the `worker_pool` branch** of `orchestrator_stack.py` to fix gemma4 MTP idle spin. The other launch branches (default frontdoor, architect, ingest, vision, embedding) were never updated. When I restarted frontdoor as `--only frontdoor` in last night's Session 6, all 5 frontdoor instances came up *fresh and idle simultaneously* — 5× spinning OMP teams overlapping on cores 0-47/96-143 made the latency visible in this morning's autopilot run.

## Session 2 (morning, post-stack-restart) — autopilot crash fix (UnboundLocalError on `datetime`)

### Problem

User attempted to restart autopilot post-KMP_BLOCKTIME fix; crashed immediately:

```
File "scripts/autopilot/autopilot.py", line 1275, in _run_loop_inner
    timestamp=datetime.now(timezone.utc).isoformat(),
UnboundLocalError: cannot access local variable 'datetime' where it is not associated with a value
```

### Root cause

Two redundant local `from datetime import datetime, timezone` statements inside function bodies — `_run_loop_inner` line 1459 and `cmd_digest` line 1660 — shadowed the module-level import at line 37. Python's compiler treats `datetime` as function-local because of the later assignment, so the earlier reference at line 1275 (in the same function) hits unbound on every trial-eval path.

Introduced in commit `df98f185` (2026-05-20, "feat: final_schema_validation flag + autopilot delegation expansion + daily digest"). Latent bug — only fires when autopilot reaches the trial-eval timestamp code at line 1275 (which trial 328's gepa_optimize path does on first dispatch).

### Change

| Repo | File | Change |
|---|---|---|
| epyc-orchestrator | `scripts/autopilot/autopilot.py:1459` | Removed redundant `from datetime import datetime, timezone` inside `_run_loop_inner` daily-digest branch |
| epyc-orchestrator | `scripts/autopilot/autopilot.py:1660` | Removed redundant `from datetime import datetime, timezone` inside `cmd_digest` |

The module-level import at line 37 (`from datetime import datetime, timezone`) already provides both names. Both local imports were pure shadow.

### Mid-session correction logged

Initially overstated in my final report: "the Sherry algorithm + STQ1_0 llama.cpp kernel (PR #22836) is a generic sub-2-bit weight-quant lever applicable to any worker model." User correctly challenged this. Walked it back:

**The kernel is generic. The algorithm is not.** Sherry is a QAT method requiring ~10B tokens on UltraFineWeb-style data — NOT a PTQ calibration like GPTQ/AWQ. Applying Sherry to an arbitrary post-trained worker requires a QAT pipeline we don't have. Today the only public Sherry-QAT'd weights are `Hy-MT1.5-1.8B`, `HY-1.8B-2bit`, and (now) `Hy-MT2-1.8B-1.25bit`. STQ1_0 kernel without those is "a decoder for weights only Tencent currently produces, and only at 1.8B class."

### Conceptual user question: "could justify use as a translation tool? Models prefer certain languages? Translation through latent representation is free?"

Answered with research-grounded nuance: not "negative" — **contingent on actual content mix**. LLMs do pivot through near-English-shaped mid-layer representations (Wendler et al.), but the pivot is lossy and asymmetric — drops more for low-resource languages, domain terminology, structured content. Existing Qwen-family models are China-trained (strong on zh, weaker on EU low-resource); gemma4 is the opposite. The question is empirical.
