### Problem

User restarted autopilot, observed multi-second latency when models delegated/escalated to each other (e.g. `frontdoor → worker_explore`). Asked: "We didn't seem to have this previously. What's the matter?"

### Root cause

**16 of 17 llama-server instances were launched without `KMP_BLOCKTIME` set.** Audit:

| Server | Port(s) | KMP_BLOCKTIME |
|---|---|---|
| worker_general (gemma4 MTP) | 8072 + 8082/8182/8282/8382 | ✓ 10 |

| embedder (×6) | 8090–8095 | ✗ |

Under `OMP_WAIT_POLICY=active` without `KMP_BLOCKTIME`, AOCC's libomp keeps worker teams **busy-waiting between OMP regions**. With 5 frontdoor instances + architect + ingest + 2 visions + 6 embedders all idle, `%Cpu(s) us` sat at **49.7%** with no inference — ~48 cores' worth of pure idle spin. When delegation fired, the receiving server had to evict spinning idle teams to claim CPU on cores 0-95, manifesting as multi-second per-delegation latency.

**Why this is the first time it bit hard**: per memory `feedback_ik_llamacpp_omp_idle_spin.md` (2026-05-09), `KMP_BLOCKTIME=10` was wired in **only on the `worker_pool` branch** of `orchestrator_stack.py` to fix gemma4 MTP idle spin. When I restarted frontdoor as `--only frontdoor` in last night's Session 6, all 5 frontdoor instances came up *fresh and idle simultaneously* — 5× spinning OMP teams overlapping on cores 0-47/96-143 made the latency visible in this morning's autopilot run.

## Session 2 (morning, post-stack-restart) — autopilot crash fix (UnboundLocalError on `datetime`)

### Problem

User attempted to restart autopilot post-KMP_BLOCKTIME fix; crashed immediately:

```
File "scripts/autopilot/autopilot.py", line 1275, in _run_loop_inner
    timestamp=datetime.now(timezone.utc).isoformat(),
UnboundLocalError: cannot access local variable 'datetime' where it is not associated with a value
```

### Root cause

Two redundant local `from datetime import datetime, timezone` statements inside function bodies — `_run_loop_inner` line 1459 and `cmd_digest` line 1660 — shadowed the module-level import at line 37. Python's compiler treats `datetime` as function-local because of the later assignment, so the earlier reference at line 1275 (in the same function) hits unbound on every trial-eval path.

Introduced in commit `df98f185` (2026-05-20, "feat: final_schema_validation flag + autopilot delegation expansion + daily digest"). Latent bug — only fires when autopilot reaches the trial-eval timestamp code at line 1275 (which trial 328's gepa_optimize path does on first dispatch).

- `scripts/maintenance/verify_routing_wiring.py` — read-only pre-flight. Loads classifier + verifier, mock-constructs HybridRouter the way memrl.py will, runs synthetic route through the fast-path, confirms shadow-mode metadata populates. Ran twice with all 6 checks passing.
- `scripts/maintenance/analyze_verifier_shadow.py` — reads progress JSONL log, joins `routing_decision` events with `task_completed`/`task_failed`/`escalation_triggered`, produces per-source decision split + verifier coverage + verdict×outcome cross-tab + Brier/ROC-AUC/ECE for `verifier_p_success` vs actual outcomes. Ran against 3 days of pre-wiring logs and correctly showed `0 classifier decisions / 508 rule+learned` — confirms the prior null state.

Production env defaults updated in `orchestrator_stack.py` so the user doesn't need to set anything externally — shadow mode kicks in automatically on launch:

```
ORCHESTRATOR_FRONTDOOR_VERIFIER_GATE=1
FRONTDOOR_VERIFIER_SHADOW=1
FRONTDOOR_VERIFIER_THRESHOLD=0.5
```
