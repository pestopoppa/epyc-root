### Root cause of the recurring optimism

The 460 GB/s number is a **STREAM-class aggregate ceiling** (synthetic memcpy interleaved across 12 channels). It is **NOT** the workload-achievable per-thread × 96-thread aggregate. The gap between achieved 145-245 GB/s (production) and theoretical 460 GB/s is **structural** — per-CCD per-thread channel allocation limits — not "unused BW we could call on with the right code". Kernels are memory-bound (`compute_kernel_memory_stalled` confirmed across 4 architectural classes via CPU24), they just are not synthetic-STREAM-bound.

### What today's roofline still robustly tells us

- **FLOPS side IS genuinely idle** (0.03%-0.35% FP-only, ~20-40% int8 estimated from IPC 0.80). Robust. This motivates Variant B / spec-dec stacking.
- **BW side is "saturated against per-token cost within achievable per-thread budget"** — the 93% per-token cross-check confirms this. The 245 GB/s achieved at healthy gemma4 IS the practical ceiling for that workload, not 53% of it.
- **There is no untapped BW lever to pull.** CPU1/CPU2/CPU15/CPU25 already proved that.

### ik_llama broken binary linkage — root cause + fix

**Diagnosis**:
- `ldd /mnt/raid0/llm/ik_llama.cpp/build/bin/llama-bench` resolved `libllama.so` to mainstream `/mnt/raid0/llm/llama.cpp/build/bin/libllama.so` — which has since been updated to drop the `llama_set_offload_policy` symbol that ik_llama's bench expects.
- ik_llama's own `libllama.so` (at `/mnt/raid0/llm/ik_llama.cpp/build/src/libllama.so`) HAS the symbol (`nm -D` confirms).
- The binary had **`RUNPATH`** (DT_RUNPATH) set to its own build directories — but `RUNPATH` loses to `LD_LIBRARY_PATH` per the Linux dynamic linker order.
- System `LD_LIBRARY_PATH` includes `/mnt/raid0/llm/llama.cpp/build/bin` (likely set in shell rc) — this beat the RUNPATH.

**Latent production-server bug**: the same `RUNPATH`-vs-`LD_LIBRARY_PATH` issue affects `llama-server`. Production gemma4 servers worked only because they don't reach the missing symbol code path AND because the orchestrator launcher explicitly prepended ik_llama's lib paths to `LD_LIBRARY_PATH` at launch time (per `project_gemma4_mtp_launch_recipe`). A code path requiring `llama_set_offload_policy` at runtime would have crashed any production server silently linked to mainstream libllama.

**Fix**: rebuild ik_llama with `-Wl,--disable-new-dtags` linker flag → produces `DT_RPATH` (legacy) instead of `DT_RUNPATH`. `DT_RPATH` beats `LD_LIBRARY_PATH`, so the binary resolves to its own libs regardless of polluted env.

```bash
cd /mnt/raid0/llm/ik_llama.cpp/build
cmake .. -DCMAKE_EXE_LINKER_FLAGS="-Wl,--disable-new-dtags" \
         -DCMAKE_SHARED_LINKER_FLAGS="-Wl,--disable-new-dtags"
cmake --build . -j 32
```

**Verified** post-rebuild:
- `readelf -d` shows `RPATH` (not `RUNPATH`) on all bench/server/lib binaries
- `ldd` resolves to `/mnt/raid0/llm/ik_llama.cpp/build/{src,ggml/src}/lib*.so` without any env overrides
- 32 source files newer than libllama.so → full rebuild syncs to current source tip (2026-05-26)
- ALL ik_llama binaries (llama-bench, llama-server, llama-cli, etc.) benefit from the RPATH fix

## DeepSeek-V4 Phase 1.2 — scope re-estimated (decision needed)

### Trigger
Operator authorized DeepSeek-V4 Phase 1.2 manual port to run after the reproducibility hardening. Yesterday's handoff estimated "1-2 day manual hand-merge" of a 6-file, 1649-LOC delta from antirez/main into `feature/deepseek4-port`. Starting the work surfaced that the estimate was wrong.

### Finding: the API gap is paradigm-level, not header-level

Survey of `git show antirez/main:src/models/deepseek4.cpp` against ik_llama's source layout reveals:

- **Header layout**: antirez uses `src/models/<arch>.cpp` (mainstream llama.cpp recent refactor); ik_llama uses flat `src/llama-<name>.cpp` (older mainstream era). 4 included headers (`models.h`, `llama-kv-cache-iswa.h`, `llama-memory-hybrid-iswa.h`, `llama-memory-recurrent.h`) don't exist in ik_llama.
- **V4 actually needs only `models.h` content** for one base class `llm_graph_context`; the other 3 headers are transitive includes serving Mamba/ISWA/recurrent archs that V4 doesn't touch.
- **The actual gap**: V4 uses `llm_graph_context` (parent class), `llm_graph_input_i` (input interface), `llm_graph_params` (constructor struct) — all from mainstream's graph-context paradigm. ik_llama uses an OLDER paradigm: `class llm_build_context` with member functions (e.g. `build_deepseek2()` member at line ~2399 of the 2915-LOC `llama-build-context.cpp`).
- **These are not the same shape.** Translating 1392 LOC of `deepseek4.cpp` from graph-context idiom to build-context idiom is a structural rewrite, not a hand-merge. The V4-specific logic (CSA + HCA + indexer + compressor + manifold-constrained HC) stays the same; the API surface around it does not.

### Revised scope: 3-5 days of API-translation work, not 1-2 days

The arch enum + hparams deltas (+52/+30/+8/+12 LOC across 4 files) ARE simple hand-merges; the real work is the 1347-LOC model file with per-call-site mapping needed for every graph operation.
