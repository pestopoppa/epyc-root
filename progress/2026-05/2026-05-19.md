## How to make performance reliable (response to "what went wrong + how to make this reliable")

### What went wrong

1. **Multi-week uptime degradation** — root cause of the 12.5 t/s "regression". 6+ days of sustained inference workload accumulates kernel state (likely page-table fragmentation, TLB pressure under continual mmap/mlock churn, NUMA balancing state, or mglru list reshuffling). At ≥1 week uptime, `sync && drop_caches=3` (the 2026-05-06 primary fix) **stops being sufficient**. Reboot is the only restoration lever currently identified.

2. **Detection lag** — the throttle was discovered only because someone manually re-ran a bench retest in 2026-05-16. Production throughput silently halved for an unknown number of days. `host_health.py` (added 2026-05-16) is wired into `safety_gate.py` but has not yet alerted on a real regression.

3. **The 26 vs 23 t/s ambiguity** is **not** a separate failure — it's a documentation/measurement artifact (bench-era used corrupted TIDE state). Now correctly attributed in 2026-05-16 wrap-up + this doc.

### Reliability plan (recommended)

**Detection** — already partially in place; needs activation:
- Confirm `host_health.py` is firing on real throughput regressions, not just preflight check. Today's session re-verified the `KMP_BLOCKTIME` fix via direct measurement (gemma4 idle = 0); host_health should auto-publish similar metrics.
- Add a daily 1-sample isolation bench (cron-driven, 1 model, n_predict=512, exact bench recipe) so throughput drift is visible on a dashboard *before* it doubles. Persist to `data/host_health/YYYY-MM-DD.json`.

**Mitigation** — cheap, high-confidence:
- **Scheduled weekly reboot** (e.g., Sunday 03:00, 1-min downtime). This session demonstrated reboot is the only known full-recovery lever; weekly cadence keeps uptime under the ~1-week drop_caches threshold. Cron + autopilot reboot-awareness needed.
- **Pre-reboot drop_caches sanity check**: if `sync && drop_caches=3` restores within X% of canonical baseline, skip reboot; else reboot. Keeps the lever cost minimal.

**Investigation** — deeper but optional:
- Bisect uptime against throughput on a controlled host. At what uptime exactly does drop_caches stop being sufficient? What kernel state correlates (TLB miss rate via `perf stat`, page-table compaction stats via `/proc/vmstat`, NUMA balancing pte_updates)? This would let us replace blanket reboot with a targeted sysctl reset.
