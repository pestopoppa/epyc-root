# 2026-05-09 — gemma4 worker_general OMP idle-spin discovery + fix

## Summary

Day-after the worker_general → gemma4-26B-A4B Q4_K_M MTP swap landed: discovered that ik_llama.cpp PR #1744's gemma-mtp branch does NOT release OMP threads during idle slots when `OMP_WAIT_POLICY=active`. The 96-thread OMP team busy-waits indefinitely, pinning ~96 cores forever with zero in-flight inference (load average 97). Production llama.cpp's OMP integration releases idle threads correctly with `active`; ik_llama.cpp's fork point regresses this. Fix: targeted env override (`OMP_WAIT_POLICY=passive`) for the `binary_override` launch path only — so other roles keep their canonical recipe untouched. Commit `5eafe2f` pushed.

## Diagnosis: delta-CPU sampling isolated the culprit

Read `/proc/<pid>/stat` for `utime+stime`, slept 5s, re-read, computed `(after - before) / (5 * 100)` to get **live cores busy** in the last 5 seconds:

| PID | cores busy in last 5s |
|---|---|
| 3401375 (worker_general) | **95.13** |
| 3402988 (ingest_long_context) | 0.00 |
| 3401712 (architect_general) | 0.00 |
| 3404267 (vision_escalation) | 0.00 |
| 3400386 (frontdoor) | 0.00 |
| 3413419 (sd-server) | 0.00 |
| 3401072 (coder_escalation) | 0.00 |

Single isolated culprit: **gemma4 worker_general spinning 95 cores with zero in-flight inference**. All other servers idle.

Verified the drive wasn't external traffic:
- `ss -tnpa | grep :8072` — zero EST connections
- llama-server log on port 8072 showed only `slots_idle` + `/health` 200 + `/props` 200; `n_processing_slots=0` for all recent task IDs
- `pgrep autopilot.py` — empty (autopilot was dead from earlier in the session, never restarted)

So nothing was driving the worker. It was spinning idle.

## Root cause: ik_llama.cpp PR #1744 + OMP_WAIT_POLICY=active

The canonical recipe (`build_launch_env` → `_CANONICAL_OMP_ENV`) sets `OMP_WAIT_POLICY=active` for ALL roles. With `active`, OMP threads busy-wait between dispatches instead of sleeping — a latency optimization (no thread-wakeup cost on first token of next request). For the production llama.cpp build, the OMP team handler manages this gracefully — when the inference slot is idle, the team threads release back to a sleeping pool. For **ik_llama.cpp's gemma-mtp branch (PR #1744)**, the team threads do NOT release; they busy-spin forever.

Why only this role: worker_general is the ONLY role whose binary is `ik_llama.cpp/build/bin/llama-server` (per the Phase 2 `runtime_requirements.binary_dir` override added 2026-05-08). All other roles use `/mnt/raid0/llm/llama.cpp/build/bin/llama-server` (production), which doesn't have the regression.

User-confirmed via stop-test: `orchestrator_stack.py stop server_8072` → load average dropped from 97 → ~5 (OS background only). Diagnosis confirmed.

## Fix

Targeted env override in `start_server` worker_pool branch, gated by `if binary_override:` (so other roles unaffected):

```python
env["OMP_WAIT_POLICY"] = "passive"
print(f"    [binary_override] OMP_WAIT_POLICY=passive (gemma4 MTP idle-spin fix)")
```

`passive` means OMP threads sleep when idle; first-token latency pays a few µs of thread wakeup per inference request. Negligible vs continuous 96-core idle waste under any usage pattern that isn't 100% saturated. Under continuous-traffic workloads (autopilot, agent loops), the threads stay warm anyway.
