## Dashboard live-view synchronization repair — implemented

Follow-up to the operator's report that the dashboard no longer felt like a real-time inside view after
concurrent task management landed: CPU-region locks could show every instance ready, the live inference tap
could look idle, and the topology panel could still show an in-progress task whose detail view had no captured
output.

### Root causes found

1. **Semantic mismatch in topology in-flight state.** The snapshot path kept fresh `task_started` rows visible
   for 20s even when no `/slots` poll showed a busy model slot. That was correct for avoiding slot-poll races,
   but the UI labeled the row as green `running`, which made queued/pre-inference/local work look like active
   decode.
2. **Alias roles were not mapped to their physical topology role.** Logical roles such as `coder_escalation`
   and `worker_summarize` share the frontdoor physical pool, but `ConcurrencyAwareBackend` previously looked up
   region locks and tap instance regions by the logical role. Since `instance_topology` only has the canonical
   physical role (`frontdoor`), alias traffic could produce `instance_shape=unknown` / `instance_regions=[]` and
   fail to line up with the CPU-region lock panel.
3. **Structured tap quiet state only advanced on file writes.** If a request started and then produced no further
   JSONL event, the dashboard would not repaint it as quiet until another tap event arrived.

### Implementation

| Area | Change |
|------|--------|
| Backend role mapping | Added topology-role inference in `src/llm_primitives/backend.py`: roles with identical URL lists to a canonical topology role dispatch under that physical role for locks/placement while retaining their logical role for routing and tap display. |
| Concurrency-aware dispatch | `src/backends/concurrency_aware.py` now accepts `topology_role`, uses it for region locks, placement-state evaluation, placement policy lookup, quarter-preference topology, and tap instance-region/shape metadata. Tap metadata now carries both logical `role` and physical `topology_role`/`lock_role`. |
| Snapshot semantics | `src/api/routes/dashboard.py` now annotates JSONL-derived in-flight tasks with `live_state=decoding` when backed by a busy slot and `live_state=pending` when merely fresh/no slot yet. It also normalizes `frontdoor:direct`-style route modifiers and alias roles before comparing with `live_busy_by_role`. |
| Structured tap | `src/api/routes/dashboard_tap.py` now preserves `topology_role`/`lock_role`, computes request age/quiet age, and marks silent open requests as `quiet` with a reason. `/dashboard/events/structured_tap` now emits periodic refreshes so quiet-state aging is visible even without new tap writes. |
| Dashboard UI | `src/api/routes/dashboard.html` groups alias traffic under the physical topology row, renders `decoding` vs `pending` badges, colors quiet tap entries amber, adds `via <topology_role>` metadata for aliases, and shows a clear no-output-yet hint in task detail instead of an empty stream pane. |
