### Implementation details

- **Insertion point** is `stack_commands.cmd_start` line ~498 (after `--numa-mode` filter, before `[2] Checking target ports`). Chosen so prewarm scope tracks `--only`, `--include-warm`, and `--numa-mode` filtering exactly — never warms an unrelated WARM model.
- **Path resolution** reuses the same `build_server_command` dispatcher the launcher uses, then scans the resulting argv for `-m` / `-md` / `--mmproj`. This guarantees the warm set always matches the launch set without duplicating the role → path mapping (which is split across `validate_model_paths`, `WORKER_POOL_MODELS`, `VISION_*_MODEL`, etc.).
- **Inode dedupe**: `(st_dev, st_ino)` keys, not paths. Folds shared-GGUF roles (frontdoor's 5 quarters → 1 file, ingest's 5 → 1, etc.) onto a single warm operation. On the live stack: 39 server entries → 10 unique GGUFs.
- **Size-ordered warm** (largest first) so a small file's elapsed time isn't muddied by a big file still streaming in the kernel page cache.
- **Failure mode**: `prewarm_all` returns 1 if any warm fails, but the callsite continues startup. Warm failure is advisory — the launch path's own `mlock` will still pull pages from disk, just without the interleave guarantee.

### Deferred

- **P5 controlled cold-start validation.** Must wait for the J6 soak window to close — `stop --all && sudo sync && drop_caches && start` would kill the running daemon. After the soak: expect `[1.5]` cold-cache aggregate ~30 s, `numa_maps` showing N0=N1=N2=N3 balance for each shared GGUF, and first autopilot trial in the 55-70 t/s envelope. The 2026-05-28 manual recovery already validated steps 4-6 of this path end-to-end (trial 94 = 70.33 t/s); P5 only proves the codified `[1.5]` reproduces that result without manual intervention.
- **Sibling-script prewarm.** Other Python entry points (`reembed_episodic_store.py`, `repair_episodic_embeddings.py`) launch their own llama-servers and could in principle hit the same collapse. Out of scope for this handoff; reopen if a maintenance run shows the symptom.
- **`MAP_POPULATE` follow-up.** If a future llama.cpp change adds `MAP_POPULATE` to its mmap flags, the prewarm becomes ~redundant (pages get faulted under whatever NUMA policy `mmap` was called under). At that point this phase should become a no-op + a deprecation banner, not a removal — the warm is still cheap enough to be belt-and-suspenders.
