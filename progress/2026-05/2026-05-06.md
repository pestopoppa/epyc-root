## Gemma 4 MTP drafter — research-intake → patch → bench → upstream

User invoked `/research-intake` on Google's blog announcement (https://blog.google/innovation-and-ai/technology/developers-tools/multi-token-prediction-gemma-4/) for pre-trained MTP drafters paired with Gemma 4 (31B Dense, 26B-A4B MoE, E4B/E2B multimodal; Apache 2.0). Session expanded into a full G0.4 directional bench, two upstream contributions to ik_llama.cpp PR #1744, and an in-house GGUF conversion of the 26B-A4B drafter that no community had yet produced.

### Pre-bench discovery

- **All Gemma 4 assistant drafters share one architecture class** (`Gemma4AssistantForCausalLM`, `model_type=gemma4_assistant`): 4 layers, 3 sliding + 1 full attention, `num_kv_shared_layers=4` (Q-only KV shared with target), `enable_moe_block=false` even for MoE targets. Variants differ only in width (31B 1024×32/16, 26B 1024×16/8, E4B 256×4/2). Earlier framing of "31B = conventional small-LM, E4B = Q-only-shared-KV" was wrong — same arch, different widths. **One llama.cpp port covers all three.**
- **ik_llama.cpp PR #1744 (DRAFT, opened 2026-05-06 02:08 UTC by Samuel/Radamanthys11)** adds `LLM_ARCH_GEMMA4_MTP` + new `src/graphs/build_gemma4.cpp` (190 lines) + tensor mapping. Foundational MTP infrastructure already exists in ik_llama.cpp from PR #1270 (GLM-4.x MoE, +813/-199, Feb 2026); PR #1744 adds the dual-model-pattern Gemma 4 case on top.
- **PR #1744 covers Gemma 4-only**: ik_llama.cpp also has Qwen 3.5 MTP (PR #1698, *dense models only* — independent third-party confirmation of our `completed/mtp-speculative-decoding.md` 0.56× hybrid finding).

### Patch 1 — chicken-and-egg gate fix in `params_use_gemma4_external_mtp`

**Problem**: First MTP attempt segfaulted on first `/completion` request. Root cause:

```c
static bool params_use_gemma4_external_mtp(const gpt_params & params_base) {
    return params_base.has_mtp &&
        params_base.speculative.type == COMMON_SPECULATIVE_TYPE_MTP &&  // <— precondition
        model_has_arch(params_base.speculative.model_dft, "gemma4_mtp");
}
```

The `params_base.speculative.type == COMMON_SPECULATIVE_TYPE_MTP` precondition is itself set as a *consequence* of this helper returning true (slot init at L357 sets the type inside the `if (has_external_mtp)` branch). With the precondition still in place, the helper always returned false → gate at L356 fell through to the "MTP enabled via flag, but model has 0 NextN layers. Disabling speculative." branch → downstream NULL deref → segfault (27 GB core dump, deleted; `ulimit -c 0` set per `feedback_no_core_dumps.md`).

**Fix**: 1-line removal of the type-precondition. `has_mtp` flag plus drafter arch `gemma4_mtp` is sufficient evidence of intent. Posted upstream as [PR #1744 comment 4388461769](https://github.com/ikawrakow/ik_llama.cpp/pull/1744#issuecomment-4388461769).

### Patch 2 — cosmetic Oops warning silencing

After Patch 1, two `Oops: tensor with strange name mtp_post_proj.weight / mtp_pre_proj.weight` warnings remained from the size-accounting iteration at `src/llama.cpp:2509-2515`. The iteration special-cases `token_embd.weight`, `output.weight`, `output_norm.weight` and warns on anything else; the two Gemma4Assistant top-level projection tensors weren't recognized. Loading itself was unaffected (handled by `create_gemma4_mtp_tensors`).

Added a 4-line skip block matching the four documented top-level Gemma4Assistant tensor names (`mtp_pre_proj`, `mtp_post_proj`, `mtp_centroids`, `mtp_token_ordering` — last two defensive future-proofing). Posted as [PR #1744 comment 4388596615](https://github.com/ikawrakow/ik_llama.cpp/pull/1744#issuecomment-4388596615) along with the 26B-A4B bench results below.

Out of scope but flagged in the comment: a separate pre-existing `Oops: tensor with strange name rope_freqs.weight` fires for the target-side gemma4 base GGUF — same code path, different arch, would fire on main too.

### G0.4 bench — pure-CPU EPYC canonical

`OMP_PROC_BIND=spread OMP_PLACES=cores OMP_WAIT_POLICY=active` + `numactl --interleave=all` + `taskset -c 0-95` + `-t 96 -fa 1 --no-mmap`, deterministic prompt, `--temp 0 --top-k 1 --top-p 1 --seed 42`, `n_predict=128`. Baseline matched prior `data/cpu_optimization/2026-05-04-q6k-default-on-validation/a2-gemma4_31b_q4km-q6kavx0.json` 6.85 t/s within 3% noise.

| Variant | Baseline | + MTP draft-max=3 | Acceptance | Speedup | Verdict |
|---------|----------|-------------------|------------|---------|---------|
| **Gemma 4 31B Dense** Q4_K_M target + Q8_0 drafter | 7.05 t/s | **21.02 t/s** | 84.3% per-token (91/108), 100% per-batch (36/36) | **2.98×** | architect-tier candidate |
| **Gemma 4 26B-A4B MoE** Q4_K_M target + in-house Q8_0 drafter | 41.49 t/s | 44.12 t/s | 58.7% per-token (81/138), 73.9% per-batch (34/46) | 1.06× | tier X — slower than existing Coder-30B-A3B 49.1 t/s |
| (PR #1744 mixed-CPU/GPU, threads=24, ngl=99, batch=128) | 21.7 | 48.6 | 74% | 2.3× | reference |

**31B Dense pure-CPU 2.98× exceeds the PR mixed-CPU/GPU 2.3×.** Mechanism: the small ~500 MB drafter amortizes well against the slow BW-bound dense target on CPU; on GPU the target is already fast so the relative drafter cost is larger.

**26B-A4B + MTP only 1.06× — empirically confirms lilting.ch contradicting-evidence finding** (intake-527 `contradicting_evidence`): MoE batch=1 sees marginal gains because (a) the smaller 16/8-head drafter struggles to predict MoE expert routing (acceptance dropped 84.3% → 58.7%), and (b) the verifier loads up to K×8 distinct experts per K accepted tokens, eroding the bandwidth saving that makes MTP win on dense. The dense-vs-MoE asymmetry of MTP gains is now a documented quantitative finding, not just a third-party warning.

### In-house GGUF conversion — 26B-A4B drafter

Before our work, only the 31B drafter had a community GGUF (`Radamanthys11/Gemma-4-31B-it-assistant-GGUF`); 26B-A4B and E2B/E4B did not. Converted `google/gemma-4-26B-A4B-it-assistant` HF safetensors (840 MB BF16, single file, 48 tensors) using PR #1744's `convert_hf_to_gguf.py` `Gemma4AssistantModel` class (which already had `pre_projection.weight → mtp_pre_proj.weight` and `post_projection.weight → mtp_post_proj.weight` mappings) → 839 MB BF16 GGUF → 441 MB Q8_0 via `llama-quantize`. No conversion errors. Output structurally identical to 31B drafter modulo width parameters.

Artifact at `/mnt/raid0/llm/models/gemma-4-26B-A4B-it-assistant-Q8_0.gguf` (441 MB). Demonstrated PR #1744's converter handles arbitrary Gemma4Assistant variants — not just the 31B that the PR author tested.
