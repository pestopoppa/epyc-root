## CPU REGION LOCKS panel — reported regression NOT reproduced (read-only diagnosis)

### Trigger

Operator reported a dashboard regression: the **CPU REGIONS LOCK** panel appeared to show "no locks held; per-region locks ENABLED" while the live inference tap simultaneously showed an active frontdoor stream (screenshot timestamped `2026-05-31T22:55`). Investigation was conducted live with AutoPilot running, strictly read-only (no restarts, no disruption to the run).

### Method + evidence (every layer verified against live AutoPilot traffic)

| Layer | Probe | Result |
|---|---|---|
| Feature flag | `tr '\0' '\n' < /proc/<pid>/environ` for the 7 live orchestrator workers (pids 4104652–8, started 06:39) | `ORCHESTRATOR_PER_REGION_LOCKS=1` + `PLACEMENT_STATE_MACHINE=1` + `CROSS_ROLE_DISJOINT_PLACEMENT=1` — **enabled** |
| Lock acquisition | Polled `/proc/locks` vs `cpu_region.*.lock` inodes for 35 s during AutoPilot's frontdoor run | `frontdoor.q0`+`frontdoor.q1` (half0) held **continuously** — lock path works |
| API endpoint | `GET /dashboard/api/region_locks` (port 8000) while frontdoor lock held | Returned `frontdoor: active_instance_idxs:[0], held:['q0','q1'], shape half0` — **correct** |
| Other roles | Same synchronized disk-vs-endpoint compare loop | `ingest_long_context` (q0+q1) and `architect_general` (full q0–q3) both tracked disk truth exactly |
| Frontend JS | `updateRegionLocks()` in `dashboard.html` (`actualActiveLockCount` ← `active_instance_idxs`; 1.5 s poll) | Count → "active" status + 🔒 cells — logic correct |

### Root cause assessment

No defect found in the live stack. Hypotheses raised and **falsified**: stale process; flag-not-in-effect; streaming lock-lifetime escape (`LlamaServerBackend.infer_stream_text` is eager — consumes the whole stream via `on_chunk` inside the `_dispatch()` lock block, so the region lock spans the full stream); endpoint dropping holders; JS miscount.

Most likely explanations for the screenshot (both non-bugs):
1. **Screenshot predated the flag taking effect in that process.** The screenshot is from `05-31 22:55`; current workers were reloaded `06-01 06:39` with the per-region flags on. Before the reload, frontdoor used the legacy global `heavy_model.lock` (not a `cpu_region.*.lock` file), so the panel correctly read "no locks held" while "ENABLED" reflected the dashboard worker's own env. (Note: the earlier same-day session's dashboard-hardening + orchestrator reload is the relevant state change.)
2. **Cross-panel timing skew.** The live-tap panel repaints completed/"quiet" requests rather than clearing them, so it can still show "frontdoor streaming" for a request that already finished — at which instant the region lock is legitimately released and the lock panel correctly shows nothing. The two panels sample independently.

### Outcome

- **No code change made.** Patching would have modified a path proven correct. The orchestrator working tree contains only a parallel agent's WIP (`chat.py` `request_id/trial_id/batch_id`, `requests.py`, `primitives.py`, `test_chat_delegation.py`) — explicitly **not** committed by this session.

### Deferred / next step if it recurs

- If the empty panel reappears **post-reload on the current stack**, capture one synchronized sample: panel state + `grep <inode> /proc/locks` for the `cpu_region.frontdoor.*.lock` files at the same instant (every such sample this session was consistent). The only render path not exercised by a failing case is `buildTapInferredRegionLocks()` (tap-inferred holders) — instrument that if a genuine disagreement is captured.

**One correction in the positive direction:** CPU MXFP4 MoE `#41922` kernels are **AVX-512, not AMX** (`csrc/cpu/sgl-kernels/vec.h`: "vectorized with avx512") → ISA-capable on Zen5. AMX GDN `#42707` is confirmed AMX-gated (`torch.cpu._is_amx_tile_supported()` in `platforms/cpu.py`) → no Zen5 path.

### MXFP4 availability scan (HF) + llama.cpp native-support finding

- Our fork (`production-consolidated-v5`) **already supports MXFP4 natively**: `GGML_TYPE_MXFP4 = 39` (+ `NVFP4 = 40`), real AVX2 `ggml_vec_dot_mxfp4_q8_0` (`arch/x86/quants.c:760`) and an `__AVX512F__`/`__AVX2__`-guarded `ggml_gemm_mxfp4_8x8_q8_0` repack. **So vLLM `#41922` is moot for us — we could bench MXFP4 GGUFs on the existing stack with zero new code.**
- Models we already run that exist in MXFP4 GGUF: **gemma-4-26B-A4B** (`noctrex/…-MXFP4_MOE-GGUF`), **Qwen3.6-35B-A3B** (incl. an MTP variant `noctrex/Qwen3.6-35B-A3B-MTP-MXFP4_MOE-GGUF`). Native-MXFP4 reference: `openai/gpt-oss-20b`/`-120b` (GGUF via ggml-org/unsloth). Specialists: `gpt-oss-safeguard-20b/120b` (safety classifiers), `RedHatAI/gpt-oss-20b-speculator.eagle3` (EAGLE3 drafter). AMD publishing a broad MXFP4 collection (DeepSeek-R1, Kimi-K2.5, GLM-5/5.1, Qwen3.5-397B).

### Decision

**Parked the MXFP4 requant angle** (operator call). Rationale: post-hoc MXFP4 requant of models we already run is unlikely to be additive on a BW-bound CPU — similar bpw (~4.25 vs Q4_K_M ~4.5), coarser scales (single E8M0/32 vs k-quant hierarchical superblock), decode is BW-bound so speed ≈ flat, and MXFP4's hardware-native edge needs FP4 units (GPU/NPU) we don't have; our x86 MXFP4 repack is also AVX2-width, not the hand-tuned AVX-512BW we built for Q8_0 (`project_x86_kquant_repack_gaps`). **gpt-oss-native MXFP4 remains a separate open question** (model onboarding, not a quant choice) — not actioned.
