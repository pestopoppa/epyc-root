## AutoPilot seeder convergence restart persistence

### Trigger

After the memory-count and dead-lever fixes, the remaining concern was that a fresh autopilot process could still read a mature routing-memory store but believe seeding was "not converged yet", making `train_routing_models` look ineligible after restart.

### Root cause / findings

- `Seeder` kept convergence state in process-local fields only:
  - `_batch_count`
  - `_consecutive_converged`
  - `_td_errors`
- Only `td_errors` were partially persisted in `autopilot_state.json`, and even that happened only on plot intervals, not every real trial.
- Startup restored only the legacy `td_errors` list and did so by mutating `seeder._td_errors` directly; it never reconstructed `_batch_count` or `_consecutive_converged`.
- Net effect: a restarted process could see `memory_count >= 500` but still present `is_converged=False` and `batch_count=0`, biasing planning away from `train_routing_models`.

### Changes made

| Repo | File / artifact | Change |
|---|---|---|
| `epyc-orchestrator` | `scripts/autopilot/species/seeder.py` | Added `Seeder.export_state()` / `Seeder.restore_state()` for explicit persistence of `td_errors`, `batch_count`, and `consecutive_converged`. |
| `epyc-orchestrator` | `scripts/autopilot/species/seeder.py` | Added legacy reconstruction fallback: if only `td_errors` exist, infer `batch_count` from history length and rebuild `consecutive_converged` from the trailing below-threshold TD-error streak. |
| `epyc-orchestrator` | `scripts/autopilot/autopilot.py` | Added `seeder_state` to the default state shape, restored it on startup, and now persist both `seeder_state` and `td_errors` on every metric-bearing trial instead of only at plot intervals. |
