## KV Cache Quantization — Handoff Audit, Bug Fixes, Long-Context Validation (Session 2026-03-28c)

### Session Overview

Picked up `handoffs/active/kv-cache-quantization.md`. Audited all phases (0-3d), fixed 4 critical bugs in hybrid buffer, benchmarked at 32K long context, determined production disposition.

### Bug Fixes (hybrid buffer path, `hadamard-kv-smoothing` branch)

1. **Mask always -inf for old cells** (`llama-graph.cpp:424`): Old KV data never attended to.
2. **n_evicted not reset on clear** (`hybrid-prec.cpp:clear`): Stale counter caused cross-request corruption.
3. **Eviction during prefill** (`hybrid-prec.cpp:init_update`): Multi-token batches triggered premature eviction. Added `in_prefill` guard.
4. **GQA shape mismatch** (`llama-graph.cpp:1928`): Missing permute for GQA broadcasting in split attention.

Also switched old K cache from turbo_q3 to q4_0 (PolarQuant dequant too lossy), and replaced non-flash split attention with flash attention + concat (cast q4_0->f16, concat, `build_attn_mha`).

### Long-Context Benchmark (32K, Qwen2.5-7B native context)

| Config @ 32K | RSS (MB) | Wall (s) | Quality |
|---|---|---|---|
| f16/f16 | 16,386 | 549.8 | correct |
| q8_0/q4_0 | 15,322 | 598.2 (+9%) | **correct** |
| q4_0/q4_0 | 15,098 | 561.1 | **garbage** |

**q4_0/q4_0 degrades at 32K on 7B model.** q8_0/q4_0 production config validated. Note: per session 2026-03-28b findings, this may be model-size dependent — TQ3/q4_0 work better at 35B+.

### Perplexity (512 ctx)

f16 = 1.5375, hybrid q4_0+Hadamard = 1.5375. **Identical.**

### Hybrid Buffer Architecture: Dead End

The dual-cache design allocates 2x full context KV caches = memory-NEGATIVE. Standard `-ctk q8_0 -ctv q4_0` is strictly better.

### TurboQuant/QJL Post-Mortem

Why Google succeeded and we got PPL 16K:
- **S=256 uniform** (paper: S=512 for initial 25% layers). SNR at S=256 = 1.13 (noise ~ signal).
- **PPL metric** (paper: task-based evals where softmax suppresses noise).
- **At competitive bit rates** (S=512 + outliers = ~4.5 bits effective), q4_0+Hadamard matches quality with zero complexity.
- **Not a hardware limitation** — same math on CPU and GPU.

### Production Disposition

- **Hadamard (Phase 1)**: Cherry-pick to production-consolidated-v3.

## TurboQuant Norm Correction + 32B Decision Gate (Session 2026-03-28e)

### Norm Correction

Restructured `ggml-turbo-quant.c` dequant to factor norm as final multiplier. Quantize computes unit-norm reconstruction, stores `||x|| / ||unit_recon||`. Qwen2.5-7B PPL: 9.017 → **2.13** (77% gap reduction, still not competitive with q4_0 1.54).

### Decision Gate: Coder-32B Q4_K_M

| Config (10 chunks, ctx=512) | PPL | vs f16 |
|---|---|---|
| f16 V | 1.3861 | — |
| turbo_q3 V (norm corrected) | 1.4676 | **+5.9%** |
| q4_0 V | 1.3875 | +0.001% |

**TQ3 ABANDONED.** q4_0 is strictly better even on 32B. Fused dequant work cancelled.

### Flash Attention V Dequant — Root Cause Found

Benchmarked 4 KV configs at 4K ctx on Coder-32B Q4_K_M:

| Config | Time/chunk | vs f16/f16 |
|---|---|---|
| f16/f16 | 37.91s | baseline |
| **q4_0/f16** | **37.42s** | **-1% (faster!)** |
| q8_0/f16 | 40.57s | +7% |
| q8_0/q4_0 | 64.87s | **+71%** |

**V=q4_0 is the ENTIRE prefill bottleneck.** K dequant is free (bandwidth savings offset cost). The `v_to_float → f32 buffer → vec_mad_f32` path in flash attention is 71% slower than the native f16 `vec_mad_f16` path.

**Production config changed**: `q4_0 K / f16 V` for pure-attention models (coder, architect). Updated `orchestrator_stack.py` and `model_registry.yaml`. Frontdoor/ingest keep q4_0/q4_0 (hybrid SSM amortizes the V dequant cost).

**Q35 frontdoor q4_0/q4_0 at 4K**: PPL f16=1.2510, q4_0/q4_0=1.2466, q4_0/f16=1.2333. All within noise. Hybrid model absorbs KV quant completely. **VALIDATED.**

**Coder-32B q4_0/f16 at 4K**: PPL 1.0034 (vs f16 1.0033). Speed 45.56s (vs f16 46.08s = 1% faster, vs q8_0/q4_0 56.61s = 23% faster). **VALIDATED.**

## Nemotron-Cascade 2 CPU Benchmark (Session 2026-03-28h)

### Summary

Downloaded and benchmarked Nemotron-Cascade 2 (30B-A3B, Mamba2 hybrid MoE) Q4_K_M on EPYC 9655 CPU. **Single-instance: 3.3x faster than Qwen3.5. NUMA 4-way: 16% slower. NOT replacing frontdoor.**

### Model Properties

- File: `nvidia_Nemotron-Cascade-2-30B-A3B-Q4_K_M.gguf` (24 GB, 6.26 BPW)
- Architecture: `nemotron_h_moe`, 52 layers, 31.58B total params
- 128 experts, 6 active/token, 1 shared expert
- **88% Mamba2 layers** (46/52) — only 6 attention layers (vs Qwen3.5's 75% Delta Net)
- KV cache: 24 MB (6 attn layers × 2 KV heads). Recurrent state: 190 MB.
- Tokenizer: `tekken` (Mistral-family) — incompatible with Qwen drafters

### CPU Benchmark Results

| Config | Nemotron t/s | Qwen3.5 t/s | Delta |
|--------|-------------|-------------|-------|
| 192t interleave | **19.4** | 7.25 | **+167%** |
| 96t node0 | **38.7** | 13.39 | **+189%** |
| 48t quarter | **40.6** | ~12.4 | **+227%** |
| **NUMA 4×48t agg** | **41.7** | **49.7** | **-16%** |

### Key Findings

1. **Single-instance dominance**: Mamba2 is 3.3x faster per instance — fewer attention layers, smaller KV cache, more compute-efficient recurrent kernel.
2. **NUMA 4-way collapse**: Per-instance drops from 40.6 t/s (solo) to ~10 t/s (4-way) — **4x degradation**. Qwen3.5 shows near-zero degradation. Root cause: 4×24 GB mmap'd copies saturate memory bandwidth; Mamba2's 190 MB recurrent state (read+written every token) adds constant bandwidth pressure.
3. **Speculation blocked**: `tekken` tokenizer incompatible with all available Qwen drafters. No Nemotron-family draft models at 0.5-1B scale.
4. **GPU advantage doesn't translate to CPU NUMA**: The 67% GPU speed advantage (187 vs 112 t/s) becomes irrelevant in multi-instance CPU deployment where memory bandwidth is the bottleneck.

### NUMA Scaling Analysis (follow-up)

Investigated why 4×48t degraded so badly. Ran 1×48t, 2×48t (same NUMA), 2×96t (cross-NUMA), 4×48t.

| Config | Agg t/s | Per-inst | Efficiency |
|--------|---------|----------|------------|
| 1×48t | 40.9 | 40.9 | 100% |
| **2×48t same NUMA** | **51.1** | 25.6 | 63% |
| 2×96t cross-NUMA | 35.6 | 17.8 | 43% |
| 4×48t | 40.6 | 10.2 | 25% |

**Root cause**: Cross-NUMA memory access, not instance count. 2×48t on the same NUMA node retains 63% efficiency and yields 51.1 t/s aggregate — **beating Qwen3.5's 4×48t (49.7 t/s) while using only half the CPU cores.**

### Revised Verdict

**FRONTDOOR CANDIDATE with 2×48t deployment.** 51.1 t/s on one NUMA node, freeing the other 96 cores for architect/coder. Quality benchmark is the deciding factor.
