## Prompt Lookup (`--lookup`) Re-ported to llama.cpp production-consolidated-v2

### Problem

The `--lookup` flag (commit `8e35dbc01` on `production-consolidated`) was lost during the v1→v2 rebase. This blocked all lookup-based benchmark configs (`lookup`, `moe_lookup`, `spec_lookup`, `moe_spec_lookup`).

### Solution

Manual port across 5 files (88 insertions). Cherry-pick failed with 10 conflicts — manual port was cleaner.

| File | Changes |
|------|---------|
| `common/arg.cpp` | `--lookup` CLI flag definition |
| `common/common.h` | `bool lookup` in `common_params` |
| `tools/server/server-task.h` | `bool lookup` in `task_params` |
| `tools/server/server-task.cpp` | JSON serialization (2 `to_json` blocks), defaults, `json_value` parsing |
| `tools/server/server-context.cpp` | `ngram_cache_context` + `lookup_enabled` in `server_slot`; cache clear in `reset()`; `can_speculate()` gate; slot init from task params; spec-first/lookup-fallback draft generation (~70 lines); ngram cache update after normal decode; cache init from prompt tokens; cache update after speculative acceptance |

### Architecture

- **Spec-first, lookup-fallback**: When both `-md` and `--lookup` are active, draft model is tried first. If no draft model loaded, prompt lookup generates draft tokens from ngram cache.
- **Per-slot isolation**: Each slot has its own `ngram_cache_context`, enabling parallel request support.
- **Cache lifecycle**: Built from prompt tokens at `SLOT_STATE_GENERATING`, updated incrementally after each decode and after speculative token acceptance.

## Qwen3.5 Benchmark Quality Scoring & Thinking Quirks

### Claude-as-Judge Quality Scoring

Scored 8 model baselines using 0–3 scale (3=correct/complete, 2=mostly correct, 1=partial/truncated, 0=wrong/degenerate):

| Model | Questions | Avg Score | Notes |
|-------|-----------|-----------|-------|
| qwen35_2b_q4km | 30 | 2.00 | Best 2B variant for task-oriented work |
| qwen35_2b_q6k | 30 | 1.63 | Higher quant → more think tokens → truncation |
| qwen35_2b_q8 | 30 | 2.07 | |
| qwen35_4b_q4km | 30 | 1.80 | Results deleted — rerun with 4096 max_tokens |
| qwen35_4b_q6k | 30 | 1.90 | Results deleted — rerun with 4096 max_tokens |
| qwen35_4b_q8 | 30 | 1.47 | Results deleted — rerun with 4096 max_tokens |
| qwen35_9b_q4km | 61 | 2.25 | Best dense worker candidate |
| qwen35_9b_q6k | 53 | 2.13 | |

**Key finding**: Higher quants score LOWER due to think-block truncation at 2048 max_tokens. The model spends tokens inside `<think>` blocks, and better quants produce more verbose reasoning that exhausts the budget before producing output.

### Token Budget Fix — max_tokens 2048 → 4096

Raised `max_tokens` in 6 suite YAMLs to prevent think-block truncation:
- `agentic.yaml`, `coder.yaml`, `general.yaml`, `math.yaml`, `thinking.yaml`, `long_context.yaml`
- `instruction_precision.yaml` stays at 512 (short structured output), `vl.yaml` stays at 1500

### Thinking Disable Infrastructure (`thinking_quirks`)

Discovered that Qwen3.5 produces `<think>` blocks even in raw `/completion` mode. For task-oriented suites (agentic, instruction_precision) where reasoning wastes tokens, the trick `<think>\n</think>\n` appended to the prompt forces the model to skip thinking.

**Implementation:**
- `registry.py`: new `get_thinking_disable_trick(role, suite)` method
- `run_benchmark.py`: applied in both `_run_quality_question` and `_run_speed_question`
- `model_registry.yaml`: `thinking_quirks` block on 28 models (trick + disable_suites)

**Models with thinking_quirks:**
- 18 Qwen3.5 models (all variants)
- DeepSeek-R1-Distill-Qwen-32B, 14B-Q6_K_L, 7B
- Qwen3-4B-Thinking-2507
- 7 deprecated models (MiniMax M2.1, Phi-4, Qwen3-Next-80B, Qwen3-30B-Thinking)
