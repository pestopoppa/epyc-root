# Progress Report — 2026-03-09

## SpecExec Verification Profiling — Complete

Empirically validated (partially refuted) the SpecExec thesis on EPYC 9655 hardware. All 3 phases complete with publication-quality data, plots, and documentation.

### Key Findings

1. **SpecExec thesis partially refuted**: Verification cost scales ~4-5x from N=1 to N=64 for Q4_K_M models (dequantization overhead). Only f16 models show near-flat behavior (1.69x at N=64).
2. **Linear K=16 is optimal**: Increasing `--draft-max` from 16 to 256 provides zero throughput benefit. Acceptance rate decay neutralizes verification cost savings.
3. **Draft model speed varies 4x**: Qwen2.5-Coder-0.5B (185 t/s) vs Qwen3.5-0.8B (44 t/s). Drafter selection matters more than K tuning.
4. **Tree speculation estimated 1.5-2.5x gain** over linear K=16, tempered from original 5-9x hypothesis.

## Remove KV Cache Quantization for Architect Models

### Change
Removed `_KV_QUANTIZED_ROLES` and associated `--cache-type-k q8_0` flag from `orchestrator_stack.py`. Both architect models (`architect_general` Qwen3-235B-A22B and `architect_coding` Qwen3-Coder-480B-A35B) now use f16 for K and V cache, matching all other roles.

### Rationale
With 1TB+ system RAM, the VRAM savings from q8_0 K-cache quantization are unnecessary. K-cache is more sensitive to quantization than V-cache — precision loss in attention logits can affect expert routing in large MoE models and degrade long-context recall. Removing it eliminates any potential quality risk.

### Expected Impact
- **Quality**: Removes any precision loss from K-cache quantization on the two largest models
- **Speed**: ~2-5% slower generation (larger KV reads during attention), negligible with available memory bandwidth
- **Memory**: ~2x KV cache memory for architects (acceptable given 1TB+ headroom)

## Orchestrator Stack Startup Fix

### Problem
Stack failed to start — lean model registry was at `orchestration/model_registry.yaml` (the default path), which has no `model.path` entries. Servers started with `-m ""` (empty model path), resulting in healthy but empty llama-server processes using only ~48GB RAM instead of ~927GB.

### Root Cause
The lean registry (designed for FOSS deployments without local model infrastructure) was placed at the default registry path. The `RegistryLoader` loads `orchestration/model_registry.yaml` by default, which was the lean file. Additionally, the lean registry referenced draft roles (`draft_qwen3_coder_0_75b`, `draft_qwen2_5_coder_0_5b`, `draft_qwen3_0_6b`) that weren't defined in it.

### Fix
1. Renamed lean registry to `model_registry_lean.yaml`
2. Symlinked full registry from `epyc-inference-research` as `model_registry.yaml`
3. Added missing draft role definitions to lean registry (for future lean-mode use)
4. Fixed `architect_general` draft reference: `draft_qwen3_0_6b` → `draft_qwen3_0_6b_q8_0` (Q2_K deprecated)

### Smoke Test Results
All 18 components healthy with real models loaded (~927GB RSS total):
- frontdoor (Qwen3-Coder-30B-A3B): 34 t/s
- coder_escalation (Qwen2.5-Coder-32B): 15 t/s
- worker pool (Qwen2.5-7B): 32 t/s
- architect_general (Qwen3-235B-A22B): 9.4 t/s
- architect_coding (Qwen3-Coder-480B-A35B): 3.8 t/s

## OpenAI-Compat Endpoint Fix (`/v1/chat/completions`)

### Problem
The `/v1/chat/completions` endpoint always returned empty responses (`"content": ""`, 0 completion tokens) even with `mock_mode=False`. The endpoint was unusable for OpenAI-compatible clients (Aider, etc.).

### Root Cause
The endpoint used `state.llm_primitives` (created at startup without `server_urls`), so `LLMPrimitives` had no backends configured. It would enter the real inference path but immediately fail with `RuntimeError("No backend configured")`. The REPL caught this and produced a `FINAL()` call that yielded an empty response.

The `/chat` endpoint works because it creates its own `LLMPrimitives(server_urls=get_config().server_urls.as_dict())` per request.

### Fix
Updated `openai_compat.py` to create `LLMPrimitives` with `server_urls` from config (matching the `/chat` endpoint pattern), rather than reusing the startup singleton.
