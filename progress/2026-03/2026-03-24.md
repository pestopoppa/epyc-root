### Decision: Q4_K_M Confirmed

f16 offers **zero quality improvement** over Q4KM (identical 74% pass rate, identical 133/183 raw score) while being 41% slower and using 3.5x more RAM. Q8 gains only +2 questions at 24% speed cost. ### Key Findings
- Prompt echo: Both Q8 and f16 echo full prompts before responding (likely chat template issue at lower TPS)
- Repetition loops: f16 showed more degeneration than Q4KM on some questions
- Math improvement: Q8 solved horse proof and probability theory correctly (Q4KM/f16 failed both)
- RAM savings: Switching from f16 to Q4KM saves **186 GB** across 4 NUMA instances (260->74 GB)

## Round-Robin Routing for NUMA Multi-Instance — Implemented

### Problem
frontdoor and coder_escalation each have 4 NUMA-pinned instances (ports 8080/8180/8280/8380 and 8081/8181/8281/8381) but all API traffic was going to the primary port only. 75% of NUMA capacity was idle.

### Solution
- **`src/backends/round_robin.py`** — `RoundRobinBackend` wraps multiple `CachingBackend` instances with a thread-safe counter
- **`src/llm_primitives/backend.py`** — `_init_caching_backends` detects comma-separated URLs and creates `RoundRobinBackend`
- **`src/config/__init__.py` + `src/config/models.py`** — Default URLs updated to comma-separated for frontdoor, coder, coder_escalation, worker_summarize
- **`src/llm_primitives/inference.py`** — `_primary_url()` helper extracts first URL for admission control, health tracking, and inference locks

### Design
- Comma-separated URLs in config: `"http://localhost:8080,http://localhost:8180,..."`
- Override via env: `ORCHESTRATOR_SERVER_URLS_FRONTDOOR=http://localhost:8080` (single URL disables round-robin)
- Admission control keys on primary URL (total queue depth across instances)
- Round-robin is at the backend layer — transparent to callers

## 35B NUMA Benchmark + Lookup Segfault Investigation

### Benchmark Results (measured)
- Single 48t moe6: **12.7 t/s** per instance (was estimated 19.6 with lookup)
- 4×48t concurrent: **~50.8 t/s aggregate** (near-linear NUMA scaling: 4.0x)
- Lookup barely activates on short prompts (11-33 token ngram caches)

### Segfault Diagnosis

| Test | Flags | Result |
|------|-------|--------|
| baseline | none | PASS |
| moe6_only | `--override-kv ...=6` | **5/5 PASS** |
| lookup_only | `--lookup` | **CRASH after 1-2 prompts** |
| moe6+lookup | both | CRASH after 2-3 prompts |

**Root cause**: `--lookup` on Qwen3.5 hybrid SSM models causes segfault when prompt cache accumulates recurrent state across requests. Related to llama.cpp PR #13194 (hybrid/recurrent memory cache invalidation). moe6 alone is clean.

**Action taken**: Disabled `lookup: false` in model_registry.yaml for frontdoor. Updated deployment docs, "What NOT To Do" section, and launch configs. Added rule #7: "Don't use --lookup on Qwen3.5 hybrids until fixed."

**REAP connection**: If REAP permanent pruning removes experts from the model weights (vs moe6's runtime override-kv), the recurrent state would be consistent with the pruned architecture, potentially avoiding the state corruption that triggers the segfault.
