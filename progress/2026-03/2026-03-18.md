### What

### COMPLETE Results

| Config | Instances | Threads/inst | CPU Binding | Per-inst t/s | Aggregate t/s | vs Baseline |
|--------|-----------|-------------|-------------|-------------|---------------|-------------|
| A (baseline) | 1 | 192 | all CPUs | 7.25 | 7.25 | 1.0x |
| B (single-node) | 1 | 96 | node 0 | 13.39 | 13.39 | **1.85x** |
| C (2-way seq) | 2 | 96 each | 1 per node | n0=13.3, n1=11.4 | **24.68** | **3.4x** |
| C (2-way conc) | 2 | 96 each | 1 per node | n0=9.4, n1=10.0 | **19.38** | **2.7x** |
| D (4-way) | 4 | 48 each | quarter | n0=13.2-13.6, n1=11.3-11.6 | **49.66** | **6.9x** |

### Critical Findings

1. **Single-node is 1.85x FASTER than all-cores**: Running 96 threads on one NUMA node (13.39 t/s) far outperforms 192 threads across both nodes (7.25 t/s). Cross-NUMA memory access penalty is devastating for this model.

2. **4-way parallel is optimal**: 48 threads per instance achieves nearly identical per-instance throughput as 96 threads. The model's compute (MoE with ~3B active params + Delta Net recurrent) saturates at 48 threads. 4 independent instances give **6.9x aggregate throughput** vs naive 192-thread baseline.

3. **NUMA node 1 is ~85% of node 0**: Consistent across all configs. Likely first-touch page cache bias (node 0 loads first, OS caches pages there).

4. **Concurrent vs sequential**: When both nodes generate simultaneously (Config C concurrent), per-instance throughput drops ~25% (13.3→9.4 t/s) due to inter-node traffic. Sequential queries to alternating nodes avoid this.

### Implications
- Production hybrid models (Qwen3.5-35B-A3B) should use `taskset` to pin to NUMA nodes
- **4 independent instances** on EPYC 9655 can serve ~50 t/s aggregate for this model (vs 7.25 single-instance)
- This REOPENS the SSM hybrid acceleration handoff: parallel serving across NUMA nodes is vastly more effective than forcing all cores to work on one request
- The orchestrator should route requests round-robin across NUMA-pinned instances

| Model | Size | 1×192t | 1×96t | 4×48t agg | Draft | Best |
|-------|------|--------|-------|-----------|-------|------|
| 27B dense | 16 GB | 12.3 | 11.8 | **24.4** | 8.6 | **2.0x** |
| 35B-A3B MoE | 19 GB | 7.25 | 13.4 | **49.7** | — | **6.9x** |
| 122B-A10B MoE | 69 GB | 12.6 | 12.5 | — | 11.2 | ~1.0x |
| 397B-A17B MoE | 205 GB | 12.4 | 11.4 | — | — | ~1.0x |

### Key Findings
1. **All Qwen3.5 hybrids converge to ~12 t/s** decode speed regardless of model size — Delta Net recurrent layers bottleneck completely
2. **35B-A3B MoE is the outlier**: 6.9x with 4-way because MoE (~3B active params) saturates at 48 threads AND cross-NUMA penalty was huge at 192t (7.25 t/s)
3. **Dense 27B**: 2.0x with 4-way — needs more threads per instance than MoE, so 48t gives only 6.1 t/s per instance
4. **Large hybrids (122B, 397B)**: NUMA makes no difference — they're recurrent-bottlenecked at ~12 t/s regardless of thread count
5. **Draft hurts ALL hybrids** — consistent with S3 findings

### Implication
The 35B-A3B MoE is unique in the Qwen3.5 family because its MoE architecture makes it CPU-efficient (few active params) but NUMA-sensitive (cross-node access penalties). All other Qwen3.5 models are recurrent-dominated and insensitive to NUMA configuration.

## Quantization Scaling on Hybrid Models (Session 2026-03-18d)

### What
Tested whether Qwen3.5 hybrid decode speed changes with quantization (Q4→Q6→Q8). If recurrent state update dominates, higher quants should barely affect speed.

### Results: Higher quants DO reduce speed

| Model | Q4_K_M | Q6_K | Q8_0 | Q8/Q4 |
|-------|--------|------|------|-------|
| 9B dense | 12.2 t/s | 11.2 t/s | 7.5 t/s | **61%** |
| 27B dense | 12.4 t/s | 9.3 t/s | — | — |
| 35B-A3B MoE | 11.4 t/s | — | 9.5 t/s | **83%** |

### Analysis
- **MoE hybrids** retain 83% speed at Q8 — fewer active params = less weight reading
- **Dense hybrids** retain only 61% at Q8 — all params read every token
- The ~12 t/s "ceiling" at Q4 is a mix of recurrent state update (~60-80%) + weight reading (~20-40%)
- **Q4_K_M is strongly preferred** for Qwen3.5 hybrids: marginal quality gain from Q8 costs 17-39% speed

Script: `bench_hybrid_quant_scaling.sh`
Data: `epyc-inference-research/data/hybrid_quant_scaling/`
