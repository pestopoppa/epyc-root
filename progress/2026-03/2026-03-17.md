## MTP-1 Speculative Decoding — Step 6 Complete: 78.5% Acceptance

### Summary
Completed the full MTP logits plumbing pipeline and acceptance rate measurement for Qwen3.5-35B-A3B. Found and fixed a critical token alignment bug. MTP acceptance is excellent at 78.5-78.9% exact match.

### Key Finding: Token Alignment Bug
The MTP head is trained with `embed(token_{n+1}) + hidden_state(n) → predict token_{n+2}`. The initial implementation incorrectly used `embed(token_n) + hidden_state(n)` (same step), yielding only 5% acceptance. Systematic diagnosis:
- Tested without transformer layer: 2% → confirmed concat+project is the problem
- Tested Q8_0: 0% → ruled out quantization
- Verified conversion correctness → ruled out tensor corruption
- Identified root cause: wrong embedding input (token_n instead of token_{n+1})

### Fix: Cached Hidden State
Cache the pre-norm hidden state from step N to CPU. On step N+1, when the sampled token_{n+1} is the input:
- `embed(token_{n+1})` = current input embedding (already correct)
- `hidden_state(n)` = cached from previous step (new)
- MTP correctly receives `embed(token_{n+1}) + hidden_state(n)`

Implementation: `llm_graph_input_mtp_hidden` class, graph params threading, CPU-side cache buffer, async extraction after decode.

### Also Fixed: Acceptance Test Off-by-One
With cached hidden, MTP predicts the SAME step's output token (not the next step's). Changed acceptance test from delayed comparison to same-step comparison.

## MTP-1 Speculative Decoding — Step 7 Complete: NOT VIABLE (0.56x throughput)

### Summary
Implemented full MTP speculation loop and standalone `llama_decode_mtp()` API. Despite 70% draft acceptance, net throughput is 0.56x baseline due to 2-token verification batches costing 3-4x single decode on the hybrid recurrent architecture.

### Implementation
1. **`llama_decode_mtp()` API**: Standalone MTP-only eval using new `LLM_GRAPH_TYPE_MTP_EVAL` graph type. Extracted `build_mtp_head()` as reusable method. ~10ms cost (5% of full decode), 100% match with in-graph MTP logits.
2. **Speculation loop** (`tools/mtp-speculation/`): MTP draft token → 2-token verification batch → accept/reject cycle.

### Bugs Fixed
- **`llm_graph_result::reset()` stale pointers**: `t_logits_mtp` and `t_mtp_hidden_out` survived graph rebuilds → use-after-free crashes. Fixed by clearing all MTP fields in `reset()`.
- **In-graph MTP guard**: Skip MTP head when `n_outputs > 1` (verification batches have dimension mismatch with single cached hidden state).

### Timing Results (Qwen3.5-35B-A3B Q4_K_M, 192 threads)

| Operation | Latency |
|-----------|---------|
| Baseline single-token decode | ~220ms (~4.5 t/s) |
| MTP-only eval | ~10ms |
| 2-token verification batch | 560-816ms (3-4x single) |
| Draft acceptance | 70.3% |
| **Net throughput** | **0.56x baseline** |

### Root Cause
Same as tree speculation: 75% Delta Net recurrent layers process tokens sequentially regardless of batch size. 2-token batch = ~2x recurrent cost + attention overhead = 3-4x total.

### Conclusion
MTP-1 speculation is NOT VIABLE on hybrid recurrent models. Would be viable on dense attention-only models where multi-token batches have near-1x cost. All speculation approaches now exhausted for Qwen3.5-35B-A3B.

### Deep Dive Findings (`research/deep-dives/dflash-dart-diffusion-speculation.md`)

**Portability to llama.cpp**: YES for the drafter architecture (standard transformer, GGUF-convertible). Requires new `llama_get_hidden_state()` API for per-layer hidden state extraction. Block-sparse attention is the hardest component but causal fallback worth testing first. Estimated 13-20 days.
