### Problem

All Qwen3.5 dense-FFN models (0.8B, 2B, 4B, 9B, 27B) were labeled `architecture: dense` in the model registry since 2026-03-03. This was wrong — ALL Qwen3.5 variants (0.8B–397B) are hybrid Delta Net with 3:1 recurrent:attention ratio. The "dense" label means dense FFN (not MoE), not pure attention.

The mislabeling caused:

The term "dense" was used in two senses that got conflated:

The benchmark agent (2026-03-03) set labels based on pre-discovery understanding. The hybrid nature was confirmed on 2026-03-15 during tree speculation Phase 8, but subsequent agents worked around it (coining "dense hybrid") instead of fixing the source data.

## KV Cache Quantization Phase 0 — Benchmarked (Session 2026-03-25b)

### Objective
Benchmark existing llama.cpp quantized KV cache support (`-ctk`, `-ctv` flags) on EPYC 9655 with flash attention. No code changes — purely CLI configuration.

### Benchmark Matrix
- **Models**: Qwen3.5-35B-A3B Q4KM (hybrid, 25% attn), Qwen2.5-Coder-32B Q4KM (pure attn)
- **KV configs**: f16/f16 (baseline), q8_0/q8_0, q8_0/q4_0, q4_0/q4_0
- **Context lengths**: 4K, 16K, 65K
- **Server config**: numactl --interleave=all, 96 threads, -ub 8192, --flash-attn on

### Key Results

**Qwen3.5-35B-A3B (frontdoor, hybrid)**:
- KV quantization is **free** — all configs within noise (~14-15 t/s gen, ~150 t/s prompt)
- KV cache is tiny (25% attention layers): 80 MiB at 4K, 1280 MiB at 65K
- Prefill speed unaffected by quantization (~200 t/s at 4K, ~184 t/s at 16K)
- **Deploy q4_0/q4_0 — zero cost**

**Qwen2.5-Coder-32B (coder, pure attention)**:
- **Generation speed neutral**: ~8.5-9.5 t/s across all configs at all context lengths
- **Prefill REGRESSION**: q8_0 KV: 34 t/s vs 111 t/s f16 at 16K (3.3x slower!). q4_0: ~47 t/s (2.4x slower)
- **Memory savings at 65K**: f16 = 16.4 GB KV -> q4_q4 = 4.6 GB (saves 11.8 GB RSS)
- Short prompt processing (42 tok) unaffected — regression only at long prefill (>2K tokens)

### Prefill Regression Root Cause
The ggml CPU flash attention kernel (`ggml_flash_attn_ext`) has significant dequantization overhead during prompt processing. During generation (single token), dequant cost is negligible vs model weight I/O. During prefill (large batch), dequant becomes a bottleneck. This is specific to the CPU implementation — GPU reports don't show this pattern.

### Deployment Decision
| Model | Config | Rationale |
|-------|--------|-----------|
| Qwen3.5-35B-A3B | q4_0 K / q4_0 V | Zero throughput cost, free memory savings |
| Qwen2.5-Coder-32B | q8_0 K / q4_0 V | Best gen speed, acceptable prefill (coder prompts typically short) |

## KV Cache Quantization Phase 2: PolarQuant — Complete (Session 2026-03-25b cont.)

### Implementation
Added `GGML_TYPE_POLAR_Q4` (type 40) to ggml: polar coordinate quantization with random orthogonal preconditioning, recursive polar transform, and k-means codebook quantization. 50 bytes per 128 elements (5.12x compression, 3.1 bits/element).

### Bugs Fixed
1. NULL `from_float` in CPU traits → segfault in KV cache write (needed both base AND CPU trait tables)
2. Uninitialized KV cache NaN propagation (added norm==0 guard)
3. Thread-safety race in init (atomic spinlock)
4. Block size mismatch: d=64 models incompatible (only d=128+ works)

### Perplexity (10 chunks, Qwen2.5-7B)

|--------|-----|--------|-------------|
| f16 V | 7.166 | — | 1x |
| q4_0 V | 7.172 | +0.007 | 3.56x |
| polar_q4 V | 7.394 | +0.229 | 5.12x |

PPL gap (+0.229) is inherent to 3.1 bits — codebooks calibrated but 2-bit resolution on upper levels compounds error.

## KV Cache Quantization Phase 3: TurboQuant (QJL) — Complete (Session 2026-03-25b cont.)

### Implementation
Added `GGML_TYPE_TURBO_Q3` (type 41): Stage 1 PolarQuant (2 bits/angle, 4 levels, 32 bytes) + Stage 2 QJL (1-bit sign projections + residual norm, 24 bytes) = 56 bytes per 128 elements (3.5 bits).

### Key Finding: QJL Requires Custom Attention Kernel
QJL residual correction does NOT work via ggml's `to_float` dequant path. QJL preserves **inner products** (attention scores) via JL lemma, not individual vectors. Reconstructing vectors from sign bits adds noise (PPL 197K). A custom `ggml_flash_attn_ext` that reads QJL-packed V directly would be needed — future work.

### Final Comparison (Qwen2.5-7B, 10 chunks)

| polar_q4 V | 7.394 | +0.229 | 5.12x |
| turbo_q3 V (stage1) | 9.017 | +1.85 | 4.57x |

**Production recommendation unchanged**: Hadamard q4_0 (Phase 1) at 3.56x compression, +0.017 PPL on Coder-32B.

## KV Cache Quantization Phase 3b: QJL Attention Kernel — In Progress (Session 2026-03-25b cont.)

### Why Phase 3 Failed
Phase 3 implemented TurboQuant (PolarQuant + QJL) but the QJL residual correction doesn't work via ggml's `to_float` dequant path. QJL preserves **inner products** (attention scores) via the JL lemma, not individual vectors. Reconstructing vectors from sign bits adds noise (PPL 197K).

### What Phase 3b Does
Custom flash attention dispatch for QJL-packed K cache. Computes scores directly from sign bits via Goemans-Williamson/SimHash estimator: `score = ‖q‖·‖k‖·cos(π·(1 - agree/sketch_dim))`.

### Results
Kernel works mechanically (correct dispatch, no crashes). Score accuracy good for large dot products (ratio ~1.03), poor for small (5-10x error). PPL ~16K — unusable. Root cause: sign-bit estimator's constant variance dominates for near-orthogonal Q·K pairs (most of attention). Paper uses outlier correction (top-8 dims at full precision) which we didn't implement — adding outliers makes the block 96 bytes (6 bits/element), losing compression advantage over q8_0.

**Phase 3c update**: Switched to Gaussian JL matrix (paper's approach, was incorrectly using Rademacher ±1). Added outlier correction (top-8 key dims at f16, exact dot product). Large-score accuracy improved (ratio 1.14) but small-score noise still dominates (10-17x error). PPL 15K → still unusable. Confirmed: hybrid precision buffer (recent tokens at full precision) is the critical missing piece.
