## Qwen3.5 Architecture Correction — Complete (Session 2026-03-25a)

### Problem

All Qwen3.5 dense-FFN models (0.8B, 2B, 4B, 9B, 27B) were labeled `architecture: dense` in the model registry since 2026-03-03. This was wrong — ALL Qwen3.5 variants (0.8B–397B) are hybrid Delta Net with 3:1 recurrent:attention ratio. The "dense" label means dense FFN (not MoE), not pure attention.

The mislabeling caused:
- RESULTS.md showing stale/invalid spec decode speedups (e.g. 9B Q4KM "25.1 t/s" from lookup, 27B "13.6 t/s" from drafter)
- Benchmark executor generating spec decode + lookup configs for models where these are net-negative
- `orchestrator-stack-audit.md` claiming 27B is "spec decode compatible!"
- The `constraints.forbid` list only blocked `eagle`, not `speculative_decoding` or `prompt_lookup`

### Root Cause

The term "dense" was used in two senses that got conflated:
1. **Dense FFN** (not MoE) — correct for 9B/27B
2. **Dense attention** (not hybrid) — wrong, the `architecture` registry field has this meaning

The benchmark agent (2026-03-03) set labels based on pre-discovery understanding. The hybrid nature was confirmed on 2026-03-15 during tree speculation Phase 8, but subsequent agents worked around it (coining "dense hybrid") instead of fixing the source data.

## KV Cache Quantization Phase 0 — Benchmarked (Session 2026-03-25b)

### Objective
Benchmark existing llama.cpp quantized KV cache support (`-ctk`, `-ctv` flags) on EPYC 9655 with flash attention. No code changes — purely CLI configuration.

### Benchmark Matrix
- **Models**: Qwen3.5-35B-A3B Q4KM (hybrid, 25% attn), Qwen2.5-Coder-32B Q4KM (pure attn)
- **KV configs**: f16/f16 (baseline), q8_0/q8_0, q8_0/q4_0, q4_0/q4_0
- **Context lengths**: 4K, 16K, 65K
- **Server config**: numactl --interleave=all, 96 threads, -ub 8192, --flash-attn on

### Key Results

**Qwen3.5-35B-A3B (frontdoor, hybrid)**:
- KV quantization is **free** — all configs within noise (~14-15 t/s gen, ~150 t/s prompt)
- KV cache is tiny (25% attention layers): 80 MiB at 4K, 1280 MiB at 65K
- Prefill speed unaffected by quantization (~200 t/s at 4K, ~184 t/s at 16K)
- **Deploy q4_0/q4_0 — zero cost**

**Qwen2.5-Coder-32B (coder, pure attention)**:
- **Generation speed neutral**: ~8.5-9.5 t/s across all configs at all context lengths
- **Prefill REGRESSION**: q8_0 KV: 34 t/s vs 111 t/s f16 at 16K (3.3x slower!). q4_0: ~47 t/s (2.4x slower)
- **Memory savings at 65K**: f16 = 16.4 GB KV -> q4_q4 = 4.6 GB (saves 11.8 GB RSS)
- Short prompt processing (42 tok) unaffected — regression only at long prefill (>2K tokens)

### Prefill Regression Root Cause
The ggml CPU flash attention kernel (`ggml_flash_attn_ext`) has significant dequantization overhead during prompt processing. During generation (single token), dequant cost is negligible vs model weight I/O. During prefill (large batch), dequant becomes a bottleneck. This is specific to the CPU implementation — GPU reports don't show this pattern.

### Deployment Decision
| Model | Config | Rationale |
|-------|--------|-----------|
| Qwen3.5-35B-A3B | q4_0 K / q4_0 V | Zero throughput cost, free memory savings |
| Qwen2.5-Coder-32B | q8_0 K / q4_0 V | Best gen speed, acceptable prefill (coder prompts typically short) |

### Key Design Decisions

1. **Autoresearch is model-agnostic**: Framework tests any registry model. New models become candidates automatically.
2. **Tier assignment is empirical**: HOT vs WARM vs COLD determined by autoresearch, not hardcoded. ~540 GB headroom enables all-HOT but may not be optimal.

4. **Stack simplification is a hypothesis**: Whether fewer models with more instances outperforms many specialists is an open question for autoresearch.
5. **Git-based ratchet**: Every improvement committed, degradations reverted. Best config always recoverable.
