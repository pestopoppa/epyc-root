## REPL Token Cap Fix (768 ‚Üí 5000)

### Problem: REPL Restart Loops on Complex Questions

Observed on `gpqa_Chemistry_0423`: model sets up the problem, starts writing reasoning/code, then the REPL tap suddenly resets. Each turn restarts from scratch instead of continuing ‚Äî 6 identical turns, each generating exactly 768 tokens of prose, never completing code.

### Root Cause

Two token caps in `src/graph/helpers.py` defaulted to 768:
- `_repl_turn_token_cap()` ‚Üí `ORCHESTRATOR_REPL_TURN_N_TOKENS` (applied when `state.tool_required` and no explicit `n_tokens`)
- `_frontdoor_repl_non_tool_token_cap()` ‚Üí `ORCHESTRATOR_FRONTDOOR_REPL_NON_TOOL_N_TOKENS` (applied to frontdoor when `tool_required=False`)

768 tokens is far too short for complex reasoning. The model writes prose reasoning, gets truncated mid-expression, the REPL sees no executable code (`extract_code_from_response` finds nothing), and the next turn starts fresh with the same prompt.

### Fix

Raised both defaults from 768 ‚Üí 5000. Applies to all models through the REPL graph execution path.

**File:** `epyc-orchestrator/src/graph/helpers.py`

## Tool Permission Fix (frontdoor web_search PermissionError)

### Problem

`PermissionError: Role 'frontdoor' cannot use tool 'web_search'` in REPL ‚Äî model tried to call `CALL("web_search", ...)` and was denied.

### Root Cause (3-layer failure)

1. **`features()` singleton** calls `get_features()` without `production=True` ‚Äî defaults to test mode
2. **`orchestrator_stack.py`** sets many `ORCHESTRATOR_*` env vars but was missing `ORCHESTRATOR_CASCADING_TOOL_POLICY=1`
3. **Legacy permission path** (`cascading_tool_policy=False`): denies access when role has no `tool_permissions` in `model_registry.yaml`. No role has `tool_permissions` defined ‚Äî so ALL roles are denied ALL tools via the legacy path.

With cascading policy enabled: empty policy chain ‚Üí `resolve_policy_chain` starts with `all_tools` and applies no narrowing ‚Üí everything allowed. This is correct for frontdoor.

### Fix

1. **Persistent**: Added `env["ORCHESTRATOR_CASCADING_TOOL_POLICY"] = "1"` to `orchestrator_stack.py` startup env vars
2. **Immediate**: Enabled via live `POST /config` endpoint on running API

**File:** `epyc-orchestrator/scripts/server/orchestrator_stack.py`

### Architectural Insight: features() Production Mode Gap

`get_features()` has a `production` parameter that enables all validated features, but the `features()` singleton never passes `production=True`. Instead, `orchestrator_stack.py` manually sets individual `ORCHESTRATOR_*` env vars. This creates a maintenance gap ‚Äî new features validated in the production defaults dict (`cascading_tool_policy`, `side_effect_tracking`, `structured_tool_output`, etc.) won't be active unless manually added to `orchestrator_stack.py`.

## Seeding Stall Diagnosis: Circuit Breaker Cascade

Two seeding stalls (0 tokens, empty RESPONSE) traced to cascade failure:
1. API started without `CASCADING_TOOL_POLICY` ‚Üí tool calls denied
2. Failures accumulated ‚Üí circuit breakers opened on port 8080
3. Subsequent requests fast-failed by circuit breaker ‚Üí 0 tokens
4. `CancelledError` crash killed API entirely

Also confirmed `ChatRequest.mock_mode` defaults to `True` ‚Äî seeding correctly sends `real_mode: True` to override.

## New Tool: `web_research` (Deep Web Search with Worker Synthesis)

**Problem:** `web_search` only returns DuckDuckGo snippets (title/URL/snippet). Model gets no actual page content to reason with.

**Solution:** New `web_research` compound tool that any model can invoke:
1. Calls `web_search()` to get top URLs
2. Fetches top N pages in parallel via `ThreadPoolExecutor`
3. Sends each page to explore worker (port 8082, Qwen2.5-7B) for query-focused synthesis
4. Returns dense summaries per source

**Files:**
- `src/tools/web/research.py` ‚Äî new compound tool
- `src/tools/web/__init__.py` ‚Äî registration

- `orchestration/prompts/root_lm_system.md` ‚Äî updated guidance
- `orchestration/prompts/rules.md` ‚Äî updated examples

**Note:** DuckDuckGo HTML scraper currently triggers bot detection (CAPTCHA). Tool pipeline is wired and will work once search backend returns results.

## Incremental Code Fix: Solution File Persistence

**Problem:** On REPL error, model rewrites code from scratch every turn because:
1. Its previous code is never shown back to it
2. Error feedback just says "Fix the error and try again."
3. On escalation, the next role has no access to the previous role's code

**Solution:** Auto-persist code to `/mnt/raid0/llm/tmp/{task_id}_solution.py` each REPL turn.
