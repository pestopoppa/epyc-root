## REPL Token Cap Fix (768 → 5000)

### Problem: REPL Restart Loops on Complex Questions

Observed on `gpqa_Chemistry_0423`: model sets up the problem, starts writing reasoning/code, then the REPL tap suddenly resets. Each turn restarts from scratch instead of continuing — 6 identical turns, each generating exactly 768 tokens of prose, never completing code.

### Root Cause

Two token caps in `src/graph/helpers.py` defaulted to 768:
- `_repl_turn_token_cap()` → `ORCHESTRATOR_REPL_TURN_N_TOKENS` (applied when `state.tool_required` and no explicit `n_tokens`)
- `_frontdoor_repl_non_tool_token_cap()` → `ORCHESTRATOR_FRONTDOOR_REPL_NON_TOOL_N_TOKENS` (applied to frontdoor when `tool_required=False`)

768 tokens is far too short for complex reasoning. The model writes prose reasoning, gets truncated mid-expression, the REPL sees no executable code (`extract_code_from_response` finds nothing), and the next turn starts fresh with the same prompt.

### Fix

Raised both defaults from 768 → 5000. Applies to all models through the REPL graph execution path.

**File:** `epyc-orchestrator/src/graph/helpers.py`

## Tool Permission Fix (frontdoor web_search PermissionError)

### Problem

`PermissionError: Role 'frontdoor' cannot use tool 'web_search'` in REPL — model tried to call `CALL("web_search", ...)` and was denied.

### Root Cause (3-layer failure)

1. **`features()` singleton** calls `get_features()` without `production=True` — defaults to test mode
2. **`orchestrator_stack.py`** sets many `ORCHESTRATOR_*` env vars but was missing `ORCHESTRATOR_CASCADING_TOOL_POLICY=1`
3. **Legacy permission path** (`cascading_tool_policy=False`): denies access when role has no `tool_permissions` in `model_registry.yaml`. No role has `tool_permissions` defined — so ALL roles are denied ALL tools via the legacy path.

With cascading policy enabled: empty policy chain → `resolve_policy_chain` starts with `all_tools` and applies no narrowing → everything allowed. This is correct for frontdoor.

### Fix

1. **Persistent**: Added `env["ORCHESTRATOR_CASCADING_TOOL_POLICY"] = "1"` to `orchestrator_stack.py` startup env vars
2. **Immediate**: Enabled via live `POST /config` endpoint on running API

**File:** `epyc-orchestrator/scripts/server/orchestrator_stack.py`

### Architectural Insight: features() Production Mode Gap

`get_features()` has a `production` parameter that enables all validated features, but the `features()` singleton never passes `production=True`. Instead, `orchestrator_stack.py` manually sets individual `ORCHESTRATOR_*` env vars. This creates a maintenance gap — new features validated in the production defaults dict (`cascading_tool_policy`, `side_effect_tracking`, `structured_tool_output`, etc.) won't be active unless manually added to `orchestrator_stack.py`.

## REPL Session Log — Processing Journal Across Turns

**Problem:** Each REPL turn builds a fresh prompt with only `last_output`/`last_error` from the previous turn. All earlier turns are invisible. Models restart from scratch, repeat failed approaches, and lose context across escalation boundaries.

**Solution:** Append-only session log (`session_{task_id}.md`) captures every turn as a `TurnRecord` (code hash, outcome, tool calls, error/output previews). Worker-generated summary injected into prompt every 2 turns.

**Key features:**
- Anti-loop detection via code hash deduplication in summary

- Feature-flagged for rollback
- Thread-safe, fail-silent (follows repl_tap.py pattern)
- Session log travels with TaskState across escalation/delegation

## WS1: SSM State Checkpointing for Speculative Decoding

### Context

Qwen3.5-35B-A3B is a hybrid SSM+MoE model (75% Mamba layers, 25% attention). Current llama.cpp can't do speculation with SSM models because `seq_rm` on recurrent memory fails for partial ranges — rejected draft tokens corrupt the recurrent state which encodes cumulative history.

### Implementation (on `llama.cpp-experimental` branch)

**Approach**: Checkpoint the full recurrent state (~63 MB for Qwen3.5-35B-A3B) before speculation, restore on rejection, then re-advance through accepted tokens.

**Files modified (8 files, +312 lines)**:

| File | Change |
|------|--------|
| `src/llama-memory-recurrent.h` | Added `llama_memory_recurrent_checkpoint` struct + `checkpoint()`/`restore()` method declarations |
| `src/llama-memory-recurrent.cpp` | Implemented `checkpoint()` (saves cell metadata + tensor data via `ggml_backend_tensor_get`) and `restore()` (restores via `ggml_backend_tensor_set`) |
| `src/llama-memory-hybrid.h` | Added `checkpoint_recurrent()`/`restore_recurrent()` to hybrid memory |
| `src/llama-memory-hybrid.cpp` | Delegating implementation + **changed `seq_rm`** to not bail early when recurrent partial removal fails (allows KV cache cleanup to proceed) |
| `include/llama.h` | Added C API: `llama_memory_has_recurrent()`, `llama_memory_checkpoint_save/restore/free()` |
| `src/llama-context.cpp` | Implemented C API functions with `dynamic_cast` to detect hybrid/recurrent memory |
| `common/speculative.cpp` | Modified `common_speculative_is_compat()` to allow hybrid models when recurrent checkpointing is available |

## Session Scratchpad Memory — IMPLEMENTED

Implemented model-extracted semantic insights for the session log. The existing `worker_fast` summary call (every 2 turns) now also extracts structured `INSIGHT|category|text` lines — zero additional inference calls.

### Design
- `ScratchpadEntry` dataclass with 5 categories: `bug_location`, `approach_eliminated`, `constraint_discovered`, `user_intent`, `dependency_found`
- Category-based pruning: newer entry in same category supersedes older, max 8 entries
- `[Key Insights]` block prepended before `[Session History]` in prompt injection
- Entries passed through `EscalationContext` and injected into escalation prompts as `## Previous Insights`
- Feature flag: `session_scratchpad` (production=True, test=False, env=`ORCHESTRATOR_SESSION_SCRATCHPAD`)

## Fast-RLM Budget Controls — IMPLEMENTED

Implemented two new budget controls from the Fast-RLM paper:

### Worker Call Budget
- `state.repl_executions` tracks total REPL execute() calls per task
- Default cap: 30 (env: `ORCHESTRATOR_WORKER_CALL_BUDGET_CAP`)
- Feature flag: `worker_call_budget` (env: `ORCHESTRATOR_WORKER_CALL_BUDGET`)
- Pressure warning at ≤3 remaining calls

### Per-Task Token Budget
- `state.aggregate_tokens` tracks cumulative completion tokens from `_last_inference_meta`
- Default cap: 200K (env: `ORCHESTRATOR_TASK_TOKEN_BUDGET_CAP`)
- Feature flag: `task_token_budget` (env: `ORCHESTRATOR_TASK_TOKEN_BUDGET`)
- Pressure warning at <15% remaining

### Design
- Budget checked before `_execute_turn()` in all 7 node types (saves wasted LLM call)
- `_rescue_from_last_output()` attempts graceful termination before hard fail
- Both flags enabled in production via `orchestrator_stack.py`, disabled in tests
- Step 1 (escalation depth) confirmed already implemented (`max_escalations=2`, `detect_role_cycle()`, `max_turns=15`)
- Step 4 (REPL truncation benchmark) deferred to seeding runs
