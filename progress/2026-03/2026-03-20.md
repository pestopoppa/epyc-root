### Deep Dive: ColBERT vs Our Retrieval Architecture

Full analysis of Reason-ModernColBERT (150M ColBERT, MaxSim late-interaction) against our TwoPhaseRetriever (BGE-large 1024-dim dense vectors + FAISS IndexFlatIP + Q-value reranking).

**Initial hypothesis**: ColBERT's +7.3 NDCG@10 late-interaction advantage over dense retrieval could improve our episodic memory retrieval.

**Conclusion: NOT applicable to episodic memory.** Five reasons:
1. **Scale mismatch**: BRIGHT benchmark tests corpus-scale retrieval (millions of docs). Our store has 500–2000 memories. FAISS flat IP is <1ms at this scale.
2. **Query format mismatch**: Our queries are structured serializations (`"type:coding | objective:implement sort | priority:high"`), not natural-language reasoning. BRIGHT tests natural reasoning queries.
3. **Q-value dominance**: Phase 2 Q-value ranking has 0.7 weight vs 0.3 for similarity. Improving Phase 1 similarity has diminishing returns.
4. **Storage overhead**: ColBERT stores 128-dim per token × ~50 tokens = 6400 floats vs 1024 for dense. 6x storage for marginal benefit.
5. **Latency**: MaxSim requires all-pairs token comparison. Not problematic at 2000 memories but unnecessary given FAISS <1ms.

### Revised Recommendations (post deep-dive)

**Recommendation 1 — Hard-negative training data for routing classifier MLP**: ReasonIR's synthetic hard-negative methodology ("plausibly related but ultimately unhelpful" documents) maps to a known routing failure mode. `extract_training_data.py` currently generates pairs from Q-values with no explicit hard negatives. Proposed: for each (task, best_role), find nearest-neighbor task where a *different* role had highest Q-value. ~2-day experiment in `structural_lab.py`.

**Recommendation 2 — ColBERT reranker for web_research**: 150M ColBERT reranker is 50x cheaper than 7B models, runs on CPU in ~5ms for 10-20 pages, doesn't compete for llama-server inference slots. Late-interaction advantage is largest on reasoning-heavy queries — exactly the domain where web_research matters most. Integration: rerank fetched pages before passing to explore worker synthesis.

### Summary

BackLite is the first algorithm to exploit dynamic attention sparsity for backward-pass acceleration during training. It tracks which attention matrix tiles are negligible during the forward pass (via per-tile LSE statistics) and masks them during backpropagation, reducing backward-pass FLOPS while maintaining gradient quality (cosine similarity >0.99 at >50% sparsity).

**nanochat validation**: Integrated into karpathy's nanochat as highest-priority attention backend. Achieved 2.10% wall-clock speedup (1.76h vs 1.80h) on GPT-2 training, with CORE score still above threshold (0.2586 vs 0.2565). PR is CLOSED (labeled "leaderboard candidate").

**LiteAttention** (arXiv:2511.11062): Foundational work by same authors. Inference-only sparse attention for video diffusion transformers using temporal sparsity coherence across denoising steps.

### Why Not Applicable

- Requires H100/H200 GPUs with FlashAttention 3 — EPYC has no training-capable GPUs
- Training-time optimization; EPYC stack is CPU inference on AMD EPYC 9655
- The forward-pass sparsity detection concept is elegant but has no CPU inference analogue
- Cross-ref to doc-to-lora handoff is tangential (llama.cpp CPU fine-tuning, not FA3)
