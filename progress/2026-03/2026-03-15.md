## Tree Speculation Phase 4 — f16 Target Validation (CONFIRMED)

Benchmarked Qwen2.5-Coder-32B **f16** + 0.5B-f16 drafter (pair 10, added to `bench_tree_speculation_server.sh`).

### Results

| n_predict | Linear | Best Tree | Delta |
|-----------|--------|-----------|-------|
| 256 | 5.01 t/s (80.1% accept) | 5.80 t/s (p=0.3) | **+15.8%** |
| 512 | 5.75 t/s (81.0% accept) | 6.36 t/s (p=0.05) | **+10.6%** |

Biggest tree win yet. All tree p_split configs (0.05–0.3) tightly clustered — tree finds the same useful alt paths regardless of threshold. +47 extra accepted tokens (256-run) translate almost entirely to throughput because f16 verification scaling is near-flat (1.69x at N=64).

### Why f16 Wins Decisively

Tree overhead (~41ms) is **draft-side only** (KV copies + sampler clones on draft model), independent of target. Two factors determine tree viability:

1. **Target verification latency**: 32B f16 ≈ 200ms/round → 41ms is ~20% tax. Q4 32B ≈ 50ms/round → 41ms is ~80% tax.
2. **Verification cost scaling with N candidates**: f16 is near-flat (bandwidth-bound — 62GB weight loading dominates). Q4_K_M scales 4-5x at N=64 (per-token dequant overhead from 4-bit → f16 unpacking).

Both conditions favor f16. Key insight: f16 isn't special — any target where `round_latency >> 41ms` AND `scaling ≈ flat` benefits. Q8_0 has simpler dequant (VNNI int8 dot product, one f32 scale per block) — predicted ~2-2.5x scaling. Q8 benchmark in progress to validate.

## Tree Speculation Handoff — Phase 8 Revised (Delta Net, Not STree)

Added Phase 8 for hybrid model tree speculation. **Critical architecture correction**: Qwen3.5/Qwen3-Next use **Delta Net** (gated linear attention, `delta-net-base.cpp`), NOT Mamba2 (`ggml_ssm_scan`). STree's log-space tree mask trick doesn't apply — Delta Net has a nonlinear state dependency (`s_old^T k` in the delta term) that prevents conversion to tree-masked cumulative sum.

Two approaches designed:
- **Approach A** (primary): Sequential state replay — checkpoint/restore recurrent state per tree path, replay autoregressive Delta Net. Exact, ~40-60ms overhead for 8 paths.
- **Approach B** (stretch): Linearized Delta Net — freeze `s_old^T k` at base state, apply tree-masked cumulative decay. Single forward pass but approximate; ~40% viability risk.

STree reference preserved for true Mamba2 models (Jamba, Falcon-H1, PLaMo2/3 — not in our production stack). Updated validation checklist, dependency graph, files summary, and complete research references (10 papers).

### Root Cause

Frozen recurrent state means tree paths diverge only via attention layers (~25% of layers in Qwen3.5). Attention-only path diversity is insufficient — acceptance rate drops 12-22pp across all models/quants. Combined with ~41ms tree construction overhead, throughput collapses -53% to -62%.

**Key finding**: Unlike dense models where Q4→Q8→f16 progression improves tree viability, hybrid models show no quant sensitivity. The 35B Q8 (35GB, slower per-round) is actually worse than Q4 (19GB) because the acceptance rate collapse dominates any reduction in relative overhead.

`!has_recurrent` guard restored with updated comment documenting empirical results. Two improvements retained for dense models:
- `kv_unified` auto-enable (dense models no longer need explicit `--kv-unified` flag)
- `n_seq_max` auto-bump (convenience)

## Phase 8 Approach A: Per-Path Sequential Recurrent Replay — IMPLEMENTED

Implemented Approach A for hybrid tree speculation on Delta Net models. Instead of frozen multi-path (which collapsed acceptance 12-22pp), each tree path is verified with exact recurrent state via checkpoint/restore + sequential decode.

### Changes (7 sites in `tools/server/server-context.cpp`, 170 lines net)

1. **New field**: `std::vector<llama_tokens> tree_replay_paths` on `server_slot` — stores path tokens for hybrid replay
2. **Checkpoint before freeze**: `llama_memory_checkpoint_save()` before `llama_set_freeze_recurrent(ctx, true)` — captures exact pre-speculation recurrent state
3. **Tree guard opened**: Removed `!has_recurrent` from tree entry condition — hybrid models now enter tree block
4. **Hybrid branch**: When `has_recurrent`, store path tokens in `tree_replay_paths` instead of adding to shared batch. Attention KV still forked via `seq_cp`.
5. **Per-path replay**: During verification, for each path: restore checkpoint → unfreeze → build mini-batch (sampled + draft tokens) → decode → sample acceptance → re-freeze. Best path wins.
6. **KV swap**: If alt path wins, swap attention KV (same as dense tree code).
7. **Post-verification**: Restore checkpoint before re-advancing accepted tokens through recurrent state.

Dense tree path completely unchanged — only hybrid models use per-path replay.

### Build

Zero warnings. `cmake --build build -j$(nproc) --target llama-server` clean.

- **Acceptance rate**: Recovery to ~50-65% (from 28-33% frozen)
- **Replay cost**: ~5-15ms per path × 8 paths = ~40-120ms total
- **Net throughput**: TBD — depends on acceptance gain vs replay cost
- **Gate 1**: Verification overhead must be <100ms/round
- **Gate 2**: Acceptance rate must exceed 55% for net positive

## Phase 8 Approach C: Checkpoint + Clone Cell for Hybrid Tree (2026-03-16 session)

Implemented and benchmarked the third approach to tree speculation on hybrid (Delta Net) models.

### Implementation

New APIs and changes across 3 files:
- **`llama_memory_recurrent_clone_cell()`** (`src/llama-context.cpp`, `include/llama.h`): Clones a recurrent cell from one seq_id to another with independent tensor data. Uses CPU staging buffer for cross-cell tensor copy.
- **`llama_memory_checkpoint_save/restore()`**: Extended checkpoint system (already existed from Phase 8 Approach A) with proper save/restore cycle per speculation round.
- **Batch allocator fix** (`src/llama-context.cpp`): Constrained `n_seq_max` to `cparams.n_seq_max` when model has recurrent memory + kv_unified, preventing `LLAMA_MAX_SEQ=256` from exhausting the 9-cell recurrent cache.
- **Server integration** (`tools/server/server-context.cpp`): Tree → checkpoint save → seq_cp + clone_cell per alt path → batched exact-logit decode → acceptance sampling → checkpoint restore → re-advance accepted tokens.
- **M-RoPE fix**: Re-advance must remove ALL speculation positions (accepted + rejected) from KV before re-inserting, because M-RoPE validates `X < Y` on positions.

### Critical Bug: clone_cell Tail Pointer Corruption

The `cells[]` array in recurrent memory serves dual purpose: `cells[i]` is a physical cell AND `cells[i].tail` is sequence metadata for seq_id `i`. When `clone_cell` allocated physical cell 2 for seq 3, the line `cells[dst_cell].tail = -1` overwrote `cells[2].tail` which was the tail pointer for seq 2 (pointing to cell 1). This corrupted seq 2's metadata, causing find_slot to fail on subsequent tree verification rounds. Fix: don't set `cells[dst_cell].tail` — only set `cells[dst_seq_id].tail = dst_cell`.

### Conclusion

All three approaches to exact-state tree speculation on hybrids are now ruled out:
- **Approach 0** (frozen): -53% to -62% from acceptance degradation
- **Approach A** (per-path replay): -60% to -66% from O(N_paths × full_forward)
- **Approach C** (checkpoint + clone): -60% from checkpoint/clone/batched overhead

The fundamental problem: maintaining exact recurrent state per path requires O(N_paths) compute through the recurrent layers, and for hybrid models where recurrent layers are 75% of the model, this overhead exceeds any tree branching benefit.

Only Approach B (linearized Delta Net, ~40% viability, approximate) remains. Production config for hybrid models stays at linear speculation + freeze-recurrent.
