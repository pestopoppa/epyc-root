# Progress Report ‚Äî 2026-03-05

## Inference Lock Starvation ‚Äî Root Cause Found and Fix Validated

### Root Cause

`PrefixRouter(num_slots=4)` assigned `id_slot` values 2-3 to requests, but llama-server runs with `-np 2` (slots 0-1 only). Requesting non-existent slots causes llama-server to hang waiting indefinitely, holding the inference lock and blocking all subsequent requests.

### Evidence

Perfectly periodic 2-OK, 2-HANG pattern confirmed via `id_slot` logging:
- Slots 0,1: headers received in 50-116ms, inference completes normally
- Slots 2,3: no headers ever received, lock held for full timeout (59-84s)

### Fix Applied

Changed `num_slots` default from 4 to 2 across all config layers:
1. `src/config/__init__.py` ‚Äî ServerSettings class default
2. `src/config/__init__.py` ‚Äî config factory env var default
3. `src/config/models.py` ‚Äî ServerConfigData default
4. `src/llm_primitives/primitives.py` ‚Äî LLMPrimitives constructor default
5. `src/api/routes/chat_pipeline/routing.py` ‚Äî explicit `num_slots` from config

### Validation Results

- **G7** (60s timeout, 10 questions): 9/10 OK (1 legitimate timeout from 85s generation)
- **G8** (120s timeout, 10 questions): **10/10 OK**, avg 22.9s, zero hangs
- Baseline pre-fix: 6/10 OK with periodic 2-OK, 2-HANG pattern

### Isolation Testing Methodology

14 test variants with 200+ total requests ruled out all other hypotheses:
- Direct llama-server (batch + streaming): 40/40 OK
- Client lifecycle patterns: 50/50 OK
- asyncio.to_thread + flock interaction: 50/50 OK
- Exact orchestrator payloads direct: 25/25 OK
- Single-worker orchestrator: reproduced hang ‚Üí not multiprocessing

### Defense-in-Depth Mitigations Retained

1. Lock hold watchdog (daemon thread, force-release after 130s)
2. Streaming fallback for all inference paths
3. Cancel check in streaming callback
4. Tighter httpx timeouts (read capped at 120s, pool wait 30s)
5. Dead code removal (`llm_call_stream()` generator-in-lock anti-pattern)

### Cleanup Completed

- Reverted ephemeral httpx.Client experiment
- Reverted WARNING-level diagnostic logs to INFO
- Re-enabled keep-alive connections (root cause was slot mismatch, not connection reuse)
- Retained INFO-level `id_slot` logging for ongoing monitoring

### Key Fixes

1. **Compound config gap**: MoE models previously only got compound configs (spec+MoE, lookup+MoE) at moe4. Now generates at every expert count including baseline ‚Äî e.g. moe4+spec, moe6+spec, baseline+spec.
2. **New `moe_spec_lookup` config type**: Triple compound (MoE override + draft model + lookup) ‚Äî server-mode only.
3. **`--speed-questions N`**: Runs acceleration configs on N slowest baseline questions instead of a fixed prompt, giving realistic speed measurements.
4. **Temperature overrides**: Per-suite temperatures from Qwen3.5 model cards (0.6 for coder, 1.0 for thinking/math, etc.).
