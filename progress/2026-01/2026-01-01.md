## 1. Benchmark Error Reporting Improvements

### Problem
Benchmark errors were displaying unhelpful messages like:
```
[ERROR] general_glm_4_6/lookup_n3/t3_q1: llama_model_loader: - kv  36: tokenizer.ggml.model str
```

This was caused by:
- `raw_output` containing both stdout and stderr concatenated
- When model fails, stderr (containing build banner) was shown as "error"

### Solution

**`scripts/lib/executor.py`:**
- Added `stderr: str = ""` field to `InferenceResult` dataclass
- Separated stdout (model output) from stderr (logs/errors)
- Extract timing info from stderr and append to raw_output for parsing
- Capture partial output on timeout

**`scripts/benchmark/run_benchmark.py`:**
- Updated error extraction to search from end of stderr (errors usually last)
- Skip common log prefixes: `build:`, `main:`, `llama_model_loader:`, `print_info:`, `load_`
- Only match real error patterns: `error:`, `error `, `failed`, `fatal`, `abort`, `segfault`, `exception`

### Result
```
# Before:
[ERROR] .../t3_q1: llama_model_loader: - kv  36: tokenizer.ggml.model str

# After:
[ERROR] .../t3_q1: exit=1
# or actual error message if present:
[ERROR] .../t3_q1: ggml_cuda_op_mul_mat failed
```

## 2. llama.cpp PR #18239 CI Fix

### PR: ggml-cpu: parallelize tensor repacking with OpenMP
**URL:** https://github.com/ggml-org/llama.cpp/pull/18239

### CI Failures Investigated

| Job | Error | Root Cause |
|-----|-------|------------|
| `ubuntu-cmake-sanitizer-riscv64-native (THREAD)` | Runner lost communication | Infrastructure issue (self-hosted runner), has `continue-on-error: true` |
| `ubuntu-latest-cmake-sanitizer (THREAD)` | `error: ignoring '#pragma omp parallel' [-Werror=unknown-pragmas]` | OpenMP pragmas without guards + `FATAL_WARNINGS=ON` |

### Fix Applied

Wrapped all OpenMP pragmas with `#ifdef GGML_USE_OPENMP` guards (matching existing codebase pattern in `ggml-cpu.c`):

```cpp
#ifdef GGML_USE_OPENMP
    #pragma omp parallel for
#endif
    for (int bg = 0; bg < n_row_groups; bg++) {
```

### Verification
- Builds successfully with `-DGGML_OPENMP=OFF -DLLAMA_FATAL_WARNINGS=ON`
- Builds successfully with `-DGGML_OPENMP=ON`

## Key Findings

### OpenMP Guard Pattern in llama.cpp
- Codebase uses `GGML_USE_OPENMP` (CMake-defined), NOT `_OPENMP` (compiler-defined)
- This is consistent across `ggml-cpu.c`, `ggml-blas.cpp`, `amx/common.h`
- `#ifdef GGML_USE_OPENMP` is the standard, idiomatic approach

### THREAD Sanitizer Build
- Uses `-DGGML_OPENMP=OFF` specifically to avoid TSan false positives with OpenMP
- Combines with `-DLLAMA_FATAL_WARNINGS=ON` which turns warnings into errors
- This configuration is intentional for clean TSan testing
