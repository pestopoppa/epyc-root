## Bug 1: Batch Initialization

### Problem

```cpp
// lookup.cpp line 109 - params.n_ctx defaults to 0 since Dec 2025
llama_batch batch_tgt = llama_batch_init(params.n_ctx, 0, 1);
```

PR #16653 changed `n_ctx` default to 0 (sentinel for GPU auto-fitting). Examples using raw `params.n_ctx` create zero-sized batches.

### Fix (PR #18729)

```cpp
- llama_batch batch_tgt = llama_batch_init(params.n_ctx, 0, 1);
+ llama_batch batch_tgt = llama_batch_init(llama_n_ctx(ctx), 0, 1);
```

Affects: `lookup.cpp:109`, `lookahead.cpp:121`

---

## Bug 2: Lookahead n_parallel

### Problem

After fixing batch init, lookahead still crashes:
```
llama_batch_allocr: error: invalid seq_id[0][1] = 1 >= 1
```

Lookahead needs 31 sequences (W + G + 1) for Jacobi iteration, but `n_seq_max` defaults to 1 (from `params.n_parallel`).

### Fix (PR #18730)

```cpp
// Add after W, N, G definitions
params.n_parallel = W + G + 1;
params.kv_unified = true;
```

## Server Mode for Speculative Decoding (Session 2)

### Problem

`moe_spec` speed tests for Qwen3-Coder-480B (271GB) were timing out at 120s. Root cause: subprocess mode reloads the entire model for each K value test.

### Key Discovery

llama-server supports speculative decoding with **per-request K values**:

| Parameter | Scope | Restart Needed? |
|-----------|-------|-----------------|
| Draft model (`-md`) | Startup | Yes (but target in page cache) |
| K value (`speculative.n_max`) | **Per-request** | **No** |

### Solution

Extended ServerManager to support draft models:

1. **`scripts/lib/executor.py`**
   - `ServerManager.start()`: Added `draft_model_path` and `draft_max` params
   - `ServerManager.run_inference()`: Added `speculative_n_max` param (per-request K)

2. **`scripts/benchmark/run_benchmark.py`**
   - Added `current_server_draft` state tracking
   - Draft restart logic: restarts server only when draft model changes
   - Updated `use_server` decision to include `"spec", "moe_spec"` types
   - Speed tests now use server mode when draft is loaded

### New Benchmark Flow

For architect_coding (Qwen3-Coder-480B + jukofyork draft):

| Step | Config | Server State | Restart? |
|------|--------|--------------|----------|
| 1 | baseline | target only | Initial start |
| 2 | moe4 | target + 4 experts | Yes (MoE change) |
| 3 | moe4_spec_k8 | target + draft | Yes (add draft) |
| 4 | moe4_spec_k16 | target + draft | **No** (K per-request) |
| 5 | moe4_spec_k24 | target + draft | **No** (K per-request) |

**Result:** 271GB model stays loaded. Only ~1GB draft model added once. Different K values tested via HTTP without restarts.

### Files Modified

| File | Changes |
|------|---------|
| `scripts/lib/executor.py` | ServerManager draft support, per-request K |
| `scripts/benchmark/run_benchmark.py` | Draft restart logic, server mode for spec decode |

### Also Added: Dynamic Subprocess Timeout (Fallback)

For cases where server mode can't be used (VL models, etc.):
- `timeout=max(180, int(size_gb * 3) + 120)` for subprocess speed tests
- Matches server startup timeout formula

### Memory Savings Implementation (COMPLETED)

After implementing dynamic block allocation, added **KV cache memory reduction**:

#### New Environment Variable: `LLAMA_PAGED_ATTN_MAX_BLOCKS`

When both `LLAMA_PAGED_ATTN` and `LLAMA_PAGED_ATTN_MAX_BLOCKS` are set:
- KV cache is reduced to `(max_blocks × block_size)` tokens
- Memory savings can exceed 80% for large context models

#### Test Results (Qwen3-1.7B)

| Configuration | KV Buffer | Cells | Memory Savings |
|---------------|-----------|-------|----------------|
| Baseline | 4480 MiB | 40960 | - |
| **100 blocks × 64** | **700 MiB** | **6400** | **84.4%** |

Total host memory: 6524 MiB → 2748 MiB (58% overall reduction)

**Performance**: 66.66 t/s (same as baseline) with coherent output.
