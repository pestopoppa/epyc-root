## Session 3: AVX-512 GELU Exploration (2026-01-09)

### Exploration

Investigated whether GELU/RoPE vectorization could provide performance gains.

| Target | Attempt | Result |
|--------|---------|--------|
| GELU (standard) | Implemented tanh-based vectorization | ❌ Segfaults - numerical edge cases |
| GELU (quick) | Vectorized sigmoid approximation | ✅ Works but only used by CLIP |
| RoPE | Not attempted | <1% of inference time |

### Key Finding

**Micro-optimizations not worth pursuing.** Inference time breakdown:
- Matrix multiplications: **90%+** (already optimized)
- Activations (GELU, SiLU, RoPE): **<1%** each

Even 2x speedup on activations = <1% overall improvement.

### Conclusion

Algorithmic improvements (spec decode 11x, MoE reduction 48%, prompt lookup 12x) >> micro-optimizations.

GELU changes reverted. Moving on.

## Session 4: Gemma-3 SWA + Speculative Decoding Investigation (2026-01-09)

### Problem Statement

Handoff reported Gemma-3 models crash with speculative decoding due to SWA (Sliding Window Attention) incompatibility.

### Investigation Results

| Test | Result |
|------|--------|
| Upstream master (b7684) + Gemma-3-27B + 1B draft | ✅ **WORKS** (20.8% acceptance) |
| MTP branch (b7508) + Gemma-3-27B + 1B draft | ❌ Crashed (before fix) |
| MTP branch (fixed) + Gemma-3-27B + 1B draft | ✅ **WORKS** (66.7% acceptance) |

### Root Cause

**The bug was MTP-branch specific, NOT upstream.**

The MTP branch added code that incorrectly casts `llama_kv_cache_iswa_context*` to `llama_kv_cache_context*`:
```cpp
// MTP branch code (problematic):
kvd->last_main_model_sinfos =
    static_cast<llama_kv_cache_context*>(mctx.get())->get_sinfos();
// This code does NOT exist in upstream master
```

### Fix Applied (MTP Branch Only)

Three commits on `mtp-branch`:
```
9eeaaa306 llama : fix speculative decoding crash with ISWA models
8037ef743 kv-cache : add unified slot mode for ISWA with --swa-full
c7f834f4c speculative : warn when ISWA models used without --swa-full
```

### Lessons Learned

1. **Always verify bugs on upstream master first** before developing fixes
2. **Check pending PRs** that might already address the issue
3. **Document which branch** the bug was found on

### Files Updated

| File | Change |
|------|--------|
| `handoffs/active/gemma3-swa-spec-decode-fix.md` | Marked RESOLVED, documented MTP-specific nature |
| `orchestration/model_registry.yaml` | Restored `draft_gemma3` as working, updated quirks |

### Conclusion

## Session 5: Phase 2 SWA Memory Optimization (2026-01-09)

### Problem

With `--swa-full` flag, the SWA cache mirrors base cache size (10240 MiB for 27B model), wasting memory when sliding window only needs a fraction.

### Solution: Forward-Looking SWA Masking

Modified `find_slot()` in `src/llama-kv-cache.cpp` to check cell masking against the **new batch's max position** instead of cached max. This allows reusing cells that will be outside the attention window *after* batch insertion.

### Implementation

Two changes in `llama-kv-cache.cpp`:
1. Compute `pos_batch_max` from `ubatch.pos[s * n_tokens + n_tokens - 1]` before while loop
2. Use `pos_batch_max + 1` instead of `cells.seq_pos_max(seq_id_cell) + 1` for SWA masking

### Results

| Metric | Before | After |
|--------|--------|-------|
| `--swa-full` required | Yes | No |
| SWA cache (27B) | 10240 MiB | 624 MiB |
| Memory reduction | — | **94%** |
| Acceptance rate | N/A (crash) | 42-81% |
| Tokens generated | 0 | 103+ |

### Test Verification

```bash
# Test 1: Short generation (50 tokens)
n_drafted = 48, n_accept = 39, accept = 81.250%

# Test 2: Longer generation (103 tokens)
n_drafted = 152, n_accept = 64, accept = 42.105%
speed: 12.264 t/s

# SWA cache confirmed at 624 MiB (not 10240 MiB)
```

### Files Modified

| File | Change |

| `llama.cpp/src/llama-kv-cache.cpp` | Forward-looking SWA masking (~15 lines) |
