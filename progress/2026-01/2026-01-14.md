## Memory Crash Fix

### Incident

Previous agent session (~00:52) crashed the machine while running orchestration tests. The agent was likely running pytest to validate existing code before implementing their planned changes (Tool/Script Registry wiring).

**Observed**: Multiple pytest processes visible in htop, consuming all available RAM.

**Root cause**: `src/api.py` eagerly initialized MemRL components (TaskEmbedder, QScorer, HybridRouter) at API startup, even for `mock_mode=True` tests. With pytest-xdist `-n auto` on a 192-thread machine, ~192 workers spawned, each loading the 0.5B embedding model.

### Fix Applied

| Component | Change |
|-----------|--------|
| **Lazy MemRL Loading** | Added `_memrl_initialized` flag; components only load on first `real_mode=True` request |
| **Memory Guard** | `tests/conftest.py` fails if <100GB free RAM, warns about `-n auto` on 192-thread machine |
| **Safe Parallelism** | Added warning comment to `pyproject.toml` about max 4 workers |
| **Pre-test Check** | `make check-memory` target, `test-all` depends on it |
| **Documentation** | Added "Test Memory Safety" section to CLAUDE.md |

### Files Modified

## Lazy Loading Architecture

The MemRL lazy loading follows the HOT/WARM/COLD memory pool design from `research/ESCALATION_FLOW.md`:

| Tier | Components | Loading |
|------|------------|---------|
| HOT (~35GB) | Frontdoor, draft models | Always resident |
| WARM (~460GB) | TaskEmbedder, specialists | Load on demand via mmap |
| COLD | Archived models | Disk only |

TaskEmbedder is now correctly in the WARM tier - only loaded when actual Q-scoring is needed, not on every API startup.

## RLM Phase 3: Escalation Integration (COMPLETE)

Implementation of automatic error recovery with role escalation in the orchestration loop.

### Summary

The Root LM loop in `src/api.py` now supports:
- **Error classification** via `_classify_error()` - Maps errors to ErrorCategory (FORMAT, CODE, LOGIC, TIMEOUT, etc.)
- **FailureRouter integration** - Consults FailureRouter on each error for routing decisions
- **Role switching** - Escalates from worker → coder → architect_general on repeated failures
- **Escalation prompts** - Builds context-aware prompts for escalated roles via `_build_escalation_prompt()`
- **Escalation logging** - All escalations logged via `progress_logger.log_escalation()`

### Key Changes

| Component | Change |
|-----------|--------|
| **Role tracking** | `current_role`, `consecutive_failures`, `role_history` variables |
| **Error classification** | `_classify_error()` helper function |
| **Escalation prompt** | `_build_escalation_prompt()` helper function |
| **Routing decision** | `FailureRouter.route_failure()` returns RoutingDecision |
| **Action handling** | Handles "retry", "escalate", "fail" actions |

### Flow

```
Error occurs in REPL execution
    ↓
Classify error → ErrorCategory
    ↓
Create FailureContext (role, error, category, failure_count)
    ↓
FailureRouter.route_failure() → RoutingDecision
    ↓
If action == "escalate":
    - Switch current_role to decision.next_role
    - Build escalation_prompt with failure context
    - Log escalation
    ↓
If action == "fail":
    - Stop with error message
    ↓
If action == "retry":
    - Continue with same role
```
