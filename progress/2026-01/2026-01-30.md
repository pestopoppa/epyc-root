## Phase 4: Input Formalizer Integration — COMPLETE

Implemented feature-flagged input formalization preprocessing. Complex prompts (optimization, proofs, algorithms, vague specs) are detected via keyword heuristics and sent to cold-tier MathSmith-8B for formal specification extraction before reaching the specialist.

### Problem

Complex prompts go to specialists as raw text. Specialists must simultaneously understand the problem AND solve it. Formalization separates concerns: a cheap 8B model extracts structure (variables, constraints, edge cases, acceptance criteria), then the specialist gets a well-defined spec alongside the original prompt.

### Keyword Detection Heuristics

| Pattern | Problem Type |
|---------|-------------|
| optimize, minimize, maximize, constraint, feasib... | optimization |
| prove, proof, verify, invariant, theorem... | proof |
| algorithm, data structure, complexity, O(, concurrent, lock-free... | algorithm |
| "build a...", "create a...", "design a..." (>50 words, no specifics) | ambiguous_spec |

### Design Decisions

1. **Subprocess, not HTTP**: MathSmith-8B is cold tier. `generate_command()` builds one-shot llama-cli. ~8s total (5s load + 3s generate).
2. **Context injection, not replacement**: Specialist sees BOTH original prompt AND `[FORMAL SPECIFICATION]` block.
3. **Keyword heuristics, not classifier**: Zero latency detection. Conservative — false negatives acceptable.
4. **Fallback chain**: `formalizer` (Q8_0) → `formalizer_q4` (Q4_K_M) → silent continue without spec.

## Architect Delegation to Tool-Wielding Specialists — COMPLETE

Implemented multi-loop architect delegation: the architect (235B @ 6.75 t/s) formulates investigation briefs in TOON format, a fast specialist (32B @ 39 t/s) runs ReAct tool loops to gather information, and the architect synthesizes. Multiple delegation loops capped at N=3.

### Problem

Architects are DIRECT_ONLY (no tool access) because they're too slow for ReAct loops. But some questions genuinely need investigation (file reads, web searches) before an architect-quality answer. Previously, the only option was to skip the architect entirely and use a weaker model with tool access.

### Solution

Architect decides → specialist investigates → architect synthesizes. TOON-encoded decisions for efficiency. Two delegation modes: `react` (investigation) and `repl` (document drafting).

### TOON Decision Protocol

Architect emits compact decisions:
- `D|<answer>` — direct answer, no delegation
- `I|brief:<text>|to:<role>` — investigate via ReAct (default)

Falls back to JSON or bare text if architect ignores TOON instruction.

## Session 4 (2026-01-30 evening): Vision Specialist Integration into Phase 3

### Problem

Vision models (Qwen2.5-VL-7B port 8086, Qwen3-VL-30B port 8087) were excluded from MemRL routing. The `vl` debug suite existed (42 OCRBench+ChartQA questions) but seeding, learning loop, and specialist routing all ignored vision roles. OCR was always-on API pre-chaining — the vision model never decided whether OCR was needed.

### Solution

Vision models = **specialists** (MemRL-routed LLMs). OCR = **tool** (invoked by specialist via ReAct). Two execution modes: `direct` (existing OCR pre-chain + VL) and `react` (new: VL model drives multi-turn loop, calls OCR on demand).

### Design Decisions

| Decision | Choice |
|----------|--------|
| Vision characterization | Specialists (not tools) — LLMs making judgments, learnable routing |
| OCR invocation | Specialist-decides via ReAct (worker_vision is agentic) |
| VL baseline for seeding | frontdoor:direct (text-only → trivial +1.0 bootstraps Q-values) |
| Feature flag | Folded into `ORCHESTRATOR_SPECIALIST_ROUTING` |
| Escalation triggers | MemRL-learned only (no keyword heuristics) |
| ReAct code path | Separate `_vision_react_mode_answer()` — text ReAct uses `primitives.llm_call()`, vision needs direct httpx to VL backend |

### Implementation (7 Steps)

| Step | File | Changes |
|------|------|---------|
| 1 | `seed_specialist_routing.py` | `VISION_ROLES`, `VISION_MODES`, `vl` suite, `image_path` forwarding, smart combo filtering |
| 2 | `run_phase3_validation.sh` | Ports 8086/8087/9001 health check, `vl` suite + vision roles in seeding/learning |
| 3 | `memrl_learning_loop.py` | `vl` in `DEFAULT_SUITES`, `image_path` in prompt loading + API forwarding |
| 4 | `tool_registry.yaml` | `ocr_extract` tool (category: vision, POST to port 9001) |
| 5 | `prompt_builders.py` | `VISION_REACT_TOOL_WHITELIST = REACT_TOOL_WHITELIST | {"ocr_extract"}` |
| 6 | `chat.py` | `force_server` on `_handle_vision_request()`, `_vision_react_mode_answer()`, `_execute_vision_tool()`, vision routing block |
| 7 | `orchestrator-quality-roadmap.md` | Vision specialist integration docs |

### New Functions in chat.py

- **`_vision_react_mode_answer()`**: Multimodal ReAct loop via httpx to VL backend. Image in first message only, subsequent turns text-only. Max 5 turns + forced final answer.
- **`_execute_vision_tool()`**: Routes `ocr_extract` → port 9001 via httpx, `calculate` → ast.parse eval, date/time inline.
- **`_handle_vision_request(force_server=)`**: Server constraint param for forced routing to specific VL port.

### Mode Constraints

| Role | Modes | Rationale |
|------|-------|-----------|
| worker_vision | direct, react | Agentic — supports tool calls |
| vision_escalation | direct only | 0% agentic — cannot do tool calls |
