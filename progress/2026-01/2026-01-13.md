## RLM (Recursive Language Model) Research & Orchestrator Roadmap

### Research Completed

**Paper Analyzed:** [arXiv:2512.24601](https://arxiv.org/abs/2512.24601) - "Recursive Language Models"
- **Authors:** Alex L. Zhang, Tim Kraska, Omar Khattab (MIT OASYS Lab)
- **Published:** December 2025

### Key Findings

The RLM paradigm treats long context as an **external environment** that models programmatically explore rather than stuffing into context windows. Our orchestration system already implements 80% of the RLM pattern.

**What we have that RLM has:**
- REPL sandbox (`src/repl_environment.py`)
- Context-as-variable pattern
- `llm_call()`, `llm_batch()` for sub-LM spawning
- `peek()`, `grep()`, `FINAL()` exploration tools

**What we have that RLM doesn't:**
- RadixAttention prefix caching (80% hit rate verified)
- Hierarchical tier system (A/B/C/D)
- Quality gates (`GateRunner`)
- Failure escalation (`FailureRouter`)

**Gaps to address (from RLM paper):**
1. Async parallel sub-LM calls (priority: HIGH)
2. Forced exploration validation (priority: HIGH)
3. Configurable recursion depth (priority: MEDIUM)
4. Per-query cost tracking (priority: MEDIUM)

## TTT (Test-Time Training) Research - CLOSED

### Technical Fixes Implemented

Fixed three cascading failures in llama.cpp training infrastructure:

1. **SET_ROWS view tensor assertion**: Added to `ignore_src` in `ggml_build_backward_expand()`
2. **FLASH_ATTN_EXT no backward pass**: Added to `ignore_src` (treat attention as gradient boundary)
3. **Graph overflow**: Increased `graph_max_nodes()` multiplier from 8x to 32x in `llama-context.cpp`

**Result**: `llama-finetune` and `llama-ttt` now complete successfully on Llama-3.2-1B-F32.

### Benchmark Results: TTT vs Baseline (Needle-in-Haystack)

| Metric | Baseline | TTT |
|--------|----------|-----|
| Context tokens | 3485 | 3481 |
| Needle retrieved? | **YES** | **NO** |
| Answer | Exact key extracted | "not specified in documentation" |
| Total time | ~30s | ~30s |
| Training accuracy | N/A | 90% |

**Critical Finding**: TTT trained to 90% next-token accuracy but FAILED to retrieve specific factual information.

### Why Research Paused (NO-GO)

| Constraint | Impact |
|------------|--------|
| FP32 models required | 4.6GB for 1B, ~100GB for 7B - impractical |
| Custom checkpoints needed | TTT-E2E paper used meta-trained models |
| FFN-only insufficient | Attention-inclusive training likely required |
| No benefit demonstrated | Baseline outperformed TTT |

## MemRL Episodic Memory Implementation

### Paper Analyzed

**MemRL: Self-Evolving Agents via Runtime Reinforcement Learning on Episodic Memory**
- arXiv:2601.03192 (Zhang et al., 2025)
- Key technique: Non-parametric RL on episodic memory with two-phase retrieval

### Implementation Completed

Implemented full MemRL-inspired episodic memory system for learned orchestration:

| Component | File | Purpose |
|-----------|------|---------|
| EpisodicStore | `repl_memory/episodic_store.py` | SQLite + numpy memory storage |
| TaskEmbedder | `repl_memory/embedder.py` | Task embedding via 0.5B model |
| TwoPhaseRetriever | `repl_memory/retriever.py` | Semantic + Q-value retrieval |
| HybridRouter | `repl_memory/retriever.py` | Learned + rule-based routing |
| ProgressLogger | `repl_memory/progress_logger.py` | Structured JSONL logging |
| QScorer | `repl_memory/q_scorer.py` | Async Q-value updates |

### Architecture Highlights

1. **Async Q-Scoring**: Decoupled from inference path - no latency impact on interactive routing
2. **Two-Phase Retrieval**: Semantic filtering (Phase 1) + Q-value ranking (Phase 2)
3. **Cold Start Strategy**: Defaults to rules while Q-database builds (Day 0-30 bootstrap)
4. **Claude-as-Judge Ready**: Benchmark suite for graded rewards

### Key Design Decisions

| Challenge | Solution |
|-----------|----------|
| Latency on Tier-A | Async Q-scoring agent (runs offline) |
| Cold start problem | Fall back to rules, bootstrap for 30 days |
| Memory scaling | 1.13TB RAM makes this a non-issue |
| Reward signal | Gate pass/fail + optional Claude-as-Judge |

## Q-Scorer Integration (Real-Time + Idle Cleanup)

### Architecture

**Two-Phase Q-Scoring:**
1. **Immediate Scoring**: After each task completion, `_score_completed_task()` updates Q-values in real-time (no inference, just DB reads/writes)
2. **Background Cleanup**: When server is idle (no active requests), processes any backlog of unscored tasks

### Components Added to API

| Component | Purpose |
|-----------|---------|
| `_score_completed_task()` | Immediate Q-value update after task completion |
| `_background_cleanup()` | Async task for opportunistic batch scoring when idle |
| `AppState.q_scorer` | QScorer instance for scoring operations |
| `AppState.episodic_store` | SQLite + numpy memory storage |
| `AppState.active_requests` | Counter for idle detection |

### Configuration

```python
ScoringConfig(
    use_claude_judge=False,  # Basic mode - no LLM required
    min_score_interval_seconds=30,  # Background cleanup interval
    batch_size=10,  # Tasks per cleanup round
)
```

### Q-Value Computation (Basic Mode)

```
reward = base_reward - gate_penalty - escalation_penalty

Where:
- base_reward: success=+1.0, failure=-0.5, partial=+0.3
- gate_penalty: -0.1 per failed gate
- escalation_penalty: -0.15 per escalation
```

## HybridRouter Integration (MemRL + RLM)

### Architecture

Combines MemRL two-phase retrieval with RLM REPL pattern:

```
Task → HybridRouter
         │
         ├─[Has enough samples + high confidence]→ Learned Routing
         │     └── TwoPhaseRetriever
         │           ├── Phase 1: Semantic filtering (embedding similarity)
         │           └── Phase 2: Q-value ranking (learned utility)
         │
         └─[Cold start / low confidence]→ Rule-Based Fallback
               └── RuleBasedRouter (from model_registry.yaml)
```

### Components Added

| Component | File | Purpose |
|-----------|------|---------|
| `HybridRouter` | `retriever.py` | Combines learned + rule-based routing |
| `TwoPhaseRetriever` | `retriever.py` | MemRL semantic + Q-value retrieval |
| `RuleBasedRouter` | `retriever.py` | Fallback from routing_hints |
| `DispatchResult.routing_strategy` | `dispatcher.py` | Tracks "learned" vs "rules" |

### Configuration

```python
RetrievalConfig(
    semantic_k=20,           # Candidates in Phase 1
    min_similarity=0.3,      # Similarity threshold
    q_weight=0.7,            # Q-value weight (vs similarity)
    confidence_threshold=0.6, # Min to use learned routing
)
```

### Cold Start Strategy

During cold start period (~30 days):
1. `should_use_learned()` returns False (not enough samples)
2. Falls back to RuleBasedRouter
3. Q-values accumulate via immediate + idle scoring
4. Eventually transitions to learned routing when confident
