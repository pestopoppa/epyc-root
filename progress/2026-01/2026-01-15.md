3. **Sequential startup** - Avoids 192-thread CPU overload during tensor repack
4. **WARM = mmap** - Pre-load at startup with `--include-warm`, no lazy loading
5. **State file** - JSON at `logs/orchestrator_state.json` for reload support

## Role Mapping Bug Fix (COMPLETE)

Fixed critical bug where `str(Role.CODER_PRIMARY)` returned `"Role.CODER_PRIMARY"` instead of `"coder_primary"`, breaking failure routing and escalation.

### Root Cause

Python's `str, Enum` mixin doesn't make `str()` return the value - it returns the enum name with class prefix.

### Fixes Applied

| File | Change |
|------|--------|
| `src/roles.py` | Added `__str__` method to return `.value` |
| `src/failure_router.py` | Added `_normalize_role()` function + belt-and-suspenders normalization |
| `src/failure_router.py` | Added `task_type` and `code_generated` fields to `FailureContext` |

### Code Changes

**roles.py:**
```python
def __str__(self) -> str:
    """Return the role value string."""
    return self.value
```

**failure_router.py:**
```python
def _normalize_role(role: str) -> str:
    """Normalize role to string value."""
    if hasattr(role, "value"):
        return role.value
    if isinstance(role, str) and role.startswith("Role."):
        return role[5:].lower()  # "Role.CODER_PRIMARY" -> "coder_primary"
    return role
```

## Phase 5: Proactive Delegation Implementation (MOSTLY COMPLETE)

Implemented the proactive delegation workflow in `src/proactive_delegation.py`.

### Components Created

| Component | Description |
|-----------|-------------|
| `IterationContext` | Tracks review loops, prevents infinite iterations (max 3/subtask, 10 total) |
| `ArchitectReview` | Review result dataclass with decision, feedback, score |
| `ReviewDecision` | Enum: APPROVE, REQUEST_CHANGES, ESCALATE, REJECT |
| `SubtaskResult` | Result from specialist executing a subtask |
| `AggregatedResult` | Final combined output from all specialists |
| `AggregationService` | Combines outputs: concatenate, merge_code, structured |
| `ArchitectReviewService` | Architect reviews specialist outputs against spec |
| `ProactiveDelegator` | Main workflow: delegate → execute → review → iterate → aggregate |

### Key Design Decisions

1. **Iteration Limits** - Prevents infinite loops:
   - `max_iterations=3` per subtask
   - `max_total_iterations=10` across all subtasks

2. **Review Prompt** - JSON format for structured feedback:
   ```json
   {
     "decision": "approve|request_changes|escalate|reject",
     "score": 0.0-1.0,
     "feedback": "Specific feedback",
     "suggested_changes": ["change 1", "change 2"]
   }
   ```

3. **Aggregation Strategies**:
   - `concatenate` (default): Section headers with `---` separators
   - `merge_code`: Deduplicates imports, adds provenance comments
   - `structured`: JSON with summary statistics

### Workflow Flow

```
TaskIR with plan.steps
    ↓
For each step:
    ↓
Build specialist prompt (action + inputs + outputs)
    ↓
Call specialist (via LLMPrimitives)
    ↓
Architect reviews output
    ↓
If APPROVE: next subtask
If REQUEST_CHANGES: iterate with feedback
If ESCALATE: switch to higher-tier role
If REJECT: mark failed, continue
    ↓
Aggregate all results
```

### Complexity-Aware Routing

Added `classify_task_complexity()` to avoid expensive architect calls for simple tasks:

| Complexity | Action | Example |
|------------|--------|---------|
| TRIVIAL | Frontdoor answers directly | "What's 2+2?" |
| SIMPLE | REPL execution (no architect) | "List files in src/" |
| MODERATE | Single specialist delegation | "Write a quicksort function" |
| COMPLEX | Architect generates TaskIR | "Design Byzantine consensus" |

### Escalation Triggers

Two types of manual escalation triggers (similar to Claude Code's `ultrathink`):

| Trigger | Effect | Model | Use Case |
|---------|--------|-------|----------|
| `/architect`, `/plan` | Forces COMPLEX | architect_general | Task decomposition |
| `/think`, `ultrathink` | Routes to thinking model | thinking_reasoning | Deep reasoning |

**Key distinction:**
- **Architect escalation**: Changes WHAT happens (task breakdown into subtasks)
- **Thinking escalation**: Changes HOW it happens (deeper reasoning for same task)

### Token Efficiency Optimizations

- Abbreviated JSON keys: `d`, `s`, `f`, `c` (not `decision`, `score`, etc.)
- Review output: ~30-50 tokens, max 128
