### Session 4: llama.cpp GGUF Optimization

13. **Discovered llama.cpp Native Support**
    - `PROJECTOR_TYPE_LIGHTONOCR` already exists in llama.cpp
    - `LightOnOCRVisionModel` converter at `convert_hf_to_gguf.py:10680`
    - Uses Pixtral vision graph builder

14. **GGUF Conversion**
    ```bash
    # Text model (F16 → Q4_K_M)
    python3 convert_hf_to_gguf.py /mnt/raid0/llm/hf/LightOnOCR-2-1B-bbox/ \
      --outfile /mnt/raid0/llm/models/LightOnOCR-2-1B-bbox-F16.gguf --outtype f16
    llama-quantize LightOnOCR-2-1B-bbox-F16.gguf LightOnOCR-2-1B-bbox-Q4_K_M.gguf Q4_K_M

    # Vision encoder (mmproj)
    python3 convert_hf_to_gguf.py /mnt/raid0/llm/hf/LightOnOCR-2-1B-bbox/ \
      --mmproj --outfile /mnt/raid0/llm/models/LightOnOCR-2-1B-bbox-mmproj-F16.gguf --outtype f16
    ```

18. **Parallel FastAPI Server**
    - Created `src/services/lightonocr_llama_server.py`
    - Worker pool with semaphore-controlled concurrency
    - 4 workers × 24 threads each = 96 total threads
    - Endpoints: `/health`, `/v1/document/ocr`, `/v1/document/pdf`

19. **Worker Config Optimization (8-page quick bench)**
    | Config | Total | Per-page | Throughput | vs 4×24 |
    |--------|-------|----------|------------|---------|
    | **8×12t** | **47.2s** | 40.6s | **0.170 pg/s** | **+15%** |
    | 4×24t | 54.6s | 25.8s | 0.147 pg/s | baseline |
    | 2×48t | 83.2s | 20.2s | 0.096 pg/s | -35% |
    | 1×96t | 123.4s | 15.4s | 0.065 pg/s | -56% |

    **Key insight**: More workers > more threads per worker. Updated server default to 8×12.
