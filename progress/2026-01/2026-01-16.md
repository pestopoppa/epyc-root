## Engram Tokenizer Compression Experiment (COMPLETE - Negative Result)

Tested whether Engram-style tokenizer compression (from DeepSeek's Jan 2025 paper) could improve Prompt Lookup N-gram match rates.

### Background

**Source**: "Conditional Memory via Scalable Lookup: A New Axis of Sparsity for LLMs" (DeepSeek-AI, Jan 2025)

Engram's `CompressedTokenizer` achieves 23% vocabulary reduction by normalizing semantically equivalent tokens:
- NFKC + NFD Unicode normalization
- Strip accents
- Lowercase
- Collapse whitespace

**Hypothesis**: Applying this normalization during Prompt Lookup matching could catch additional N-gram matches (e.g., "Alexander" = "ALEXANDER" = "alexander").

### Experiment Design

Created `scripts/experiments/compressed_tokenizer_analysis.py` to measure:
1. Build N-gram index with raw token IDs (current Prompt Lookup behavior)
2. Build N-gram index with normalized token text
3. Count additional speculatable positions from compression

### Results

| Test Case | 3-gram | 4-gram | 5-gram | Verdict |
|-----------|--------|--------|--------|---------|
| Case variations ("Alexander"/"ALEXANDER") | 0% | 0% | 0% | No benefit |
| Code naming (getUserData/get_user_data) | +42.9% | +200% | 0% | Only code patterns |
| Exact phrase repeats | 0% | 0% | 0% | Expected |
| Documentation headings | +50% | 0% | 0% | Minor |
| Technical docs (realistic) | 0% | 0% | 0% | No benefit |

### Why It Doesn't Work

1. **BPE tokenization encodes case differently**: "Alexander" and "ALEXANDER" become completely different token ID sequences at the byte level
2. **Text normalization can't unify tokens split differently**: Normalizing decoded text doesn't help when the underlying byte sequences differ
3. **Code naming is the exception**: camelCase vs snake_case sometimes share normalized forms, but absolute gains are tiny (7→10 positions)

### Key Insight

Engram's tokenizer compression works because they **retrained the tokenizer** to emit the same token IDs for normalized equivalents. Applying normalization at inference time to an existing BPE tokenizer cannot achieve this.

### Conclusion

**Do NOT pursue llama.cpp modifications** - the approach is fundamentally limited by tokenizer architecture. Phase 2 (implementation) not warranted.

## K-Value Optimization Analysis (COMPLETE)

Analyzed 76 model+draft combinations across K={4,8,16,24} to understand optimal K (draft_max) parameter selection.

### Key Findings

| Category | Count | Percentage |
|----------|-------|------------|
| K=24 optimal (still increasing) | 18 | 24% |
| K<=8 optimal | 47 | 62% |
| K=16 optimal | 11 | 14% |

### Pattern: Model Size Determines Optimal K

**Large dense (70B+)**: ALL show monotonically increasing performance up to K=24:
- Llama-3.1-70B + PARD-1B-Q8: 4.7 → 10.3 t/s (K=4 → K=24)
- Qwen2.5-7B + Coder-0.5B: 22.8 → 46.6 t/s (K=4 → K=24) - 2x gain!

**MoE models**: ALL peak at K=8:
- Qwen3-Coder-30B + 0.75B: 31.4 (K=8) → 14.0 (K=24) - DECREASING

**Small thinking models**: Peak at K=4 or K=8

### Theoretical Explanation

Optimal K = f(target_inference_time / draft_inference_time)
- Large models: Target slow, draft overhead negligible → higher K better
- MoE/small: Target fast, draft overhead dominates at high K

### Recommendations

1. For 70B+ dense models: Test K={8,16,24,32,48}
2. For MoE models: Only test K={4,8}
3. For small dense: Only test K={4,8,16}

### Priority Models for K>24 Testing

1. `general_qwen2_5_7b + coder_0_5b`: 46.6 t/s at K=24 (still climbing)
2. `worker_summarize + qwen25_coder`: 21.3 t/s at K=24 (jumped from 8.0)

## Role Enum Migration (COMPLETE)

Migrated key orchestration files from magic role strings to `Role` enum (`src/roles.py`) for type safety and IDE autocomplete.

### What Was Done

Replaced magic strings like `"frontdoor"`, `"coder"` with `Role.FRONTDOOR`, `Role.CODER_PRIMARY`:

| File | Changes |
|------|---------|
| `src/api.py` | Added `Role` import, updated `ESCALATION_ROLES` dict, updated 4 role usages |
| `src/api/services/orchestrator.py` | Updated `ESCALATION_ROLES` dict |
| `src/api/routes/chat.py` | Updated 7 role usages (routing, REPL init, LLM calls) |
| `src/api/routes/openai_compat.py` | Updated model-to-role mapping |

### Why It Works Transparently

The `Role` enum inherits from `str`:
```python
class Role(str, Enum):
    FRONTDOOR = "frontdoor"
    CODER_PRIMARY = "coder_primary"
    ...
```

This means:
- `str(Role.FRONTDOOR)` returns `"frontdoor"`
- `Role.FRONTDOOR` can be used wherever a string is expected
- Existing code that accepts strings continues to work

### What's NOT Migrated (Intentional)

- `llm_primitives.py` - Uses string defaults in function signatures (works via str subclass)
- `failure_router.py` - Already has `_normalize_role()` that handles both strings and Role enums
- Pydantic model defaults - Stay as strings for JSON serialization

### Benefits

1. **IDE autocomplete** - `Role.` shows all available roles
2. **Compile-time typo detection** - `Role.FRONTTDOOR` fails immediately
3. **Centralized documentation** - Each role has docstring explaining its purpose
4. **Type safety** - Function signatures can use `Role` type hints
