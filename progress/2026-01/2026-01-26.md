## Summarization Model Comparison Testing

### Objective
Evaluate non-thinking models for document summarization tasks, following up on previous work that found thinking models (Qwen3-4B-Thinking, Qwen3-30B-A3B-Thinking) output excessive chain-of-thought content even with `--reasoning-budget 0`.

### Test Setup
- **Input**: Twyne V1 Whitepaper (20 pages, ~10K chars context from figure descriptions)
- **Task**: Generate executive summary covering thesis, innovations, mechanics, benefits, target audience
- **Server Config**: llama-server with flash attention, 16K context

### Results

| Model | Acceleration | Elapsed | Tokens | t/s | Summary Len | Quality |
|-------|--------------|---------|--------|-----|-------------|---------|
| Qwen2.5-Coder-32B | Spec K=24 | 63.1s | 355 | 5.63 | 1945 chars | Good |
| Qwen3-32B | Spec K=8 | 62.3s | 349 | 5.60 | 2035 chars | Good |
| **Qwen3-Next-80B-A3B** | **MoE4** | 73.8s | 464 | **6.29** | **2560 chars** | **Best** |

### Key Findings

1. **All three models produce clean summaries** - no thinking artifacts, professional output
2. **Qwen3-Next-80B-A3B (SSM+MoE4) recommended for summarization**:
   - Most comprehensive output (2560 chars vs 1945-2035)
   - Highest detail coverage of all requested topics
   - 6.29 t/s - competitive speed despite 80B parameters
   - No speculative decoding needed (SSM incompatible anyway)

3. **MoE4 vs MoE2**: Previous config used MoE2 for `ingest_long_context` role. MoE4 provides better quality while maintaining good speed.

4. **Context processing overhead dominates**: All models show similar speed (~5.6-6.3 t/s) for this 10K context task. Actual generation is fast but input processing takes significant time.

**Updated `model_registry.yaml`:**
- `ingest_long_context` role: MoE expert count 2 → 4 for better quality
- Added `summarization` as candidate role for Qwen3-Next-80B-A3B

## Complete Executive Summary Comparison

### Objective
Compare all available models on the full executive summary task (300-400 words covering thesis, innovations, mechanics, benefits, target audience).

### Test Setup
- **Input**: Twyne V1 Whitepaper figure descriptions (~10K chars context)
- **Task**: Full executive summary prompt (same as large model tests)
- **Server Config**: llama-server with flash attention

### Results (Full Executive Summary)

| Model | Architecture | t/s | Tokens | Summary Len | Quality |
|-------|-------------|-----|--------|-------------|---------|
| **Qwen3-Coder-30B-A3B** | 30B MoE (6 active) | **15.34** | 322 | 1874 chars | Good |
| GLM-4.7-Flash (no thinking) | 30B MoE (4 active) | 12.25 | 297 | 1687 chars | Good |
| Meta-Llama-3-8B | 8B dense | 8.53 | 315 | 1742 chars | Good |
| **Qwen3-Next-80B-A3B** | 80B MoE (MoE4) | 6.29 | 464 | **2560 chars** | **Best** |
| Qwen3-32B | 32B + spec K=8 | 5.60 | 349 | 2035 chars | Good |
| Qwen2.5-Coder-32B | 32B + spec K=24 | 5.63 | 355 | 1945 chars | Good |

### GLM-4.7-Flash Thinking Mode Fix

**Problem**: GLM-4.7-Flash is a thinking model (deepseek2 architecture). Default mode outputs meta-analysis after the actual summary, wasting tokens and producing verbose output.

**Solution**: Use `chat_template_kwargs: {"enable_thinking": false}` in the API request.

```python
response = client.post(
    "http://localhost:8086/v1/chat/completions",
    json={
        "model": "GLM-4.7-Flash",
        "messages": [{"role": "user", "content": prompt}],
        "max_tokens": 800,
        "temperature": 0,
        "chat_template_kwargs": {"enable_thinking": False}  # Critical!
    },
)
```

**Result**: Clean output at 12.25 t/s with no meta-commentary.

### Key Findings

1. **Qwen3-Coder-30B-A3B (frontdoor) is fastest** at 15.34 t/s - 2.4x faster than Qwen3-Next-80B-A3B
2. **GLM-4.7-Flash works well** with `enable_thinking=false` at 12.25 t/s
3. **Dense models with spec decode are slower** (~5.6 t/s) for context-heavy tasks
4. **Meta-Llama-3-8B is slower than expected** at 8.53 t/s (no spec decode configured)
5. **Qwen3-Next-80B-A3B produces most comprehensive output** (2560 chars vs 1687-1945)

### Recommendations

**For fast summarization (worker tier):**
- **Primary**: Qwen3-Coder-30B-A3B (15.34 t/s) - can share port with frontdoor
- **Alternative**: GLM-4.7-Flash with `enable_thinking=false` (12.25 t/s)

**For high-quality comprehensive summarization:**
- **Primary**: Qwen3-Next-80B-A3B (MoE4) at 6.29 t/s - best detail coverage

**Avoid:**
- GLM-4.7-Flash without `enable_thinking=false` (outputs meta-commentary)
- Thinking-variant models (output `<think>` tags even with `--reasoning-budget 0`)

## Two-Stage Summarization Pipeline Implementation

### Objective
Implement two-stage summarization: fast draft from frontdoor (15 t/s) + quality review from large model (6 t/s) using reduced context.

### Design
- **Stage 1**: Frontdoor (Qwen3-Coder-30B-A3B) produces draft summary from document
- **Stage 2**: Large model (Qwen3-Next-80B-A3B) reviews draft + key excerpts
- **Trigger threshold**: 5K tokens (~20K chars)
- **Context reduction**: ~88% (14.8K → 1.8K tokens)
- **Caching**: Content-addressed cache for Stage 1 results (SHA256 hash key, 24h TTL)

### Files Created/Modified

| File | Change |
|------|--------|
| `src/services/draft_cache.py` | **NEW** - Content-addressed cache for Stage 1 results |
| `src/repl_environment.py` | Added `_grep_hits_buffer`, `get_grep_history()`, `clear_grep_history()` |
| `src/prompt_builders.py` | Added `build_stage2_review_prompt()` function |
| `src/api/routes/chat.py` | Added `_should_use_two_stage()`, `_run_two_stage_summarization()` |
| `orchestration/model_registry.yaml` | Added `two_stage_summarization` config section |

### Test Results (Twyne Whitepaper - 59K chars, ~15K tokens)

| Metric | Value |
|--------|-------|
| Total time | 183s |
| Stage 1 time | ~48s |
| Stage 2 time | ~135s |
| Context reduction | 14,850 → 1,815 tokens |
| Pipeline | Automatic (triggered by threshold) |

### Key Implementation Details

1. **Automatic trigger**: Detects summarization tasks + context > 5K tokens
2. **Stage 1 timeout**: 120s (frontdoor)
3. **Stage 2 timeout**: 300s (large model, extended for slower inference)
4. **Cache location**: `/mnt/raid0/llm/cache/drafts/`
5. **Fallback**: Falls back to standard orchestration on two-stage failure

### Expected Speedup Analysis

For 15K token documents:
- Single-stage (large model reads full doc): ~250s estimated
- Two-stage (implemented): 183s
- **Speedup**: ~1.4x

Speedup increases with document size due to prefill cost dominance.
