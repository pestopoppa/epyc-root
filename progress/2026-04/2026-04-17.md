## Session 1 — Autopilot Seeder Refactor: 3-Way → Dynamic Per-Role Evaluation

### Problem
Autopilot quality was frozen at q=1.138 for ~180 trials (since trial 1 of AR-3 Run 3, 2026-04-12). Investigation revealed three compounding issues:

1. **Eval tower routing collapse**: `eval_tower._eval_question()` sends `force_role=""` for end-to-end quality measurement. This goes through HybridRouter → learned routing (96% uniform Q-values, skipped) → rule-based fallback → `return ["frontdoor"]`. Every sentinel question routed to frontdoor, producing identical 11/29 correct = q=1.138 on every trial.

2. **3-way seeder too abstract**: The seeder's `evaluate_question_3way()` tested 3 hardcoded paths (SELF:direct, SELF:repl, ARCHITECT) instead of individual models. Q-values were keyed by abstract actions ("SELF:direct") not role names ("frontdoor", "architect_general"), so routing never learned per-model preferences. This was a pre-autopilot simplification that became stale.

3. **Blacklist poisoning**: 6 blacklist entries seeded from program.md included 4 server-launch-time flags not tunable at runtime (tree_speculation, dflash, hybrid_speculation), 1 model not in the active registry (nemotron_mamba2), and 1 entry (tool_permissions_legacy) potentially poisoned by recent REPL/tool routing bug fixes.

### Root Cause
The 3-way eval was designed before autopilot existed, as a simplification of the original all-roles `evaluate_question()`. It collapsed per-model signal into 3 coarse categories, preventing Q-values from learning which role handles which question type best. Combined with the eval tower's end-to-end measurement (correct by design, but useless when routing can't learn), autopilot mutations to prompts on architect_general.md, coder_escalation.md, etc. were invisible because no questions ever reached those models.

### Design Decisions

1. **Eval tower unchanged** — `force_role=""` is correct by design. It measures end-to-end system quality. Once per-role Q-values accumulate from the refactored seeder, routing will start working and eval tower will reflect real quality variation.
2. **No mode forcing** — `force_mode=""` lets the orchestrator's `_select_mode()` choose naturally (direct for simple, repl for complex). Each role handles questions its own way. Verified: `chat.py:409` treats empty force_mode as None → natural selection.
3. **Binary rewards** — `success_reward(passed)` returns 1.0/0.0, same as 3-way. Q-values converge to P(success|role) for each role.
4. **Dynamic discovery** — `discover_active_roles()` reads `server_mode` section of model_registry.yaml. Only `seeding_types.py` needs manual updates when stack changes (port mappings, exclusions, key-to-role aliases).

Deferred until sufficient per-role Q-values accumulate. During transition, old 3-way memories coexist with new per-role memories; HybridRouter.route() uses embedding similarity which finds both.
- **specialist_routing flag**: Still disabled (`ORCHESTRATOR_SPECIALIST_ROUTING` not set in stack). The per-role Q-values need to accumulate first. Once `should_use_learned()` passes (>3 similar memories with q_confidence above threshold), learned routing will activate.
