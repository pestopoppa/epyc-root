### Phase 3.0 IPC prototype validates inter-process EP

Per user's clarified ask: 4 NUMA-pinned llama.cpp instances cooperating on a SINGLE generation stream, with BOTH the router and the active expert pass split across instances. This is the handoff's Phase 2 scope, slightly stronger.

The synchronization primitive (busy-spin on cacheline-separated atomic state per worker, with NUMA pinning) is essentially free; the remaining work is model-state sharding across processes, not sync optimization.

### Honest framing

The intra-process attempts (Phase 1a/1b/2) were correct + env-gated + PPL-preserved scaffolding but didn't deliver throughput. The fundamental limitation is ggml's sequential-per-op graph executor: even with sharding, all threads still run op N together with global barriers, so per-NUMA parallelism wasn't actually achievable within one process.

Inter-process EP escapes that constraint: 4 small models running in parallel, each in its own ggml engine, connected by IPC primitives. The prototype validates the IPC overhead is negligible (<1% of token time even at the busiest configurations).

### Phase 3.1 implementation

Library committed at `/mnt/raid0/llm/cpu-ep-prototype/` commit `f47bec4`. ~770 LOC across `ep_dispatcher.{h,cpp}` + `test_ep_dispatcher.cpp` + `Makefile` + migrated `ipc_bench.cpp`.

### Critical bug fix vs monolithic prototype

The original `ipc_bench.cpp` had `ep_session` MAP_SHARED'd, which meant per-process role/instance_id propagated across processes via shared memory (last writer wins). The new library malloc's the session struct (so each process gets a COW copy at fork) and only the inner pointers (`ctl`, `broadcast`, `gather`) point into MAP_SHARED regions.

### Performance improvement

Per-cacheline state isolation (one `ep_state_slot` = full 64-byte cacheline per worker, vs the original's packed `atomic<uint32_t> state[16]`) eliminates false sharing as master writes `state[0..N]=GO` in succession.

**RTT 0.73 μs at 4 NUMA-pinned workers** vs monolithic's 0.86 μs = **15% faster**.

### Test results (5/5 pass)

| Test | Result |
|------|--------|
| `rtt` | 5 configs, 0.73 μs floor, payload-bound at large hidden dims (29.6 μs at 128K bytes) |
| `bitexact` | 0 failures across 5 rounds × 1024 ints × 4 workers |
| `stress` | 1M iterations, 5.71 s, no state-machine races |
| `fail` | Worker 3 exits at iter 5, master detects via ESRCH at iter 4 (sub-iteration latency via `ep_master_reap_dead` polling every 4096 spins inside `ep_wait_workers`) |
| `pdeath` | Master crashes, workers detect via `PR_SET_PDEATHSIG=SIGTERM`, exit cleanly with no zombies |

### Public API surface (final)

- **Lifecycle**: `ep_session_create_master`, `ep_session_destroy`, `ep_session_role/instance_id/n_workers`
- **Master**: `ep_broadcast`, `ep_wait_workers`, `ep_gather`, `ep_master_broadcast_buffer`, `ep_master_reap_dead`
- **Worker**: `ep_worker_recv`, `ep_worker_send_done`, `ep_worker_wait_go`, `ep_worker_signal_done`, `ep_worker_broadcast_buffer`, `ep_worker_gather_buffer`

C-callable so it links cleanly into llama.cpp without C++ runtime considerations beyond the .cpp file.

## CPU15 Phase 3.2(b+c) — env-var-driven master/worker fork at `ggml_cpu_init`

Wired the dispatcher into the ggml-cpu init path so that `GGML_EP_ROLE=master` + `GGML_EP_N_INSTANCES=N` triggers a fork of N-1 worker llama.cpp processes (master is implicit instance 0) before any threads or locks are taken. Worker children enter a passive `wait_go` / `signal_done` loop and never return to the caller's `main()`; on master EXIT (or master crash, via `PR_SET_PDEATHSIG=SIGTERM`) they `_exit()` cleanly without running master's atexit handlers.

### New files

- `ggml/src/ggml-cpu/ggml-ep-bootstrap.h` — public C API (`ggml_ep_bootstrap_if_requested`, `ggml_ep_get_session`)
- `ggml/src/ggml-cpu/ggml-ep-bootstrap.cpp` — env-var parsing, fork wiring, master `atexit` cleanup

### Hook

- `ggml_cpu_init` (in `ggml/src/ggml-cpu/ggml-cpu.c`) calls `ggml_ep_bootstrap_if_requested()` as its first action, gated by an internal `std::atomic<bool>` guard so subsequent calls are no-ops.

## CPU15 Phase 3.2(d.0) — IPC harness wired around `ggml_compute_forward_mul_mat_id`

Master signals via `ep_broadcast` at the top of every MoE op and waits via `ep_wait_workers` at the end. Workers stay in the bootstrap passive ack loop and contribute no compute — master still does the full op locally, so PPL is bit-exact to baseline. Validates the IPC harness inside the graph executor and establishes the structure for step (d.1) to slot actual work distribution into.

### Code

`ggml/src/ggml-cpu/ggml-cpu.c::ggml_compute_forward_mul_mat_id`:
- Top: `ep_session * ep_sess = ggml_ep_get_session(); if (master + n_workers>0 && ith==0) ep_broadcast(NULL,0)`
- Bottom: `if (master_active) { ggml_barrier; if (ith==0) ep_wait_workers(); }`
- Single-thread IPC: only ith==0 issues the calls; ggml_barrier gates other threads of this process.
- Active only when `GGML_USE_OPENMP` is undefined (matches the existing CPU15 Phase 1/2 intra-process EP paths).

### Smoke test (gemma-4-26B-A4B-it Q4_K_M, MoE, master+2 workers, 12 threads, prompt = "The capital of France is", 8 tokens, --seed 42)

| Mode | Output | Generation t/s |
|------|--------|---------------|
| Baseline (no env vars) | `[Start thinking]\nThe user is asking for` | 25.8 |
| EP master + 2 workers | `[Start thinking]\nThe user is asking for` | 26.5 |

Output bit-identical. Throughput within noise (the +0.7 t/s delta is run-to-run variance — master is doing the same compute path, just gated by ~62 sync rounds × 1 μs ≈ 60 μs overhead per token, or 0.04% of the 38 ms decode budget). No zombie workers — `PR_SET_PDEATHSIG=SIGTERM` cleans them up when master is killed.

### Status

Phase 3.2(d.0) lands the harness skeleton. Workers are still passive: they wake on every GO, immediately ack DONE, but contribute no compute. Master computes the full op alone. PPL is bit-exact by construction.

Commit: `8d53675fe` on `feature/cpu-ep-inter-process`.

### Phase 3.2(d.1) plan (next session — substantial)

To get actual speedup, workers must:
1. Exit the bootstrap passive loop and have model weights accessible. Cleanest path: move the fork hook from `ggml_cpu_init` to after `llama_load_model`, so workers inherit master's model mmaps via copy-on-write.
2. Redirect worker stdout/stderr to /dev/null and pin to a NUMA node (env-var `GGML_EP_NUMA_PIN=1` populates `cfg.worker_cpus` / `worker_numa_nodes`).
3. Run a custom worker compute loop OR run llama.cpp normally with stdio suppressed; both have tradeoffs (custom loop is surgical but needs new RPC protocol; full llama.cpp is simpler but workers redundantly execute non-MoE ops).
4. Inside `mul_mat_id`, both master and workers skip experts not assigned to my instance (`cur_a % n_instances != my_instance_id`).
5. After expert loop, gather + sum-reduce: master pulls workers' partials via `ep_gather` and adds into local `dst`; second broadcast round sends merged `dst` back to workers so their next op sees the same input.

Estimated effort: 4-6 hours focused. Phase 3.2(e) GGUF expert-shard loading then drops worker memory from ~16 GB to ~4 GB per instance (gemma-26B-A4B numbers). PPL gate is the correctness contract; throughput gate D3' (≥+20% over 6.16 t/s on REAP-246B Q4_K_M) is the production criterion.

## CPU15 Phase 3.2(d.1.a + d.1.b + d.1.c) — workers contribute, slicing works, NUMA pinning ready

Three sub-steps landed in sequence on `feature/cpu-ep-inter-process`. The cumulative result: inter-process Expert Parallelism is functionally working with bit-exact output across all configurations tested (master + 1, 3, 7 worker instances).

### (d.1.a) `385ee1d5c` — Workers exit passive loop, sync at MoE op boundaries

`ggml/src/ggml-cpu/ggml-ep-bootstrap.cpp`: workers no longer enter the passive ack loop. They redirect stdin/stdout to /dev/null (stderr stays attached for diagnostics) and return from bootstrap. They then run llama.cpp normally — independently mmap the same GGUF (kernel page-cache de-dup means physical pages are shared across processes), execute the same forward pass deterministically, and synchronise with master at every MoE op via the EP path inside `ggml_compute_forward_mul_mat_id`.

`ggml/src/ggml-cpu/ggml-cpu.c::ggml_compute_forward_mul_mat_id`: worker-side hooks symmetric to master's d.0 hooks — `ep_worker_wait_go` at op top, `ep_worker_signal_done` at op bottom, gated by `ggml_barrier`.

Both master and workers compute the FULL op (no slicing yet). Validates that N independent processes can run inference in lockstep and converge at every MoE boundary without deadlock.

### (d.1.b) `f9d7fe4c1` — Actual expert slicing + gather + sum-reduce + merged broadcast

The headline change. `mul_mat_id` now distributes work:

- Each instance skips experts not assigned (`cur_a % ep_n_inst != ep_my_id`).
- `dst` is zero-initialised (single thread, `ith==0`) so slots not handled by my instance read as 0 into the cross-instance sum.
- Workers `memcpy` partial `dst` to `ep_worker_gather_buffer`, signal DONE.
- Master `ep_wait_workers` + `ep_gather` (pointers into shared mem stored in static `g_ep_partials`), then ALL master threads parallel sum-reduce a contiguous slice of `dst` floats by adding workers' partials.
- Second sync round: master `memcpy`s merged `dst` to `ep_master_broadcast_buffer`, `ep_broadcast`s, `ep_wait_workers`. Workers `ep_worker_wait_go`, `memcpy` from `ep_worker_broadcast_buffer` to local `dst`, `ep_worker_signal_done`.

Dispatcher buffer sizes bumped from 1 MiB to 32 MiB each (broadcast + per-worker gather) to comfortably cover both decode dst (~72 KiB on gemma-26B-A4B) and prompt-processing dst at typical batch sizes.

Smoke tests (gemma-4-26B-A4B-it Q4_K_M, `--seed 42`, prompt = "The capital of France is", -n 8):

| Config | Generated text | Generation t/s |
|--------|----------------|----------------|
| Baseline (no EP, 12t) | "[Start thinking]\nThe user is asking for" | 23.6 |
| EP N=2 unpinned, no slicing (d.1.a, 12t each) | bit-exact same | 15.7 |
| EP N=2 unpinned, slicing (d.1.b, 12t each) | bit-exact same | 18.1 |
| EP N=4 unpinned, slicing (d.1.b, 6t each) | bit-exact same | 15.8 |

Slicing recovered ~+15% vs unsliced redundant compute. Output is **bit-identical** at every configuration.

### (d.1.c) `e2b8c5834` — NUMA pinning, one instance per NUMA node

`GGML_EP_NUMA_PIN=1` triggers per-instance NUMA pinning in the bootstrap: master to node 0, worker w to node `(w+1) % n_nodes`. Each instance is `sched_setaffinity`'d to all CPUs in its node (parsed from `/sys/devices/system/node/nodeN/cpulist`) and gets `MPOL_PREFERRED` for future heap/anonymous allocations.

This pairs cleanly with the existing `GGML_NUMA_WEIGHTS=1` + `--numa distribute` path: model weights interleaved across all 4 nodes, each instance has dedicated cores + dedicated DDR channels for its KV cache and scratch buffers.

| Config (gemma-4-26B-A4B-it Q4_K_M, -n 8) | Generation t/s |
|--------|----------------|
| Baseline (no EP, 12t) | 23.6 |
| EP N=4 + NUMA_PIN only | 11.8 |
| EP N=4 + NUMA_PIN + WEIGHTS=1 + --numa distribute (24t each) | **19.9 (84% of baseline)** |

NUMA pinning without weight distribution **regresses** to 11.8 t/s because pages first-touched by master on node 0 are read remotely by workers on nodes 1/2/3. Pinning compute without distributing data inverts the locality benefit. The two-flag pairing is the correct configuration; we don't auto-set the inner flags here because they have semantics independent of EP.
