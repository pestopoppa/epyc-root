### Phase 3.0 IPC prototype validates inter-process EP

Per user's clarified ask: 4 NUMA-pinned llama.cpp instances cooperating on a SINGLE generation stream, with BOTH the router and the active expert pass split across instances. This is the handoff's Phase 2 scope, slightly stronger.

Standalone prototype at `/mnt/raid0/llm/cpu-ep-prototype/` (commit `4901cc7` in that repo, ~330 LOC C++):

| Configuration | Per-iter RTT | Token IPC (62 × 3 rounds) | % of 162 ms baseline |
|---|---|---|---|
| **Pure IPC (no fake compute)** | **0.86 μs** | **0.16 ms** | **0.1%** |
| With 5120-float fake compute | 3.42 μs | 0.64 ms | 0.4% |
| Hidden=32K | 16.22 μs | 3.02 ms | 1.9% |
| Hidden=1024 | 1.60 μs | 0.30 ms | 0.2% |
| 8 workers (oversubscribed) | 3.89 μs | 0.72 ms | 0.4% |
| NUMA-unpinned (control) | 3.41 μs | 0.64 ms | 0.4% (no diff at this scale) |

**Decision: GO for full Phase 3 implementation.** Pure IPC RTT is ~200× under the viability threshold (was ≤200 μs; achieved <1 μs). The synchronization primitive (busy-spin on cacheline-separated atomic state per worker, with NUMA pinning) is essentially free; the remaining work is model-state sharding across processes, not sync optimization.

### Honest framing

The intra-process attempts (Phase 1a/1b/2) were correct + env-gated + PPL-preserved scaffolding but didn't deliver throughput. The fundamental limitation is ggml's sequential-per-op graph executor: even with sharding, all threads still run op N together with global barriers, so per-NUMA parallelism wasn't actually achievable within one process.

Inter-process EP escapes that constraint: 4 small models running in parallel, each in its own ggml engine, connected by IPC primitives. The prototype validates the IPC overhead is negligible (<1% of token time even at the busiest configurations).

### Phase 3.1 implementation

Library committed at `/mnt/raid0/llm/cpu-ep-prototype/` commit `f47bec4`. ~770 LOC across `ep_dispatcher.{h,cpp}` + `test_ep_dispatcher.cpp` + `Makefile` + migrated `ipc_bench.cpp`.

### Critical bug fix vs monolithic prototype

The original `ipc_bench.cpp` had `ep_session` MAP_SHARED'd, which meant per-process role/instance_id propagated across processes via shared memory (last writer wins). The new library malloc's the session struct (so each process gets a COW copy at fork) and only the inner pointers (`ctl`, `broadcast`, `gather`) point into MAP_SHARED regions.

### Performance improvement

Per-cacheline state isolation (one `ep_state_slot` = full 64-byte cacheline per worker, vs the original's packed `atomic<uint32_t> state[16]`) eliminates false sharing as master writes `state[0..N]=GO` in succession.

**RTT 0.73 μs at 4 NUMA-pinned workers** vs monolithic's 0.86 μs = **15% faster**.

### Test results (5/5 pass)

| Test | Result |
|------|--------|
| `rtt` | 5 configs, 0.73 μs floor, payload-bound at large hidden dims (29.6 μs at 128K bytes) |
| `bitexact` | 0 failures across 5 rounds × 1024 ints × 4 workers |
| `stress` | 1M iterations, 5.71 s, no state-machine races |
| `fail` | Worker 3 exits at iter 5, master detects via ESRCH at iter 4 (sub-iteration latency via `ep_master_reap_dead` polling every 4096 spins inside `ep_wait_workers`) |
| `pdeath` | Master crashes, workers detect via `PR_SET_PDEATHSIG=SIGTERM`, exit cleanly with no zombies |

### Public API surface (final)

- **Lifecycle**: `ep_session_create_master`, `ep_session_destroy`, `ep_session_role/instance_id/n_workers`
- **Master**: `ep_broadcast`, `ep_wait_workers`, `ep_gather`, `ep_master_broadcast_buffer`, `ep_master_reap_dead`
- **Worker**: `ep_worker_recv`, `ep_worker_send_done`, `ep_worker_wait_go`, `ep_worker_signal_done`, `ep_worker_broadcast_buffer`, `ep_worker_gather_buffer`

C-callable so it links cleanly into llama.cpp without C++ runtime considerations beyond the .cpp file.
