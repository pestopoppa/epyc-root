# 2026-04-29 — ColGREP CLI Bridge Unblocked (v1.0.6 → v1.2.0)

## Summary

Triggered by a question about whether the orchestrator REPL would benefit from a more powerful semantic search tool. Reviewed the actual REPL surface (`peek`/`grep`/`list_dir`/`code_search` via NextPLAID + combined ops including `_workspace_scan`) — semantic search is already present via NextPLAID multi-vector retrieval. The next-step CLI bridge (ColGREP, intake-355) had been blocked on an upstream Rust panic during ONNX/GPU init in v1.0.6 since the EPYC box has no CUDA/cuDNN.

Upstream ColGREP v1.2.0 (released 2026-04-10) replaces the panic with a fallback message and adds `--force-cpu` / `NEXT_PLAID_FORCE_CPU`. Validated locally and installed.

#### 3. CPU4 op-coalesced barriers (Phase 0 + Phase 1)

Phase 0 manual op-chain analysis on Qwen3 MoE estimated 24-29% per-token barrier-count reduction (Q/K/V MUL_MAT chain coalescable). Phase 1 implementation falsified the estimate.

**Critical discovery**: smoke test at COALESCE=1 with MUL_MAT in allowlist produced GARBLED output ("Failed to parse input at pos 0: GD Sw\n…\nCompatibility几..."). Root cause: `ggml_compute_forward_mul_mat` writes src1 quantization to shared `params->wdata` BEFORE its internal barrier (`ggml-cpu.c:1467-1487`). Coalescing two MUL_MATs lets op N+1 clobber wdata while op N's chunk-loop still reads it. **Phase 0 missed this constraint.**

After tightening allowlist (excluded MUL_MAT/MUL_MAT_ID), smoke passes bit-exact. PPL chunk-3 bit-exact verified on Coder + REAP. But achievable barrier-count reduction drops from 24-29% to ~5% (1 skippable barrier per layer in attention block: ROPE-Q → RMS_NORM-K only).

Phase 1 throughput measurement:

| Model | n_total | Δ_pct |
|---|---|---|
| Coder-30B Q4_K_M | 20 reps (5 + 15 alternated) | -10 to -20% (high CV; aggregate -19.7% across 3 trials) |
| Next-80B Q4_K_M | 5 | -6.2% (noisy) |
| REAP-246B Q4_K_M | 5 | -2.3% (clean signal) |

Gate (≥+5% on 2 of 3 sync-bound Q4_K_M models): NOT MET. Patch (~80 LOC) committed in `9f6191581` to llama.cpp-experimental, env-gated `GGML_BARRIER_COALESCE=1` default off (mirrors slot-promotion dispatcher v1 treatment).

CPU20 bundles: `2026-04-29-cpu4-op-coalesced-barriers-phase0/`, `2026-04-29-cpu4-op-coalesced-barriers-phase1/`.

**Lesson preserved for future Phase 0 analyses**: MUST check buffer-sharing constraints (params->wdata, params->wsize, threadpool state), not just dependency-graph independence. The wdata-shared-buffer hazard caused the 5× overestimate.
