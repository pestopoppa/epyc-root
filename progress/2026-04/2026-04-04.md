2. **Trial 43**: Autopilot detected corruption, attempted git restore — restore also failed

### Root Cause: 5 Safety Gaps

| # | Gap | Location | Fix |
|---|-----|----------|-----|
| 1 | `ast.parse()` accepts syntactically valid but semantically destroyed code | `prompt_forge.py:486` | New `_validate_code_mutation()`: syntax + shrinkage + public names + import test |
| 2 | No semantic validation (missing classes/functions not detected) | `prompt_forge.py:486` | Public name preservation check — all module-level defs must survive |
| 3 | Simplicity criterion only blocks growth, not destruction | `autopilot.py:499` | Added >50% shrinkage guard for both code and prompt mutations |
| 4 | T0 eval too shallow to detect escalation breakage | `autopilot.py:491` | Known limitation — needs larger eval or targeted import check (not fixed yet) |
| 5 | Reverts not committed to git | `prompt_forge.py:479` | Both `revert_code_mutation()` and `revert_mutation()` now git commit |

### Fixes Implemented

**`_validate_code_mutation()` in `prompt_forge.py`** — 4-layer deep validation:
1. **Syntax**: `ast.parse()` (existing)
2. **Catastrophic shrinkage**: Reject if >60% of lines removed
3. **Public name preservation**: All top-level classes/functions from original must exist in mutated version
4. **Import test**: Actually write file, `importlib.import_module()`, restore original — catches circular imports, missing deps

**Simplicity criterion in `autopilot.py`** — bidirectional:
- Existing: block growth >20% with <0.02 quality gain
- New: block shrinkage >50% unconditionally (for both code and prompt mutations)

**Revert commits** — both `revert_code_mutation()` and `revert_mutation()` now auto-commit the revert so corrupted state is never left as HEAD.

### Verification
Tested with the actual garbage that caused the incident:
- 454→3 lines: `valid=False, reason=catastrophic shrinkage: 454→3 lines (99% removed)`
- Missing names: `valid=False, reason=missing public names: {EscalationPolicy, ...}`
- Identity (no change): `valid=True, reason=ok`

## 1. Goose (block/goose) — Architecture Deep Dive

**What it is**: Rust/TypeScript autonomous AI coding agent (58% Rust, 34% TS). Desktop app + CLI.
**Repo**: github.com/block/goose

### Core Architecture

Three components: **Interface** (CLI/desktop) -> **Agent** (core loop) -> **Extensions** (MCP tools).

The agent loop is a standard tool-use cycle:
1. User request + available tool list -> LLM provider
2. LLM returns tool call (JSON)
3. Goose executes tool, gathers result
4. Result -> back to LLM
5. Repeat until LLM gives final response (no more tool calls)

**Error recovery**: Errors (invalid JSON, missing tools, execution failures) are sent back to the model as tool responses rather than crashing the loop. The LLM gets the error information and attempts self-recovery. Known weakness: some MCP tool errors cause unrecoverable states where the session shows the error repeatedly.

### Multi-Model Routing: Lead/Worker

This is Goose's most interesting pattern for us.

**Configuration**:
- `GOOSE_LEAD_MODEL` (e.g., claude-opus) -- complex reasoning, planning
- `GOOSE_MODEL` (e.g., claude-sonnet) -- execution, simpler tasks
- `GOOSE_LEAD_TURNS` -- number of initial turns using the lead model (default: 3)
- `GOOSE_LEAD_FAILURE_THRESHOLD` -- when to fall back to lead after worker failures
- `GOOSE_LEAD_FALLBACK_TURNS` -- how many turns to use lead during fallback

**Switching logic**: Start with lead for first N turns (planning phase), then hand off to worker (execution phase). If the worker gets confused or loops, Goose pulls the lead model back in. Once recovered, worker takes over again.

**Extended roles** (from issue #4036 and docs):
- `lead` -- planning, complex reasoning
- `worker` -- execution, code writing
- `oracle` -- second opinion
- `planner` -- dedicated planning model (separate from lead)

**Planner configuration**: `GOOSE_PLANNER_PROVIDER` + `GOOSE_PLANNER_MODEL` allow a different model specifically for the plan-creation phase. Example: GPT-4.1 for planning, Claude Sonnet for execution.

### Context Management: Smart Compaction

**Two-tiered approach**:
1. **Auto-compaction**: Triggered at 80% of token limit (configurable via `GOOSE_AUTO_COMPACT_THRESHOLD`). Summarizes older conversation while keeping recent turns in full detail.
2. **Content revision**: After 10+ tool calls, older tool call outputs are summarized in background while recent calls stay verbatim.
3. **Fallback strategies** (if still over limit): summarize, truncate, clear, or prompt user.

**Session persistence**: SQLite database (`sessions.db`) since v1.10.0. Supports session resume, fork (branch from a specific message), and multi-session switching.

### MCP Extension System

Extensions implement MCP (Model Context Protocol) and expose tools, prompts, and resources. Types:
- **Stdio extensions**: Launch external process, communicate via stdin/stdout MCP protocol. Goose manages the process lifecycle.
- **Built-in extensions**: Compiled into the binary.

1,700+ MCP servers available. Extensions are configured in `config.yaml`.

### What's NOT Transferable

- Goose calls cloud APIs exclusively. No local model management, no server lifecycle, no NUMA awareness.
- The lead/worker switching is turn-count-based, not task-complexity-based. Our routing intelligence (SkillRL, domain detection) is more sophisticated.
- MCP is designed for API-calling agents. Our tool registry with behavioral guidance (from CC analysis, intake-249) is more tightly integrated.

## 2. Clido (clido-ai/clido-cli) — Architecture Deep Dive

**What it is**: Local-first, multi-provider CLI coding agent with TUI interface.
**Repo**: github.com/clido-ai/clido-cli (newer/smaller project, limited web indexing)

**Note**: Clido has minimal public documentation outside its README. Web search engines return very few results for it, suggesting it is either very new or low-traction. The analysis below is based on the README content successfully fetched.

### Profile System

Profiles stored in `~/.config/clido/config.toml` (global) or `.clido/config.toml` (project-local, takes precedence).

Each profile specifies:
- Provider (Anthropic, OpenAI, Google, DeepSeek, Mistral, xAI, Groq, Together, Fireworks, Cerebras, Perplexity, MiniMax, Alibaba Cloud, Kimi, Ollama -- 16 total)
- Model identifier
- API key env var reference
- Custom base URL override
- Optional "fast" sub-profile for utility tasks

**Switching**: `--profile` flag, `CLIDO_PROFILE` env var, or in-TUI model picker (Ctrl+M).

**Key insight**: The "fast" sub-profile is interesting -- it allows a lightweight model for utility operations (summarization, formatting) while keeping the main profile for reasoning. This maps to our frontdoor/worker/explore role split.

### Cost Tracking

Real-time token usage and cost display in TUI header. Commands:
- `/cost` -- current session expenditure
- `/tokens` -- input/output token breakdown
- `--max-budget-usd` -- per-session spending cap (hard limit)

Session costs tracked in audit log across sessions.

### YAML Declarative Workflows

`workflow run <FILE>`, `workflow validate`, `workflow inspect` commands.
- Structured multi-step task execution
- Step dependencies and sequencing
- Preflight validation checks

No detailed YAML schema found in public docs.

### Permission Model

Three modes:
1. **default** -- prompts for confirmation on file modifications
2. **accept-all** -- auto-approves all operations
3. **plan** -- creates plans without execution

Per-prompt quick-select: Once, Session-wide, Workdir-wide, Deny, Deny with feedback.

### Tool Surface

23 tools across categories:
- Read: file inspection, search, codebase indexing
- Edit: file modification, refactoring
- Execute: bash with optional sandboxing (macOS `sandbox-exec` / Linux `bwrap`)
- Git: commit, push, branch, PR creation
- Analysis: semantic search, memory management

### What's NOT Transferable

- Clido is a pure API-calling agent. Its "multi-provider" is about switching between cloud APIs, not managing local inference servers.
- Cost tracking assumes per-token API pricing. Our cost model is GPU-hours and memory bandwidth, not dollars-per-token.
- Profile switching is manual. No automatic routing based on task analysis.

## 3. Comparative Analysis vs EPYC Orchestrator

### Multi-Model Routing

| Aspect | Goose | Clido | EPYC Orchestrator |
|--------|-------|-------|-------------------|
| Routing trigger | Turn count (first N = lead, rest = worker) | Manual profile switch | Task complexity analysis (SkillRL + domain detection) |
| Model roles | lead, worker, planner, oracle | main, fast sub-profile | frontdoor, worker_fast, worker_slow, explore |
| Failure fallback | Worker failure count -> fall back to lead | Manual switch | Escalation pipeline (fast -> slow -> explore) |
| Cost optimization | Cheap worker for most turns | Budget cap per session | None explicit (local models = fixed GPU cost) |
| Sophistication | Low (turn-count heuristic) | None (manual) | High (learned routing, quality signals) |

**Verdict**: Goose's lead/worker is a simpler version of what we already have. Our routing is more sophisticated. However, Goose's failure-threshold-based fallback is a clean pattern -- a simpler alternative to our escalation chain for cases where the full pipeline is overkill.

### Cost Management

| Aspect | Goose | Clido | EPYC Orchestrator |
|--------|-------|-------|-------------------|
| Per-session budget | None | `--max-budget-usd` | None |
| Real-time tracking | None visible | TUI header (tokens + dollars) | inference_tap.log (raw, no aggregation) |
| Historical tracking | None | Audit log | Session log (turns, not costs) |

**Verdict**: Clido's per-session budget cap is the most directly transferable pattern. Even though our "cost" is GPU-time not API-dollars, a per-session token budget (max input + output tokens before the session must compact or terminate) would prevent runaway sessions that consume excessive inference time.

### Context Management

| Aspect | Goose | Clido | EPYC Orchestrator |
|--------|-------|-------|-------------------|
| Compaction trigger | 80% of token limit | Not documented | 50% threshold (planned, B2) |
| Strategy | Summarize old turns + tool outputs | Not documented | Protected zones (first N + last M, planned) |
| Tool output handling | Summarize after 10+ calls | Not documented | None (full output retained) |
| Session persistence | SQLite | Resumable checkpoints | session_store.py (SQLite) |
| Session forking | Yes (branch from message) | Not documented | None |

**Verdict**: Goose's tool-output-specific summarization (after 10+ tool calls) is a useful refinement we should adopt. Our planned B2 context compression covers the general case, but adding a separate tool-output summarization pass would reduce context bloat from verbose REPL outputs and file reads.

### Tool Permissions

| Aspect | Goose | Clido | EPYC Orchestrator |
|--------|-------|-------|-------------------|
| Model | MCP-defined per extension | 3-mode (default/accept-all/plan) | Cascading tool policy |
| Granularity | Per-extension | Per-operation type | Per-tool per-role |
| User approval | Not documented per-tool | Once/Session/Workdir/Deny | None (policy-based) |

**Verdict**: Clido's quick-select approval model is a good UX pattern for interactive sessions but irrelevant for our autonomous pipeline. Our cascading policy is more appropriate for local inference.

## 4. Actionable Patterns for EPYC

### Pattern 1: Per-Session Token Budgeting -- HIGH VALUE

**Source**: Clido's `--max-budget-usd`
**Adaptation**: Implement `--max-session-tokens` (or env `ORCHESTRATOR_MAX_SESSION_TOKENS`)

Even with local inference, runaway sessions waste GPU time. A token budget would:
- Cap total tokens (input + output) per session
- Trigger compaction at 70% of budget (before the hard limit)
- Hard-stop at 100% with a summary of work completed
- Track cumulative tokens in `session_log.py` (we already track turns)

**Effort**: Low. Add token counting to session_log, add threshold checks in main loop.
**Files**: `src/graph/session_log.py`, `src/graph/helpers.py`, `src/config/__init__.py`
**Maps to**: B5 (Session Analytics) in orchestrator-conversation-management.md

### Pattern 2: Tool Output Summarization -- MEDIUM VALUE

**Source**: Goose's "summarize tool outputs after 10+ calls"
**Adaptation**: After N tool calls (configurable, default 8), use `worker_explore` to summarize older tool outputs in-place, keeping the last 3 tool call/result pairs verbatim.

**Key refinement over Goose**: Goose summarizes ALL old tool outputs. We should be selective:
- REPL outputs: Always summarize (verbose, often redundant between iterations)
- File reads: Replace with "Read file X (N lines)" stub after use
- Web search results: Summarize to key facts
- Error outputs: Keep verbatim (critical for debugging)

**Effort**: Medium. Requires hooking into context compression pipeline.
**Files**: `src/context_compression.py` (new, planned in B2), `src/graph/helpers.py`
**Maps to**: B2 (Context Compression) in orchestrator-conversation-management.md

### Pattern 3: Failure-Count Escalation Trigger -- LOW VALUE (we have better)

**Source**: Goose's `GOOSE_LEAD_FAILURE_THRESHOLD`
**Assessment**: We already have escalation based on quality signals. However, a simple failure counter as a FAST PATH trigger (3 consecutive tool errors = escalate immediately without waiting for quality scoring) could reduce latency for obvious failure modes.

**Effort**: Trivial. Deferred -- current escalation works.

### Pattern 4: Session Forking -- LOW VALUE

**Source**: Goose's `--fork` flag
**Assessment**: Interesting for exploration but our orchestrator is not interactive enough to benefit. Note for future.

### Pattern 5: Declarative Workflow YAML -- NOT APPLICABLE

**Source**: Clido's `workflow run`
**Assessment**: Our task graph is implicit in the LangGraph state machine + role routing. The autopilot's species/action system already fills the structured-pipeline niche.

### Deep Dive: Multiscreen Attention (intake-256)

**Architecture**: Replaces softmax with absolute query-key screening. Keys evaluated independently against learned threshold. Gated Screening Tile = unified attention+FFN (no separate FFN block). d_K=16, d_V=64 (tiny projections). MiPE = RoPE on 2/16 dims, self-disables for wide windows.

**Results**: 40% param savings, 2.3-3.2x latency at 100K, stable training at lr=1, near-perfect long-context retrieval. NOT truly sub-quadratic — uses learned per-head windows (95.3% finite) → effectively linear.

**EPYC relevance**: CPU-friendly (no exp/log, bounded ops, SIMD-friendly). KV cache 69% smaller per head. Spec decode should work (non-recurrent, independent key eval). **But**: no implementation exists, no weights, largest model 4B, paper is 3 days old.

**Verdict**: WATCH item. Handoff stub created: `multiscreen-attention-evaluation.md`.

### Deep Dive: Gemma 4 Evaluation (intake-251/252)

**Critical finding: Gemma 4 26B-A4B is a PURE TRANSFORMER MoE.** No Delta Net, no recurrent layers. This is the single most important difference from Qwen3.5-35B-A3B.

| Property | Gemma 4 26B-A4B | Qwen3.5-35B-A3B |
|----------|-----------------|------------------|
| Active params | 3.8B | 3B |
| Architecture | Pure transformer | 75% Delta Net recurrent |
| Experts | 128+1 shared | MoE hybrid |
| SWA | 5:1 local/global (1024 win) | N/A |
| Context | 256K | 131K |
| Spec decode | Should work (non-recurrent) | ALL approaches failed |
| GGUF Q4_K_M | 16.8 GB | 19 GB |
| Quality (MMLU Pro) | 82.6% | ~80-85% est |

**Speed projections**: Baseline ~25-35 t/s (pure attn, no recurrent bottleneck). With E2B draft (65-80% acceptance est) → ~40-70 t/s. NUMA 4-way + spec → ~150-200 t/s aggregate. Current Qwen3.5 aggregate: 50.8 t/s.

**Risks**: SWA + spec decode interaction (Gemma 3 had this bug), `<unused24>` eval bug (#21321), community reports of slow inference on some platforms.

**Draft situation**: E2B (5B/2.3B effective, same vocab 262K) is natural draft. Gemma-3-1B (1B, same vocab) is cross-gen candidate. Both need empirical testing.

**Benchmark plan**: 5-phase plan designed (smoke test → baseline → spec decode → combined → quality). Decision gate: >100 t/s agg + quality ≥2.4 → replace Qwen3.5 as frontdoor.

### Deep Dive: mesh-llm (intake-253)

**MoE expert sharding**: Full trunk + overlapping expert subsets per node. Zero cross-node traffic because each node's local router selects from available experts. Gate mass analysis (same technique as our REAP) identifies critical experts for replication.

**Single transferable finding: Asymmetric KV cache quantization** (Q8_0 keys / Q4_0 values). Keys need more precision for routing, values are less sensitive. Could reduce KV cache ~35% with minimal quality loss. Action: benchmark on Qwen3.5 and Gemma 4.

**Other findings**: Blackboard agent collaboration (lightweight gossip, not applicable to single-node), SSD expert streaming from flash-moe (5.5 t/s on 397B with 48GB RAM — reference for future), demand-aware model unloading (relevant if we add more roles).

### Deep Dive: Think Anywhere (intake-258)

**Mechanism**: Two-stage pipeline (cold-start LoRA SFT on 5K Gemini-generated samples + GRPO RL on 14K coding problems, 8x A100). Model learns to emit `<thinkanywhere>` tokens at high-entropy positions during code generation. Hierarchical reward: `R = 0.1 * R_struct + R_correct`.

**Key findings**:
- Ablations: cold-start alone -21.5pp, RLVR alone -6.0pp, no upfront thinking only -2.8pp (inline blocks do most work)
- Padding tokens at think positions: -1.8pp — even pausing helps (implicit computation)
- Token efficiency: 238-306 fewer tokens than GRPO, upfront thinking shrinks 35-50%, inline blocks avg ~23 tokens
- Position analysis: model converges on assignment + return statements (highest entropy positions)
- **Zero-shot math transfer**: AIME 2024: 5.3% → 17.3% (trained only on code!) — thinking capability is domain-general
- Gaps: no failure analysis, no comparison with reasoning compression methods, no inference latency, only 7B scale

**EPYC relevance**: Text-based version works with existing vocab (no llama.cpp changes). But we don't train models, and spec-decode interaction is problematic (high-entropy mode switches tank draft acceptance). Main value: entropy-at-position finding could enhance `difficulty_signal.py`.

### Deep Dive: RTK (intake-259)

**Architecture**: 6-phase pipeline, 42 command modules, 12 filtering strategies (stats extraction, failure focus, tree compression, deduplication, etc.). Strategy per command is hardcoded, not heuristic.

**Key findings**:
- Real performance: 80% overall session savings, 99% on test runners, 94% on large git diffs
- Claude Code integration: PreToolUse hook rewrites Bash commands. **Only intercepts Bash, not built-in tools (Read/Grep/Glob).**
- **Security unresolved** (Issue #640): shell injection via `sh -c`, telemetry on by default, plaintext secrets in tracking DB, CI trust bypass
- **Compensation problem** (Issue #582): one user measured 18% cost *increase* — compressed output forced more LLM output tokens
- **EAGAIN errors under heavy tool use** (Issue #968) — directly relevant to our autopilot
- 376 open issues including P1-critical bugs (multi-file cat, git push timeouts, broken JSON output)
- Token counting uses rough char/4 approximation, not actual tokenizer

**Verdict**: Token savings real but deploy-cautious. Security surface and bug load suggest evaluating in sandbox first, then potentially reimplementing highest-value strategies as native hooks.
