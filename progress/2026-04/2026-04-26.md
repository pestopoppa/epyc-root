### Implementation: parallel-warm via collective threadpool barrier (commit `43c65b926`)

Approach chosen over the loader-hook design: rather than registering tensors at model-load time and pre-allocating up-front, the new `ggml_ep_shard_warm_parallel()` function is called from the top of `ggml_compute_forward_mul_mat_id` by ALL threads of the calling op. On first encounter of each tensor:

1. ith==0 acquires the shard map mutex, allocates the anon-mmap buffer (mbind'd to current process's NUMA mask), inserts a `ShardEntry` with `ready=false`, releases the mutex.
2. `ggml_barrier(threadpool)` publishes the allocation to all threads of the op.
3. Each thread reads the entry under the mutex briefly, then computes its disjoint slice of the n_kept experts via `k = ith, ith+nth, ith+2*nth, ...` and memcpy's from `src0->data` to `buf` in parallel.
4. `ggml_barrier(threadpool)` ensures all chunks committed.
5. ith==0 atomically marks `ready=true`.

Steady-state cost is one mutex acquire per thread for the cache hit. Cold first-call cost: from ~250 ms single-threaded → ~250 μs across 96 threads (memcpy bandwidth scales with threads up to memory-system limit).

Files: `ggml/src/ggml-cpu/ggml-ep-shard.{h,cpp}` + 3-line call site in `ggml-cpu.c::ggml_compute_forward_mul_mat_id`. ShardEntry got an `std::atomic<bool> ready` field; existing lookup function refactored to use `try_emplace + assignment` because std::atomic isn't copy-assignable.

### REAP-246B with full stack — the headline test

Tested with all the relevant flag combinations:

| Config | Generation t/s | Δ vs baseline 6.89 |
|--------|----------------|---------------------|
| Baseline single-instance 96t --numa distribute | 6.89 | 100% |
| EP N=4 master-all-nodes + park + drone + shard + eager-warm (master 96t, workers 24t) | timed out at 600s, ~partial | — |
| **EP N=4 single-node-per-instance + drone + shard + eager-warm (24t per inst)** | **3.26 ± 0.01** | **−53%** |

The master-all-nodes + park config did NOT deliver the predicted +93%. Investigation found: master_park's "spinning parker" threads contend for cores with workers even when notionally idle. With master at 96t pinned to all 4 nodes (192 cpus including HT) and workers at 24t each on single nodes, the 72 master-parker threads are sched_setaffinity'd to all 192 cpus including worker territory — causing per-tensor warm cost to balloon from the predicted ~2 ms to ~20 seconds (OS scheduler thrashing on shared cores). Bench timed out before reaching steady state.

The simpler N=4 single-node-per-instance config — master pinned to node 0 alone, workers each to their own node, no master_all_nodes or master_park — avoids that contention. Eager-warm completed in ~minutes (vs ~hours for lazy-shard with 4 instances), and steady-state throughput came out at **3.26 t/s = −53% vs baseline**.

### Why the architectural prediction failed

The math predicted +93% from "master keeps full bandwidth for non-MoE + workers parallelise MoE". Two issues turned out to dominate:

1. **Master-parker thread contention** in the master_all_nodes + park config: 72 spinning master threads on cores that workers need. Even spinning-without-DRAM-traffic costs CPU cycles that interfere with workers' bandwidth-bound work. Real solution would be futex-based parking (deferred earlier as 4-6h work). Spin-park alone is insufficient.

2. **Cross-NUMA expert reads in single-node-per-instance config**: GGUF first-touched by master (pinned to one node) puts pages on that node's NUMA. Workers on other nodes cross-NUMA-read for their MoE work. Effective per-instance bandwidth is much less than node-local DDR, plus the 4 instances contend on master's node when reading. Net per-instance bandwidth is ~25% of baseline single-instance.

GGML_NUMA_WEIGHTS=1 + --numa distribute on baseline single-instance spreads the pages across all 4 nodes and threads across all 4 nodes simultaneously, achieving full system bandwidth. EP with single-node-per-instance loses this property without recovering compute parallelism (workers are bandwidth-bound, not compute-bound, on REAP-246B).

### Conclusion: EP doesn't help bandwidth-saturated >150B-class MoE on EPYC NPS4

Cross-validated:
- REAP-246B: −53% (N=4 single-node + eager-warm, this session)
- REAP-246B: −98% (N=4 master-all-nodes + park, would not finish)
- M2.7: −23% (yesterday's measurement)

The fundamental issue: when single-instance --numa distribute already saturates 100% of EPYC NPS4's aggregate DDR bandwidth, partitioning into N instances divides bandwidth without recovering useful compute parallelism. The crossover point on this hardware is ~150B total params.

**Synthetic bandwidth estimate**: Q4_K_M, A35B active params → ~17.5 GB of weight reads per token × 6.92 t/s ≈ **121 GB/s**. Aggregate DDR ceiling on EPYC 9655 NPS4 ≈ **460 GB/s** (12 channels DDR5-6000 at ~80% efficiency). Per-NUMA-node ceiling ≈ **115 GB/s** (12 ch / 4 nodes × ~38 GB/s/ch). So:

- **Aggregate utilization: 26-32%** — confirms NOT aggregate-bandwidth-saturated
- **Per-node utilization** depends on locality. Uniformly distributed: ~30 GB/s per node (well under 115). Unbalanced: could saturate one node

**Decision matrix per plan Phase D outcome:**

The combination of:
- **49.3 / 96 CPUs utilized** (half the threads blocked at any moment)
- IPC = 0.50 (memory or sync stalls)
- 13% sys time (kernel overhead, possibly NUMA migration)
- Memory bandwidth at only 26-32% aggregate

strongly suggests the bottleneck is **synchronization (barriers between ops + expert-loop serialization)**, NOT aggregate DRAM bandwidth. With 80 experts × top-8 active = 8 active experts per token across 96 threads = 12 threads per expert. Imbalance + barrier waits between ops → half the threads finish early and idle.

**Plan Phase N decision** (L3aaN reboot):
- L3aaN provides finer NUMA granularity (12 nodes × 1 CCD vs 4 nodes × 3 CCD), allowing per-CCD weight pinning. Helpful only if the bottleneck is **per-node BW saturation** or **cross-NUMA cache-line latency**.
- Phase D evidence points to **barrier/sync overhead**, which L3aaN cannot help. L3aaN would actually make sync coordination harder (more nodes → more cross-node coherence).
- **Recommendation: SKIP Phase N (L3aaN reboot) for now.** Real next lever is **CPU4 hierarchical sync primitive** to reduce barrier cost on REAP-246B-class workloads. Re-evaluate L3aaN only if CPU4 work also doesn't move the needle.

### P3: CPU1 stack instability — per-flag isolation

Tested each flag individually on Coder-30B Q4_K_M with `-r 5`:

| Config | Coder-30B Q4_K_M | Verdict |
|--------|-------------------|---------|
| canonical | 43.37 ± 0.10 | reference |
| +CCD_POOLS only | 43.44 ± 0.06 | **safe, stable** (+0.2%) |
| +NUMA_WEIGHTS only | **32.91 ± 22.18** | **UNSTABLE** — root cause |
| +CCD_WORK_DIST only | 43.66 ± 0.18 | **safe, stable** (+0.7%) |
| +BARRIER only | 43.88 ± 0.15 | **safe, stable** (+1.2%) |
| **3-flag stack (no NW)** | **44.15 ± 0.13** | **+1.8%, stable** |

Cross-confirmed on Qwen3.6-35B Q8_0:

| Config | Qwen3.6-35B Q8_0 |
|--------|------------------|
| canonical | 14.63 ± 0.01 |
| +NUMA_WEIGHTS only | 9.07 ± 3.91 |
| 3-flag stack (no NW) | 14.39 ± 0.03 |

**Smoking gun**: `GGML_NUMA_WEIGHTS=1` is the entire cause of the CPU1-stack instability. The other 3 flags work fine.

**Fix attempt (commit `8cb04da9d`)**: replaced process-wide `set_mempolicy(MPOL_INTERLEAVE)` with per-region `mbind()` after mmap. Correct scope fix (no longer leaks INTERLEAVE policy into KV cache, threadpool stacks, scratch buffers). However, **measurement after fix shows instability persists** — `MPOL_INTERLEAVE` itself behaves unstably on shared file-cache multi-NUMA hosts under fragmented memory, regardless of how the policy is applied.

**Hypothesis (unverified)**: shared file-backed mmap + INTERLEAVE policy + page cache state across processes creates non-deterministic placement. Each new process starts with whatever pages already live in the page cache (from prior runs of any process), and the kernel can only enforce policy on newly-faulted pages. Multi-rep within a single process also faults pages incrementally; placement evolves over the run, creating per-run variance.

**Decision**: `GGML_NUMA_WEIGHTS=1` is **deprecated** until a different mechanism is implemented. Candidates:
- Private anon mmap + custom file-load (complete control over page allocation)
- mlock + mbind + MOVE_PAGES for deterministic placement
- Per-tensor mbind (the "stripe" mode, never implemented)

**Remaining usable CPU1 stack** for production: `GGML_CCD_POOLS=1 GGML_CCD_WORK_DIST=1 GGML_BARRIER_LOCAL_BETWEEN_OPS=1` (NO NUMA_WEIGHTS). Stable, +1.8% on Coder-30B, parity on Qwen3.6-35B Q8_0. Worth shipping default-off as a research substrate; modest +1.8% production gain available via opt-in.

### P4: CPU4 hierarchical sync — design AND implement (NEGATIVE RESULT)

Implemented per [cpu-hierarchical-barrier.md](../../handoffs/active/cpu-hierarchical-barrier.md): moved CCD setup out of `#ifndef GGML_USE_OPENMP` so `GGML_CCD_POOLS=1` activates in OpenMP builds; modified `ggml_barrier` to use the existing 2-level CCD-aware sense-flip barrier when CCD state is initialized, with per-thread state via `tp->workers[omp_get_thread_num()]`.

Build green. Init logs `[GGML_CCD_POOLS] enabled: 12 CCDs x 8 threads/CCD (total 96, core_base=0)` — the new code path runs.

**Measurements (consistently net-negative):**

|-------|--------|-----------|--------------|
| Coder-30B Q4_K_M | + GGML_CCD_POOLS=1 (hier. active) | 40.25 ± 0.12 | -4.3% |

| REAP-246B Q4_K_M | + GGML_CCD_POOLS=1 + OMP_PROC_BIND=close | 5.06 ± 0.00 | -25% (binding catastrophic) |

**Why it doesn't win** (most likely → least): (1) libgomp's omp barrier is highly optimized; the 30% cycles in libgomp.so seen in Phase D `perf record` are NOT pure waste, much is productive scheduling. (2) Cache-locality assumption requires strict pinning; without it the hierarchical barrier's per-CCD cachelines are remote. (3) `OMP_PROC_BIND=close` itself regresses -7% on canonical — pinning interferes with libgomp's NUMA-aware scheduling. (4) The 49/96 idle threads is **structural MoE top-K imbalance**, not a barrier defect — no faster barrier can fix it.

**Disposition**: code reverted; `ggml-cpu.c` back to baseline. Design + measurements preserved in `cpu-hierarchical-barrier.md` for future reference.

**Strong negative result for the Q4_K_M sync-bound class.** No remaining software lever in this round. Future avenues not falsified but out of scope:

- Pre-shuffled MoE quantization layouts (offline tooling)

L3aaN stays closed off for the Q4_K_M class (confirmed sync-structural, not NUMA-locality).
