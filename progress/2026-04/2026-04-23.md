### Key design choices

- **Separate handoffs, not one umbrella**: TP sharding, system tuning, and GEMV ukernels each have distinct phase structures, prior art, and falsification conditions. Merging would bury the single-largest lever (TP) under config-knob audit work. Index document ties them together without conflating.
- **Baseline-first gate**: CPU3 Phase 0 (measure current BW utilization + barrier cost) gates all three handoffs. Cheap to do (days); decides which levers have headroom.
- **NPS mode as a shared dependency**: TP sharding's stretch gain (3.5–5×) and weight-replication's gain both need NPS4 or L3-as-NUMA. BIOS reboot work batched into system-tuning handoff Phase 2; cross-handoff coordination documented.
- **Composition explicit in index**: the CPU index includes a composition matrix showing CPU1/CPU2/CPU3/KV/Speculation are multiplicative and the 460 GB/s memory BW ceiling is the physical cap. Realistic stack (2.5× TP × 1.75× ukernel × 1.25× tuning = 5.5×) would saturate ceiling on most production models; further gains require reducing weights-read-per-token (KV, speculation, sparsity).

### Why this matters

Deployment has leaned hard on NUMA 4-way multi-instance aggregate throughput (6.7× frontdoor). This makes concurrent-session aggregate great but leaves single-session interactive use at 14.2 t/s on 30B-A3B — the user-visible slow path. The new backlog targets single-session specifically. If the TP lever delivers even the conservative end (2×), that's 14 → 28 t/s single-session on frontdoor; stretch (5× combined with ukernels) is 14 → 70+ t/s, matching what DGX Spark / RTX 3090 class systems deliver for this model size. The lever is credible because the bandwidth is physically there (460 GB/s socket) and currently only reached via parallel instances.

## Perf-gap analysis: Qwen3.6-27B at 4.8 t/s on EPYC 9655 — why

Session discussion produced a grounded explanation of why we see 4.8 t/s on Qwen3.6-27B Q8_0 while consumer GPUs (RTX 3090 ~70 t/s) and Mac Studio M-series (~60–70 t/s) report much higher on the same architecture:

| Platform | Mem BW | Q4_K_M BW-bound ceiling | Spec-dec (hybrid DeltaNet) | Typical achieved |
|---|---|---|---|---|
| EPYC 9655 | ~460 GB/s (2-socket effective) | ~30 t/s | 0.56× (dead on CPU) | **4.8 t/s Q8, ~25% of roofline** |
| RTX 3090 | 936 GB/s | ~62 t/s | 3–5× (DFlash+DDTree) | 70–207 t/s |
| Mac Studio M2 Ultra | 800 GB/s | ~53 t/s | ~1.3× (MLX parallel scan) | 60–70 t/s |
| DGX Spark GB10 | 273 GB/s | ~18 t/s | 5× verified | 91 t/s |

Two independent gaps, not one: **~2× bandwidth** vs consumer GPUs (unavoidable on CPU), and **~3–5× speculative-decoding** gap because DeltaNet sequential recurrence forces sequential verification on llama.cpp's CPU backend. The parallel-scan primitive on GPU/MLX is the hinge; CPU physically cannot match it on hybrid SSM. 27B dense + 75% DeltaNet is a pathological CPU case — 27B all-active weights AND sequential recurrence simultaneously.

**Documented for session reference, not persisted as deep-dive** — the analysis is captured in the new ukernel handoff's "Case for / case against" section and is traceable via the 35B-A3B vs 27B dense clarification recorded here.
