## NUMA_MIRROR Phase 1a + 1b (CPU25) — per-node pointer plumbing + TLS setter

Picked up from the Phase 0a/0b accessor migration (committed earlier this session as `9b1dbf4dd` and `b9920cc44`) and pushed into the actual Phase 1 framework on `feature/cpu-ep-inter-process` in `/mnt/raid0/llm/llama.cpp-experimental`.

### What landed

**Phase 1a — per-node pointer plumbing** (commit `ca39cb80a`):

- `struct ggml_tensor` gains `data_per_node[GGML_NUMA_MAX_NODES]` (default `GGML_NUMA_MAX_NODES = 8`) under `#ifdef GGML_NUMA_MIRROR`.
- `tensor_data(t)` now reads `t->data_per_node[ggml_current_numa_node]` with bounds check + fallback to slot 0 when TLS is unset.
- `tensor_set_data(t, p)` writes ALL N slots identically and keeps the raw `t->data` mirror synchronized for unmigrated callers.
- New `tensor_set_data_per_node(t, node, p)` API exposed for the upcoming Phase 1c per-node mmap path.
- `ggml_new_tensor_impl` populates `data_per_node[]` consistently with the raw `data` field.

Two debug iterations were needed:
1. Macro placement bug — `data_per_node[GGML_NUMA_MAX_NODES]` was used in the struct definition before `#define GGML_NUMA_MAX_NODES 8`. Fixed by hoisting the `#ifndef GGML_NUMA_MAX_NODES / #define / #endif` block to before `struct ggml_tensor`.
2. Positional struct initializer in `ggml.c`'s `ggml_new_tensor_impl` was missing the new field; added a `#ifdef GGML_NUMA_MIRROR` guarded `/*.data_per_node=*/ { 0 }` slot, plus a follow-up loop that writes the same `d` pointer into all N slots.

**Phase 1b — TLS setter at graph-compute entry** (commit `90a17af62`):

- Each compute thread, immediately after `set_numa_thread_affinity()` inside `ggml_graph_compute_thread()`, now calls `getcpu(2)` (with a `SYS_getcpu` syscall fallback for old glibc) and writes the resulting node index to `ggml_current_numa_node`. Both the OMP path (line 4092) and the pthread fallback (line 4108) funnel through this function.

### Phase 1c next — buffer-level mirror is mandatory

Hugepages are NOT provisioned on this host (`HugePages_Total = 0` for both 2 MB and 1 GB sizes, all 4 nodes). Phase 1c will use anon mmap + `mbind(MPOL_BIND)` on regular 4 KB pages (THP will opportunistically promote to 2 MB) — no kernel reboot needed.

Critical scope finding: a mmap-only mirror is insufficient. For Coder-30B Q4_K_M the model is ~17 GB total but **13.4 GB lives in CPU_REPACK** (CPU2's AVX-512BW 8x8 interleaved layout) and only ~4.3 GB remains in the original mmap. Mirroring just the file mmap leaves 79% of weight reads cross-NUMA and would not deliver the Phase 2 +25% gate. Phase 1c needs to mirror at the **buffer** level — both the mmap source AND the CPU_REPACK output.

Design notes recorded in `handoffs/active/numa-mirror-integration.md` (Phase 1c design section).

## NUMA_MIRROR Phase 1c (CPU25) — buffer-level mirror, DECISIVE NEGATIVE

Pushed straight into Phase 1c (per-NUMA-node weight replication, the actual mmap+mbind path) on `feature/cpu-ep-inter-process` in the same session. Implementation is correct and bit-exact, but the throughput gate failed.

### What landed

**Phase 1c** (commit `29a69599a`): CPU_REPACK buffer-level mirror.

- `ggml/src/ggml-cpu/repack.cpp`: per-buffer side-table (`std::unordered_map<ggml_backend_buffer_t, cpu_repack_mirror>` under a mutex) tracks N anon-mmap+mbind replicas per buffer. Allocator re-mbinds primary to MPOL_BIND-node-0 (instead of the existing MPOL_INTERLEAVE) and adds N-1 fresh `mmap(MAP_ANONYMOUS|MAP_NORESERVE) + mbind(MPOL_BIND, n)` replicas.
- `init_tensor` fans out `data_per_node[0..N-1]` to `(replica_n_base + tensor_offset)`. `set_tensor` (post-repack) copies the freshly-written tensor bytes from primary to each non-primary replica at the same offset, so all N hold byte-identical content. `free_buffer` munmaps replicas and `ggml_aligned_free`s the primary.
- Trigger decoupled from `ggml_is_numa()`: fires whenever `GGML_NUMA_MIRROR>=2` AND buffer ≥ 1 MiB. Compatible with the proper canonical (`numactl --interleave=all + OMP_PROC_BIND=spread + OMP_PLACES=cores`) — `--numa distribute` is NOT required and actually regresses the baseline.
- **Hot-path migration** (Phase 0a missed this file): `forward_mul_mat` and `forward_mul_mat_id` in `repack.cpp` were still reading `src0->data` / `src1->data` / `dst->data` / `data_ptr` directly (5 sites). Migrated to `tensor_data()`. Without this, Phase 1c regressed −45% from baseline because every thread hit primary on node 0.

### Validation

- PPL on Coder-30B Q4_K_M wiki.test, chunks 1-12, fa=1, --no-mmap, mirror=4: chunk1 = 7.4537, final = **11.1215**. Byte-identical to a clean `-march=znver5` baseline build (no mirror). The znver5-baseline rebuild after the migration also produced byte-identical PPL, confirming the migration itself is a no-op when MIRROR is off.
- Mirror correctly fires: `cpu-repack-mirror: 13.1 GiB primary on node 0 + 3 node replicas (mirror=4)`.
- One-shot per-thread debug log (since removed) confirmed each of 96 threads `getcpu()`s a CPU in its expected node range (cores 0-23 → node 0, ..., 72-95 → node 3) and writes the matching value to TLS `ggml_current_numa_node`.

### Throughput

Proper canonical: `OMP_PROC_BIND=spread OMP_PLACES=cores OMP_WAIT_POLICY=active taskset -c 0-95 numactl --interleave=all -t 96 -fa 1 -mmp 0`.

| Model | Quant | tg128 baseline (znver5) | tg128 mirror=4 | Δ |
|---|---|---|---|---|
| Coder-30B-A3B | Q4_K_M | 48.16 ± 0.15 | 47.66 ± 0.04 | **−1.0%** (noise) |
| Qwen3.6-35B-A3B | Q8_0 (tg64) | 23.30 ± 0.02 | 23.45 ± 0.02 | **+0.6%** (noise) |

**Phase 2 gate (≥ +25% on Coder-30B): NOT MET.** Mirror is within run-to-run noise.

### Root cause

Single-socket NPS4 EPYC 9655 is **DRAM-channel-bound**, not fabric-bound, at 96-thread saturation. Per-thread BW share is 460 GB/s / 96 = 4.79 GB/s/thread regardless of NUMA placement. With mirror, each NPS4 node has 3 channels (115.2 GB/s) and 24 threads → 115.2 / 24 = identical 4.79 GB/s/thread. There is no fabric headroom to recover.

CPU24's perf-record finding ("compute kernels memory-stalled INSIDE on cross-NUMA loads at 96 threads with 4.8 GB/s/thread BW share") was correctly identifying memory-stall but could NOT distinguish fabric-stall from DRAM-channel-stall. Phase 1c cleanly rules out the fabric-stall hypothesis on this hardware.

The vproxy-tools fork's reported gains (+62% QwQ-32B FP16, +34% DeepSeek-R1 671B Q8) were measured on **two-socket** 2× EPYC 9275F configurations where cross-SOCKET fabric (going through inter-package coherence links) is significantly slower than DRAM channels and IS the binding constraint. Single-socket NPS4 has intra-package fabric (Infinity Fabric inside the same package) that is fast enough that DRAM channels saturate first.

### Disposition

- `GGML_NUMA_MIRROR` is left default-OFF (compile-time gated). Production stack should NOT enable it.
- The accessor migration (Phase 0a/0b/1a/1b) is preserved as zero-overhead infrastructure. `tensor_data()` compiles to direct field access in default builds.
- The buffer-level mirror (Phase 1c) infrastructure is preserved for any future hardware where fabric IS the binding constraint (2-socket EPYC, multi-package, GPU-CPU NUMA).
- **Reopen only if a 2-socket configuration becomes relevant.** This is the largest concrete throughput lever from the CPU24 analysis; if even this doesn't deliver, the software-level CPU optimization runway on this hardware is materially exhausted.
