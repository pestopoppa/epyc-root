## Session 4 — Fix architect_general Think-Loop: Drop --jinja Flag

### Problem

architect_general (Qwen3.5-122B-A10B hybrid SSM+MoE) continued entering infinite `<think>` block loops during autopilot operation, despite `--reasoning off` server flag (commit 0591952). User observed the loops live in the TUI tap.

### Root Cause

The `--jinja` server flag loads Qwen3.5's native chat template, which includes `<think>`/`</think>` block scaffolding. The template itself primes the hybrid model into think mode **before** the `--reasoning off` flag can suppress it. The `--reasoning` flag acts downstream of template application — by the time it intervenes, the model has already seen the thinking scaffolding and enters the loop.

## Session 5 — GPU Handoff Update: vLLM DDTree+Dflash Speculation Plan

### Problem

User saw a community benchmark posting: DDTree + Dflash on vLLM with Qwen3.5-27B AWQ achieving **91.08 tok/s accepted** (96.4% acceptance rate) on a DGX Spark GB10. Needed to understand the setup (what "draft model" is used) and persist the optimization opportunity in the GPU acceleration handoff before it gets forgotten.

### Key Finding

**No separate draft model.** The setup uses:
- **DDTree** — tree-based multi-candidate verification strategy
- **Dflash** — block diffusion drafting (generates candidate token blocks via iterative denoising, conditioned on target model hidden states)

This reopens speculative decoding for Qwen3.5 hybrid models **on GPU only**. Our exhaustive CPU testing (DFlash, MTP-1, tree speculation — all concluded not viable) failed because Delta Net recurrent layers process tokens sequentially regardless of batch size. On GPU, parallel scan handles the recurrent state, removing the bottleneck.

Our own research already validated this direction: DFlash paper reports τ=6.49 on Qwen3.5-35B-A3B (GPU), and our C++ forward pass was verified correct (hidden states match HF to <0.01). The problem was never the algorithm — it was CPU sequential verification cost.

## Session 2 — Learned Routing Controller: Design + Phase 1 Implementation

### Problem

The orchestrator's routing system uses episodic KNN retrieval (FAISS + Q-ranking, 10-50ms/request) for every routing decision, even obvious ones. The existing `RoutingClassifier` MLP was feature-flagged OFF and trained on only 94 memories (FAISS index had been rebuilt, losing 175K embeddings). The 175K episodic memories had ~30 distinct action labels that needed normalization to ~5 clean routing targets.

### Design Discussion

Explored whether routing intelligence could become a learned neural controller. Key conclusions:

- **MLP on BGE embeddings** is the right Phase 1 architecture (~200K params, <1ms CPU inference)
- **Episodic memory becomes write-only** during inference — shifts from runtime query target to experience replay buffer for periodic retraining
- **Hidden-state probing on frontdoor** is viable Phase 2 — but SSM hybrid (Jamba) requires probing only attention layers, mean-pooling across positions (not last-token)
- **Phase 1.5 logit probe** validates the "piggyback on frontdoor" concept before llama.cpp fork changes
- **Decision surfaces**: role selection, mode selection, escalation prediction, context injection budget, multi-turn budget. Spec-decode excluded (hardware-bound, not task-dependent)
- **Independent models per surface**, not shared trunk — routing has 174K labels, others have 10K or zero
- **SkillBank is complementary**: SkillBank = "what model should do", MLP = "which model does it"
- Existing infrastructure discovered: `routing_classifier.py`, `train_routing_classifier.py`, `extract_training_data.py`, HybridRouter fast-path (line 767), feature flag — all pre-existing
