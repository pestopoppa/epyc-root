## Session 7 — Answer-Tag Stop Sequences (Reasoning Compression Action 16)

### Problem

During specialist routing benchmarks, models waste massive amounts of think tokens on post-answer rumination. After reaching the correct answer and emitting `#### September 1, 2013`, the model continues with "Wait, let me check... Actually... Let me verify..." loops that generate zero new insight but burn generation budget.

Root cause: the `####` answer marker was in the prompt but NOT configured as a stop sequence. The model had no signal to stop generating after the answer.

Naive fix (stop on `####`) would halt generation *before* the answer content. Required a format change.

### Solution

Replaced `####` marker with XML `<answer></answer>` tags across all benchmark prompts. The stop sequence `</answer>` fires *after* the answer content is fully emitted.

**Flow**: Model generates `...reasoning... <answer>September 1, 2013</answer>` → `</answer>` triggers stop → zero post-answer tokens wasted → scorer extracts via `<answer>(.*?)</answer>`.

### Changes

| Component | Change |
|-----------|--------|
| `dataset_adapters.py` | All 7 prompt instructions: `"after ####"` → `"inside <answer></answer> tags"`. All extract patterns updated. |
| `direct_stage.py` | `</answer>` added to default stop sequences. Request-level `stop_sequences` merge support. |
| `debug_scorer.py` | Default patterns updated to `<answer>` extraction. Legacy `####` fallback preserved for backward compat. |
| `requests.py` | `stop_sequences: list[str] | None` field added to `ChatRequest` |
| `seeding_orchestrator.py` | `stop_sequences` param plumbed through `call_orchestrator_forced` and `_call_orchestrator_with_slot_poll` |
| `sentinel_questions.yaml` | All 9 `####` references updated to `<answer></answer>` format |

### Expected Impact

- **Direct mode**: ~50-80% reduction in wasted tokens on scored benchmarks (all post-answer rumination eliminated)
- **REPL mode**: Prompt change helps model structure output; REPL stopping still via FINAL() mechanism
- **Non-benchmark requests**: Zero impact — `</answer>` stop sequence never fires if tags aren't in the output
- **Backward compatibility**: Scorers fall back to `####` extraction for any in-flight results with old format

## Session 10 — WS-3 Root Cause Fix + Arm B Checkpoint Analysis + A' Launch

### WS-3 Was Not Firing

Arm B checkpoint analysis (104/400 questions from JSONL) revealed **100% web search rate** in REPL mode across all 8 suites — WS-3 cascading tool policy was not denying web tools.

**Root cause**: `routing.py:56` hardcodes `task_type: "chat"` in `task_ir`. The WS-3 check in `repl_executor.py:299` compares `task_type` against `NO_WEB_TASK_TYPES` (`math`, `coder`, `thinking`, `instruction_precision`), but since it's always `"chat"`, the `no_web` context is never set. All WS-3 plumbing (tool registry, REPL environment, policy layers) was correct — the source signal was missing.

**Fix**: Added role→task_type derivation in `routing.py` after routing completes:
- `worker_math` → `"math"`
- `coder_escalation` / `coder_primary` → `"coder"`
- `thinking_reasoning` → `"thinking"`

When these roles are selected, `task_ir["task_type"]` is updated, which flows through to the `NO_WEB_TASK_TYPES` check and sets `no_web=True` in the REPL tool context.

**Scope**: Fixes production routing path. Seeding eval with forced `frontdoor:repl` won't trigger the deny (correct — frontdoor users should have web access). 32/32 tool policy tests pass.
