## Hybrid SSM follow-up on Qwen3-Next-80B-A3B

Quick 1-model × 3 configs × 3 shapes × n=5 follow-up to test whether Probe B's +8.9% pp512 c3 finding on Nemotron-9B (dense Hybrid SSM) generalizes to Qwen3-Next-80B-A3B (MoE Hybrid SSM).

### Result — DOES NOT GENERALIZE

| Shape | Nemotron c3 | Next-80B c3 |
|---|---|---|
| tg64 | +2.1% | +0.4% |
| pp512 | **+8.9%** | **+1.7%** |
| pp2048 | +3.7% | +0.9% |

Direction is preserved (positive) but magnitude is uniformly ~5× smaller on Next-80B. The MoE structure (mul_mat_id dispatcher with per-token-varying active expert subset) defeats CPU1's CCD-aware partitioning advantage.

### Revised deployment recommendation

| Hybrid SSM sub-class | Recommendation |
|---|---|
| Dense Hybrid SSM (Nemotron-class) | c3 (CPU1+mbind-off) opt-in for prefill-heavy workload |
| MoE Hybrid SSM (Qwen3-Next-80B-A3B) | Default v5 stack — c3 effect is within practical noise floor |

The "Hybrid SSM benefits from c3" generalization from Probe B was overstated. It's actually a "dense-arch + barrier-heavy + bandwidth-bound prefill" pattern, not a hybrid-SSM pattern per se.

Decay is ALiBi-style `(2^-0.5)^h` slopes for `n_heads=16`, scaled per-linear-layer by `1 - (l-1)/(L-1) + 1e-5`, sign-flipped. ## markdownfs intake deep-dive — intake-520 corrected, three patterns extracted

User trigger: `/research-intake https://github.com/subramanya1997/markdownfs`, then "deep dive the intake please. There must be more worthwhile information."

### What changed

The original intake-520 entry was based on a single shallow README pass and contained two factual misreads. The deep dive read the full source surface — README, all 11 docs, `Cargo.toml`, `tests/perf_comparison.rs` (1,035 lines), GitHub API metadata, full commit history — and corrected them.

### Two architectural caveats the original intake missed

1. **MCP server runs as `uid=0` root with NO per-user authentication.** Documented inside the project at `docs/mcp-guide.md` ("All MCP operations run as root (uid=0, gid=0). There is no per-user authentication within the MCP protocol — the agent has full access") and acknowledged as a known follow-up in `docs/demo-readiness.md`. The wheel/agent-token/chmod permission model is **HTTP-only**. The "user-to-agent permission delegation" framing in the README and in our original intake is aspirational at the MCP boundary.
2. **Single-writer per `state.bin`.** CLI + HTTP + MCP cannot share a workspace concurrently — multi-process must funnel through the HTTP server. So mdfs cannot host multiple agents on a shared workspace via MCP alone. Documented in `docs/mcp-guide.md`. The "shared durable memory across agents" framing is structurally not implemented.

### The 102.8× speedup decoded

Reproducible (real benchmark in `tests/perf_comparison.rs`) but methodologically biased: native FS in the bench uses `std::env::temp_dir()` which is `/tmp` = tmpfs (RAM) on Linux, so both sides are RAM. The number measures kernel VFS + syscall overhead saved (per-call dentry/inode/syscall round-trip vs in-process `BTreeMap` mutation), NOT in-memory-vs-disk. Average is unweighted arithmetic mean across 16 ops; per-op character — hot-cat 100–1000×, touch/stat 50–500×, large writes 2–5×, grep/find 3–20×. Irrelevant for any LLM-inference-bound workload (FS syscalls nowhere near saturated at agentic cadence).
