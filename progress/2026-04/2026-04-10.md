## Session 1 — Answer-Tag Port to epyc-inference-research

### Problem

The `#### → <answer></answer>` migration (Reasoning Compression Action 16, 2026-04-09) only landed in `epyc-orchestrator`. The research repo's `dataset_adapters.py` and `debug_scorer.py` — used by the 3-way routing eval and all standalone research benchmarks — still instructed models to format answers with `####`. Since the orchestrator's `direct_stage.py` fires `</answer>` as a stop sequence, models receiving `####`-format prompts from the research repo never triggered early stopping, continuing to waste tokens on post-answer rumination.

Discovered during a live 3-way eval run where model output showed "ensure the final answer is after ####" in the inference stream.

### Root Cause

The two repos have independent copies of `dataset_adapters.py` and `debug_scorer.py` (no cross-imports). The research copy has diverged significantly (additional adapters for long-context, physics, AIME, OlympiadBench, etc.), so the orchestrator commit (`2876a53`) couldn't be cherry-picked — it required manual porting.

### Key Findings

**GPU-specific (future):**
- rocWMMA enables flash attention via `-DGGML_HIP_ROCWMMA_FATTN=ON` — up to 2x prefill speedup. ONLY for prefill, NOT decode.
- hipBLASLt grouped GEMM is highest-impact for MoE models (29% improvement, 10x API call reduction).
- CPU+GPU hybrid MoE expert offloading (`-ot "exps=CPU"`) is the most promising GPU path — leverages existing NUMA infrastructure.
- AITER achieves 17x MLA decode, 14x MHA prefill, 3x fused MoE on MI300X (vLLM/SGLang only, no llama.cpp).
- RDNA3 performance cliff: decode regresses 58% at 65K ctx. Community fix exists but not mainlined.

**CPU-relevant (actionable now):**
- Decode phase budget: 85-92% weight GEMMs (memory-BW), 7-12% attention (short ctx), 25-35% (long ctx), >50% (very long). Validates NUMA 4-way strategy.
- Multi-instance (TP1×8) independently validated by AMD's MI300X benchmarking team — mirrors our NUMA 4-way.
- Attention transitions to compute-bound at long sequences (16K+) — KV cache compression has throughput impact at long context, not just memory.
- Per-shape GEMM tuning (TensileLite) shows 1.6-2.6x for decode-phase matrices — unexplored on CPU via AMX/AVX-512.

## Session 5 — v3 Spec Decode Bug Fix (Qwen2.5-Coder-32B)

### Problem

Speculative decoding with any draft model on Qwen2.5-Coder-32B returned `HTTP 500: Invalid input batch` on every prompt on the v3 binary. Coder_escalation (primary escalation path) was running without spec decode as a workaround.

### Root Cause

Tree speculation multi-path verification (default `p_split=0.1`) adds alternative draft paths to the batch with seq_ids computed as `n_parallel + slot.id * 8 + k`. With `-np 1`, these seq_ids (1, 2, 3...) exceeded `n_seq_max = 1`. The auto-configuration that bumps `n_seq_max` and enables `kv_unified` was gated on the user having explicitly passed `--kv-unified`, leaving the default path broken.

Debug log evidence: `init: invalid seq_id[3][0] = 2 >= 1`

**Not architecture-specific** — affects all models with draft + default p_split when `kv_unified=false`. The original isolation tests for Qwen3/Qwen3.5 likely used different configurations.

### Fix

Auto-enable `kv_unified` and bump `n_seq_max` when tree speculation is active (draft model + `p_split > 0`), removing the requirement for users to manually pass `--kv-unified`. The existing `has_recurrent` guard at the verification site prevents tree multi-path from firing on hybrid/SSM models, so `kv_unified=true` is safe to set unconditionally.

**File**: `tools/server/server-context.cpp` (load_model, ~line 675)
