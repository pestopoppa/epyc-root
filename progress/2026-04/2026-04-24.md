## Summary

Task #10 (96t-single-NUMA-node production sweep) expanded into a full concurrent-split study after user prompts to try 8×24t, 16×12t, 32×6t, 48×4t. Single-session latency paths remain on 1×48t or 1×96t single-instance configurations.

## Why the concurrent split works

Hypotheses consistent with Phase-0 perf data:
1. **Barrier cost is O(threads-per-instance)**: perf showed 32-45% of cycles in libomp barriers at 96t. 48 small barriers in parallel beat 4 large ones.
2. **CCD locality**: EPYC 9655 has 8 cores per CCD × 12 CCDs = 96. 4-thread instances use ≤1 CCD entirely; 48-thread instances span 6 CCDs with cross-CCD coherence traffic.
3. **Page cache coherence**: all instances mmap the same GGUF → shared weight reads, no extra memory pressure.
4. **BW channel interleaving**: finer-grained instance competition gives better DDR5 channel time-division.

For dense Q4 models (smaller compute/BW ratio), instances become compute-bound before BW-bound at 4t → 32×6t is the peak. For Q8 models (more compute per byte), instances stay BW-bound further → 48×4t or possibly finer.

## Late addition — CPU1 Phase 0 microbench validation

After the concurrent-split sweep user-requested pivot: "ok 4x48t multi-instance findings are great but what I really care about is getting aggregate throughput numbers on a single instance ... proceed with the sharding technique".

Wrote a standalone C++ microbench (`/mnt/raid0/llm/cpu-tp-prototype/tp_gemv_bench.cpp`) to validate the core CPU1 pattern before committing to the week-scale ggml integration. Compares 5 thread-pool structures on single-threaded GEMV:

| Mode | Description |
|---|---|
| A | Flat OpenMP 192t, ephemeral team per call (current ggml pattern) |
| C | Flat 96t ephemeral |
| D | Flat 96t persistent (region hoisted outside iter loop) |
| E | Flat 192t persistent |
| **B** | **TP 12×16: 12 pthreads pinned per-CCD + inner 16t OpenMP teams** |

At DRAM-bound 15 GB GEMV:
- A (flat 192 ephemeral): 74.82 ms/iter, **204 GB/s** — baseline matching ggml pattern
- E (flat 192 persistent): 74.58 ms/iter, 205 GB/s — `#pragma omp barrier` instead of team-respawn
- **B (TP 12×16): 70.82 ms/iter, 215 GB/s** — **+5% over flat persistent, +14% over flat ephemeral**

Numerical validation: bit-exact across all modes.

### Key findings

1. **TP 12×16 pattern WORKS** on DRAM-bound GEMV. +5% to +28% over flat OpenMP at realistic working-set sizes. Not the 2-5× the handoff projected for end-to-end decode — but a single-op microbench systematically under-estimates decode gains because the chain of ops compounds barrier savings across layers.

2. **Persistent team hoisting alone captures most of the "easy" gain** (flat-persistent = 205 vs flat-ephemeral = 184 = +11%). This suggests the CPU1 Phase 1 work is naturally splittable:
   - **Phase 1a**: persistent thread pool (replace ephemeral per-op team with one long-lived team + barriers). Smaller surgery (~1 day), likely +10-15% end-to-end.
   - **Phase 1b**: add per-CCD structure on top of Phase 1a. ~1 week; additional +3-10% end-to-end.

3. **192t is fine when persistent**. Mode E at 204 GB/s ≈ Mode B at 215 GB/s. Real decode's 192t losses come from the ephemeral spawn pattern, not from 192 threads per se. This suggests even simpler wins are available before full TP sharding.

### Revised Phase 1 plan (after ggml source inspection)

Initially I hypothesized a Phase 1a "persistent thread pool" (easy win) + Phase 1b "per-CCD structure" (harder). **Inspection of `ggml/src/ggml-cpu/ggml-cpu.c:467-503` reveals ggml already has a persistent threadpool** (`struct ggml_threadpool` with long-lived workers, `n_graph`/`n_barrier` atomic coordination, per-thread `cpumask` for affinity). So Phase 1a doesn't exist — the "flat OpenMP ephemeral vs persistent" gap in the microbench is a comparison of OpenMP runtime behaviors, not ggml's.

The 32-45% barrier cost in yesterday's perf profile is therefore **ggml's flat 192-way spin-wait on `n_barrier_passed`**, not team-respawn overhead. Phase 1 becomes a single task again:

- **Phase 1 — Per-CCD sub-team structure in ggml** (~1 week): replace the flat 192-way threadpool with 12 CCD-pinned sub-pools (16 threads each). Each sub-pool has a local barrier. Work distribution per op routes shards to sub-pools. For `GGML_OP_MUL_MAT` column-shard across output N; for `GGML_OP_MUL_MAT_ID` shard experts across CCDs. Cross-CCD coordination only when output dim is smaller than 12 or when attention ops need cross-head reduction.

  Key code sites to modify:
  - `ggml/src/ggml-cpu/ggml-cpu.c:467` — `struct ggml_threadpool` — add sub-pool array
  - `ggml/src/ggml-cpu/ggml-cpu-impl.h:532` — `ggml_barrier()` — add per-sub-pool variant
  - `ggml/src/ggml-cpu/ggml-cpu.c:~620` — `ggml_barrier` implementation — split flat counter into per-CCD + cross-CCD tree
  - Thread launch / cpumask application — set per-worker CCD affinity at pool init

  Gate: bit-exact output vs existing ggml on 5-model smoke test; ≥1.3× single-instance decode speedup on 30B-A3B Q4.

- **Env-var flag**: `GGML_CCD_POOLS=1` enables the sub-pool structure. Default off so production stays untouched during A/B testing.

**Note on microbench transferability**: the +5-14% gain measured on a standalone OpenMP microbench is a lower-bound on the ggml-integrated gain. Real decode has many more barriers per token (dozens of ops × 64 layers), so the cumulative barrier-cost savings is larger than a single-op microbench shows. But microbench *structural* validation — CCD-pinned + sub-team barrier reduces end-to-end GEMV time at DRAM scale — does transfer.

### Why NPS4 ≠ NPS2 even with `--interleave=all`

The 12 memory channels are physically unchanged. Software interleave does route pages round-robin, yet the recovery is only partial. Three reasons:

1. **Remote-access ratio grows.** Under NPS4 + interleave, each thread pulls 75% of pages from remote nodes (3 of 4 are non-local). Under NPS2 + interleave it was 50%. That's a 1.5× hit on hop-latency cycles at identical BW.
2. **Within-node channel stripe shrinks from 6 to 3.** A single-thread sequential burst gets narrower local BW; interleave only relocates the *next* page's source, not the current page's channel count.
3. **Directory coherency overhead scales with domain count.** EPYC uses directory-based coherency; 4 domains means more snoop/lookup paths than 2.

Verified: `numa_balancing=1` did NOT help (20.42 t/s vs 21.03 with it off) — migration overhead during active decode costs more than it saves.

### Decision gates (runbook Step 6)

Two gates fired simultaneously with opposite verdicts:
- ✅ **CPU1 Phase 1.0+1.1 delivers +12.5%** → gate for "proceed to Phase 1.2/1.3".
- ❌ **Multi-instance 4×48t production regresses −26%** → gate for "rollback to NPS2".

Four paths forward (awaiting user decision):

1. **Rollback to NPS2.** Safest. Loses the Phase 1.0+1.1 gain and the 48×4t peak.
2. **Stay on NPS4 + implement Phase 1.3 (NUMA weight mbind, 2–3 days).** Expected to close much of the single-instance gap by forcing true per-node locality with weight replication.
3. **Stay on NPS4 + reconfigure production to 48×4t concurrent.** Extracts new peak (104 t/s aggregate) immediately, but single-user latency roughly doubles.
4. **Reboot to L3-as-NUMA (12-way).** Further explores NUMA granularity; likely worsens single-instance and may or may not help concurrent.

## Session 4 — Phase 1.3 implementation (NUMA-aware weight mmap via set_mempolicy)

User approved Option 2: stay on NPS4 and implement Phase 1.3. First iteration implements the simplest lever — process-wide `MPOL_INTERLEAVE` set via `set_mempolicy` before mmap, env-gated by `GGML_NUMA_WEIGHTS=1`. This is equivalent to running `numactl --interleave=all` but inline, so individual llama.cpp instances can opt in without a wrapper. Intended to interact correctly with kernel file-readahead (which bypasses per-region `mbind` but honors process mempolicy).

### Implementation

File: `src/llama-mmap.cpp`

- New code block inside `llama_mmap::impl::impl(file, prefetch, numa)`, executed before the `mmap()` call when `GGML_NUMA_WEIGHTS != 0`.
- Counts NUMA nodes by enumerating `/sys/devices/system/node/nodeN`.
- Calls `syscall(SYS_set_mempolicy, MPOL_INTERLEAVE, mask, 64)` with a mask covering nodes 0..n−1.
- Also suppresses `MAP_POPULATE` so pages fault in under the new policy rather than pre-faulting under the parent process's default.
- Bit-exact with prior behavior when the env var is unset or `=0`.

First-draft attempt used per-region `mbind()` on the mapped address; that failed because file-backed MAP_SHARED pages don't honor per-region mbind once the kernel's readahead thread has queued them (reproducibly placed most of a 17 GB file on one node). Switching to process-wide `set_mempolicy` before mmap fixed it — pages distribute evenly across the 4 nodes (verified via `numastat -m`: 4566 / 4641 / 4531 / 4609 MB).

### Results (Qwen3-Coder-30B-A3B Q4_K_M, 96t, `-p 0 -n 64 -r 3`, fresh cache)

| Config | NPS4 t/s |
|---|---|
| (A) noOMP flat | 14.53 |
| (B) noOMP + `GGML_CCD_POOLS=1` | 15.45 |
| (C) noOMP + `GGML_NUMA_WEIGHTS=1` | **34.84** |
| (D) noOMP + `GGML_CCD_POOLS=1` + `GGML_NUMA_WEIGHTS=1` | **39.59** |

- Phase 1.3 alone lift: **+140%** (A → C)
- CPU1 Phase 1.0+1.1 on top of Phase 1.3: **+13.6%** (C → D)
- Combined lift over baseline: **+156%** (A → D)
- **88% of NPS2 pre-reboot best** (NPS2 CPU1 Phase 1.0+1.1 = 44.85; gap closed from −38% to −12%)
- Inline `set_mempolicy` is also **+16% better than external `numactl --interleave=all`** (noOMP+CCD: 39.59 vs 27.86 with numactl). Hypothesis: the external numactl policy interacts differently with the llama-bench fork/exec and initial thread creation path.

### Why Phase 1.3 jumps so much vs prior numactl-interleave runs

Prior re-bench under `numactl --interleave=all` got 27.86 with noOMP+CCD — today's Phase 1.3 path gets 39.59 with the same build. The difference is that `numactl` sets the policy on the process at exec time, but llama.cpp's mmap code path sets `POSIX_FADV_SEQUENTIAL` and uses `MAP_POPULATE` which populate pages through the kernel readahead thread — and *that* thread's policy may not inherit the user policy reliably. Disabling `MAP_POPULATE` inside the mmap constructor (done as part of this patch) lets the reshaped per-task policy govern all placement decisions, including readahead's.

### Decision

Keep NPS4. NPS4 + Phase 1.3 + CPU1 Phase 1.0+1.1 = 39.59 t/s single-instance (88% of NPS2 baseline) while preserving the 48×4t concurrent peak of 104 t/s. The net trade is −12% single-instance vs +100%+ concurrent aggregate — clear win given user supports orchestrator rework for concurrent deployment.

## Session 5 — Phase 1.2 (CCD work distribution) + Phase 1.3 'local' mode scaffolding

User approved continuing into Phase 1.2 + Phase 1.3 v2. Committed as `61b00eb53` on `cpu-optimization/backlog-2026-04-23`.

### Phase 1.2 (ggml-cpu.c: `ggml_compute_forward_mul_mat`)

Added CCD-block-contiguous work distribution path. Gated by `GGML_CCD_WORK_DIST=1` (requires `GGML_CCD_POOLS=1`). Under `GGML_USE_OPENMP=off` only.

Each thread computes a contiguous sub-range:
- CCD c gets `[c * big / n_ccd, (c+1) * big / n_ccd)` of the larger output dim
- Thread ith within CCD c gets a strided slice of that range
- Falls through to the default chunked path when disabled or when the matmul is too small

Bit-exact with prior behavior when env unset.

### Phase 1.3 `local` mode (llama-mmap.cpp)

Added alongside the existing `GGML_NUMA_WEIGHTS=1` (interleave) mode:

- `GGML_NUMA_WEIGHTS=local`: disables `MAP_POPULATE` and sets `POSIX_FADV_RANDOM` so pages fault on demand under the touching thread's default LOCAL policy. No global `set_mempolicy`.

### Measurements (NPS4, Qwen3-Coder-30B-A3B Q4, 96t, `-p 0 -n 64 -r 3`)

| Config | t/s (warm run) |
|---|---|
| Phase 1.0+1.1 + NW=1 (reference) | **40-41** |
| Phase 1.0+1.1 + Phase 1.2 + NW=1 | **40.18 – 40.96** |
| Phase 1.0+1.1 + Phase 1.2 + NW=local (no warm-up) | 18.4 – 18.6 |

**Phase 1.2 is net neutral under page-interleave**. Static partitioning loses some of the work-stealing benefit of the default chunk-steal loop; the per-node locality gain is zero when weights are 4 KB-interleaved across 4 nodes.

**`NW=local` without a warm-up touch pass fails to distribute weights**. `numastat` showed **~15 GB on node 0 alone, ~1 GB on each of nodes 1/2/3** — the main thread that runs init_mappings and model-load pulls most of the file into cache before CCD threads start running inference, placing all those pages on its node.

### Why Phase 1.2 and `local` don't yet deliver standalone gains

Phase 1.2 requires per-node weight locality to benefit. Interleave doesn't provide locality. `local` mode would provide locality *IF* each CCD's threads were the first to touch their assigned rows — but in practice the main init thread touches them first.

Both mechanisms are correct foundations. The missing piece is a **per-CCD pinned warm-up touch pass** that first-touches weights from the correct thread/node.

### Phase 1.3 v2 proper (design for next session)

1. After `init_mappings()` completes AND when `GGML_NUMA_WEIGHTS=local` + `GGML_CCD_POOLS=1` are set:
2. Read CCD geometry (same env vars the pool uses).
3. Spawn N pinned pthreads, one per CCD, each bound to that CCD's CPU set + NUMA node (`sched_setaffinity` + `numa_run_on_node` equivalent).
4. Each thread iterates `weights_map`. For each tensor it reads one byte from every page in row range `[c*N/n_ccd, (c+1)*N/n_ccd)` — matching Phase 1.2's partitioning.
5. First-touch places those pages on the thread's NUMA node.
6. Threads exit; model is now per-node-cached for all subsequent inferences.

Estimated effort: 1-2 days. Expected: 44+ t/s single-instance, closing or exceeding NPS2 baseline.

- **`GGML_CCD_POOLS=1 GGML_NUMA_WEIGHTS=1` (Phase 1.0+1.1 + Phase 1.3 v1)**: **40-41 t/s single-instance**, **89% of NPS2 baseline**.
- **48×4t concurrent NPS4-native**: 104 t/s aggregate — new peak, still orchestrator-rework-pending.

## Session 7 — CCD pool cpuset fix (concurrent hang debugged)

Discovered the CCD-pool concurrent hang root cause: Phase 1.1's
CCD-cpumask pinning hardcoded core IDs assuming cpus 0-95 all available.
Under `taskset -c 24-47` (e.g. for instance 1 of a 4-way concurrent), the
pinning targeted core 0 with no intersection → `Cpus_allowed_list: 0`
→ all 24 worker threads piled onto core 0 → deadlock-like stall
(2.6 GB RSS growth, 610% CPU on 1 core, zero progress).

**Fix** (commit `0ade7bd4d`, tightened in `69b4c3fa4`): at
`ggml_threadpool_new_impl`, read `sched_getaffinity` to detect the
process's cpuset. Compute every pin target we'd emit and verify all are
`CPU_ISSET`. If the cpuset doesn't fit cleanly (e.g. non-contiguous
phys+SMT ranges under NPS4-node cpusets), disable core pinning while
keeping the 2-level barrier. New threadpool field: `int ccd_core_base`
(0 = default, first-allowed-core for offset, -1 = no pinning).

New threadpool struct field (minor ABI change contained to fork):
`ccd_core_base` between `ccd_threads` and the sync state.

Post-fix aggregate throughput (NPS4, 30B-A3B Q4, CCD+NW=1 + per-node

## Session 8 — Phase 1.4 v1 (axis-0 partitioning + safe CCD-local barrier)

PPL-verified safe Lever B via per-op partition alignment.

**What was tried first (all UNSAFE, PPL 10.8-11.1 vs baseline 9.43)**:

**Root cause of unsafe heuristics** (diagnosed via PPL + code trace):
  1. MUL_MAT and ADD pick partition axis independently. In prefill,

fallback in ADD/MUL/SCALE/UNARY f32 when `ggml_nrows == 1 && ne0 > nth`.
Every thread takes `ne0/nth` contiguous axis-0 elements, partition
exactly matches MUL_MAT Phase 1.2. Between-op barrier downgrades to
CCD-local only when next op is decode-shape pointwise

## Session 9 — op fusion attempt (partial)

Built ggml infrastructure for MUL_MAT + residual ADD fusion. Graph-side
NOT enabled due to memory aliasing.

**Landed** (commit `b2154f3f3`):
  - New API: `ggml_mul_mat_add_residual(ctx, a, b, residual)` in
    `ggml/include/ggml.h` + `ggml/src/ggml.c`. Builds a MUL_MAT with

the same pointer (0x634608adce40). Retroactive fusion tried to read
inpSA WHILE writing cur to the SAME buffer → in-place race → NaN.

**Real fix (deferred)**: build fused op at graph construction, not via
post-hoc mutation. Requires `build_attn` to expose the o-projection
weight OR restructuring qwen3moe.cpp to build attention manually and
call `ggml_mul_mat_add_residual` directly. Deeper surgery (hours).
