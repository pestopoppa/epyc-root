### Model-Specific Issues Discovered and Fixed

| Model | Issue | Root Cause | Fix |
|-------|-------|-----------|-----|
| All Gemma4 | Empty output (1000000 tps) | `/completion` endpoint doesn't apply chat template; Gemma4 needs it | `use_chat_api: true` → routes to `/v1/chat/completions` |
| All Gemma4 | Degenerate repetition (70-83% of responses) | Known upstream bug (google-deepmind/gemma#622) | `repeat_penalty: 1.05` per-request |
| All Gemma4 | `<\|turn>thought` leakage costing 1pt/answer | Model enters thinking mode by default | `reasoning: off` server flag |
| All Gemma4 | KV cache quality | Default f16 suboptimal | `kv_cache: q8_0/q8_0` |
| Qwen3.6 | Stuck in `<think>` loops (0% on all runs) | Thinking model needs template; `/completion` doesn't apply it | `use_chat_api: true` + `reasoning: off` |
| Qwen3.6 | f16 KV clips dynamic range | Model trained in bf16, f16 degrades PPL | `kv_cache: q8_0/q8_0` |
| M2.7 | Training data leakage (37% of responses) | MiniMax not in llama.cpp built-in template list; generic template applied | `--jinja` flag auto-enabled for all chat API models |
| M2.7 | repeat_penalty=1.05 caused regression (52%→27%) | Penalty pushed M2.7 into memorized sequences instead of preventing repetition | Removed repeat_penalty for M2.7 |
| M2.7 | All tokens consumed by think blocks, empty answers | Model enters thinking mode, exhausts token budget | `reasoning: off` server flag |
| M2.7 | KV cache q6_0 crashed server | `q6_0` not in our llama.cpp build's supported types | Changed to `q8_0/q8_0` |
| SG4-26b Q4KM | Irrecoverable degeneration (16%) despite all fixes | Q4_K_M quantization fundamentally degrades MoE expert routing | Model deprecated and GGUF deleted (16 GB freed) |
| All chat models | `/v1/chat/completions` crash with multimodal content format | `_run_vl_inference` sent `[{"type":"text","text":...}]` for text-only | Fixed: sends plain string for text-only, multimodal array only with images |

### Lookup Ngram Sweep Upgrade

Replaced fixed list `[3, 4, 5]` with binary peak search over `[2, 128]`. Sentinel config `lookup_n0` triggers `_sweep_lookup_ngram()` which divides and conquers, finding the optimal ngram value in ~7 steps. Results stored per-step for curve analysis.
