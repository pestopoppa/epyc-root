## Session 3: Eval Log Observability — Prefill/Generation Timing Breakdown

### Problem

Eval log showed `ARCHITECT → PASS (19.9s, 0.2 t/s, 3 tok)` — misleading because 0.2 t/s conflates ~18s of prompt evaluation with ~1.5s of generation. The model correctly answered a GPQA multiple-choice question with 3 tokens, but the wall-clock t/s metric made it look broken.

### Root Cause

`eval_log_format.py` computed t/s as `tokens_generated / wall_elapsed`, mixing prefill and generation time. Both `prompt_eval_ms` and `generation_ms` were already tracked end-to-end (llama-server → `InferenceResult` → `LLMPrimitives` → `ChatResponse` → `RoleResult`) but never surfaced in the eval log output.

### Changes

**`scripts/benchmark/eval_log_format.py`** — Added `_timing_str()` helper, appended to all three formatters:
- `format_self_direct`, `format_self_repl`, `format_architect_result` now emit timing breakdown
- Prefill shows `pp/s` (prompt processing speed) when `cache_stats.total_prompt_tokens` is available
- Generation shows `t/s` computed from `tokens_generated / generation_ms`
- Example output: `ARCHITECT → PASS (19.9s, 0.2 t/s, 3 tok) [prefill=18.5s 1520 pp/s, gen=1.5s 2.0 t/s]`

**`scripts/benchmark/seeding_eval.py`** — Added `prompt_eval_ms` to all three `cost_metrics` dicts (SELF:direct, SELF:repl, ARCHITECT).

**`scripts/benchmark/seeding_injection.py`** — Added `prompt_eval_ms` to reward injection context for MemRL/Optuna cost optimization.

### Design Notes

- Prefill cost is fixed per question (same prompt length regardless of routing action) — useful for infra optimization (KV cache, prompt caching)
- Generation cost is the variable lever for routing decisions — the model controls thinking/answer length
- The existing `compute_comparative_rewards` already correctly uses `generation_ms` for cost penalty (line 96), not total elapsed
- `prompt_eval_ms` in injection metadata enables future Optuna to factor prefill cost into routing decisions across models with different prefill speeds

1. **intake-263 (ACP server)**: 2-day-old fork of upstream `agentclientprotocol/claude-agent-acp`. ACP is an IDE/editor protocol (Zed, JetBrains), orthogonal to our llama.cpp inference stack. Downgraded to not_applicable.
