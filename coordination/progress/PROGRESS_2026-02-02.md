# Orchestration Progress: 2026-02-02

## MemRL Mode-Advantage Task Enrichment

### Summary

Added 90 hand-written mode-advantage tasks + 3 external dataset adapters to the seeding pipeline, shifting MemRL reward distribution from weak (both-correct ~60%) to strong (specialist-wins ~25-35%). Pre-downloaded CRUXEval (800) and BigCodeBench (1,140) from HuggingFace. GAIA (165) requires gated access approval.

### Problem

The debug suite (327 single-turn QA) produced weak comparative rewards — all tasks solvable by direct inference. Evidence: frontdoor:direct 80.8% ≈ coder_primary:direct 80.8% on pilot run. MemRL learned cost-awareness but NOT routing quality.

### Solution

**90 mode-advantage YAML tasks** (`benchmarks/prompts/debug/mode_advantage.yaml`):
- 15 computation-gated (react >> direct): modular arithmetic, SHA-256, base64, statistics
- 15 iterative-fix (repl >> direct): bug fixes with test suites, code tracing, regex
- 15 multi-step (delegation >> direct): chained calculations, Dijkstra, topological sort
- 15 escalation-gated (specialist >> frontdoor): A*, trie, Union-Find, KMP, Bloom filter
- 30 mini-SWE (repl+delegation >> direct): broken Python class + failing tests + known fix

**16 exemplars** tagged `mode_advantage: true` in math (+4), coder (+4), agentic (+4), long_context (+4).

**3 HuggingFace dataset adapters** (`scripts/benchmark/dataset_adapters.py`):
- GaiaAdapter: 165 questions, exact-match, L1-L3 tiers (gated — needs HF access)
- CRUXEvalAdapter: 800×2 tasks (output + input prediction)
- BigCodeBenchAdapter: 1,140 multi-library coding tasks

### Persona Note

The seeding script (`seed_specialist_routing.py`) sends raw prompts via `/chat` with `force_role` and `force_mode`. Persona prompts (from `persona_loader.py` / `persona_registry.yaml`) are NOT applied — they only activate when explicitly passed at the `llm_primitives` layer. This is intentional: persona variance would confound the routing signal during MemRL seeding.

### Seeding Run Fix: Slot Erase on Timeout

First seeding run revealed architect_general (port 8083) timing out with 0 tokens on mode-advantage tasks. The llama-server continued processing the stale request, causing cascading timeouts.

**Fix**: Added `_erase_slots(port)` to `seed_specialist_routing.py`. After a timeout/error on a heavy port with 0 tokens generated, the script now POSTs `/slots/{id}?action=erase` to force-cancel the stuck inference. Trigger: `error AND target_port in HEAVY_PORTS AND tokens_generated == 0`.

Safety analysis: `force_role` bypasses escalation, seeding is sequential per question — no risk of erasing a concurrent legitimate request.
