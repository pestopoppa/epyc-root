## 1. Parallel Tensor Repacking Patch

### Problem
Model loading was bottlenecked by single-threaded tensor repacking for AVX-512 optimization. For a 271GB model, ~120s of 150s load time was spent in `repack_*` functions.

### Solution
Added `#pragma omp parallel for` to 6 repack functions in `ggml/src/ggml-cpu/repack.cpp`:
- `repack_q4_0_to_q4_0_4_bl`
- `repack_q4_K_to_q4_K_8_bl` (most common - Q4_K_M models)
- `repack_q2_K_to_q2_K_8_bl`
- `repack_q4_0_to_q4_0_8_bl`
- `repack_iq4_nl_to_iq4_nl_4_bl`
- `repack_iq4_nl_to_iq4_nl_8_bl`

### Benchmark Results

| Model Size | Before | After | Speedup |
|------------|--------|-------|---------|
| 6.8GB Q4_K | 5.0s | 3.3s | **1.5x** |
| 19GB Q4_K | 11.9s | 5.3s | **2.2x** |
| 271GB Q4_K | ~150s | ~60s | **~2.5x** (estimated) |

- Diff: `patches/llama-cpp-parallel-repack.diff`

## 5. Benchmark Script MoE Expert Count Fix

### Problem
Benchmark showed MOE2 slower than baseline (5.63 vs 6.47 t/s) when manual tests showed MOE2 faster (11.51 vs 6.53 t/s). Investigation revealed the server was started with a fixed 4-expert count and reused for ALL MoE configs.

### Root Cause
In `run_benchmark.py`, line 302: `moe_override = f"{moe_key}=int:4"` started the server with 4 experts, and all MoE configs (MOE2, MOE4, MOE6, MOE8) ran on the same 4-expert server. Speed differences were just variance.

### Solution
Added server restart logic when MoE expert count changes between configs:
1. Track `current_server_experts` (None = default)
2. For baseline: restart with no override (default experts)
3. For MoE configs: restart with specific expert count
4. Check if restart needed: `required_experts != current_server_experts`

### Investigation: On-the-fly Expert Changes
Investigated whether llama-server supports changing expert count per-request:
- **Result**: NOT possible
- `n_expert_used` is baked into `hparams` at model load time
- Inference graph is constructed using these hparams
- No API endpoint to modify model parameters at runtime
- Unload/reload is the only way to change expert count

### Files Modified
- `scripts/benchmark/run_benchmark.py` - Added server restart logic for MoE configs

### Impact
All MoE models will now be benchmarked correctly with their actual expert counts.

## 7. Long Context Benchmark Fixes (2025-12-22)

### Problem 1: Context Size Not Applied
Server was starting with 65K context instead of model-specific 131K for architect_coding.

**Root Cause**: `active_server.start()` wasn't passing `role` parameter, so registry couldn't look up model-specific `max_context`.

**Fix**: Added `role=role` to both `active_server.start()` calls in `run_benchmark.py`.

### Problem 2: Timeout Loses All Output
HTTP client timeout discarded all generated tokens, wasting computation.

**Fix**: Implemented streaming with partial output collection:
- Server uses `stream: true` for SSE responses
- Executor collects tokens incrementally via `iter_lines()`
- On timeout, returns partial output instead of empty string
- Benchmark saves partial output (>50 chars) for quality review

### Problem 3: Timeouts Still Too Short
20-minute timeout insufficient for 80K token prompts at 1 t/s generation.

**Fix**: Increased long_context base timeout from 1200s to 3600s (60 min).

### Problem 4: Slow Prompt Processing
80K token prompt taking 2.7 hours at ~8 t/s.

**Fix**: Added server optimizations:
- Flash attention (`-fa on`) - ~2x speedup for long contexts
- Larger ubatch (`-ub 8192`) - faster batch processing
- Made flash attention conditional via registry (`constraints.flash_attention: false`)

**Result**: Prompt processing improved from ~8 t/s to **~205 t/s** (25x faster).

### Files Modified
| File | Change |
|------|--------|
| `scripts/benchmark/run_benchmark.py` | Added `role=role` to server start, partial output saving |
| `scripts/lib/executor.py` | Streaming inference, flash attention, ubatch size |
| `scripts/lib/registry.py` | `get_flash_attention()`, `get_ubatch_size()` methods |
| `orchestration/model_registry.yaml` | `flash_attention`, `ubatch_size` in server_defaults |
| `benchmarks/prompts/v1/long_context.yaml` | Timeout 1200s → 3600s |

### Registry Override Example
To disable flash attention for a model that doesn't support it:
```yaml
role_name:
  constraints:
    flash_attention: false
```
