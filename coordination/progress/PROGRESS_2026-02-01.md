# Orchestration Progress: 2026-02-01

## Vision Pipeline → Document Pipeline Integration

### Summary

Wired vision requests (`/chat` with image/file inputs) through the full document processing pipeline (DocumentPreprocessor → DocumentChunker → FigureAnalyzer → DocumentREPLEnvironment) instead of the old bare `ocr_image()` + VL model direct-answer path.

### Problem

VL OCR tests all FAIL in phase 3 baseline. Root cause: `_execute_vision()` bypassed the entire document pipeline that was built and tested (48 integration tests, Twyne V1 whitepaper validation). It did a bare `client.ocr_image()` call that failed silently, hardcoded `tools_used=1`, and returned a single-shot VL model answer.

### Architecture Change

**Before:**
```
/chat + image → _execute_vision() → bare ocr_image() → VL model → EARLY RETURN
                 (bypasses pipeline)     (fails silently)   (hardcoded tools=1)
```

**After:**
```
/chat + image → _execute_vision() → DocumentPreprocessor.preprocess_file()
                                      ├─ PDFRouter (deterministic text + bbox)
                                      ├─ LightOnOCR fallback
                                      ├─ DocumentChunker (sections)
                                      └─ FigureAnalyzer → VL model describes figures
                → stores on routing.document_result, returns None
                → mode selection forces "repl" + FRONTDOOR
                → _execute_repl() creates DocumentREPLEnvironment
                    ├─ sections(), figures(), search_sections() tools
                    ├─ Text model synthesizes answer via REPL loop
                    └─ tools_used tracked properly by REPL infrastructure
```

### Files Modified

| File | Change |
|------|--------|
| `src/api/routes/chat_utils.py` | Added `document_result: PreprocessingResult \| None` to RoutingResult |
| `src/api/routes/chat_pipeline.py` | Rewrote `_execute_vision()` to preprocess not answer; modified `_execute_repl()` to use DocumentREPLEnvironment |
| `src/api/routes/chat.py` | Force REPL mode + FRONTDOOR role when document_result is set |
| `src/services/document_preprocessor.py` | Added base64 image support to `_extract_document_paths()` |
| `src/api/routes/chat_vision.py` | No changes — kept for backward compat (imports preserved) |

### Key Design Decisions

1. **Preprocessing, not answering**: `_execute_vision()` no longer builds a ChatResponse. It runs DocumentPreprocessor and stores the result on `routing.document_result`, then returns None to fall through.
2. **REPL mode forced**: When `document_result` is set, mode selection forces `execution_mode="repl"` with `Role.FRONTDOOR` so a text model orchestrates the REPL with document tools.
3. **DocumentREPLEnvironment**: Extends REPLEnvironment with `sections()`, `figures()`, `search_sections()`, `document_info()` — the model can explore the document structure programmatically.
4. **Base64 → temp file**: Base64 images are saved to `/mnt/raid0/llm/tmp/` with correct extension (detected from magic bytes) for DocumentPreprocessor to process.
5. **Graceful fallback**: If preprocessing fails, a context note is injected and the request falls through to standard text orchestration.

## Unified Evaluation Script — One Script To Rule Them All

### Summary

Consolidated 5+ eval scripts into a single canonical script (`seed_specialist_routing.py`) for continuous comparative evaluation. Added continuous mode, HF dataset sampling (31K+ questions), JSONL checkpointing, escalation chain detection, smart auto-launch, and offline stats. Deprecated 4 superseded scripts.

### Problem

5+ eval scripts existed, none doing the right thing continuously:
- `orchestrator_eval.py`: On-policy only, binary rewards, can't discover faster routes
- `seed_specialist_routing.py`: Correct comparative methodology but one-shot, YAML-only (327 questions)
- `memrl_learning_loop.py`: Fixed iterations, no counterfactual
- `compare_orchestrator_direct.py`: A/B comparison only
- `run_phase3_validation.sh`: Pipeline orchestrator for the above

User requirement: ONE script that runs for days collecting comparative routing data for MemRL.

### Solution

Extended `seed_specialist_routing.py` (the only script with correct comparative methodology) with:

1. **`--continuous` mode** — Runs batches indefinitely with SIGINT graceful shutdown. `--continuous-interval N` between batches. Auto-incrementing seed per batch.
2. **HF dataset sampling** — Imports `dataset_adapters.py` for 31K+ questions (MMLU 14K, ARC+HellaSwag 11K, GSM8K+MATH 1.8K, HumanEval+MBPP 664, IFEval 541). Falls back to YAML for agentic/long_context.
3. **JSONL checkpointing** — Per-question checkpoint to `benchmarks/results/eval/seeding_{session_id}.jsonl`. Ctrl+C safe.
4. **`--resume SESSION_ID`** — Continues where left off after restart.
5. **`--stats`** — Aggregate accuracy by role×mode across all sessions. Shows MemRL confidence coverage (combos with ≥3 observations).
6. **`--preflight`** — Health check + smoke test. Auto-launches stack if nothing running; launches API-only if model servers are up but API is down.
7. **Escalation chain detection** (on by default) — Detects cheap-fail→expensive-pass patterns, injects `escalate:from→to` rewards (+0.8) into MemRL.
8. **Global dedup** — `seen_questions.jsonl` tracks all evaluated questions across sessions. 3x oversampling compensates for dedup filtering.

### Reward Methodology (unchanged)

```
specialist correct & baseline wrong    → +1.0
specialist wrong & baseline correct    → -0.5
both correct                           → 0.5 - 0.15 × max(0, cost_ratio - 1.0)
both wrong                             → -0.3
escalation (cheap fail → expensive pass) → +0.8
```

### CLI

```bash
# THE command. Launch and leave for days.
python scripts/benchmark/seed_specialist_routing.py \
  --continuous --suites all --sample-size 10 --cooldown 2.0 --preflight

# Resume after restart
python scripts/benchmark/seed_specialist_routing.py \
  --continuous --resume seeding_20260201_143022

# Quick stats
python scripts/benchmark/seed_specialist_routing.py --stats

# One-shot batch
python scripts/benchmark/seed_specialist_routing.py --suites thinking coder --sample-size 5
```

## API Stability Hardening for Multi-Day Continuous Evaluation

### Summary

Fixed 5 bugs preventing `seed_specialist_routing.py --continuous` from running reliably for days. Root causes: Starlette POST deadlock, event loop blocking, signal handler crashes, lazy MemRL initialization, and resource leaks in long-running connections.

### Bugs Fixed

| Bug | Root Cause | Fix |
|-----|-----------|-----|
| All POST requests hang | `BaseHTTPMiddleware` body-consumption deadlock | Rewrote `rate_limit.py` as pure ASGI middleware |
| REPL crashes in worker threads | `signal.signal(SIGALRM)` raises ValueError in non-main thread | Wrapped in try/except, track `_alarm_set` flag |
| Background Q-scorer blocks event loop | `score_pending_tasks()` called synchronously in async function | `await asyncio.to_thread()` |
| Per-request scoring blocks event loop | `_score_task()` does sync HTTP to embedder port 8090 | Fire-and-forget via ThreadPoolExecutor(max_workers=4) |
| First requests timeout after API restart | `ensure_memrl_initialized()` lazily loads TaskEmbedder on first request | Moved to lifespan startup (eager init) |
| httpx connection leak in seed script | Bare `httpx.get()` per poll creates ephemeral connections | Module-level `httpx.Client` with connection reuse |
| No SIGTERM handling in seed script | Process killed without cleanup | Added SIGTERM signal handler |
| REPL timeout silently fails in threads | `signal.alarm()` only works in main thread | `asyncio.wait_for()` wrapping `to_thread()` |

### Files Modified

| File | Change |
|------|--------|
| `src/api/rate_limit.py` | Complete rewrite: BaseHTTPMiddleware → pure ASGI middleware |
| `src/api/services/memrl.py` | ThreadPoolExecutor for scoring; `asyncio.to_thread` for background cleanup |
| `src/api/__init__.py` | Eager MemRL init in lifespan; shutdown_scoring() on exit |
| `src/repl_environment/environment.py` | Signal handler non-main-thread safety |

### Failure Mode Analysis

Analyzed 8 potential failure modes for multi-day operation:
- **P1 (p=0.60)**: httpx connection leak → socket exhaustion. **Fixed.**
- **P2 (p=0.25)**: Unbounded daemon threads → thread exhaustion. **Fixed.**
- **P3 (p=0.20)**: Progress logger disk growth (~14MB/day). Acceptable on 4TB RAID.
- **T1 (p=0.08)**: SQLite WAL lock contention. Mitigated by single-threaded access pattern.
- **T2 (p=0.05)**: SIGALRM race in REPL threads. **Fixed** (asyncio.wait_for).
- **T3 (p=0.04)**: OOM killer targeting API. Unlikely at 535GB/1130GB utilization.
- **T4 (p=0.03)**: FAISS index corruption on crash. Low risk, non-critical path.
- **T5 (p=0.02)**: TCP TIME_WAIT exhaustion. **Fixed** (connection reuse).
